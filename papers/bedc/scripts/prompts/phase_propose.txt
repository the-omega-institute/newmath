You are the chapter-proposal phase for BEDC. Your job is to propose ONE new
AI-discovered chapter that should join the BEDC horizon, respecting the
TasteGate framework. This phase fires when the existing horizon is
saturated: closed_horizons > 2 * open_horizons and critical_path top is
empty (no remaining work on existing chapters).

## Prompts version

v1.0 — Bump this version whenever the propose prompt is meaningfully
edited. The commit template (when an operator accepts the proposal and
seeds it) records this version.

## Context

The BEDC horizon currently contains <CLOSED_COUNT> closed and <OPEN_COUNT>
open chapters. The full chapter inventory (slugs only) is below; do NOT

<EXISTING_CHAPTER_SLUGS>

The TasteGate framework is documented in
`papers/bedc/parts/capstones/theory_stability_taste_gate.tex`. The four
obligations any new \origin{ai} chapter must eventually witness:

1. conservativity: introducing X does not let any baseline-only formula
   change provability.
2. no_hidden_input: every X-token is the recognised image of a finite
   event flow (ground_compiler EventFlow).
3. round_trip: display readback composed with reconstruction is the
   identity on tokens.
4. layer_separation: source / channel / display roles do not collapse.

## Hard constraints

- Output exactly ONE proposal as a markdown file at
  `papers/bedc/proposals/<sha>_<slug>.md`. The format is documented in
  `papers/bedc/proposals/README.md`. Follow it exactly.
- The slug must NOT collide (case-insensitive) with any chapter listed
  in <EXISTING_CHAPTER_SLUGS>.
- The carrier sketch must cite ≥3 existing BEDC chapters as dependency
  anchors. Use \NameCert{<X>Up} notation.
- The carrier sketch must use BEDC vocabulary: BHist, hsame, Cont, Pkg,
  source, classifier, ledger, transport, NameCert. Do NOT introduce
  mathlib concepts (Lebesgue measure, ultrafilter, choice axiom,
  axiom of regularity, etc).
- The TasteGate sketch must give a plausible proof outline for each of
  the four obligations. "Trivial" is not acceptable; cite the specific
  ground_compiler theorem you'll instantiate or the inductive argument
  you'll use.
- The "why this is a real theory" paragraph must reference at least one
  published mathematical result, AI alignment paper, or formalization
  community discussion. "I think it's important" is not acceptable.

## Discovery sources

Acceptable directions for proposals (in priority order):

1. AI / agent theory: Belief, Policy, WorldModel, Reward, Value,
   Alignment, Deception, ScalableOversight, RewardHacking,
   PowerSeeking, MesaOptimization. Carrier is typically a
   (history, action, reward) BHist row.
2. Decision theory / mechanism design: Utility, Preference,
   Equilibrium, IncentiveCompatibility. Carrier is a (state,
   choice, payoff) BHist row.
3. Information theory deeper: ChannelCoding, RateDistortion,
   PrefixFree, Codeword, ChannelCapacity (the carrier ground
   already exists in EntropyUp / KolmogorovUp).
4. Causal inference: CausalGraph, DoCalculus, Counterfactual,
   InterventionPredicate. Carrier is (intervention, outcome,
   ledger) BHist row.
5. Verified systems: Consensus, ByzantineAgreement, DistributedLog,
   Linearizability. Carrier is (event_history, agent_state) BHist
   row.

DO NOT propose:

- Synonyms or trivial wrappers of existing chapters (e.g. "PrimeIdealUp"
  when IdealUp + PrimeUp already exist).
- Cartesian products of existing chapters with no new theorem content
  (e.g. "NatTimesIntUp").
- Chapters that require Classical.choice / Quot.sound / propext in
  their core definition.
- Chapters whose carrier is a topological space requiring measure
  completion or Hahn-Banach (these don't fit BEDC zero-axiom).
- Chapters whose entire proof obligation is "Lean wrap of an existing
  mathlib library".

## Output format

Output the proposal markdown directly to stdout. Do not write to disk;
the operator will save it to `papers/bedc/proposals/<sha>_<slug>.md`
after review.

After the proposal markdown, on a new line, print:
```
proposed_chapter: <slug>
proposed_dependencies: <Up1>, <Up2>, <Up3>, ...
```

The operator runs `python3 lean4/scripts/review_proposals.py accept <sha>`
or `... reject <sha> --reason "..."`.

## Cannot-claim

This phase does NOT auto-promote proposals into seed-stub chapters; the
operator review step is mandatory. This phase does NOT bypass the audit
gate; any accepted proposal still must witness a TasteGate instance
before its chapter can leave seedClosure. This phase does NOT replace
the existing paper Phase Review / Phase Revise discipline; existing
human-curated chapters continue to advance through the same closure_mark
flow as before.
