## TOP-LEVEL HARD GATE — orphan-chapter repair (cross-chapter lattice integrity)

**Background (2026-05-13 dossier audit)**: 447 of 652 `parts/concrete_instances/` chapters (69%) had ZERO `\autoref{ch:concrete-instances-<sibling>-namecert}` or `\autoref{ch:visions-<slug>}` cross-references — every newly-generated chapter was a kernel-only float, citing only its own slug labels. The dependency graph degenerated into a star fan-out from kernel with no horizontal lattice. Two coordinated rules fix this:

**Rule 1 — every new chapter MUST cross-reference at least one sibling** (enforced by the `orphan-new-chapter` gate in `phase_paper_gates.py`, hard-fail at verify). When you propose a `new_chapter` target (vision_concretization / automath_concretization / human_chapter_extension), `proposed_change` MUST instruct Phase Revise to insert at least one of:

- `\autoref{ch:concrete-instances-<sibling-slug>-namecert}` to a sibling concrete-instances chapter whose carrier / classifier / NameCert obligation row this new chapter consumes or extends
- `\autoref{ch:visions-<vision-slug>}` to the originating vision chapter (always required for `vision_concretization` targets — the `\concretizedIn{...}` marker on the vision side is separate from the inbound `\autoref` and both must exist)
- `\autoref{thm:<sibling-slug>-...}` or `\autoref{def:<sibling-slug>-...}` resolving to a different slug than the new chapter's own slug

Pure self-refs (`\autoref{thm:<own-slug>-...}`) DO NOT count. The chapter must show how it RELATES to existing chapters — which sibling's carrier it transports through, which obligation row it discharges as a consumer, which vision narrative it instantiates.

**Rule 2 — orphan-chapter repair as a first-class candidate kind** (`kind = "orphan_repair"`). The repair pool consists of every existing concrete-instances chapter whose dossier `out_degree == 0` AND whose `\autoref{thm/def:...}` block resolves only to self-slug labels. To enumerate the repair candidates locally without rebuilding the dossier:

```
grep -lE '\\autoref\{ch:concrete-instances-' papers/bedc/parts/concrete_instances/*_namecert_construction.tex > /tmp/has_sibling_ref.txt
# Files NOT in that grep output are orphan candidates; pick one whose own slug
# names a concept that genuinely depends on another chapter's carrier.
```

For an `orphan_repair` target, `paper_files[0]` is the orphan chapter file; `proposed_change` describes the surgical edit:

- Identify ONE existing sibling chapter whose carrier or classifier or NameCert obligation row the orphan chapter's content actually depends on (a SUBSTANTIVE dependency — not a fabricated link).
- Insert one paragraph or sentence in the orphan chapter that names the sibling and the specific dependency relationship, with one `\autoref{ch:concrete-instances-<sibling>-namecert}` or `\autoref{thm:<sibling>-...}` cite.
- If the orphan chapter genuinely has no substantive sibling dependency (it sits directly on kernel primitives only — BHistCarrier, NameCert, hsame, ProbeBundle), the correct repair is to identify the FIRST DOWNSTREAM consumer (another chapter that uses this chapter's results) and add a reverse forward link there instead (`paper_files[0]` becomes the downstream chapter, the orphan becomes the cite target). Either direction satisfies the dossier lattice — what matters is that the orphan moves out of `out_degree == 0` OR `in_degree == 0` state.

**HARD GATE — orphan_repair quota**: when `\count_orphan_chapters` (the count from the `grep -L` above) is ≥ 50, AT LEAST 1 of the round's emitted targets MUST be `kind = "orphan_repair"`. Cap remains 1 per round so Phase Revise is not flooded with surgical edits.

## What counts as a valid review target
Pick targets that are HIGH-VALUE — each should be something a careful reader would flag during peer review. Acceptable categories are substantive theory/paper-quality categories; splitting may appear only as an attached implementation detail when required to land such a target:

1. **Logical gap or unjustified step.** A theorem statement that is used downstream but whose proof skips a non-trivial case, or a definition that is invoked before being properly fixed.
2. **Notational or definitional inconsistency.** Same symbol used with two meanings; a definition restated differently in two chapters; an undefined macro or label dangling.
3. **Dead/redundant material.** A definition or proposition that is never referenced; a section whose content is fully subsumed by another; theorem hypotheses that are vacuous or implied.
4. **Missing companion result.** A claim made informally but not stated as a theorem; a proof obligation that the Lean side already satisfies (`\leanchecked` exists) but the paper has no `\leanchecked{X}` marker; a concrete instance for an abstract construction that is asserted to exist but is not exhibited.
5. **Unclear or imprecise statement.** A theorem whose hypotheses must be tightened (or relaxed) for the conclusion to hold as currently written; an "obvious" appeal that hides a real argument.
6. **Stronger statement available.** The paper proves a weaker form when the same proof yields more (e.g. uniform bound proved pointwise; restriction to finite case when general case is identical).
7. **Cross-reference / structural improvement.** A theorem block that is mis-located (would be easier to follow earlier or later); a result that should be promoted from remark to numbered statement, or vice versa.
8. **Content-bearing split support.** If a substantive theorem/definition/proof/certificate target would push a chapter file near the line cap, include a split plan as an implementation detail: extract the relevant content into a child file under a same-theme subdirectory, and make the parent file a pure index containing only ordered `\input` lines. Do not emit split-only targets; file movement by itself is maintenance, not theory growth.
9. **Theory extension awaiting formalization.** Propose a NEW theorem statement (not yet in the paper, not yet in Lean) that follows from the existing kernel and is worth formalizing. Acceptable shapes:
   - **Composite consequence.** A non-trivial conclusion combining 2+ existing `\leanchecked` theorems, where the combined statement is not already in the paper.
   - **Constructor inversion / determinism / no-confusion lemma** for an existing `inductive` (BMark / BHist / ProbeBundle / SigRel / psame / external binary words / derived-carrier inductives) that does not yet have one.
   - **Generalization of an existing checked theorem** with a strictly broader hypothesis or a strictly stronger conclusion, provable by the same proof skeleton.
   - **New derived predicate or function** over a kernel object (e.g. a structural property of `BHist`, a new well-founded measure, a closure operator) plus its key well-formedness theorem.
   - **NameCert spec for a new canonical math object** in the BEDC licensing horizon that is not yet covered (BoolUp / OptionUp / ProdUp / SumUp / ListUp / IntUp / RatUp / RealUp / ComplexUp / FieldUp / ModuleUp / VecSpaceUp / MatrixUp / PolynomialUp / S1Up / MetricSpaceUp / CompactUp / CategoryUp / FunctorUp / NatTransUp). State carrier, classifier, and the at-least-one structural property the certificate must satisfy.
   - **Coverage / completeness theorem** stating that some existing construction exhausts a class (e.g. every admitted history under domain D admits an InGap witness; every constructor pair gives the same psame outcome; etc.).

   The proposal MUST be provable from the existing Lean kernel without `axiom`, `sorry`, `Mathlib.*`, `Classical.*`, or `Quot.*`. The statement must mention concrete BEDC kernel/setup objects (BMark, BHist, msame, hsame, Ext, Cont, ProbeBundle, SigRel, sameSig, Pkg, InGap, NameCert, SemanticNameCert, BaseReflectionSetup, AskSetup, PackageSetup, DomainSetup, NameCertSetup, or a `BEDC.` namespace symbol).

   The revise phase will INSERT this as a new `\begin{theorem}` / `\begin{lemma}` / `\begin{definition}` block at a thematically correct chapter location. It will write a brief PROSE proof sketch in `\begin{proof}…\end{proof}`. It will NOT add a `\leanchecked` marker — that is the next-Lean-round's job. It will NOT add `\leanvariant`. The Lean side runs an independent formalize pipeline that scans `papers/bedc/parts/` for theorem sites WITHOUT a `\leanchecked` marker and picks them up.

   INVALID shapes (skip silently):
   - Re-statement of an existing checked theorem with trivial reformulation.
   - A theorem whose proof reduces to one of `exact T` / `apply T` / `T.symm` / `T.mp` / `T.mpr` for a single existing `\leanchecked` theorem T (those are variants, never extensions).
   - A claim about an object that does not yet exist anywhere in the kernel.
   - Vague conjectures with no precise statement.

Reject these as low-value (do NOT propose):
- Pure typographical fixes, spelling, missing periods, kerning.
- Comment changes / inline notes (the project bans inline comments anyway).
- Adding "TODO", "future work", or version annotations.
- Edits that only relabel without changing logical content.
- Renaming for cosmetic uniformity unless name collision causes ambiguity.
- Pure prose smoothing whose expected diff does not change theorem hypotheses, theorem conclusions, proof dependency, marker alignment, definition scope, cross-reference correctness, or structural placement under the roadmap.
- Scope-clarification targets that only rename a theorem, soften a claim, or add a caution sentence, unless the existing statement is mathematically false or causes a concrete downstream contradiction.
- A second target that is merely explanatory cleanup bundled with a stronger theorem target. Mixed-strength bundles dilute the round; emit only the strongest target.

## Anti-parameter-echo (HARD GATE)

Reject candidates whose theorem statement is a mechanical "carrier/classifier equivalence transport". Concrete forbidden shapes:

1. **Source-equivalence echo.** A theorem of the form: assume $A \Leftrightarrow B$ as carriers and $\sim_A \Leftrightarrow \sim_B$ as classifiers, conclude that every previously proved $A$-side property holds on the $B$-side (e.g. $\mathsf{XSpineRepCoherent}_{A,\sim_A} \Rightarrow \mathsf{XSpineRepCoherent}_{B,\sim_B}$, $\mathsf{SemanticNameCert}_A \Leftrightarrow \mathsf{SemanticNameCert}_B$, or paired carrier/pattern/ledger $\Leftrightarrow$ chains). The proof is "transport hypothesis through the equivalence on each field" — that is parameter packaging, not theory growth.

2. **NameCert renaming.** Restating an existing certificate with $A$/$B$ subscripts and asserting their fields are pairwise equivalent. The down-stream consumer can already invoke the original certificate under either source presentation; the renamed version is bookkeeping.

3. **Indexed-everywhere theorem.** Statements that introduce abstract carriers $A,B:\mathsf{BHist}\to\mathsf{Prop}$ and abstract classifiers $\sim_A, \sim_B$ as universally quantified parameters with no concrete instantiation, and whose conclusion lists $\Leftrightarrow$ between every field of two parametric structures. If the theorem cannot be falsified by a concrete BEDC kernel construction (BMark / BHist / specific Pkg / specific NameCert), it is parameter packaging.

4. **Coherence transport sequels.** Once a chapter has a `<X>_source_equivalence` theorem proved, do not propose a `<X>_under_source_equivalence`, `<X>_finite_<sub>_source_equivalence`, `<X>_<field>_source_equivalence`, or other -source-equivalence sibling that differs only in which sub-structure of the same parametric pair is being equated. The first such theorem absorbs the rest.

A `theory_extension` candidate that looks like any of the above is INVALID even if it would compile and even if it advances the open phase. Skip it and pick a target whose conclusion is a closure / inversion / coverage / determinacy claim about a CONCRETE BEDC inductive construction (specific BMark/BHist/ProbeBundle/SigRel pattern, specific Pkg / NameCert instance, specific cons-boundary witness, etc.), not a $\Leftrightarrow$ chain over abstract parameters.

Valid replacements when you were tempted by an echo target:
- A constructor-inversion / no-confusion lemma for the underlying inductive (e.g. `BHist` cons vs nil, `ProbeBundle` constructor disjointness, `SigRel` signature mismatch).
- A composition theorem chaining 2+ existing checked theorems whose conclusion uses a concrete kernel object.
- A coverage / exhaustion theorem stating that every admitted instance of an existing inductive admits a witness, with the proof going by induction on constructors.
- A new derived predicate over `BHist` plus its closure under `hsame` / `Cont` / `Ext`.

## Anti-saturated-file (HARD GATE)

Track the recent activity profile and the current size of `papers/bedc/parts/` files to avoid concentrating consecutive rounds in one file or one "<X> source equivalence" axis. Also avoid picking files that are already near the 800-line cap, which would force an emergency split or oversize-fail the round.

1. Run `git log --oneline -20 -- papers/bedc/parts/` to inspect the last twenty paper-side commits. Build the multiset of touched files.
2. **File-size pre-check.** Run `wc -l <each candidate paper_files[0]>`. A candidate is INVALID if its `paper_files[0]` is **≥760 lines** unless the target is precisely a content split (see Structural split rule in `phase_revise.txt`): extract a focused sub-chapter file inside the same theme directory and leave the parent as a pure index of ordered `\input` lines. An "append a new theorem to a 760+ line file" target is rejected — the gate will fail it after Phase Revise spends a codex-exec budget for nothing.
3. **Hub-file pre-check.** A `.tex` file is a HUB if it has been previously split — its non-blank, non-comment lines are ≥80% `\input{...}` directives, and it carries no `\chapter`/`\section`/`\begin{theorem}`/`\begin{definition}`/`\begin{proof}` of its own. The actual content lives in the child files referenced by `\input`. To detect:
   ```bash
   f=<candidate paper_files[0]>
   total=$(grep -cEv '^\s*$|^\s*%' "$f")
   inputs=$(grep -cE '^\s*\\input\{' "$f")
   own=$(grep -cE '\\(chapter|section|begin\{(theorem|definition|proposition|lemma|corollary|proof)\})' "$f")
   ```
   If `total <= 30` AND `inputs * 100 >= total * 80` AND `own == 0`, the file is a HUB. A candidate whose `paper_files[0]` is a HUB is INVALID UNLESS its `proposed_change` is precisely "create new child file under the hub's subdirectory, then append a single `\input{…}` line to the parent hub" (additive split). Targets that say "add theorem / definition / proof / certificate to <hub-file>" are rejected — content does not live in the hub. Instead either (a) pick the relevant `\input`-ed child file as `paper_files[0]` and target that, or (b) create a new child file alongside the existing children and add it to the hub's `\input` list. Appending content to a hub will be reverted by the next "keep paper sites singular" cleanup round and waste a codex-exec budget.
4. If the candidate target's `paper_files[0]` already appears ≥3 times in those 20 commits, the candidate must clear an additional bar: it must either close a roadmap exit criterion, or fix a logical/notational defect that is visible by grep, or introduce a NEW kernel touchpoint not yet present in the file. A fourth same-file source-equivalence / parameter-transport target is INVALID.
5. If the candidate's `summary` contains any of `source equivalence`, `source-equivalence`, `under source equivalence`, `under_source_equivalence`, `parameter transport`, `<obj>_A`, `<obj>_B` (literal subscript), and the same paper file already contains a theorem whose label or title matches `*source-equivalence*` or `*under-source-equivalence*`, treat the candidate as a parameter-echo sibling (see Anti-parameter-echo above) and skip it.

The pipeline runs many rounds in parallel; concentrating them in one file produces label collisions and bookkeeping growth without theory growth. Spread the work.

In `notes`, briefly state (a) which files appeared most often in the last 20 commits, (b) the line counts of every file listed in `targets[*].paper_files[0]`, (c) confirm the chosen target spreads the workload and stays under 760 lines (or is a split), and (d) confirm `paper_files[0]` is not a hub (or, if it is, that the target is an additive split that creates a new child file).

## Pre-emit redundancy gate (HARD GATE)

Four consecutive paper rounds (P835/P836/P837/P838) failed with "Verify: no commits unique to this round" — Phase REVIEW emitted a target whose content was already covered in the chapter, and Phase REVISE found nothing to add and exited fast (~30-50s codex exec, no commits). To pre-empt, before emitting a `theory_extension` or `missing_companion` candidate:

1. **Label collision pre-check.** If the candidate's `proposed_change` mentions a specific `\label{thm:<X>}` / `\label{def:<X>}` / `\label{prop:<X>}`, run:
   ```bash
   grep -rE "\\\\label\{(thm|def|prop|cor|prin|lem):<X>\}" papers/bedc/parts/
   ```
   If the label already exists anywhere under `papers/bedc/parts/`, the target is INVALID — pick another or strictly rename.

2. **Title collision pre-check.** If the candidate's `proposed_change` includes a `\begin{theorem}[<title>]` / `\begin{definition}[<title>]` clause, grep the target chapter for an equivalent existing block:
   ```bash
   grep -nE "\\\\begin\{(theorem|definition|lemma|proposition)\}\[" papers/bedc/parts/<paper_files[0]>
   ```
   If a block with semantically equivalent title already exists, the candidate is redundant.

3. **Marker redundancy pre-check.** For a `missing_companion` candidate naming an existing `\leanchecked{<Lean.target>}` / `\leanstmt{<Lean.target>}`, grep the candidate's chapter for that target:
   ```bash
   grep -F "<Lean.target>" papers/bedc/parts/<paper_files[0]>
   ```
   If the marker is already there, the marker is already wired and adding it again would be a no-op.

4. **Statement subsumption.** If the candidate's `proposed_change` describes a theorem whose conclusion (after stripping macros and quantifiers) reads as the conclusion of any existing `\begin{theorem}[...]` block in the same section, the candidate is a duplicate — skip it.

If these checks empty the candidate pool for this chapter, EXPAND search to non-saturated chapters (see Critical-path discovery above) before emitting `{"targets": []}`. An empty round is preferred over a fast-empty failure (the empty round costs no Phase REVISE budget; a fast-empty failure consumes ~30-50s of REVISE codex exec for nothing).

A `theory_extension` target must include a precise theorem/lemma/definition shape in `proposed_change`, the existing definitions/theorems it depends on, a constructive proof skeleton, why it is not a trivial variant of one existing checked theorem, and the Lean namespace where it should naturally be formalized later. It must also state one downstream use: a later theorem, certificate, bridge, roadmap exit criterion, or Lean formalization target that would consume the result.

Every candidate target must pass this 0–12 quality score before it may be selected:
- `novelty`: 0–2. Is this more than a restatement or wording polish?
- `theory_value`: 0–2. Does it improve a theorem, definition, proof obligation, certificate, or roadmap object?
- `verifiability`: 0–2. Can the anchor and issue be confirmed by file inspection / grep?
- `locality`: 0–2. Can the revise phase implement it without global rewrite or broad conflict risk?
- `line_cap_safety`: 0–2. Does it stay below the 800-line cap, or include a concrete split plan?
- `downstream_use`: 0–2. Is there a concrete later theorem, bridge, certificate, roadmap exit criterion, or Lean formalization site that will consume this target?

Reject any candidate with total score < 9, with `verifiability = 0`, with `line_cap_safety = 0`, with `downstream_use = 0`, or whose only effect is explanatory wording. Empty rounds are preferred over speculative targets.

## Layered closure planning (READ FIRST — drives target shape)

Each open chapter has TWO axes of closure progress, both read from the chapter's `closurestatus` block:

- **theory_closure axis (6 grades)**: `seedClosure` → `obligationClosure` → `scopedClosure` → `publicClosure` → `bridgedClosure` → `matureClosure`. **Owned by paper P-rounds (THIS phase) + bedc-deep supervisor**.
- **formal_status axis (8 grades)**: `unformalizedV` → `formalTargetV` → `encodedDefV` → `scaffoldCheckedV` → `theoremCheckedV` → `auditCleanV` → `axiomCleanV` → `bridgeCheckedV`. Owned by lean R-rounds.

A chapter retires from the critical path only when it reaches `(scopedClosure, theoremCheckedV)` simultaneously. The critical_path JSON includes per-chapter `theory_grade` / `theory_grade_next` / `next_axis` / `next_grade_transition`. Use these to pick a target SHAPE that advances the chapter's theory grade by exactly one step.

### Theory-axis transition templates (target shape per step)

| Current → Next | Required target shape |
|---|---|
| `(none) → seedClosure` | Add a `closurestatus` block to the chapter declaring `\theoryclosure{\seedClosure}\formalstatus{\unformalizedV}` and naming the chapter's flagship lean target via `\leantarget{X}`. This wires the chapter into the layered system. |
| `seedClosure → obligationClosure` | Enumerate the chapter's full obligation set (the BEDC-required claims `<X>Up` must establish: carrier predicate, classifier, identity laws, etc.). At least 3 `\begin{theorem}…\label{thm:<X>-obligation-...}` blocks per chapter. |
| `obligationClosure → scopedClosure` | For each obligation, add the SCOPE — declare which BEDC primitives / kernel constructs it depends on (`Cont`, `hsame`, `UnaryHistory`, `NameCert`, etc.) via inline references in the proof or definition. The scope grounds the obligation against the kernel. **(retirement threshold)** |
| `scopedClosure → publicClosure` | Add a public-facing statement: a single theorem the chapter exports (e.g. `\NameCert_{<X>Up}`). Other chapters cite it with `\leanchecked` markers. |
| `publicClosure → bridgedClosure` | Add a bridge theorem connecting the chapter's concrete instance to the abstract `<X>Up` interface. Demonstrates the concrete construction satisfies the schema. |
| `bridgedClosure → matureClosure` | Add a "mature treatment" — comprehensive examples / cross-chapter references / completeness statement. Last grade. |

### closure_mark target kind (already supported in v2.6+)

A `closure_mark` review target proposes adding a `\closureat{<X>Up}{<strength>Str}[BEDC.<...>.<theorem>]` line to the chapter — that's the binary closure marker the audit reads. closure_mark targets register the transition, but the underlying content (obligation/scope/public/bridge work) must already exist. Do NOT propose closure_mark for a chapter that hasn't done the prior-grade content work.

**HARD GATE — never re-mark a matureClosure chapter**: If the chapter ALREADY has a `closurestatus` block whose `\theoryclosure` is `\matureClosure`, the theory axis is at its topmost grade — there is no further grade to raise. NEVER propose `kind = "closure_mark"` for such a chapter, even if the chapter body contains lots of new `\leanchecked` rows. The chapter is done. Pick a different chapter from the rotation list / critical_path top / root_unblocks. A round whose target is closure_mark on an already-matureClosure chapter is a no-op round (Phase REVISE will produce a zero-meaningful-diff commit) and is worse than skipping the target. To detect: grep for `\theoryclosure{\matureClosure}` in the candidate chapter file before emitting the target.

### Transition rotation (HARD GATE — drives upgrade-chain balance)

`critical_path.py` emits a `top_transitions` field with 6 ranked candidate lists, one per theory_closure transition:

```
top_transitions:
  to_seed:        [chapters without closurestatus block, ranked by fan-out]
  to_obligation:  [chapters at seedClosure, ranked by fan-out]
  to_scoped:      [chapters at obligationClosure, ranked by fan-out]
  to_public:      [chapters at scopedClosure (retired), ranked by fan-out]
  to_bridged:     [chapters at publicClosure, ranked by fan-out]
  to_mature:      [chapters at bridgedClosure, ranked by fan-out]
```

Without rotation, paper rounds dogpile `to_seed` (first-time register) because that's where unattacked chapters dominate. Result: 50+ chapters parked at `scopedClosure` with no further upgrade, the `publicClosure → bridgedClosure → matureClosure` upgrade chain stays cold.

**HARD GATE — round-number rotation (weighted by backlog)**: At LEAST 1 of your N proposed targets MUST be `kind = "closure_mark"` drawn from the round-rotation transition. Use:

```
primary_transition_index = ROUND_NUM % 6
```

with the mapping `0→to_seed, 1→to_obligation, 2→to_scoped, 3→to_public, 4→to_bridged, 5→to_mature`. Pick `top_transitions[primary][0]` (the highest fan-out candidate at that transition) as the closure_mark target.

**Backlog-weighted reinforcement**: `to_seed` chapters dominate the open horizon (typically 100+ chapters with `theory_grade=null`). To prevent the upgrade chain from being permanently dominated by the `to_seed` backlog, use this rule:

- If `len(top_transitions['to_seed']) >= 50`, emit **2 to_seed targets per round** (regardless of `ROUND_NUM % 6`). This doubles the rate of new chapter registration and clears the backlog ~2x faster.
- If `len(top_transitions['to_seed']) >= 20`, emit **1 extra to_seed target** in addition to the round-rotation primary. So a round that's already on `ROUND_NUM % 6 == 0` (to_seed primary) emits 2 to_seed; a round on any other primary emits 1 primary + 1 extra to_seed.
- Otherwise, follow the standard `ROUND_NUM % 6` rotation only.

If the primary transition list is empty (e.g. round 5 looking for `to_mature` but no chapter is at `bridgedClosure` yet), fall back to `(primary + 1) % 6`, repeat until a non-empty transition is found. Skip the rotation gate ONLY when ALL 6 transitions are empty — extremely rare.

This guarantees over a 6-round window each transition gets at least 1 paper attempt, AND lets the to_seed backlog drain faster when it's the bottleneck. Existing HARD GATEs (`top[0..2]` critical-path + `top_root_unblocks[0..2]` root-unblock) still apply for the OTHER N-1 targets in the round.

`closure_mark` from rotation can co-exist with the existing closure_mark proposals on top-3 retired-eligible chapters; both are valid `closure_mark` instances. Difference: rotation forces breadth across grades; existing closure_mark logic forces depth on critical-path top.

### Skip rule (narrow — applies only when both grades are non-null)

The skip rule only applies to chapters that ALREADY have a `closurestatus` block (`theory_grade` non-null AND `formal_grade` non-null). For those:

- If `next_axis == "formal_status"` AND `theory_grade` is non-null, the chapter's theory axis is already ahead of its formal axis — lean R-rounds need to catch up first. Skip it for paper rounds — pick a chapter where `next_axis == "theory_closure"`, OR a chapter where both grades are null (those are always paper-eligible per the `(none) → seedClosure` row above).

For chapters with `theory_grade` null (the majority today — the `closurestatus` block has not been written yet), paper P-rounds are ALWAYS eligible. The `(none) → seedClosure` template row IS the paper work for these chapters: write the closurestatus block. Do NOT skip a null-grade chapter just because `critical_path.py` happens to compute `next_axis == "formal_status"` for it (the computation defaults to `formal_status` when both grades are null because the formal-axis order is longer; this is a critical_path internal detail and does NOT mean paper should skip).

Empty-targets escape `{"targets": []}` is reserved for the genuine corner case where every accessible chapter has both grades non-null AND every accessible front is lean-bottlenecked. This is exceedingly rare today.

Each target's JSON MAY include a `grade_transition` field quoting `next_grade_transition`. Phase Revise uses it to scope the diff (don't add prose unrelated to the named transition). For null-grade chapters the transition is `"(no block) -> seedClosure"`.

## Critical-path discovery (HARD GATE)

Run before target selection:

```bash
python3 "$(dirname "$(git rev-parse --git-common-dir)")/lean4/scripts/critical_path.py"
```

The script ranks every OPEN horizon `<X>Up` chapter by `transitive_downstream` (with `1/(1+thms)` as tiebreaker) after excluding nodes that carry a closurestatus block whose theoryclosure is scopedClosure-or-above and whose formalstatus is theoremCheckedV-or-above, or whose dependencies are not yet ready. **At least 1 candidate's `paper_files[0]` MUST correspond to a chapter from `top[0..2]`** of that JSON output (e.g. `top[0].file_paper` → `papers/bedc/parts/concrete_instances/13_real_namecert_construction.tex`). Prefer `kind="theory_extension"` or `kind="missing_companion"` in those chapters — those are the kinds that grow the chapter.

**Closure proposals (`kind = "closure_mark"`)**. If a horizon `<X>Up` chapter has every NameCert clause (carrier / classifier / stability / exactness / ledger / standard bridge if applicable) with a `\leanchecked` to a Lean target that resolves under `lean4/BEDC/`, AND the chapter does NOT yet carry a closurestatus block whose theoryclosure is scopedClosure-or-above and whose formalstatus is theoremCheckedV-or-above, propose a `closure_mark` target. The proposal MUST specify both axes explicitly, plus scope / not-claimed / upgrade-path text, and MUST NOT make either axis a function of the other. The block template is:

  \begin{closurestatus}{\<X>Up}
    \theoryclosure{\scopedClosure}      % paper-side closure grade
    \scopeclosed{<one-sentence binding scope>}
    \formalstatus{\theoremCheckedV}     % Lean-side verification grade
    \leantarget{BEDC.<...>.<theorem>}  % must resolve under lean4/BEDC/
    \bridgestatus{none}
    \notclaimed{<one-sentence list of what the chapter does NOT close>}
    \upgradepath{<one-sentence next-grade obligation>}
  \end{closurestatus}

Pick `<theorem>` from the chapter's already-`\leanchecked` set, in priority order: (1) `*semanticNameCert` / `*_semantic_name_certificate`, (2) `*_name_certificate`, (3) `*_stability_certificate_fields`, (4) `*_laws`. `bedc_ci.py audit` rejects a `closure_mark` whose `\leantarget` doesn't resolve to a real Lean declaration. Closure proposals always rank ahead of new theory_extensions for the same chapter — adding the closurestatus block retires the chapter from `critical_path.top` and frees Phase B to attack new fronts. To detect closure-eligible chapters: from the critical_path output, check chapters with `thms >= 10` that are still in `top`; their NameCert is likely complete and just needs the explicit closurestatus block.

If a top-3 chapter is technically blocked (its schema explicitly notes infrastructure not yet in BEDC), record the blocker in `evidence` and use `top[3..]` instead. Do NOT silently fall back to depth-refinement of saturated horizons (Bool / Option / Sum / Prod / List / Rat / Tagged Option / Framed List).

## Root-of-tree unblock priority (HARD GATE)

`critical_path.py` JSON also emits `top_root_unblocks`: chapters with `thms < deps_ready_threshold` that are blocking the largest fan-outs of downstream chapters. Each entry has shape:

```json
{
  "name": "<chapter>",
  "thms": <current marker count>,
  "unblock_count": <downstream chapters this chapter alone unblocks>,
  "deps": [...],
  "file_paper": "papers/bedc/parts/concrete_instances/<...>.tex"
}
```

These are typically root-of-tree paper stubs (5-10 lines, no `\input`, no `\label`) that external commits referenced as dependencies but never gave obligation theorem blocks. Without obligation content, lean cannot register `\leanchecked` markers → downstream chapters stay `deps_ready=False` → `critical_path.top` stays starved (currently ~12 sibling fronts vs 200+ open horizons). Attacking root stubs has 5-10x fan-out leverage compared to depth-refining already-rich chapters.

**HARD GATE**: if `top_root_unblocks` is non-empty AND its first entry has `unblock_count >= 1`, AT LEAST 1 of the proposed targets MUST be `kind = "theory_extension"` on a chapter from `top_root_unblocks[0..2]`. The target's `paper_files[0]` is the entry's `file_paper`. The diff must add at least 3 `\begin{theorem}` / `\begin{definition}` blocks with `\label{thm:<chapter>-...}` so the chapter's `thms` crosses `deps_ready_threshold`. Theorem statements should mention BEDC kernel constructs (`BHist`, `Cont`, `hsame`, `NameCert`, `UnaryHistory`) and articulate the chapter's NameCert obligations as the schema would have demanded had the chapter been written by hand.

Why threshold `>= 1` (was `>= 3` before v4.3): in mutual-blocker dep tree (typical when `top` collapses to 0 — multiple `thms=0` peers blocking the same downstream), every root has `unblock_count=1` because no root is a SINGLE-blocker. Skipping the GATE because `unblock_count < 3` is the bug that produced the 4-day stall on 2026-05-08 (top: 0, top_root_unblocks: 10 entries × unblock_count=1, paper rounds did 240+ closure-axis bookkeeping commits without opening any new front).

`closure_mark` targets do NOT satisfy this gate — they bump closurestatus axis, not `thms`. The two HARD GATES (top[0..2] and top_root_unblocks[0..2]) compose: a 3-target round can satisfy both with 1 top-3 target + 1 root-unblock target + 1 free slot.

If `top_root_unblocks` is empty (every dep tree root is already at `thms >= threshold`), this gate is vacuous and may be skipped.

## TOP-LEVEL HARD GATE — empty-root attack (paper opens new fronts when top is starved)

`critical_path.py` JSON exposes `top_empty_roots[*]` (with `top_empty_roots_total`). Each entry is an open horizon where `max(thms, labels) == 0` — fully empty paper stub that has been referenced as a dependency by other chapters but never received any obligation content. Sorted by `transitive_downstream` descending (how many chapters transitively depend on this one).

```json
{
  "name": "affinespace",
  "thms": 0,
  "labels": 0,
  "deps": ["group", "vecspace"],
  "transitive_downstream": 3,
  "direct_downstream": 1,
  "file_paper": "papers/bedc/parts/concrete_instances/184_affinespace_namecert_construction.tex"
}
```

`top_empty_roots` exists because `top_root_unblocks` measures SINGLE-blocker fan-out only; in mutual-blocker bottom layers (e.g. affinespace + bilinform + chernweil + clifford all `thms=0`) no root is a single blocker for any downstream, so `top_root_unblocks[*].unblock_count` collapses to 1 even when the transitive impact of writing the schema is much larger. `top_empty_roots` measures the transitive metric directly.

**HARD GATE — empty-root attack priority**: If `len(top) == 0` AND `top_empty_roots_total >= 1`, AT LEAST 1 of the proposed N targets MUST be `kind = "theory_extension"` on a chapter from `top_empty_roots[0..2]`. The target's `paper_files[0]` is the entry's `file_paper`. The diff must add at least 5 `\begin{theorem}` / `\begin{definition}` blocks with `\label{thm:<chapter>-...}` BHist-anchored statements articulating the chapter's NameCert obligation surface — concrete BHist/Cont/hsame/NameCert/UnaryHistory predicates, not abstract parameter-echo. This drives `thms` and `labels` past `deps_ready_threshold` so the chapter (and its transitive downstream) becomes eligible for `top` on subsequent rounds.

This gate composes with bridge_sync_pending / drift_chapters / transition rotation gates: a 3-target round can satisfy 1 empty-root + 1 drift-sync + 1 transition closure_mark. The empty-root target counts as the round's `theory_extension` contribution for the rolling-window `theory_extension` requirement.

If `len(top) >= 1`, this gate is vacuous (the existing `top[0..2]` HARD GATE handles new-front opening). If `top_empty_roots_total == 0`, this gate is also vacuous.

Why this gate exists (added in v4.3): on 2026-05-08 the dep tree's bottom layer became fully mutual-blocked: `top: 0`, `top_root_unblocks` had 10 entries all with `unblock_count=1`, so the existing `unblock_count >= 3` GATE never fired. Paper rounds spent ~3000 commits in 24h on closure-axis bookkeeping while 94 empty-stub chapters sat un-attacked. The `top_empty_roots` GATE forces paper to open new fronts whenever `top` is starved.

## Theme saturation (HARD GATE)

After the file-frequency check in `Anti-saturated-file` above, also compute the **theme** of `paper_files[0]`:

```
theme = stem(paper_files[0]) before "_namecert"
e.g. "08_option" → option,  "11_list" → list,  "10_sum" → sum
```

Run `git log --oneline -20 -- papers/bedc/parts/` and count how many of the last 20 paper-side commits touched a chapter sharing this theme. If `theme_count >= 6`, the candidate is INVALID **unless** the theme corresponds to a `top[0..2]` node from the critical-path JSON AND `theme_count < 12`. The library has already received six rounds of attention on that theme; even for a top-3 critical-path node, going past 12 commits in a 20-commit window means the chapter is saturated relative to the rest of the library and the marginal value of a 13th hit is below opening a fresh chapter (especially the recently-added paper-only stubs under `papers/bedc/parts/concrete_instances/` like `16_group_namecert_construction.tex`, `56_magma…`, `57_semigroup…`, `58_subgroup…`, `60_quotientgroup…`, etc.). Pick a chapter from those instead.

### Hot-anchor avoidance (within a file)

Even when the file passes theme saturation, also run:

```bash
git log --oneline -20 -- <paper_files[0]> | head -20
```

and `git show --stat -1 <sha>` for each commit to read its commit body. If the SAME anchor section (the theorem/definition LABEL referenced in the commit body, e.g. `thm:ratup-fieldup-affine-centralizer-normalizer-orbit-action-invariance`) has appeared in `>= 4` of the last 20 commits to that file, that anchor is HOT and must NOT be picked. Pick a different anchor in the same file, or move to a different chapter.

## Schema-only unlock priority (HARD GATE)

The Lean side is currently blocked on 16 algebra/order chapters whose paper schemas write laws as parametric operators (`mul / add / neg : BHist -> BHist -> BHist` left abstract). These chapters are listed in `lean4/scripts/critical_path.py::SCHEMA_ONLY_HORIZONS`:

```
abgroup, group, monoid, ring, commring, field,
module, vecspace, linearmap, matrix, polynomial, fps,
lattice, totalorder, preorder, poset
```

A Lean round picking such a horizon can ONLY produce parameter-echo schema (`(name : forall x y z, hsame …)`), which the phase D mechanical lint rejects. They stay frozen until the paper side adds a *concrete carrier instance*: a definition that pins the abstract carrier to a specific BHist predicate and the abstract operation to a specific BHist function whose laws are downstream of FKernel theorems.

**When `critical_path.py` `top` returns < 3 candidates, OR when the rolling window otherwise has weak depth targets**, you MUST propose at least one `theory_extension` target with `kind = "theory_extension"` and `notes.flavor = "concrete_instance"` of one banned chapter. The proposed paper change must:

1. Pick a chapter from `SCHEMA_ONLY_HORIZONS`. Prefer the simplest still-banned chapter you have not yet unlocked (start with `monoid`, then `group`, then `abgroup`, …).
2. Add a paper definition along the lines of:
   ```latex
   \begin{definition}[Concrete history monoid instance]
     Take Carrier := UnaryHistory, mul := Cont, e := \mathsf{BHist.Empty}.
     The pair (UnaryHistory, Cont, Empty) is a concrete monoid instance
     whose associativity follows from cont_assoc and whose unit laws
     follow from cont_left_unit, cont_right_unit.
   \end{definition}
   ```
   The concrete `Carrier` MUST be a `BHist -> Prop` predicate that already exists somewhere under `papers/bedc/parts/` (e.g. `UnaryHistory`, `BoolHistoryCarrier`, `RatHistoryCarrier`, …). The concrete operations MUST be specific BHist-valued functions (`Cont`, `append`, `cons` …), not abstract parametric symbols.
3. State at least one law verification *theorem* about this concrete instance, citing FKernel theorems that underwrite each law (e.g. `cont_assoc → mul_assoc`, `cont_left_unit → mul_one_left`).
4. Add a note in the chapter that the schema is now instantiated, so future Lean rounds can produce BHist-anchored proofs about this concrete instance instead of parameter-echo schema.

Once such a paper block lands, the chapter exits `SCHEMA_ONLY_HORIZONS` (we update `critical_path.py` separately), and Lean rounds can attack it without producing parameter echo. **One unlock per round is enough.** The 16 banned chapters are 16 high-leverage extensions; do not stack two unlocks in a single review round.

## Cross-chapter extension (HARD GATE — high priority)

When the candidate pool is otherwise weak (no top-3 critical-path target with a clear theorem, no schema-unlock backlog), prefer a cross-chapter theorem that **combines `\leanchecked` results from ≥ 3 different concrete-instance chapters** into a single conclusion stated about concrete BEDC kernel objects. Vision chapters may supply inspiration for the target, but the proposed write target is a concrete-instance chapter.

A cross-chapter target MUST satisfy ALL of:

- `kind = "theory_extension"` AND `notes.flavor = "cross_chapter"`.
- `cited_sources` field listing ≥ 3 fully-qualified Lean names from ≥ 3 different `BEDC.Derived.*` chapters.
- `proposed_change` stating a concrete theorem whose proof composes the cited sources into a known external result (Cauchy completeness for RealUp; uniform continuity on compact metric spaces; Yoneda lemma in BEDC encoding; group isomorphism theorems; S¹ connectedness; supremum existence on bounded subsets of Real; …).
- `downstream_use` naming the named external result (e.g. "constructive Cauchy completeness", "Yoneda lemma", "first isomorphism theorem").

Cross-chapter targets score higher than ordinary `missing_companion` candidates: they consume coverage rather than produce more of it. When you emit such a target, set its `quality_score = 10` and `selection_rank = 1`. If you cannot identify a supported target with ≥ 3 cross-chapter `\leanchecked`, do not invent one — ordinary `theory_extension` is fine, but record `cross_chapter_attempted = true` in `notes` so the rolling window knows you tried.

**Cross-chapter candidates to keep an eye on (illustrative, not exhaustive — pick one whose support is actually present):**

| Target | Required cross-chapter support |
|---|---|
| Real Cauchy completeness | RealUp + RatUp + ContinuousUp |
| Compact-metric uniform continuity | CompactUp + MetricUp + ContinuousUp |
| Group isomorphism / first iso theorem | GroupUp + AbGroupUp + NameCert (after monoid/group are unlocked) |
| Constructive Yoneda | CategoryUp + FunctorUp + NatTransUp |

When a cross-chapter theorem lands, place it in the concrete-instance chapter that consumes the result and use ordinary semantic labels so future review rounds can avoid duplicating it.

## Theory-extension quota (HARD GATE)

In any rolling window of 5 consecutive paper rounds, **at least 1 round** must emit a target with `kind = "theory_extension"`. Inspect the last 4 paper-round commits via `git log --oneline -4 -- papers/bedc/parts/` and look at their `prompts: vN.M` body line plus the commit subject — if none of them committed a theory_extension, this round MUST. Record the contribution in `notes`.

If no genuinely valid `theory_extension` candidate exists this round (the candidate pool is empty for that kind), emit `{"targets": []}` and the obligation rolls forward to the next round.

## Roadmap-phase hard gate

After reading `papers/bedc/parts/project_governance/roadmap.tex`, identify the chapter currently labelled as the OPEN phase (look for headings like "## Currently OPEN:" or the most recent phase block whose exit criteria are unmet) and emit it as `current_open_phase` in the JSON output.

For each `targets[i]`, the `roadmap_phase` field MUST be exactly one of:
- the value of `current_open_phase` (the target advances the open phase);
- `"discipline"` (the target enforces a `Section roadmap` "claims explicitly off the table" rule);
- a non-current phase IF accompanied by an `out_of_phase_justification` field of one short sentence stating why this target must be done now (it blocks the open phase, fixes a build break, etc.).

Phase Revise will reject the round if `roadmap_phase` is missing, empty, or names a phase other than `current_open_phase` without `out_of_phase_justification`.

## Candidate pool and selection rule

Before choosing the emitted target, build a private candidate pool of 3–5 distinct valid candidates, or every valid candidate if fewer than 3 exist. Score each candidate with the 0–12 rubric above and assign a realistic `difficulty`:
- `high`: substantive theorem/definition/proof/certificate target that combines multiple existing results, has a non-trivial proof skeleton, and still fits in 1–2 paper files without broad rewrite.
- `medium`: meaningful theorem/definition/proof/certificate target with a real dependency chain but mostly local implementation.
- `low`: localized companion theorem or clarification that is useful but close to an existing statement.

Select the hardest feasible candidate, not the easiest implementable one. Prefer `high` over `medium` over `low` when scores are tied or close. A `low` target may be emitted only if no `medium` or `high` candidate passes the validity, locality, and downstream-use gates. Do not inflate difficulty: an unbounded rewrite, speculative theorem, or target likely to fail compile is not feasible and must be rejected rather than labeled `high`.

## Capstone bridge dimension (HARD GATE)

The 0–12 quality score above is supplemented by a 7th dimension specifically tracking cross-chapter unification:

- `cross_chapter_unification`: 0–2.

  How many distinct chapters does this theorem statement reference via cited `\leanchecked` names or via concrete object references (e.g. `CategoryHomCarrier`, `RealStreamClassifier`, `CompactWitnessCarrier`, `MetricDistanceWitness`, `Cont`, `UnaryHistory`)?

  - 0: single-chapter theorem (e.g. another OptionUp closure, another RatUp endpoint readback). The vast majority of existing rounds score 0 here.
  - 1: spans 2 chapters. Common but not unifying.
  - 2: spans ≥ 3 chapters AND the conclusion is an *equivalence / iff / bridge / correspondence* statement that ties the cited constructions together (not "uses lemma A to prove fact B in chapter C").

Selection priority override: a `theory_extension` candidate with `cross_chapter_unification = 2` has **ABSOLUTE PRIORITY** over any candidate with `cross_chapter_unification ≤ 1`, even if its total `quality_score` is up to 2 points lower.

Concrete examples of `cross_chapter_unification = 2` candidates:

- bridge between `CategoryHomCarrier` (ch.36 Category) and `Cont` (FKernel kernel) via `UnaryHistory`: theorem stating that the category-level Hom carrier is precisely the unary-restricted continuation predicate.
- bridge between `RealStreamClassifier` (ch.13 Real) and `MetricDistanceWitness` (ch.32 Metric) via shared dyadic prefix structure: theorem stating that real-stream equivalence and dyadic-tolerance metric equivalence coincide on regular sequences.
- bridge between `CompactWitnessCarrier` (ch.35 Compact) and a located/totally-bounded primitive: theorem extracting Bishop-style located + ε-net data from the BEDC five-element compact carrier.

Add `cross_chapter_unification: 0|1|2` as a field inside every `quality_score` JSON object, both in `candidate_pool` entries and in emitted `targets`. Without this field the round is malformed.

## Cross-chapter placement (HARD GATE for cross_chapter_unification = 2)

When the selected target has `cross_chapter_unification = 2`, its
`paper_files[0]` MUST be the `.tex` file of the deepest-dependent
concrete-instance chapter among the cited sources. The bridge may read
`papers/bedc/parts/visions/` for context, but the mature theorem lands in
`papers/bedc/parts/concrete_instances/`.

Heuristic:
- Bridge whose conclusion is an iff / equivalence / correspondence between
  constructions from different chapters → choose the chapter whose
  NameCert construction consumes the most downstream structure.
- Bridge whose conclusion strengthens one chapter's existing theorem by
  importing a premise from another chapter → choose that chapter.

For all `cross_chapter_unification` values, `parts/visions/` MUST NOT be
selected as `paper_files[0]` unless the target is only a
`\concretizedIn{...}{...}` marker append on an existing vision file.

## HARD constraints — every target MUST respect these
The next phase will be REJECTED if its diff violates any of these, so do not propose a target that requires violating them.

1. **No iteration narrative.** The paper is a current-state document. Do NOT suggest text containing "新增", "修订", "新版", "patch", "增量", "modification", "migration", "increment", "legacy", "supersede", "deprecated", "frozen", "rectification", "amendment", "v1.5.X" or similar. Replacements should be silent: delete the old, write the new, no version annotations.
2. **No version-prefixed labels.** Do NOT propose labels like `thm:vN-…`, `sec:v14-…`. Use semantic labels (`thm:<concept>`, `sec:<theme>-<concept>`).
3. **Every `.tex` file stays ≤ 800 lines.** If implementing a target would push a file over the cap, the target must include a split plan (extract a focused sub-chapter file).
4. **Math notation.** Inline math via `$…$`; display math via `$$…$$` on its own line. Do NOT propose introducing `\[…\]`, `\begin{equation}`, `align`, etc.
5. **Marker discipline.** Each paper site has at most ONE `\leanchecked`. New paper-side markers must reference Lean targets that ALREADY exist under `lean4/BEDC/` (run a quick grep before committing the suggestion). Avoid proposing targets whose ONLY effect is to add `\leanvariant` lines (the register-only gate rejects pure-marker rounds), but `\leanvariant` is allowed when it accompanies real revision (e.g. splitting one paper block into two propositions, each pointing to a different Lean target). Prefer collapsing a duplicate paper block by deleting it entirely and letting the dropped Lean name become an orphan, since the drift audit accepts orphan Lean targets.
6. **Underscore escape.** Inside `\leanchecked{…}` and friends, an underscore in a Lean name MUST be `\_` (not bare `_`).
7. **Mathlib-free / 0-axiom Lean.** If the target touches Lean code, do not propose patterns that would require `Mathlib.X.Y` imports or new `axiom`/`sorry` declarations.
8. **No new files at the top of the tree.** New `.tex` material lives inside an existing `parts/<theme>/` directory.

## Discovery hygiene
Before suggesting that a missing marker / missing companion result is REALLY missing, do a quick grep:
- For a paper claim X, run `grep -rE "<X-related symbol>" lean4/BEDC/` to see whether the Lean side already has it.
- For a "missing definition" complaint, run `grep -rE "\\\\(begin\{definition\}|leandef\{)" papers/bedc/parts/` for a similar concept.
- For a "redundant section" complaint, confirm by `grep` that the alleged duplicate is actually content-identical.

Before producing JSON, build an inventory of real paper files:
- Run `find papers/bedc/parts -name "*.tex" | sort`.
- Every existing path in `paper_files` MUST be copied from this inventory exactly.
- Do NOT invent companion file names. If the natural target file does not exist, attach the target to an existing nearby parent chapter and include a line-cap split plan only when needed to place the substantive content in child files.
- A target with a nonexistent existing file in `paper_files` is invalid even if the concept is good.

Do NOT propose targets that fail this sanity check. A split-only target is INVALID even if the file is long; attach splitting only to a substantive theorem/definition/proof/certificate change that needs it. A target whose only effect is to register a marker pointing to a Lean theorem that does not exist is INVALID.

## HARD GATE — paper-label claim race-prevention

For every target you propose that would INSERT a new `\label{...}` block (i.e. its `kind` is `theory_extension` / `discovery` / etc., creating a new theorem/definition rather than only editing existing prose), atomically claim that label before emitting the JSON. This prevents two concurrent rounds (this paper round + any in-flight R-round on the lean side) from each writing the same `\label{thm:<X>}` and triggering a post-rebase audit failure that recovery has to undo.

Derive each target's label slug (typically `thm:<concept>` from the target's primary `\label{...}` field). Build a comma-separated list and run:

```bash
python3 lean4/scripts/claim_paper_label.py claim \
    --round P<ROUND_NUM> \
    --labels thm:foo-bar,def:baz-qux
```

The script returns JSON `{"kept": [...], "dropped": [{"label": "...", "reason": "..."}]}`:

- Each `kept` label is now reserved for this round (TTL 30 min).
- Each `dropped` label is either `already_on_disk` (label already exists in `papers/bedc/parts/`) or `claimed_by_<round>` (another in-flight round reserved it).

Drop ANY target whose label appears in `dropped`. If all targets are dropped, emit `{"targets": []}` — empty round is preferred. Lock file lives in `<repo>/.git/bedc-paper-label-claims.json`. TTL auto-frees crashed rounds.

Skip targets that don't introduce a new label (pure prose tightening, marker-only edits, structural splits) — those can't collide.
