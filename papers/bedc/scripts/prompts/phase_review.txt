## What counts as a valid review target
Pick targets that are HIGH-VALUE — each should be something a careful reader would flag during peer review. Acceptable categories:

1. **Logical gap or unjustified step.** A theorem statement that is used downstream but whose proof skips a non-trivial case, or a definition that is invoked before being properly fixed.
2. **Notational or definitional inconsistency.** Same symbol used with two meanings; a definition restated differently in two chapters; an undefined macro or label dangling.
3. **Dead/redundant material.** A definition or proposition that is never referenced; a section whose content is fully subsumed by another; theorem hypotheses that are vacuous or implied.
4. **Missing companion result.** A claim made informally but not stated as a theorem; a proof obligation that the Lean side already satisfies (`\leanchecked` exists) but the paper has no `\leanchecked{X}` marker; a concrete instance for an abstract construction that is asserted to exist but is not exhibited.
5. **Unclear or imprecise statement.** A theorem whose hypotheses must be tightened (or relaxed) for the conclusion to hold as currently written; an "obvious" appeal that hides a real argument.
6. **Stronger statement available.** The paper proves a weaker form when the same proof yields more (e.g. uniform bound proved pointwise; restriction to finite case when general case is identical).
7. **Cross-reference / structural improvement.** A theorem block that is mis-located (would be easier to follow earlier or later); a result that should be promoted from remark to numbered statement, or vice versa.
8. **Theory extension awaiting formalization.** Propose a NEW theorem statement (not yet in the paper, not yet in Lean) that follows from the existing kernel and is worth formalizing. Acceptable shapes:
   - **Composite consequence.** A non-trivial conclusion combining 2+ existing `\leanchecked` theorems, where the combined statement is not already in the paper.
   - **Constructor inversion / determinism / no-confusion lemma** for an existing `inductive` (BMark / BHist / ProbeBundle / SigRel / psame / external binary words / derived-carrier inductives) that does not yet have one.
   - **Generalization of an existing checked theorem** with a strictly broader hypothesis or a strictly stronger conclusion, provable by the same proof skeleton.
   - **New derived predicate or function** over a kernel object (e.g. a structural property of `BHist`, a new well-founded measure, a closure operator) plus its key well-formedness theorem.
   - **NameCert spec for a new canonical math object** in the BEDC licensing horizon that is not yet covered (BoolUp / OptionUp / ProdUp / SumUp / ListUp / IntUp / RatUp / RealUp / ComplexUp / FieldUp / ModuleUp / VecSpaceUp / MatrixUp / PolynomialUp / S1Up / MetricSpaceUp / CompactUp / CategoryUp / FunctorUp / NatTransUp). State carrier, classifier, and the at-least-one structural property the certificate must satisfy.
   - **Coverage / completeness theorem** stating that some existing construction exhausts a class (e.g. every admitted history under domain D admits an InGap witness; every constructor pair gives the same psame outcome; etc.).

   The proposal MUST be provable from the existing Lean kernel without `axiom`, `sorry`, `Mathlib.*`, `Classical.*`, or `Quot.*`. The statement must mention concrete BEDC kernel/setup objects (BMark, BHist, msame, hsame, Ext, Cont, ProbeBundle, SigRel, sameSig, Pkg, InGap, NameCert, SemanticNameCert, BaseReflectionSetup, AskSetup, PackageSetup, DomainSetup, NameCertSetup, or a `BEDC.` namespace symbol).

   The revise phase will INSERT this as a new `\begin{theorem}` / `\begin{lemma}` / `\begin{definition}` block at a thematically correct chapter location. It will write a brief PROSE proof sketch in `\begin{proof}…\end{proof}`. It will NOT add a `\leanchecked` marker — that is the next-Lean-round's job. It will NOT add `\leanvariant`. The Lean side runs an independent formalize pipeline that scans `papers/bedc/parts/` for theorem sites WITHOUT a `\leanchecked` marker and picks them up.

   INVALID shapes (skip silently):
   - Re-statement of an existing checked theorem with trivial reformulation.
   - A theorem whose proof reduces to one of `exact T` / `apply T` / `T.symm` / `T.mp` / `T.mpr` for a single existing `\leanchecked` theorem T (those are variants, never extensions).
   - A claim about an object that does not yet exist anywhere in the kernel.
   - Vague conjectures with no precise statement.

Reject these as low-value (do NOT propose):
- Pure typographical fixes, spelling, missing periods, kerning.
- Comment changes / inline notes (the project bans inline comments anyway).
- Adding "TODO", "future work", or version annotations.
- Edits that only relabel without changing logical content.
- Renaming for cosmetic uniformity unless name collision causes ambiguity.

## HARD constraints — every target MUST respect these
The next phase will be REJECTED if its diff violates any of these, so do not propose a target that requires violating them.

1. **No iteration narrative.** The paper is a current-state document. Do NOT suggest text containing "新增", "修订", "新版", "patch", "增量", "modification", "migration", "increment", "legacy", "supersede", "deprecated", "frozen", "rectification", "amendment", "v1.5.X" or similar. Replacements should be silent: delete the old, write the new, no version annotations.
2. **No version-prefixed labels.** Do NOT propose labels like `thm:vN-…`, `sec:v14-…`. Use semantic labels (`thm:<concept>`, `sec:<theme>-<concept>`).
3. **Every `.tex` file stays ≤ 800 lines.** If implementing a target would push a file over the cap, the target must include a split plan (extract a focused sub-chapter file).
4. **Math notation.** Inline math via `$…$`; display math via `$$…$$` on its own line. Do NOT propose introducing `\[…\]`, `\begin{equation}`, `align`, etc.
5. **Marker discipline.** Each paper site has at most ONE `\leanchecked`. New paper-side markers must reference Lean targets that ALREADY exist under `lean4/BEDC/` (run a quick grep before committing the suggestion). Avoid proposing targets whose ONLY effect is to add `\leanvariant` lines (the register-only gate rejects pure-marker rounds), but `\leanvariant` is allowed when it accompanies real revision (e.g. splitting one paper block into two propositions, each pointing to a different Lean target). Prefer collapsing a duplicate paper block by deleting it entirely and letting the dropped Lean name become an orphan, since the drift audit accepts orphan Lean targets.
6. **Underscore escape.** Inside `\leanchecked{…}` and friends, an underscore in a Lean name MUST be `\_` (not bare `_`).
7. **Mathlib-free / 0-axiom Lean.** If the target touches Lean code, do not propose patterns that would require `Mathlib.X.Y` imports or new `axiom`/`sorry` declarations.
8. **No new files at the top of the tree.** New `.tex` material lives inside an existing `parts/<theme>/` directory.

## Discovery hygiene
Before suggesting that a missing marker / missing companion result is REALLY missing, do a quick grep:
- For a paper claim X, run `grep -rE "<X-related symbol>" lean4/BEDC/` to see whether the Lean side already has it.
- For a "missing definition" complaint, run `grep -rE "\\\\(begin\{definition\}|leandef\{)" papers/bedc/parts/` for a similar concept.
- For a "redundant section" complaint, confirm by `grep` that the alleged duplicate is actually content-identical.

Do NOT propose targets that fail this sanity check. A target whose only effect is to register a marker pointing to a Lean theorem that does not exist is INVALID.

## Output Format (MUST follow exactly)
Output a single fenced JSON block:

```json
{
  "targets": [
    {
      "kind": "logical_gap | notation_inconsistency | dead_material | missing_companion | unclear_statement | stronger_available | structural | theory_extension",
      "paper_files": ["papers/bedc/parts/core/05_xxx.tex"],
      "section_label": "sec:core-some-concept",
      "anchor": "thm:concrete-name | def:concrete-name | the line near `\\begin{theorem}[Title]`",
      "summary": "One-sentence summary of the problem.",
      "evidence": "Concrete quotes / line refs / grep results that establish the issue is real.",
      "proposed_change": "Precise instructions for the revise phase: what to delete, what to add, where to insert.",
      "expected_effect": "What logically improves once the change lands.",
      "lean_touchpoint": "(optional) Lean target that should be referenced or registered, e.g. `BEDC.FKernel.X.foo`. Use `\\_` for underscores when it will appear in `\\leanchecked{...}`.",
      "difficulty": "low | medium | high",
      "chapter": "core | proof_obligations | concrete_instances | concrete_hardening | hardening | formalization | proof_sprint | proof_standing | project_governance | appendices",
      "roadmap_phase": "(name of the roadmap phase this target advances, or 'discipline' if it enforces a Section-roadmap-off-the-table claim, or 'maintenance' if it is a pure discipline hygiene fix outside any phase)",
      "discovery_confirmed": true
    }
  ],
  "notes": "(optional) Anything the revise phase should know that does not fit a per-target field."
}
```

Field requirements:
- `paper_files` lists EXACTLY the .tex file(s) the revise phase must edit. Do not list main.tex unless main.tex itself is what changes.
- `proposed_change` must be precise enough that the next phase can act without re-reading the entire chapter. Include line context, surrounding sentences, label names.
- `discovery_confirmed` must be `true`, meaning you ran the grep / file inspection and the issue is real (not hallucinated from memory of an old draft).
- Do NOT propose targets that contradict the HARD constraints above.
- Do NOT propose targets that consist only of adding `\leanvariant` markers; variants are preserved as-is and not added by automated rounds.
