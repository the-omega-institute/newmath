## Multi-target round support — EXECUTION CONTRACT (mandatory)

Phase Review may emit up to 5 targets in `targets[]` (not just `targets[0]`). This is NOT a "pick your favorite" list. You MUST attempt every target in `targets[0..N-1]`. A round that commits only target 0 — leaving valid targets 1..N-1 untouched — is INCOMPLETE and wastes the round's codex-exec budget; targets 1..N-1 must be re-emitted by a later Phase Review and re-implemented, multiplying total compute cost.

The previous version of this prompt was misread as allowing "do the first one and stop". It does not. Explicit rules:

- **Process every target in array order.** After committing one target (or noting it as already-applied), continue to the next. Do NOT terminate the codex run after a single target completes successfully.
- **closure_mark / drift / bridge_sync / orphan_repair are CHEAP** (typically 1-3 line edits each). These should be processed FIRST in priority order — even before the higher-difficulty `human_chapter_extension` or `theory_extension` targets — because they close horizons immediately and have minimal merge-conflict surface.
- **One commit per target**: each applied target gets its own commit with subject formatted per Step 6 below (e.g. `P<N>: sync <X>Up formalstatus to axiomCleanV` for drift sync). This keeps merge conflict surface small (1-2 file changes per commit) and makes the recovery consumer's job easier when a single target's lake build fails.
- **Sequential apply, sequential commit, single push at end**: after ALL targets in the round have been processed (committed or skipped), the round's final push to its branch happens once. Do NOT push after each individual target commit — the orchestrator's merge step pushes the round's branch tip to BASE in one operation.
- **Failed target ≠ failed round**: if target `i` cannot apply (already done, prerequisite missing, file at line cap), skip it with a `git commit --allow-empty -m "P<N>: skip target i — <reason>"` skip-marker commit and continue. Skip markers are dropped by the orchestrator's `git filter` before merge, but produce a clear log of which targets were attempted.
- **0 targets applied is rare**: if every target was skip-able, the round may emit empty. But this should be extremely uncommon in a 4-5 target round.

Between targets:

- Re-run `make check` after applying each substantive target to confirm intermediate state compiles. Quick `closure_mark` targets (1-3 line edits) can be batched without intermediate compile.
- If target `i` fails to apply (e.g. file already at desired state, prerequisite missing), commit a skip-marker and continue to target `i+1`. Do NOT abort the whole round.
- Each applied target gets its own commit; the round's commit chain accumulates N entries (1 per applied target).

## Hub recognition rule

A `.tex` file may already be a HUB — i.e. previously split, with all substantive content moved into child files referenced by `\input`. Detect it on `paper_files[0]` before editing:

```bash
f=<paper_files[0]>
total=$(grep -cEv '^\s*$|^\s*%' "$f")
inputs=$(grep -cE '^\s*\\input\{' "$f")
own=$(grep -cE '\\(chapter|section|begin\{(theorem|definition|proposition|lemma|corollary|proof)\})' "$f")
```

If `total <= 30` AND `inputs * 100 >= total * 80` AND `own == 0`, the file is a HUB and you MUST NOT append theorem / definition / proof / certificate content to it. Two valid moves:

1. Pick the relevant `\input`-ed child file (one of the paths the hub references) and edit that child instead. The original target's chapter / section / theorem labels remain intact because they live in the child.
2. If no existing child fits the new content, create a new sibling child under the same hub-subdirectory (e.g. `papers/bedc/parts/concrete_instances/prod/<new-child>.tex`), put the new content there, and append a single `\input{...}` line to the parent hub in alphabetical / numeric order.

Appending substantive content directly to a hub will be reverted within a few rounds by a "keep paper sites singular" cleanup commit, wasting your round's codex-exec budget. Phase REVIEW should already have rejected such targets — if you see a hub as `paper_files[0]` for a non-split content target, treat it as a Phase REVIEW miss and abort the round per Step 0 rather than committing the broken edit.

## Step 0.7: Proof-prose readability (HARD GATE)

When the target adds or rewrites a `\begin{proof}…\end{proof}` block, the prose MUST be paragraph-segmented for readability. A monolithic 8–15-line wall of text running through "Apply Theorem X. Apply Theorem Y. Apply Theorem Z. ..." is rejected even if it builds.

Concrete formatting rules:

1. **Blank-line paragraph break** between logically distinct moves. Each of the following starts a new paragraph (= separate it from the previous prose by a blank line in the .tex source):
   - the forward direction vs the reverse direction of an `iff`;
   - each case of a case analysis (e.g. "absent/absent", "present/present", "mixed");
   - each induction step (base / cons / append / etc.);
   - each major lemma application chain that produces a witness used downstream.

2. **At most ~4 consecutive `Apply Theorem~\ref{...}` invocations per paragraph.** If a chain needs more, break the proof into named steps:

   ```
   We first build the spine representation. Apply ... and ... .

   Next we transport the classifier. Use ... .

   Finally, repack into the public certificate. Apply ... .
   ```

3. **Display math (`$$...$$`) goes on its own paragraph**. Do not embed a `$$...$$` block inside a single sentence followed by more prose; surround it by blank lines so it reads as a step.

4. **No paragraph longer than ~6 lines** of source text. If a proof step is genuinely longer (e.g. a delicate constructor inversion), break by inserting a fresh sentence that names the next sub-goal ("It remains to show ...", "We now turn to ...").

5. **Prefer `\begin{enumerate}` for 3+ parallel sub-claims** instead of "First, …. Second, …. Third, …." run-on prose. Each `\item` is one paragraph.

6. Existing well-formatted proofs in the same chapter are the local model — read 2–3 nearby `\begin{proof}` blocks before writing yours, and match their cadence.

A revise round whose new proof block violates rules 1–4 (one solid wall, no blank-line breaks, >4 sequential `Apply` invocations in one paragraph) is INVALID. Re-split the prose before committing.
