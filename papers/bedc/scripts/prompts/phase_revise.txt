## Hard Constraints — DO NOT violate these
The Phase Verify step will reject the round if any of these is broken.

1. **No iteration narrative anywhere in the diff.** The forbidden vocabulary list (banned even in commit body and prose) is:
   `新增`, `新版`, `修订`, `修复了上一版本`, `patch`, `migration`, `frozen`, `supersede`, `superseded`, `deprecated`, `legacy`, `increment`, `增量`, `修改记录`, `变更记录`, `变更原因`, `rectification`, `amendment`, `v1.5.X`, `v1.4.X`, `v14-…`, `v15-…` (or any `vNN-` prefix in labels).
   The paper is a CURRENT-STATE document. To replace old text with new text: delete the old, write the new, do not annotate the change.
2. **Single .tex file ≤ 800 lines after your edits.** If a substantive theorem/definition/proof/certificate edit would make a file exceed 800 lines, you MUST split: extract content into a focused child file in the same `parts/<theme>/` directory or a thematic subdirectory below it, then leave the parent as a pure index containing only ordered `\input` lines. Splitting by itself is not a valid round.
3. **Math notation.** Use only `$…$` for inline math and `$$…$$` (each `$$` on its own line) for display. Do NOT introduce `\[…\]`, `\begin{equation}`, `\begin{align}`, `\begin{equation*}`, `\begin{eqnarray}`, etc.
4. **No inline file comments.** Do not add `% TODO …` style notes, do not insert "see git log for history" lines, do not add change-tracking comments. The paper is the artifact; the diff is the audit trail.
5. **Marker discipline.**
   - At most one `\leanchecked{X}` per paper site; multiple variants use `\leanvariant{Y}`.
   - Underscores in Lean target names MUST be `\_` inside `\leanchecked{…}` / `\leanvariant{…}` / `\leanstmt{…}` / `\leandef{…}`.
   - Every marker target X you ADD must be a real declaration under `lean4/BEDC/`. The audit `python3 lean4/scripts/bedc_ci.py audit` rejects unresolved markers.
6. **Lean discipline (if you touch Lean code).**
   - No `import Mathlib.…`. Stdlib + built-in tactics only.
   - No new `axiom`. No new `sorry`. No new `constant`.
   - Do not weaken existing theorem signatures.
   - The audit `python3 tools/check-axioms.py` rejects any `axiom` keyword under `lean4/BEDC/`.
7. **No top-level new directories or files.** Place new content inside an existing `papers/bedc/parts/<theme>/` directory.
8. **Do NOT git push.** The merge and push are handled externally.
9. **Do NOT edit `papers/bedc/main.tex`'s structure.** You may edit it ONLY to add a single `\input{parts/<theme>/<file>}` if you split a chapter. Do not reorder chapters here.
10. **Do NOT edit `lean4/lakefile.lean`, `lean4/lean-toolchain`, or `papers/bedc/preamble.tex` unless a target explicitly requires a new macro.** Adding a macro must be conservative (no breaking changes to existing macros).

## Structural split rule

If an edited `.tex` file is already above 720 lines, or the target would add more than 25 lines to a file above 760 lines, prefer a split instead of appending. This is especially important for concrete instance files that accumulate certificate variants.

Allowed split pattern:
- Create child files under the same theme directory, preferably inside a thematic subdirectory such as `papers/bedc/parts/concrete_instances/list/<child>.tex` or `papers/bedc/parts/concrete_instances/option/<child>.tex`.
- Move all substantive content out of the parent into child files: headings, orientation prose, theorem clusters, definitions, proofs, certificate blocks, and cross-reference prose belong in children.
- Leave the original numbered parent file as a pure index: ordered `\input{...}` lines only, with no heading, no orientation paragraph, no theorem/proof content, and no explanatory prose.
- Preserve existing labels exactly unless the label is semantically wrong.
- Do not edit `papers/bedc/main.tex` when splitting content already included by an existing parent file.
- Use `\input` paths in the same style as the rest of the paper, relative to `papers/bedc/main.tex`.
- After splitting, every touched `.tex` file must be ≤ 800 lines.

If the target includes a line-cap split, perform the split before applying any new theorem/prose content, then apply the content in the appropriate child file. If the target is only a split with no substantive theorem/definition/proof/certificate change, it is INVALID; skip the round rather than committing file movement.

## Step 1: Apply each target
For each target in the list above:

1. Open the listed `paper_files` and locate the `anchor` (theorem/definition block or section).
2. Apply `proposed_change` exactly. Read the surrounding ~30 lines so the edit lands in the correct semantic context, not just by string match.
3. If the target adds a new theorem/definition block: place it inside an existing section, with a semantic `\label{thm:<concept>}` / `\label{def:<concept>}`. NEVER use a version-prefixed label.
4. If the target adds a `\leanchecked` / `\leanstmt` / `\leandef` marker: confirm via grep that the named Lean target exists (`grep -rE "(theorem|lemma|def|inductive|structure|class)\s+<name>" lean4/BEDC/`), then add the marker on the line immediately after `\end{theorem}` / `\end{definition}` / `\end{lemma}` / `\end{corollary}`. Escape underscores with `\_`.
5. If the target removes dead material: delete the entire block AND any references to its label elsewhere in the paper (`grep -rn "<label>" papers/bedc/`). A dangling `\autoref` is a build failure.
6. If the target tightens or generalizes a statement: update both the theorem block AND its proof to match the new statement. Do not leave a proof that proves the OLD statement attached to the NEW statement.
7. If the target requires touching Lean code: edit `lean4/BEDC/...`, write a real proof (no `sorry`, no `axiom`), and verify with `cd lean4 && lake build` that it compiles.
8. If the target's `kind` is `theory_extension` (a new theorem awaiting formalization):
   - Insert a new `\begin{theorem}[<short title>]\label{thm:<concept>}` / `\begin{lemma}` / `\begin{definition}` block at the chapter location given by `paper_files`. Pick a thematically correct subsection — adjacent to similar results.

   - Write a brief PROSE proof sketch in `\begin{proof}…\end{proof}` describing the constructive strategy (which constructor / which existing checked theorem / which induction). The sketch is informational; it does not need to be machine-checkable.
   - Do **NOT** add `\leanchecked`, `\leanvariant`, `\leanstmt`, or `\leandef`. The Lean side runs an independent formalize pipeline that scans `papers/bedc/parts/` for theorem sites without markers and picks them up. Adding a marker now would point at a non-existent Lean target and fail the drift audit.
   - Do **NOT** introduce a new `inductive` type or new `class` field merely to support the proposed theorem. The proposal must be statable using existing kernel + setup objects.
   - Do **NOT** edit `lean4/BEDC/...` for a theory_extension target. Paper proposes; Lean implements separately.

## Step 4.5: Lint the diff against parameter-echo expansion (HARD GATE)
Even after Phase Review picked the target, re-check that the implementation did not drift into a parameter-echo theorem. Run:
```bash

```
If the new `\Longleftrightarrow` / `\hookleftarrow` count exceeds 3 within a single new theorem block AND the theorem statement introduces fresh universally-quantified parametric carriers ($A,B$) and parametric classifiers ($\sim_A,\sim_B$) and concludes with paired field-by-field $\Leftrightarrow$ chains, the round is a parameter-echo restatement. ABORT: `git restore --staged papers/bedc/ && git checkout -- papers/bedc/`, then pick a different target whose conclusion is a closure / inversion / coverage / determinacy claim about a CONCRETE BEDC kernel object.

Also lint for label collision before commit:
```bash
git diff --cached -- papers/bedc/ \
  | grep -nE "^\+\\\\label\{" \
  | sed -E 's/.*label\{([^}]+)\}.*/\1/' \
  | while read L; do
      cnt=$(grep -rE "\\\\label\{${L}\}" papers/bedc/ | wc -l | tr -d ' ');
      if [ "$cnt" -gt 1 ]; then echo "DUPLICATE: $L appears $cnt times"; fi;
    done
```
Any DUPLICATE line means the proposed change collides with a label already in the tree (either a sibling round added the same label, or you're echoing an existing definition under a new name). ABORT and either pick a fresh label or drop the target.
