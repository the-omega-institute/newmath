Splitting by itself is not a valid round.
3. **Math notation.** Use only `$…$` for inline math and `$$…$$` (each `$$` on its own line) for display. Do NOT introduce `\[…\]`, `\begin{equation}`, `\begin{align}`, `\begin{equation*}`, `\begin{eqnarray}`, etc.
4. **No inline file comments.** Do not add `% TODO …` style notes, do not insert "see git log for history" lines, do not add change-tracking comments. The paper is the artifact; the diff is the audit trail.
5. **Marker discipline.**

6. **Lean discipline (if you touch Lean code).**

## Structural split rule

If an edited `.tex` file is already above 720 lines, or the target would add more than 25 lines to a file above 760 lines, prefer a split instead of appending. This is especially important for concrete instance files that accumulate certificate variants.

Allowed split pattern:
- Create child files under the same theme directory, preferably inside a thematic subdirectory such as `papers/bedc/parts/concrete_instances/list/<child>.tex` or `papers/bedc/parts/concrete_instances/option/<child>.tex`.
- Move all substantive content out of the parent into child files: headings, orientation prose, theorem clusters, definitions, proofs, certificate blocks, and cross-reference prose belong in children.
- Leave the original numbered parent file as a pure index: ordered `\input{...}` lines only, with no heading, no orientation paragraph, no theorem/proof content, and no explanatory prose.
- Preserve existing labels exactly unless the label is semantically wrong.
- Do not edit `papers/bedc/main.tex` when splitting content already included by an existing parent file.
- Use `\input` paths in the same style as the rest of the paper, relative to `papers/bedc/main.tex`.
- After splitting, every touched `.tex` file must be ≤ 800 lines.

If the target includes a line-cap split, perform the split before applying any new theorem/prose content, then apply the content in the appropriate child file. If the target is only a split with no substantive theorem/definition/proof/certificate change, it is INVALID; skip the round rather than committing file movement.

## Hub recognition rule

A `.tex` file may already be a HUB — i.e. previously split, with all substantive content moved into child files referenced by `\input`. Detect it on `paper_files[0]` before editing:

```bash
f=<paper_files[0]>
total=$(grep -cEv '^\s*$|^\s*%' "$f")
inputs=$(grep -cE '^\s*\\input\{' "$f")
own=$(grep -cE '\\(chapter|section|begin\{(theorem|definition|proposition|lemma|corollary|proof)\})' "$f")
```

If `total <= 30` AND `inputs * 100 >= total * 80` AND `own == 0`, the file is a HUB and you MUST NOT append theorem / definition / proof / certificate content to it. Two valid moves:

1. Pick the relevant `\input`-ed child file (one of the paths the hub references) and edit that child instead. The original target's chapter / section / theorem labels remain intact because they live in the child.
2. If no existing child fits the new content, create a new sibling child under the same hub-subdirectory (e.g. `papers/bedc/parts/concrete_instances/prod/<new-child>.tex`), put the new content there, and append a single `\input{...}` line to the parent hub in alphabetical / numeric order.

Appending substantive content directly to a hub will be reverted within a few rounds by a "keep paper sites singular" cleanup commit, wasting your round's codex-exec budget. Phase REVIEW should already have rejected such targets — if you see a hub as `paper_files[0]` for a non-split content target, treat it as a Phase REVIEW miss and abort the round per Step 0 rather than committing the broken edit.

## Step 0.7: Proof-prose readability (HARD GATE)

When the target adds or rewrites a `\begin{proof}…\end{proof}` block, the prose MUST be paragraph-segmented for readability. A monolithic 8–15-line wall of text running through "Apply Theorem X. Apply Theorem Y. Apply Theorem Z. ..." is rejected even if it builds.

Concrete formatting rules:

1. **Blank-line paragraph break** between logically distinct moves. Each of the following starts a new paragraph (= separate it from the previous prose by a blank line in the .tex source):
   - the forward direction vs the reverse direction of an `iff`;
   - each case of a case analysis (e.g. "absent/absent", "present/present", "mixed");
   - each induction step (base / cons / append / etc.);
   - each major lemma application chain that produces a witness used downstream.

2. **At most ~4 consecutive `Apply Theorem~\ref{...}` invocations per paragraph.** If a chain needs more, break the proof into named steps:

   ```
   We first build the spine representation. Apply ... and ... .

   Next we transport the classifier. Use ... .

   Finally, repack into the public certificate. Apply ... .
   ```

3. **Display math (`$$...$$`) goes on its own paragraph**. Do not embed a `$$...$$` block inside a single sentence followed by more prose; surround it by blank lines so it reads as a step.

4. **No paragraph longer than ~6 lines** of source text. If a proof step is genuinely longer (e.g. a delicate constructor inversion), break by inserting a fresh sentence that names the next sub-goal ("It remains to show ...", "We now turn to ...").

5. **Prefer `\begin{enumerate}` for 3+ parallel sub-claims** instead of "First, …. Second, …. Third, …." run-on prose. Each `\item` is one paragraph.

6. Existing well-formatted proofs in the same chapter are the local model — read 2–3 nearby `\begin{proof}` blocks before writing yours, and match their cadence.

A revise round whose new proof block violates rules 1–4 (one solid wall, no blank-line breaks, >4 sequential `Apply` invocations in one paragraph) is INVALID. Re-split the prose before committing.

   - **What to write** — the block has seven mandatory fields:
     ```
     \begin{closurestatus}{\<X>Up}
       \theoryclosure{\scopedClosure}
       \scopeclosed{<one-sentence binding scope>}
       \formalstatus{\theoremCheckedV}
       \leantarget{BEDC.Derived.<X>Up.<theorem>}
     ```

## Step 4.5: Lint the diff against parameter-echo expansion (HARD GATE)
Even after Phase Review picked the target, re-check that the implementation did not drift into a parameter-echo theorem. Run:
```bash
git diff --cached -- papers/bedc/ \
  | grep -nE "^\+" \
  | grep -E "Longleftrightarrow|hookleftarrow"
```
If the new `\Longleftrightarrow` / `\hookleftarrow` count exceeds 3 within a single new theorem block AND the theorem statement introduces fresh universally-quantified parametric carriers ($A,B$) and parametric classifiers ($\sim_A,\sim_B$) and concludes with paired field-by-field $\Leftrightarrow$ chains, the round is a parameter-echo restatement. ABORT: `git restore --staged papers/bedc/ && git checkout -- papers/bedc/`, then pick a different target whose conclusion is a closure / inversion / coverage / determinacy claim about a CONCRETE BEDC kernel object.

Also lint for label collision before commit:
```bash
git diff --cached -- papers/bedc/ \
  | grep -nE "^\+\\\\label\{" \
  | sed -E 's/.*label\{([^}]+)\}.*/\1/' \
  | while read L; do
      cnt=$(grep -rE "\\\\label\{${L}\}" papers/bedc/ | wc -l | tr -d ' ');
      if [ "$cnt" -gt 1 ]; then echo "DUPLICATE: $L appears $cnt times"; fi;
    done
```
Any DUPLICATE line means the proposed change collides with a label already in the tree (either a sibling round added the same label, or you're echoing an existing definition under a new name). ABORT and either pick a fresh label or drop the target.
