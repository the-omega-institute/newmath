
\chapter{v1.5.4 Manifest: Minimal Verification Contract}
\label{ch:v154-manifest}

This v1.5.4 increment is a minimal verification contract. It does not introduce new derived objects, does not expand the completion layer, and does not rewrite the frozen core. It fixes the last naming ambiguity left after v1.5.3: the active concrete package-sameness relation is the one-constructor base relation, not an implicit equivalence closure.

\section{Scope}
This increment adds only the following active refinements:
\begin{enumerate}
  \item a fixed design choice for concrete package-sameness: use \(\psame_{\Pi}^{\mathrm{base}}\) as the active relation;
  \item a separate notation for a future optional equivalence closure \(\psame_{\Pi}^{\mathrm{eq}}\);
  \item a rule that exact concrete Globalize uses the base relation unless a later checked closure-reflection theorem is recorded;
  \item a split between policy-mode token uniqueness and canonical-constructor-mode token uniqueness;
  \item a verification queue with public theorem names and acceptance criteria.
\end{enumerate}

\section{Non-goals}
This increment does not claim:
\begin{enumerate}
  \item a checked Lean, Coq, Agda, or other proof-assistant build;
  \item a new construction of \(\NatUp\), \(\AddUp\), \(\RealUp\), \(\CircleUp\), or \(\FoldUp\);
  \item a full package algebra over an equivalence-closed package-sameness relation;
  \item any revision of the frozen finite kernel.
\end{enumerate}

\section{Development rule}
The rule of v1.5.4 is:
\[
  \text{keep the active relation minimal, and name every closure as a separate obligation.}
\]
Thus the exact concrete signature package instance is based on constructor inversion. Any future use of an equivalence closure must carry a new theorem proving that reflection survives the closure.
