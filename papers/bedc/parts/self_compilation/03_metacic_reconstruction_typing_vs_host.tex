\subsection{Typing Relation vs Host CIC}\label{subsec:reconstruction-typing-vs-host}

\texttt{BEDC.MetaCIC.HasType} is an inductive proposition inside the host
calculus. It represents a typing relation for the MetaCIC fragment: an
inhabitant of $\mathrm{HasType}\ \Gamma\ t\ A$ is a derivation object built
from the five constructors listed above, not the algorithm that checks Lean
terms.

The checking algorithm belongs to the host CIC kernel. That kernel elaborates
and type-checks Lean terms inside the implementation, with executable kernel
code running in the OCaml/Rust compiler/runtime environment. BEDC does not
replace that kernel; every theorem about \texttt{HasType} is accepted only
when the host kernel checks its proof term.

The represented relation is intentionally narrower than host typing. It
describes the fragment's Pi, lambda, application, variable, and sort rules,
together with explicit beta reduction. It does not simulate the full host
reduction and elaboration stack: eta-conversion, universe polymorphism, and
inductive types beyond the represented Pi/Sort fragment are outside this local
relation.

The main sanity condition is proof-theoretic. \texttt{HasType} is axiom-free,
and each displayed \texttt{HasType} derivation is a Lean term checked by the
host kernel. The strict axiom-purity audit also verifies that the checked
MetaCIC theorem dependencies do not use \texttt{propext},
\texttt{Classical.choice}, or \texttt{Quot.sound}.
