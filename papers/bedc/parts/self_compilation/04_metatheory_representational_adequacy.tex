\section{Representational Adequacy for the MetaCIC Fragment}
\label{sec:self-comp-metacic-representational-adequacy}

The adequacy question for \texttt{BEDC.MetaCIC} is not whether BEDC replaces
the host calculus. It is whether the represented fragment has enough syntax,
typing structure, and reduction structure to stand for the corresponding host
CIC fragment without changing the derivational content of the examples under
discussion. The relevant fragment is the closed dependent-function fragment
with variables, application, lambda abstraction, Pi formation, one base sort,
and $\beta$-reduction.

\subsection{Adequacy Criterion}
\label{subsec:metacic-adequacy-criterion}

Let $t$ be a closed term of the host CIC fragment generated by one base sort,
dependent products, lambda abstraction, application, and bound variables. A
representational adequacy statement for \texttt{BEDC.MetaCIC} has three parts.
First, there is a representation map

$$
\begin{aligned}
\ulcorner - \urcorner :
  \mathrm{HostTerm}_{0,\Pi,\lambda,\mathrm{app}}
  \longrightarrow \texttt{BEDC.MetaCIC.Term}.
\end{aligned}
$$

Second, host typing derivations for that fragment correspond to represented
typing derivations:

$$
\begin{aligned}
\varnothing \vdash_{\mathrm{host}} t : T
\quad\Longleftrightarrow\quad
\texttt{BEDC.MetaCIC.HasType}\ []\ \ulcorner t \urcorner\ \ulcorner T \urcorner .
\end{aligned}
$$

Third, the correspondence is derivational rather than merely extensional:
constructor uses in the host fragment are matched by constructor uses in
\texttt{HasType}. Sort formation corresponds to \texttt{sortRule}, variable
lookup corresponds to \texttt{varRule}, Pi formation corresponds to
\texttt{piRule}, lambda introduction corresponds to \texttt{lamRule}, and
application elimination corresponds to \texttt{appRule}. Thus the represented
object is not only a pretty-printed term. It is a term together with a checked
inhabitant of the represented typing judgment.

This criterion is intentionally fragment-relative. It does not assert
adequacy for the whole host kernel, and it does not assert that the host
elaborator can be recovered from the represented syntax. It says that every
closed derivation in the specified fragment has a represented term and a
corresponding \texttt{HasType} derivation, and that every represented
derivation in the fragment reads back to a host derivation of the same
dependent-function form.

The witnesses cover the core constructor pattern. The identity examples use
lambda introduction and variable lookup under a binder. The projection
examples test multiple nested binders and the index discipline needed to
distinguish earlier and later arguments. The Pi-value identity shows that the
fragment can quantify over represented function types, not merely over the
base sort. The function-argument example combines Pi typing, binder shifting,
variable lookup, and application elimination in one derivation.

The table should be read as a finite mechanized adequacy gallery. It supports
the claim that the represented rules are aligned with the host fragment on
these canonical terms. A full adequacy theorem would quantify over all closed
terms in the fragment and prove the two readback directions by induction on
host and represented derivations.

\subsection{Substrate Differences}
\label{subsec:metacic-substrate-differences}

The representation is not textually identical to the host surface. Three
substrate differences are essential.

\paragraph{De Bruijn binders and named binders.}
The host presentation usually uses named binders, while
\texttt{BEDC.MetaCIC.Term} uses de Bruijn indices. This is an encoding choice,
not a change in binding structure. A closed named term has a de Bruijn
translation determined by the binder stack, and alpha-equivalent named terms
translate to the same de Bruijn term. Conversely, a de Bruijn term can be read
back to a named term by choosing fresh binder names. The correspondence is
therefore isomorphic up to alpha-renaming for the closed fragment.

\paragraph{Single universe and a universe hierarchy.}
The represented syntax has one constructor \texttt{Term.sort}. It therefore
covers only the universe-zero presentation of the fragment used here. Host
terms whose derivations rely on explicit universe levels, cumulative
constraints, or polymorphic universe parameters are outside this section's
adequacy criterion. Within the one-sort fragment, \texttt{sortRule} supplies
the represented sort judgment used by the witness derivations.

\paragraph{$\beta$-only reduction and $\beta+\eta$ conversion.}
\texttt{BEDC.MetaCIC.BetaStep} contains beta contraction and congruence
rules. It does not contain eta-conversion. Adequacy for this section is
therefore adequacy for a beta-conversion fragment. Host arguments that rely
on eta-equality are not represented as definitional equality in this layer.

\hline

\subsection{Coverage Boundary}
\label{subsec:metacic-coverage-boundary}

The adequacy claim is bounded by the represented grammar and judgment rules.
The following host-kernel features are not covered by the section's criterion:

\begin{center}
\begin{tabular}{p{0.34\linewidth}p{0.50\linewidth}}
\hline
Feature & Status in \texttt{BEDC.MetaCIC} \\
\hline
Universe polymorphism & Not represented; there are no universe variables,
levels, constraints, or cumulative universe rules. \\
Inductive families & Not represented; the grammar has Pi, lambda,
application, variables, and sort, but no family declarations or eliminators. \\
Propositions versus sets & Not separated; the one-sort fragment has no
distinct \texttt{Prop}/\texttt{Type} split. \\
Eta conversion & Not represented; beta is the explicit conversion relation. \\
Host elaboration & Not represented; elaborator behavior, coercions, typeclass
search, and notation expansion are outside the term grammar. \\
\hline
\end{tabular}
\end{center}

These boundaries are part of the adequacy statement. The represented calculus
is adequate only for the stated closed dependent-function fragment. It is not
an adequate representation of Lean's complete kernel language, nor of the
surface language accepted by the elaborator. Within its boundary, however, the
mechanized examples demonstrate that the core identity, projection, Pi, and
application patterns have BEDC.MetaCIC terms whose \texttt{HasType}
derivations mirror the corresponding host CIC derivations.
