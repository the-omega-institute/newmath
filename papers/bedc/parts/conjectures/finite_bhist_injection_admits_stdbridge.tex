\section{Finite-BHist-Injection Admits StdBridge}
\label{sec:conjectures-finite-bhist-injection-admits-stdbridge}

\begin{conjecture}[Finite BHist Tuple Route Admits StdBridge]
\label{conj:finite-bhist-injection-admits-stdbridge}
For every BEDC chapter $\langle X\rangle\mathsf{Up}$ equipped with a fixed finite $\BHist$ tuple presentation that
\begin{enumerate}
  \item displays a tuple carrier $(b_0,b_1,\ldots,b_k)$ of accepted $\BHist$ component rows;
  \item supplies an image row showing that every accepted carrier row is one of the displayed tuple components;
  \item supplies a readback row recovering the corresponding $\BHist$ component from each displayed image row;
  \item presents the chapter carrier as a fixed-arity $\BHist$ tuple $(b_0,b_1,\ldots,b_k)$ whose $\NameCert$ rows are transported componentwise along those readback rows;
  \item inhabits $\ChapterTasteGate$ for the displayed tuple presentation;
\end{enumerate}
there exists a $\langle X\rangle\mathsf{UpStdBridge}$ theorem in $\BEDC.Derived.\langle X\rangle\mathsf{Up}$ that bridges the displayed finite tuple's $\NameCert$ obligation surface to the standard reflection schema.
\end{conjecture}

\conjectureStatus{weakened}

\paragraph{Status notes.}
The obligation is the fixed finite $\BHist$ tuple route. The bridge follows by case analysis over the displayed tuple carrier, replaying the image row and readback row for each component, and transporting the tuple carrier's $\NameCert$ rows component by component before packaging the finite family as the chapter's standard reflection bridge.

The remaining open part is the general transport from arbitrary
regular-language recognisers to $\BHist$ tuple presentations. That route
would need a recogniser-to-carrier construction and a uniform component
transport theorem not supplied by the fixed tuple test target.

\paragraph{Active fixed-tuple test.}
The active finite test is exactly the fixed-tuple kernel route: a tuple carrier, an image row, a readback row, componentwise $\NameCert$ transport, and the $\ChapterTasteGate$ row. Recognizer-to-carrier construction is not part of this test and remains a separate boundary condition.

The exact remaining target is recognizer-to-carrier construction. It must produce a recognizer acceptance row, finite tuple extraction, $\BHist$ component readback, componentwise $\NameCert$ transport, and $\ChapterTasteGate$ from the recognizer presentation before any stronger $\StdBridge$ conjecture is reopened.

\begin{theorem}[Finite BHist injection fixed tuple route]
\label{thm:finite-bhist-injection-fixed-tuple-route}
In the fixed finite tuple case, every candidate $\StdBridge$ route must be a displayed $\BHist$ tuple route. The image row, readback row, tuple carrier, componentwise $\NameCert$ transport, and $\ChapterTasteGate$ row must all be present before the bridge is requested.
\end{theorem}
\begin{proof}
Start with the displayed tuple carrier $(b_0,b_1,\ldots,b_k)$. The image row restricts every accepted carrier read to one of these finite $\BHist$ components, and the readback row returns the corresponding component as kernel data.

Transport is then componentwise. Each $\NameCert$ obligation is replayed along the displayed readback row for its component, and the $\ChapterTasteGate$ row records that this tuple presentation is the one accepted by the chapter.

This isolates only the fixed-tuple route. It does not construct a tuple carrier from an arbitrary regular-language recogniser, so the general recogniser-to-carrier problem remains open.
\end{proof}

\begin{proposition}[Finite BHist injection fixed tuple obligation surface]
\label{prop:finite-bhist-injection-fixed-tuple-obligation-surface}
Before any $\StdBridge$ target is requested in the fixed finite tuple case, the chapter must display the tuple carrier, the image row, the readback row, the componentwise $\NameCert$ transport rows, and the $\ChapterTasteGate$ row as public preconditions. These rows are the exact obligation surface for the fixed-tuple route, and they do not supply the missing construction from arbitrary regular-language recognisers to tuple carriers.
\end{proposition}
\begin{proof}
Apply \autoref{thm:finite-bhist-injection-fixed-tuple-route}. It already requires the displayed $\BHist$ tuple carrier, the image row, the readback row, componentwise $\NameCert$ transport, and the $\ChapterTasteGate$ row before the bridge is requested.

The proposition only repacks that route as an obligation surface. It preserves the open recogniser-to-carrier problem because no row here constructs a tuple carrier from an arbitrary regular-language recogniser.
\end{proof}

\begin{theorem}[Finite BHist injection recognizer boundary]
\label{thm:finite-bhist-injection-recognizer-boundary}
A recognizer-to-carrier route can be compared with the fixed-tuple
$\StdBridge$ route only after it displays a recognizer acceptance row, a finite
tuple extraction row, a readback row into $\BHist$ components, componentwise
$\NameCert$ transport, and a $\ChapterTasteGate$ row. These five rows form the
boundary between arbitrary regular-language recognizers and the fixed finite
tuple obligation surface.
\end{theorem}
\begin{proof}
Apply \autoref{thm:finite-bhist-injection-fixed-tuple-route}. It gives the
fixed tuple, image row, readback row, componentwise $\NameCert$ transport, and
$\ChapterTasteGate$ row required before the fixed-tuple bridge route is
requested.

Now use \autoref{prop:finite-bhist-injection-fixed-tuple-obligation-surface}.
It records the same rows as the public obligation surface and keeps them
separate from the construction of tuple carriers from arbitrary recognizers.

Thus a recognizer-side comparison must first expose the acceptance row and the
finite tuple extraction row that land in that surface. This theorem only names
that boundary; it does not construct the bridge or solve the recognizer-to-tuple
transport problem.
\end{proof}

Finally apply the recognizer boundary theorem. It already separates the required boundary rows from the construction of tuple carriers from arbitrary regular recognizers, so this theorem records admissibility only after the boundary is displayed and does not construct the recognizer-to-carrier map.
\end{proof}

\begin{theorem}[Finite tuple StdBridge fixed-route closure]
\label{thm:finite-tuple-stdbridge-fixed-route-closure}
In the fixed finite tuple case, once the tuple carrier, image row, readback row, componentwise $\NameCert$ transports, and $\ChapterTasteGate$ row are displayed, the $\StdBridge$ obligation surface is closed for that fixed route. This closure is only the displayed tuple route; it leaves the arbitrary regular-language recogniser-to-carrier problem open.
\end{theorem}
\begin{proof}
Apply \autoref{thm:finite-bhist-injection-fixed-tuple-route}. It identifies the fixed tuple carrier, image row, readback row, componentwise $\NameCert$ transport, and $\ChapterTasteGate$ row as the whole route that must be present before the bridge is requested.

Use \autoref{prop:finite-bhist-injection-fixed-tuple-obligation-surface}. The proposition repacks those rows as the exact public obligation surface, so no extra row is needed for the fixed tuple route once all five displayed rows are present.

Finally apply \autoref{thm:finite-tuple-stdbridge-refutation-candidate}. Any counter-instance at the fixed route must keep the whole displayed tuple surface present and locate an additional reflection-axis obstruction. A missing tuple, image row, readback row, componentwise transport, or taste-gate row is therefore outside the closed fixed-route surface.

The theorem does not solve the arbitrary regular-language recogniser-to-carrier problem. That problem still requires a recogniser acceptance row, tuple extraction row, and uniform component transport beyond the fixed tuple route.
\end{proof}

\begin{theorem}[Finite tuple StdBridge refutation candidate]
\label{thm:finite-tuple-stdbridge-refutation-candidate}
A valid counter-instance to \autoref{conj:finite-bhist-injection-admits-stdbridge} in the fixed finite tuple case must display all five fixed-tuple rows: the tuple carrier $(b_0,b_1,\ldots,b_k)$, the image row, the readback row, the componentwise $\NameCert$ transport rows, and the $\ChapterTasteGate$ row. It must also display a reflection-axis requirement for the same chapter that is incompatible with the standard reflection schema consumed by $\StdBridge$.

Thus a failed bridge attempt caused only by a missing tuple, missing image row, missing readback row, missing componentwise transport, or missing $\ChapterTasteGate$ row is not a counter-instance to the weakened fixed-tuple route.
\end{theorem}
\begin{proof}
Start from \autoref{thm:finite-bhist-injection-fixed-tuple-route}. That theorem identifies the fixed tuple, image, readback, componentwise naming transport, and $\ChapterTasteGate$ rows as prerequisites for the route under test.

Now use \autoref{prop:finite-bhist-injection-fixed-tuple-obligation-surface}. It repacks the same rows as the exact obligation surface and separates them from the open recogniser-to-carrier problem.

Therefore the negation surface must keep the whole fixed-tuple surface present and locate the obstruction in an additional reflection-axis requirement. A missing prerequisite only shows that the fixed-tuple theorem was not reached; it does not refute the route once its premises are displayed.
\end{proof}
