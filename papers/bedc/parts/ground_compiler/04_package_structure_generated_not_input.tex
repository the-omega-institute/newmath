\chapter{Package Structure Is Generated, Not Input}
\label{ch:ground-compiler-package-structure-generated}

\autoref{ch:ground-compiler-event-flow} and \autoref{ch:ground-compiler-zeckendorf-safe-channel-encoding} establish the raw bijection
$$
  \mathsf{EventFlow}
  \;\cong\;
  \mathsf{LegalZStream}
  \;=\;
  \big(\, (0 \mid 10)^{*}\,11 \,\big)^{*}.
$$
\autoref{ch:ground-compiler-source-channel-layer-separation} then separates source-level events from channel-level bits: source-level normalization, classification, and ledger recognition act on decoded event flows, never on channel substrings.

This chapter records the next no-hidden-input constraint. Package structure is not a formal compiler input. If the compiler directly accepted a datum such as
$$
  \mathrm{Package}(\mathrm{source}, \mathrm{pattern}, \mathrm{classifier},
  \mathrm{stability}, \mathrm{ledger})
$$
or a host object such as $\mathrm{TheoryPkg}$, $\NameCert$, or $\mathrm{TheoremPkg}$, then the package fields, boundaries, roles, order, and hierarchy would have been supplied from outside the generated event flow. That is exactly the structural hidden input excluded by \autoref{def:ground-event-flow-structural-hidden-input}.

The chapter's central rule is therefore:
$$
  \boxed{\text{packages are recognized structures in event flows, not compiler inputs}.}
$$
This is the source-layer form of the package and gap discipline already used by BEDC: package tokens are generated by answer histories, package interfaces expose only the token and its ledger, and nontrivial package visibility requires a gap ledger recording which source histories enter the package token.

\section{Package recognition on event flows}
\label{sec:package-generated-recognition}

\begin{definition}[Package candidate flow]
\label{def:package-generated-candidate-flow}
\leandef{BEDC.GroundCompiler.PackageGenerated.PackageCandidateFlow}
A \emph{package candidate flow} is an event flow
$$
  S \in \mathsf{EventFlow}.
$$
The phrase does not assert that $S$ is a package. It asserts only that $S$ is available as a candidate source flow on which package recognition may be attempted.
\end{definition}

\begin{definition}[Generated package recognizer]
\label{def:package-generated-recognizer}
\leandef{BEDC.GroundCompiler.PackageGenerated.GeneratedPackageRecognizer}
A \emph{generated package recognizer} is an event flow
$$
  R \in \mathsf{EventFlow}.
$$
It is not a host-level parser, YAML schema, AST constructor, or external grammar. Its public formal carrier is itself generated from the two raw marks through the event-flow layer.
\end{definition}

\begin{definition}[Package recognition relation]
\label{def:package-generated-recognition-relation}
\leandef{BEDC.GroundCompiler.PackageGenerated.PackageRecognitionRelation}
The package recognition relation is written
$$
  \mathsf{RecognizesPkg}(R, S),
  \qquad
  R,S \in \mathsf{EventFlow}.
$$
It is read as: the generated recognizer flow $R$ recognizes the event flow $S$ as a package flow. If such an $R$ exists, then $S$ is a recognized package flow.
\end{definition}

\begin{remark}[Recognition is not input]
\label{rem:package-generated-recognition-not-input}
The judgment $\mathsf{RecognizesPkg}(R,S)$ does not mean that $S$ has been externally tagged as a package. It means that an already generated recognizer flow $R$ recognizes package-shaped structure in the source event flow $S$. The direction is therefore
$$
  \mathsf{EventFlow}
  \longrightarrow
  \mathsf{RecognizedPackageFlow},
$$
not
$$
  \Pkg
  \longrightarrow
  \mathsf{EventFlow}.
$$
\end{remark}

\begin{definition}[Recognized package flow]
\label{def:package-generated-recognized-package-flow}
\leandef{BEDC.GroundCompiler.PackageGenerated.PkgFlow}
For an event flow $S$, define
$$
  \mathsf{PkgFlow}(S)
  \;:\Longleftrightarrow\;
  \exists R \in \mathsf{EventFlow},\ \mathsf{RecognizesPkg}(R,S).
$$
Thus $S$ is a recognized package flow exactly when a generated recognizer flow recognizes it as package-shaped.
\end{definition}

\begin{principle}[No external package input]
\label{prin:package-generated-no-external-package-input}
\leanchecked{BEDC.GroundCompiler.PackageGenerated.no\_external\_package\_input}
At the formal datum boundary, a host package object is not an admissible compiler input:
$$
  \neg\,\mathsf{FormalCompilerInput}(\mathrm{hostPkg}).
$$
There is no formal compiler entry point
$$
  \mathsf{CompilePkg} \colon \Pkg \longrightarrow \mathsf{LegalZStream}.
$$
The only formal channel compiler remains
$$
  \mathsf{Compile} \colon \mathsf{EventFlow} \longrightarrow \mathsf{LegalZStream}.
$$
If an event flow $S$ is recognized as a package flow, its package code is just $\mathsf{Compile}(S)$.
\end{principle}

\begin{proposition}[Package recognition does not change channel code]
\label{prop:package-generated-recognition-preserves-code}
\leanchecked{BEDC.GroundCompiler.PackageGenerated.package\_recognition\_preserves\_code}
If $\mathsf{RecognizesPkg}(R,S)$, then the channel encoding of $S$ is still $\mathsf{Compile}(S)$. Package recognition does not add, delete, or rewrite channel bits.
\end{proposition}

\begin{proof}
The judgment $\mathsf{RecognizesPkg}(R,S)$ is a source-level recognition relation. It says only that $S$ is recognized by the generated recognizer $R$ as package-shaped. The channel encoding is given by $\mathsf{Compile}$ from \autoref{def:channel-encoding-flow-encoding}, and depends only on the event flow $S$. Since the recognition judgment does not alter $S$, it does not alter $\mathsf{Compile}(S)$.
\end{proof}

\begin{corollary}[Recognized package code is not a separate code]
\label{cor:package-generated-code-not-separate}
\leanchecked{BEDC.GroundCompiler.PackageGenerated.package\_code\_not\_separate}
If $S$ is a recognized package flow, then
$$
  \mathsf{PackageCode}(S) \;:=\; \mathsf{Compile}(S)
$$
is the ordinary channel code of the same event flow, not the output of a separate package encoder.
\end{corollary}

\section{Roles, seals, and ledgers}
\label{sec:package-generated-roles-seals-ledgers}

\begin{definition}[Package role emergence]
\label{def:package-generated-role-emergence}
\leandef{BEDC.GroundCompiler.PackageGenerated.PackageRoleEmergence}
Let $\rho$ range over the finite package-role list
\begin{center}
\begin{tabular}{l}
  source flow, \\
  pattern flow, \\
  classifier flow, \\
  stability flow, \\
  ledger flow, \\
  package-seal flow.
\end{tabular}
\end{center}
For a generated package recognizer $R$, a package candidate flow $S$, and a source-level event subflow $T$, define
$$
  \mathsf{PkgRole}_{R}(S,T,\rho)
  \;:\Longleftrightarrow\;
  \mathsf{RecognizesPkg}(R,S)
  \wedge
  T\in\mathsf{EventFlow}_{\ne\emptyset}.
$$
The role name $\rho$ is not a ground input. It is a recognition result attached to an event-flow subflow after $\mathsf{RecognizesPkg}(R,S)$ has already been witnessed.
\end{definition}

\begin{remark}[Relation to the five fields of $\NameCert$]
\label{rem:package-generated-relation-to-namecert-fields}
The five fields of a $\NameCert$,
$$
  \SourceSpec,\qquad
  \PatternSpec,\qquad
  \ClassifierSpec,\qquad
  \StabCert,\qquad
  \LedgerPolicy,
$$
are derived-interface certificate roles. $\PatternSpec$ records stable behavior on the source, $\ClassifierSpec$ records how source material is grouped, $\StabCert$ records closure, reuse, descent, composition, and seal stability, and $\LedgerPolicy$ records what a name hides and how the hidden material is tracked. This chapter does not treat those five fields as external input. It says only that, in a recognized package flow, a generated recognizer may identify subflows corresponding to those roles.
\end{remark}

\begin{definition}[Package-seal subflow]
\label{def:package-generated-seal-subflow}
\leandef{BEDC.GroundCompiler.PackageGenerated.PackageSealSubflow}
If $R$ recognizes a source-level event subflow $\sigma$ of $S$ as expressing package completion, then $\sigma$ is a \emph{package-seal subflow} of $S$, written
$$
  \mathsf{PkgSeal}_{R}(S,\sigma)
  \;:\Longleftrightarrow\;
  \mathsf{PkgRole}_{R}(S,\sigma,\mathrm{seal}).
$$
The package-seal subflow is not the channel terminator $11$. It is source-level event-flow structure, recovered through $\mathsf{Decode}$ and then recognized by $R$.
\end{definition}

\begin{proposition}[Package seal is not the channel terminator]
\label{prop:package-generated-seal-not-channel-terminator}
\leanchecked{BEDC.GroundCompiler.PackageGenerated.package\_seal\_is\_source\_subflow}
If $\mathsf{PkgSeal}_{R}(S,\sigma)$, then $\sigma$ is a source-level event subflow. In particular, its formal carrier is an element of $\mathsf{EventFlow}_{\ne\emptyset}$, whereas the channel-level terminator $11$ is a channel-code delimiter.
\end{proposition}

\begin{proof}
The channel-level terminator $11$ belongs to the encoding mechanism of $\mathsf{LegalZStream}$, as fixed in \autoref{def:channel-encoding-event-terminator}. A package seal is a subflow of $\mathsf{EventFlow}$ recognized by $R$. These live on different layers. By the layer-separation theorem \autoref{thm:source-channel-layer-separation}, a channel substring cannot be used directly as a source event or source subflow. Therefore the package seal is not the channel terminator.
\end{proof}

\begin{definition}[Package ledger subflow]
\label{def:package-generated-ledger-subflow}
\leandef{BEDC.GroundCompiler.PackageGenerated.PackageLedgerSubflow}
The \emph{package ledger subflow} predicate is
$$
  \mathsf{PkgLedger}_{R}(S,L)
  \;:\Longleftrightarrow\;
  \mathsf{PkgRole}_{R}(S,L,\mathrm{ledger}).
$$
Thus if $\mathsf{RecognizesPkg}(R,S)$ and $L$ is recognized as the ledger-role subflow of $S$, then $L$ records the source material hidden or compressed by $S$. The ledger subflow is part of the decoded source event flow, not an external provenance table.
\end{definition}

\begin{principle}[No package without ledger]
\label{prin:package-generated-no-package-without-ledger}
\leanchecked{BEDC.GroundCompiler.PackageGenerated.no\_package\_without\_ledger}
A nontrivial recognized package flow must carry a recognized ledger subflow:
$$
  \mathsf{PkgFlow}(S)
  \;\Longrightarrow\;
  \exists R,L,\ \mathsf{RecognizesPkg}(R,S) \wedge \mathsf{PkgLedger}_{R}(S,L),
$$
where the ledger witness is an event subflow identified by the same generated recognizer that recognizes $S$ as package-shaped. This is the package-level form of the gap policy: visible package tokens are not admissible for compression unless their ledgers record the histories entering the token.
\end{principle}

\begin{theorem}[Visible package code alone is insufficient]
\label{thm:package-generated-visible-code-insufficient}
\leanchecked{BEDC.GroundCompiler.PackageGenerated.visible\_code\_insufficient}
If there are no $R$ and $L$ such that
$$
  \mathsf{RecognizesPkg}(R,S)
  \wedge
  \mathsf{PkgLedger}_{R}(S,L),
$$
then $S$ is not a recognized package flow:
$$
  \neg\,\mathsf{PkgFlow}(S).
$$
\end{theorem}

\begin{proof}
By \autoref{prin:package-generated-no-package-without-ledger}, any recognized package flow supplies a generated recognizer $R$ and a ledger subflow $L$ with
$$
  \mathsf{RecognizesPkg}(R,S)
  \wedge
  \mathsf{PkgLedger}_{R}(S,L).
$$
If no such pair exists, the assumption $\mathsf{PkgFlow}(S)$ is impossible.
\end{proof}

\begin{corollary}[A package code is not merely a visible token]
\label{cor:package-generated-not-merely-visible-token}
A package code cannot consist only of a visible package-token event. It must include enough recognized ledger flow to recover provenance, unless the recognizer identifies the package as trivial and non-compressing.
\end{corollary}

\section{Compile-decode invariance}
\label{sec:package-generated-compile-decode}

\begin{definition}[Recognition-preserving compilation]
\label{def:package-generated-recognition-preserving-compilation}
\leandef{BEDC.GroundCompiler.PackageGenerated.RecognitionPreservingCompilation}
Compilation is \emph{recognition-preserving for packages} when, for every $\mathsf{RecognizesPkg}(R,S)$,
$$
  \mathsf{Decode}(\mathsf{Compile}(S)) = S
$$
and therefore
$$
  \mathsf{RecognizesPkg}(R,\mathsf{Decode}(\mathsf{Compile}(S))).
$$
\end{definition}

\begin{theorem}[Channel compilation preserves package recognition]
\label{thm:package-generated-compilation-preserves-recognition}
\leanchecked{BEDC.GroundCompiler.PackageGenerated.channel\_compilation\_preserves\_recognition}
If $\mathsf{RecognizesPkg}(R,S)$, then
$$
  \mathsf{RecognizesPkg}(R,\mathsf{Decode}(\mathsf{Compile}(S))).
$$
\end{theorem}

\begin{proof}
By the flow-level round-trip theorem \autoref{thm:channel-encoding-flow-round-trip},
$$
  \mathsf{Decode}(\mathsf{Compile}(S)) = S.
$$
Substituting this equality into the recognition judgment gives
$$
  \mathsf{RecognizesPkg}(R,\mathsf{Decode}(\mathsf{Compile}(S))).
$$
\end{proof}

\begin{corollary}[Package recognition is invariant under compile-decode]
\label{cor:package-generated-recognition-invariant}
\leanchecked{BEDC.GroundCompiler.PackageGenerated.package\_recognition\_invariant}
For every recognized package flow $S$, channel compilation followed by decoding preserves its package-recognition status.
\end{corollary}

\begin{definition}[Package-code equivalence]
\label{def:package-generated-code-equivalence}
\leandef{BEDC.GroundCompiler.PackageGenerated.PackageCodeEquiv}
For recognized package flows $S$ and $T$, define
$$
  S \equiv_{\mathrm{code}} T
  \;:\Longleftrightarrow\;
  \mathsf{Compile}(S)=\mathsf{Compile}(T).
$$
\end{definition}

\begin{proposition}[Package-code equivalence is raw-flow equality]
\label{prop:package-generated-code-equivalence-is-equality}
\leanchecked{BEDC.GroundCompiler.PackageGenerated.package\_code\_equivalence\_is\_equality}
For recognized package flows $S$ and $T$,
$$
  S \equiv_{\mathrm{code}} T
  \quad\Longleftrightarrow\quad
  S=T.
$$
\end{proposition}

\begin{proof}
If $S=T$, then $\mathsf{Compile}(S)=\mathsf{Compile}(T)$.

Conversely, suppose $\mathsf{Compile}(S)=\mathsf{Compile}(T)$. Applying $\mathsf{Decode}$ to both sides gives
$$
  \mathsf{Decode}(\mathsf{Compile}(S))
  =
  \mathsf{Decode}(\mathsf{Compile}(T)).
$$
By \autoref{thm:channel-encoding-flow-round-trip}, the left side is $S$ and the right side is $T$. Hence $S=T$.
\end{proof}

\begin{corollary}[Raw package flows are code-bijective]
\label{cor:package-generated-raw-flows-code-bijective}
\leanchecked{BEDC.GroundCompiler.PackageGenerated.raw\_flows\_code\_bijective}
Recognized package flows, considered as raw event flows, are in bijection with their legal channel codes. This is not the same as saying classified package objects are bijective with codes: a package classifier may identify two raw package flows.
\end{corollary}

\section{Classified packages as quotients}
\label{sec:package-generated-classified-quotients}

\begin{definition}[Package classifier quotient]
\label{def:package-generated-classifier-quotient}
If a package recognizer supplies a classifier relation $\sim_{\mathrm{pkg}}$ on recognized package flows, then the classified package object associated to $S$ is the equivalence class
$$
  [S]_{\sim_{\mathrm{pkg}}}.
$$
The projection
$$
  S \longmapsto [S]_{\sim_{\mathrm{pkg}}}
$$
may identify distinct raw flows.
\end{definition}

\begin{proposition}[Classified packages need not be code-bijective]
\label{prop:package-generated-classified-not-code-bijective}
If there exist recognized package flows $S \neq T$ with
$$
  S \sim_{\mathrm{pkg}} T,
$$
then the map from recognized package codes to classified package objects is not injective.
\end{proposition}

\begin{proof}
By \autoref{prop:package-generated-code-equivalence-is-equality}, $S \neq T$ implies
$$
  \mathsf{Compile}(S) \neq \mathsf{Compile}(T).
$$
But $S \sim_{\mathrm{pkg}} T$ implies
$$
  [S]_{\sim_{\mathrm{pkg}}} = [T]_{\sim_{\mathrm{pkg}}}.
$$
Thus two distinct codes map to one classified package object. The map is therefore not injective.
\end{proof}

\begin{corollary}[Raw package code and classified package object are different layers]
\label{cor:package-generated-raw-code-vs-classified-object}
Package codes are bijective with recognized raw package flows. Classified package objects are quotients of those flows:
\begin{center}
\begin{tabular}{rcl}
  raw package flow & $\longleftrightarrow$ & package code, \\
  raw package flow & $\longrightarrow$ & classified package object.
\end{tabular}
\end{center}
The first relation is bijective; the second is generally many-to-one.
\end{corollary}

\section{Recognition from channel code}
\label{sec:package-generated-recognition-from-code}

\begin{definition}[Package recognition from channel code]
\label{def:package-generated-recognition-from-code}
\leandef{BEDC.GroundCompiler.PackageGenerated.RecognizesPkgCode}
Given $c \in \mathsf{LegalZStream}$, define package recognition from code by the existence of a decoded event flow:
$$
  \mathsf{RecognizesPkgCode}(R,c)
  \;:\Longleftrightarrow\;
  \exists S \in \mathsf{EventFlow},
  \mathsf{Decode}(c)=\mathsf{some}(S)
  \wedge
  \mathsf{RecognizesPkg}(R,S).
$$
Thus package recognition from channel code always proceeds through decoding.
\end{definition}

\begin{proposition}[No package recognition by channel substring]
\label{prop:package-generated-no-recognition-by-channel-substring}
There is no valid package recognition rule that inspects arbitrary channel substrings without first decoding the channel stream into an event flow.
\end{proposition}

\begin{proof}
By \autoref{thm:source-channel-layer-separation}, channel substrings are not source events. Package recognition is defined on event flows by \autoref{def:package-generated-recognition-relation}. A rule that recognizes package structure directly from channel substrings would therefore operate on the wrong layer. Valid package recognition must proceed through $\mathsf{Decode}$.
\end{proof}

\begin{example}[Package recognition over a completion skeleton]
\label{ex:package-generated-completion-skeleton}
Consider the event flow
$$
  0 \mid 00 \mid 000 \mid 01 \mid 011 \mid 100.
$$
Its channel code is
$$
  011001100011010110101011100011.
$$
A generated recognizer $R$ may recognize this event flow as a completion-like package candidate, identifying
\begin{center}
\begin{tabular}{ll}
  $0 \mid 00 \mid 000$ & finite source-prefix flow, \\
  $01$                & repeat-boundary witness, \\
  $011 \mid 100$      & raw/normal carry ledger, \\
  $\sigma$            & package-seal subflow.
\end{tabular}
\end{center}
None of this occurs at the channel-substring layer. The channel code must first decode to
$$
  0 \mid 00 \mid 000 \mid 01 \mid 011 \mid 100,
$$
and only then can $R$ recognize package roles.
\end{example}

\begin{example}[Package recognition over an additive / fold skeleton]
\label{ex:package-generated-add-fold-skeleton}
Consider the event flow
$$
  0 \mid 00 \mid 001 \mid 0011 \mid 0100.
$$
Its channel code is
$$
  0110011001011001010110100011.
$$
A recognizer may identify
\begin{center}
\begin{tabular}{ll}
  $0$             & unit-source event, \\
  $00$            & binary-repetition witness, \\
  $001$           & additive-recursion witness, \\
  $0011 \mid 0100$ & raw/normal fold ledger.
\end{tabular}
\end{center}
The compiler itself does not call this flow addition, multiplication, or fold. Those names require a recognized package flow and, if exported as derived interfaces, the corresponding $\NameCert$, derived certificate, and ledger support.
\end{example}

\section{Conservativity and derived-interface boundary}
\label{sec:package-generated-conservativity}

\begin{theorem}[Package-recognition conservativity]
\label{thm:package-generated-conservativity}
Adding package recognition relations of the form
$$
  \mathsf{RecognizesPkg}(R,S)
$$
does not add new finite-kernel marks, histories, mark-sameness rules, history-sameness rules, continuation rules, or derived-interface formation rules.
\end{theorem}

\begin{proof}
The relation $\mathsf{RecognizesPkg}(R,S)$ is defined over event flows $R,S \in \mathsf{EventFlow}$. It does not introduce raw marks beyond $\bzero$ and $\bone$, construct new histories in the finite kernel, assert new $\msame$, $\hsame$, or $\psame$ judgments, or license derived interfaces by itself. A derived interface still requires the relevant $\NameCert$ fields, and an exported object still requires the acceptance discipline of \autoref{def:ground-event-flow-accepted-object-flow}. Therefore package recognition is conservative over the finite kernel.
\end{proof}

\begin{corollary}[Recognized package flow is weaker than derived interface]
\label{cor:package-generated-weaker-than-derived-interface}
If $S$ is recognized as a package flow, this alone does not license an upward-arrow interface. Licensing still requires the appropriate $\NameCert$ and, for export, derived-certificate and acceptance-gate support.
\end{corollary}

\begin{proof}
A recognized package flow supplies only source-level structural recognition. The typed naming discipline of \autoref{ch:core-typed-naming-certificates} requires a $\NameCert$ before an upward-arrow derived interface is licensed. Hence package recognition alone is weaker than derived-interface licensing.
\end{proof}

\begin{remark}[Outlook]
\label{rem:package-generated-outlook}
This chapter establishes that package structure is a recognition result inside an event flow, not a compiler input. The core relation is $\mathsf{RecognizesPkg}(R,S)$ with $R,S \in \mathsf{EventFlow}$: it preserves channel code, keeps package seals separate from channel terminators, requires ledger subflows for nontrivial package visibility, makes raw package flows code-bijective, treats classified packages as quotients, forces channel-code recognition through $\mathsf{Decode}$, and remains conservative over the finite kernel. The same no-hidden-input rule applies to $\NameCert$ recognition: certificate fields are recognized as event-flow substructures rather than received as a host-level five-field object.
\end{remark}
