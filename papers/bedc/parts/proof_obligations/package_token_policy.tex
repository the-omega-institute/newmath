\section{Package Token Policy and Reflection}
\label{ch:v151-package-token-policy}

The v1.5 layer replaced the function-like token notation \(\Tok_{\Pi}(s)\) by the judgment \(\TokIntro(\Pi,s,p)\). This chapter packages the necessary token behavior into a single proof interface.

\begin{definition}[Package token policy]
For a probe bundle \(\Pi\), a package token policy, written
\[
  \PkgTokPol(\Pi),
\]
consists of the following clauses for the concrete signature-token instance.
\begin{enumerate}
  \item \textbf{Token existence.} Every generated answer history \(s\) that is intended to be visible admits some token \(p\) with \(\TokIntro(\Pi,s,p)\).
  \item \textbf{Soundness.} If \(\TokIntro(\Pi,s,p)\), \(\TokIntro(\Pi,t,q)\), and \(\hsame(s,t)\), then \(\psame(p,q)\).
  \item \textbf{Reflection.} If \(\TokIntro(\Pi,s,p)\), \(\TokIntro(\Pi,t,q)\), and \(\psame(p,q)\), then \(\hsame(s,t)\).
  \item \textbf{Package-sameness closure.} On introduced tokens, \(\psame\) is closed under reflexivity, symmetry, and transitivity.
  \item \textbf{No external token equality.} Visible tokens are compared by \(\psame\), not by any imported equality judgment.
\end{enumerate}
\end{definition}

\begin{theorem}[Package-sameness is equivalent to signature-sameness]
\label{thm:v151-psame-iff-hsame}
Assume \(\PkgTokPol(\Pi)\). If
\[
  \TokIntro(\Pi,s,p),
  \qquad
  \TokIntro(\Pi,t,q),
\]
then
\[
  \psame(p,q)
  \quad\Longleftrightarrow\quad
  \hsame(s,t).
\]
\end{theorem}

\begin{proof}
The implication from right to left is the soundness clause of \(\PkgTokPol(\Pi)\). The implication from left to right is the reflection clause. No external token equality is used.
\end{proof}

\begin{corollary}[Concrete package completeness is policy-grounded]
The completeness direction of the concrete signature Globalize instance is frozen only under \(\PkgTokPol(\Pi)\). In particular, it is not a theorem for arbitrary package-sameness relations.
\end{corollary}

\begin{proof}
Completeness requires recovering \(\hsame(s,t)\) from \(\psame(p,q)\). This is exactly the reflection field of \(\PkgTokPol(\Pi)\).
\end{proof}

\begin{remark}[Proof-obligation boundary]
The token policy is deliberately narrow. It applies to the concrete signature-token package instance. Other package constructions must provide their own soundness and reflection certificates before they may inherit the same completeness theorem.
\end{remark}

\section{Inductive Package-Token Policy for Signature Packages}
\label{ch:v152-inductive-package-token-policy}

\autoref{ch:v151-package-token-policy} introduced \(\mathsf{PackageTokenPolicy}(\Pi)\). This chapter removes one remaining ambiguity: package-sameness for the concrete signature-token instance should be generated by one rule only. Package reflection should then be a constructor-inversion theorem, not an extra informal assumption.

\begin{definition}[Concrete signature-token introduction]
For the concrete signature-token instance, the judgment
\[
  \TokIntro(\Pi,s,p)
\]
means that token \(p\) is introduced as a visible token for answer history \(s\) under probe bundle \(\Pi\).
\end{definition}

\begin{definition}[Inductive package-sameness for signature tokens]
For the concrete signature-token instance, \(\psame_{\Pi}^{\mathrm{sig}}(p,q)\) is generated only by the following rule:
\[
\frac{
  \TokIntro(\Pi,s,p)
  \qquad
  \TokIntro(\Pi,t,q)
  \qquad
  \hsame(s,t)
}{
  \psame_{\Pi}^{\mathrm{sig}}(p,q)
}.
\]
No other package-sameness constructors are admitted for this concrete instance.
\end{definition}

\begin{lemma}[Package-sameness soundness]
\label{lem:v152-psame-sound}
If
\[
  \TokIntro(\Pi,s,p),\quad \TokIntro(\Pi,t,q),\quad \hsame(s,t),
\]
then
\[
  \psame_{\Pi}^{\mathrm{sig}}(p,q).
\]
\end{lemma}

\begin{proof}
This is the single constructor of \(\psame_{\Pi}^{\mathrm{sig}}\).
\end{proof}

\begin{lemma}[Package-sameness reflection]
\label{lem:v152-psame-reflect}
If
\[
  \TokIntro(\Pi,s,p),\quad \TokIntro(\Pi,t,q),\quad
  \psame_{\Pi}^{\mathrm{sig}}(p,q),
\]
then
\[
  \hsame(s,t).
\]
\end{lemma}

\begin{proof}
By inversion on the derivation of \(\psame_{\Pi}^{\mathrm{sig}}(p,q)\). Since the concrete package-sameness relation has only the displayed constructor, any such derivation is generated from introduced signatures that are \(\hsame\). Therefore \(\hsame(s,t)\), up to the token-introduction premises used to identify the displayed tokens.
\end{proof}

\begin{theorem}[Concrete PackageTokenPolicy]
\label{thm:v152-concrete-package-token-policy}
The concrete signature-token instance satisfies \(\mathsf{PackageTokenPolicy}(\Pi)\) whenever token introduction is available for generated signatures and package-sameness is interpreted as \(\psame_{\Pi}^{\mathrm{sig}}\).
\end{theorem}

\begin{proof}
Token introduction is supplied by the chosen concrete token-introduction rule. Package-sameness introduction is Lemma~\ref{lem:v152-psame-sound}. Package reflection is Lemma~\ref{lem:v152-psame-reflect}. The no-extra-constructor clause holds by definition of \(\psame_{\Pi}^{\mathrm{sig}}\).
\end{proof}

\begin{corollary}[Exact signature package]
For the concrete signature-token instance, \(\psame_{\Pi}^{\mathrm{sig}}(p,q)\) between introduced tokens is equivalent to internal sameness of the answer histories that introduced them.
\end{corollary}

\begin{proof}
The forward direction is Lemma~\ref{lem:v152-psame-reflect}; the reverse direction is Lemma~\ref{lem:v152-psame-sound}.
\end{proof}

\begin{remark}[Implementation note]
In a machine implementation, this chapter should be realized by defining package-sameness as an inductive relation with exactly one constructor. Reflection should then be proved by case analysis on that constructor.
\end{remark}

\section{Token-Introduction Uniqueness and Package Reflection}
\label{ch:v153-token-intro-uniqueness}

\autoref{ch:v152-inductive-package-token-policy} made concrete package-sameness a one-constructor relation. That is necessary but not sufficient for the strongest reflection theorem. If the same visible token can be introduced by two unrelated signatures, then package-sameness of tokens does not determine the signatures uniquely. This chapter isolates the missing requirement.

\begin{definition}[Token-introduction uniqueness]
\label{def:v153-token-unique}
For a fixed generated probe bundle \(\Pi\), define the token-introduction uniqueness obligation \(\mathsf{TokUnique}(\Pi)\) as the following principle:
\[
  \TokIntro(\Pi,s,p),\quad
  \TokIntro(\Pi,t,p)
  \quad\Longrightarrow\quad
  \hsame(s,t).
\]
This is an internal uniqueness condition up to history-sameness. It is not host-level equality of tokens or signatures.
\end{definition}

\begin{remark}[Why this is needed]
From a derivation of \(\psame_{\Pi}^{\mathrm{sig}}(p,q)\), inversion yields some generated signatures \(s_0,t_0\) introducing \(p,q\), together with \(\hsame(s_0,t_0)\). If the displayed premises use \(s,t\) instead, one still needs to compare \(s\) with \(s_0\) and \(t\) with \(t_0\). This is exactly the role of \(\mathsf{TokUnique}(\Pi)\).
\end{remark}

\begin{lemma}[Token replacement]
\label{lem:v153-token-replacement}
Assume \(\mathsf{TokUnique}(\Pi)\). If
\[
  \TokIntro(\Pi,s,p),\quad
  \TokIntro(\Pi,s_0,p),
\]
then
\[
  \hsame(s,s_0).
\]
\end{lemma}

\begin{proof}
This is Definition~\ref{def:v153-token-unique} applied to the two token-introduction judgments for the same token \(p\).
\end{proof}

\begin{theorem}[Concrete package reflection with token uniqueness]
\label{thm:v153-package-reflection-token-unique}
Assume \(\mathsf{TokUnique}(\Pi)\). If
\[
  \TokIntro(\Pi,s,p),\quad
  \TokIntro(\Pi,t,q),\quad
  \psame_{\Pi}^{\mathrm{sig}}(p,q),
\]
then
\[
  \hsame(s,t).
\]
\end{theorem}

\begin{proof}
Invert the derivation of \(\psame_{\Pi}^{\mathrm{sig}}(p,q)\). Since the concrete signature-token package-sameness relation has only the constructor displayed in \autoref{ch:v152-inductive-package-token-policy}, the derivation supplies signatures \(s_0,t_0\) such that
\[
  \TokIntro(\Pi,s_0,p),\quad
  \TokIntro(\Pi,t_0,q),\quad
  \hsame(s_0,t_0).
\]
By Lemma~\ref{lem:v153-token-replacement}, the first token-introduction premise gives \(\hsame(s,s_0)\), and the second gives \(\hsame(t,t_0)\). Symmetry and transitivity of \(\hsame\) now yield \(\hsame(s,t)\).
\end{proof}

\begin{corollary}[Exact package-sameness under token uniqueness]
\label{cor:v153-exact-package-sameness}
Assume \(\mathsf{TokUnique}(\Pi)\). For introduced tokens,
\[
  \psame_{\Pi}^{\mathrm{sig}}(p,q)
\]
if and only if their introducing signatures are internally history-same.
\end{corollary}

\begin{proof}
The forward direction is Theorem~\ref{thm:v153-package-reflection-token-unique}. The reverse direction is the single constructor of \(\psame_{\Pi}^{\mathrm{sig}}\).
\end{proof}

\begin{corollary}[Updated PackageTokenPolicy]
\label{cor:v153-updated-package-token-policy}
The concrete signature-token package policy is exact when it includes token introduction, one-constructor package-sameness, and \(\mathsf{TokUnique}(\Pi)\).
\end{corollary}

\begin{proof}
Existence and soundness are supplied by token introduction and the package-sameness constructor. Reflection is Theorem~\ref{thm:v153-package-reflection-token-unique}. The no-extra-constructor clause is the closed-generation clause for \(\psame_{\Pi}^{\mathrm{sig}}\).
\end{proof}

\section{Token-Uniqueness Modes}
\label{ch:v154-token-uniqueness-modes}

The v1.5.3 patch made \(\TokUnique(\Pi)\) explicit. v1.5.4 keeps it as the active policy field, but separates two implementation modes so that the next machine sprint has a precise target.

\begin{definition}[Policy mode]
\label{def:v154-policy-mode}
In policy mode, \(\TokUnique(\Pi)\) is a field of the concrete package-token policy:
\[
  \TokIntro(\Pi,s,p),\quad \TokIntro(\Pi,t,p)
  \quad\Longrightarrow\quad
  \hsame(s,t).
\]
All v1.5.4 concrete Globalize exactness theorems may use this field as an assumption.
\end{definition}

\begin{definition}[Canonical-constructor mode]
\label{def:v154-canonical-mode}
In canonical-constructor mode, token introduction is implemented by a canonical token constructor, written schematically as
\[
  \TokCan(\Pi,s,p).
\]
The implementation must provide the no-confusion rule
\[
  \TokCan(\Pi,s,p),\quad \TokCan(\Pi,t,p)
  \quad\Longrightarrow\quad
  \hsame(s,t).
\]
If \(\TokIntro\) is defined by \(\TokCan\), then \(\TokUnique(\Pi)\) follows.
\end{definition}

\begin{lemma}[Canonical mode implies token uniqueness]
\label{lem:v154-canonical-implies-unique}
Assume \(\TokIntro(\Pi,s,p)\) is definitionally implemented by \(\TokCan(\Pi,s,p)\), and assume the no-confusion rule of Definition~\ref{def:v154-canonical-mode}. Then \(\TokUnique(\Pi)\) holds.
\end{lemma}

\begin{proof}
Given \(\TokIntro(\Pi,s,p)\) and \(\TokIntro(\Pi,t,p)\), unfold the implementation to obtain \(\TokCan(\Pi,s,p)\) and \(\TokCan(\Pi,t,p)\). The no-confusion rule gives \(\hsame(s,t)\), which is exactly \(\TokUnique(\Pi)\).
\end{proof}

\begin{principle}[Active mode for v1.5.4]
\label{prin:v154-active-token-mode}
The active paper theorem uses policy mode. Canonical-constructor mode is an implementation target, not a paper-level replacement, until a checked build records the implementation and no-confusion theorem.
\end{principle}

\begin{corollary}[Stable package reflection contract]
\label{cor:v154-stable-reflection-contract}
The package reflection theorem may be used in later v1.5.x patches only in one of two forms:
\begin{enumerate}
  \item base-relation reflection under \(\TokUnique(\Pi)\);
  \item base-relation reflection under a checked canonical-constructor theorem implying \(\TokUnique(\Pi)\).
\end{enumerate}
It may not be used for \(\psame_{\Pi}^{\mathrm{eq}}\) unless \(\ClosureReflect(\Pi)\) is separately proved.
\end{corollary}

\begin{proof}
The first form is Theorem~\ref{thm:v154-base-reflection-active}. The second form follows from Lemma~\ref{lem:v154-canonical-implies-unique}. The final exclusion is Corollary~\ref{cor:v154-eq-not-exported}.
\end{proof}
