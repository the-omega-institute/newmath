\chapter{A Concrete Naming Certificate for $\DyadicMeshUp$}
\label{ch:concrete-instances-dyadicmesh-namecert}
\origin{human}

The $\DyadicMeshUp$ interface names finite dyadic grid packets used by real-completion and validated-numerics consumers. In the Moore--Kearfott--Cloud interval-analysis reading, a mesh is a finite grid of rational interval cells with explicit enclosure ledgers; in BEDC it is only a displayed $\mathsf{BHist}$ package over $\DyadicRatCoreUp$, $\RationalIntervalUp$, $\DyadicPrecisionUp$, $\DyadicApproximationUp$, $\RealUp$, and $\ValidatedNumericsUp$. The mesh cell row consumes the $\RationalIntervalUp$ endpoint carrier of \autoref{ch:concrete-instances-rationalinterval-namecert} and exposes its validated enclosure ledger to the $\ValidatedNumericsUp$ consumer surface of \autoref{ch:concrete-instances-validatednumerics-namecert}.

Before those interval and validated-numerics consumers read an enclosure, the finite mesh packet consumes the dyadic endpoint and radius rows licensed by \autoref{ch:concrete-instances-dyadicratcore-namecert}; $\DyadicMeshUp$ refines and packages that endpoint data into finite cell ledgers rather than treating dyadic rational endpoints as a merely adjacent surface.

\begin{definition}[DyadicMesh BHist source]
\label{def:dyadicmesh-bhist-source}
A $\DyadicMeshUp$ source is a finite $\mathsf{BHist}$ packet
$$
M=(\ell,J,I,E,H,C,P,N)
$$
with the following visible rows:
\begin{enumerate}
\item $\ell$ is a unary level row naming the displayed grid precision;
\item $J$ is the finite dyadic cell-index row at level $\ell$;
\item $I$ is the $\RationalIntervalUp$ endpoint row for the cell selected by $J$;
\item $E$ is the containment ledger against $\DyadicRatCoreUp$, recording the dyadic lower endpoint, upper endpoint, midpoint, radius, and endpoint-order reads for the cell;
\item $H$ stores componentwise $\hsame$ transport for the level, cell index, rational interval endpoints, dyadic endpoint reads, and containment ledger;
\item $C$ stores the $\Cont$ routes by which refinement, endpoint readback, and downstream enclosure consumers inspect the same finite cell;
\item $P$ is the $\Pkg$ provenance row naming $\DyadicRatCoreUp$, $\RationalIntervalUp$, $\DyadicPrecisionUp$, $\DyadicApproximationUp$, $\RealUp$, and $\ValidatedNumericsUp$;
\item $N$ is the local $\NameCert_{\DyadicMeshUp}$ registration row.
\end{enumerate}
The classifier identifies accepted mesh packets by transporting them to a common displayed refinement level, comparing the refined cell-index rows there, and then comparing the rational interval endpoints by the inherited $\RationalIntervalUp$ $\hsame$ rows. The carrier has no coordinate for a completed real cell, infinite grid, host interval tree, or solver object.
\end{definition}

\begin{theorem}[DyadicMesh cell containment obligation]
\label{thm:dyadicmesh-cell-containment-obligation}
Every accepted $\DyadicMeshUp$ packet $M=(\ell,J,I,E,H,C,P,N)$ exposes one finite dyadic cell. Its public containment surface is exactly the unary level row $\ell$, the dyadic cell-index row $J$, the rational interval row $I$, the $\DyadicRatCoreUp$ endpoint and radius reads stored in $E$, the displayed transport rows $H$, the continuation routes $C$, the package provenance $P$, and the local $\NameCert$ row $N$.
\end{theorem}
\leanchecked{BEDC.Derived.DyadicMeshUp.DyadicMeshCarrier\_cell\_containment\_obligation}
\leanchecked{BEDC.Derived.DyadicMeshUp.DyadicMeshPacket\_cell\_containment\_obligation}
\begin{proof}
Project the carrier of \autoref{def:dyadicmesh-bhist-source}. The selected cell is not reconstructed from an ambient grid object; it is the displayed pair of a level row and a finite cell-index row.

Read the rational interval coordinate $I$ and the dyadic containment ledger $E$. These two rows provide the endpoint, midpoint, radius, and endpoint-order data visible to a consumer.

The remaining coordinates route and certify those reads. The transport row $H$, continuation row $C$, package row $P$, and naming row $N$ exhaust the public containment surface, leaving no coordinate for a completed cell or external solver state.
\end{proof}

\begin{theorem}[DyadicMesh refinement stability]
\label{thm:dyadicmesh-refinement-stability}
Let $M=(\ell,J,I,E,H,C,P,N)$ be an accepted $\DyadicMeshUp$ packet. If $M'$ is obtained by appending displayed refinement evidence to $C$ so that the level and cell-index rows refine to a finer dyadic row while the covered rational interval cell is transported by the same endpoint $\hsame$ rows, then $M'$ is again an accepted $\DyadicMeshUp$ packet and covers the same real cell boundary visible to $\RealUp$ consumers.
\end{theorem}
\leanchecked{BEDC.Derived.DyadicMeshUp.DyadicMeshPacket\_refinement\_stability}
\begin{proof}
Start with the finite packet of \autoref{def:dyadicmesh-bhist-source}. Appending refinement evidence changes only the displayed continuation route by which the finer level and cell-index rows are read.

Transport the rational interval endpoint rows through $H$. Because the same endpoint $\hsame$ rows are preserved, the $\RationalIntervalUp$ boundary seen after refinement is the transported form of $I$ rather than a new completed real cell.

Repack the refined level, refined cell index, transported interval, dyadic containment ledger, updated continuation route, package provenance, and local name row. These are exactly the carrier coordinates, so the refined packet is accepted and the visible $\RealUp$ boundary is unchanged.
\end{proof}

\begin{theorem}[DyadicMesh RationalInterval coverage]
\label{thm:dyadicmesh-rationalinterval-coverage}
For every accepted $\DyadicMeshUp$ packet $M=(\ell,J,I,E,H,C,P,N)$, the rational interval row $I$ is the complete coverage object exported by the mesh cell. A $\RationalIntervalUp$ consumer may read the endpoint order, containment ledger, and refinement routes through $I$, $E$, $H$, and $C$, but it cannot read an infinite mesh, a host subset of $\RealUp$, or a global real-completion proof from the mesh packet.
\end{theorem}
\leanchecked{BEDC.Derived.DyadicMeshUp.DyadicMeshPacket\_rational\_interval\_coverage}
\begin{proof}
Project the interval coordinate $I$ from the carrier. It is the only coverage row of the packet, and its endpoint and order data are interpreted by the existing $\RationalIntervalUp$ surface.

Use the dyadic ledger $E$ only to identify the dyadic endpoint and radius rows that support the interval read. The routes $H$ and $C$ transport and expose those finite rows to the consumer.

Since the carrier contains no additional coverage coordinate, the exported coverage is precisely $I$ with its finite dyadic ledger. Infinite meshes, host subsets, and real-completion proofs are outside the packet surface.
\end{proof}
\leanchecked{BEDC.Derived.DyadicMeshUp.DyadicMeshPacket\_rationalinterval\_coverage}

\begin{theorem}[DyadicMesh NameCert obligation surface]
\label{thm:dyadicmesh-namecert-obligation-surface}
The seed $\NameCert_{\DyadicMeshUp}$ obligation surface has five rows:
\begin{enumerate}
\item carrier: the finite $\mathsf{BHist}$ packet of \autoref{def:dyadicmesh-bhist-source};
\item classifier: common-refinement comparison of level and cell-index rows followed by endpoint $\hsame$ comparison through $\RationalIntervalUp$;
\item stability: $\Cont$ appends refinement evidence while preserving the transported rational interval cell;
\item exactness: every endpoint, midpoint, radius, order, and containment read is one of the dyadic ledger rows in $E$ routed by $C$ and transported by $H$;
\item ledger: the $\Pkg$ and local $\NameCert$ rows record exactly the dependency surface over $\DyadicRatCoreUp$, $\RationalIntervalUp$, $\DyadicPrecisionUp$, $\DyadicApproximationUp$, $\RealUp$, and $\ValidatedNumericsUp$.
\end{enumerate}
These rows are the full seed naming surface for finite dyadic meshes.
\end{theorem}
\begin{proof}
The carrier row is the source packet itself. It lists the only visible coordinates from which a mesh cell can be read.

The classifier row first transports mesh packets to a common displayed refinement level. Once the cell-index rows are compared there, the rational interval endpoints are compared through the inherited endpoint $\hsame$ rows.

Stability is the refinement route of \autoref{thm:dyadicmesh-refinement-stability}. Exactness is the assertion that the dyadic endpoint, midpoint, radius, order, and containment reads are exactly the ledger entries in $E$ routed by $C$ and transported by $H$.

The ledger row is the package and local-name boundary. It records the named dependency certificates and therefore completes the seed $\NameCert$ surface without adding an analytic or solver coordinate.
\end{proof}
\leanchecked{BEDC.Derived.DyadicMeshUp.DyadicMeshPacket\_namecert\_obligation\_surface}

\begin{theorem}[DyadicMesh ValidatedNumerics handoff]
\label{thm:dyadicmesh-validatednumerics-handoff}
A $\ValidatedNumericsUp$ consumer of an accepted $\DyadicMeshUp$ packet may read exactly the finite enclosure package consisting of the level row, dyadic cell index, $\RationalIntervalUp$ cell coverage row, $\DyadicRatCoreUp$ containment ledger, displayed $\DyadicPrecisionUp$ refinement route, $\DyadicApproximationUp$ endpoint readback, $\RealUp$ boundary token, $\hsame$ transport, $\Cont$ routes, $\Pkg$ provenance, and local $\NameCert$ row. The handoff supplies a finite enclosure mesh cell and does not assert solver correctness, floating-point hardware semantics, classical real completeness, or a global mesh refinement theorem.
\end{theorem}
\begin{proof}
Read the mesh packet through \autoref{thm:dyadicmesh-cell-containment-obligation}. The level, cell index, interval row, dyadic containment ledger, transport, continuation, provenance, and local name rows are the only public coordinates.

Apply \autoref{thm:dyadicmesh-rationalinterval-coverage}. The coverage object exported to the validated-numerics side is the finite $\RationalIntervalUp$ row, supported by the dyadic endpoint and radius ledger.

Use \autoref{thm:dyadicmesh-refinement-stability} for precision movement. Refinement is a displayed continuation route and therefore preserves the same covered cell boundary visible to the real-name consumer.

Thus the validated-numerics handoff is a finite enclosure mesh cell with named dependency rows. No row supplies solver correctness, hardware semantics, classical completeness, or a global refinement theorem.
\end{proof}
\leanchecked{BEDC.Derived.DyadicMeshUp.DyadicMeshPacket\_validatednumerics\_handoff}

The public $\DyadicMeshUp$ export is exactly the finite packet rows of
\autoref{def:dyadicmesh-bhist-source} together with the containment obligation
of \autoref{thm:dyadicmesh-cell-containment-obligation}, refinement stability

$\NameCert$ surface of
\autoref{thm:dyadicmesh-namecert-obligation-surface}, and the
$\ValidatedNumericsUp$ handoff of
\autoref{thm:dyadicmesh-validatednumerics-handoff}.

This public boundary exports finite mesh-cell rows only. It does not export an
infinite mesh, a host subset of $\RealUp$, solver correctness,
floating-point semantics, or classical real completeness.
\end{theorem}
\begin{proof}
Project the source packet of \autoref{def:dyadicmesh-bhist-source}. Its visible
rows are the level, finite cell index, rational interval cell, dyadic
containment ledger, transport rows, continuation routes, package provenance,
and local name row.

Apply \autoref{thm:dyadicmesh-cell-containment-obligation} and
\autoref{thm:dyadicmesh-refinement-stability}. These identify the exported
cell surface and show that displayed refinement changes only the finite
reading route while preserving the transported cell boundary.

Next use \autoref{thm:dyadicmesh-rationalinterval-coverage} and
\autoref{thm:dyadicmesh-namecert-obligation-surface}. The coverage row is the
finite $\RationalIntervalUp$ cell, and the naming surface records the carrier,
classifier, stability, exactness, and ledger rows without adding an analytic
mesh object.

Finally apply \autoref{thm:dyadicmesh-validatednumerics-handoff}. The same
finite rows are the only public enclosure rows available to downstream
$\ValidatedNumericsUp$ consumers. Since every row is projected from the finite
packet, infinite meshes, host subsets, solver correctness, floating-point

\end{proof}

\begin{theorem}[DyadicMesh common refinement endpoint determinacy]
\label{thm:dyadicmesh-common-refinement-endpoint-determinacy}
Let
$M=(\ell,J,I,E,H,C,P,N)$ and
$M'=(\ell',J',I',E',H',C',P',N')$ be accepted $\DyadicMeshUp$
packets satisfying \autoref{def:dyadicmesh-bhist-source}. Suppose both
packets are displayed at a common refinement level and, after transport to
that level, agree on the refined finite cell-index row and on the
$\RationalIntervalUp$ endpoint $\hsame$ rows. Then every
$\DyadicApproximationUp$ endpoint read and every $\RealUp$ boundary token
exported through the two packets factors through the same finite
$\RationalIntervalUp$ row, endpoint-radius-order ledger, $\Cont$ route,
$\Pkg$ provenance row, and local $\NameCert$ row.

Equivalently, the consumer-visible rows for these three handoffs are the
same finite $\mathsf{BHist}$ rows routed through the common endpoint
$\hsame$, $\Cont$, $\Pkg$, and $\NameCert$ coordinates. The theorem does not
assert a host subset of $\RealUp$, a completed real cell, solver state,
floating-point semantics, or an infinite mesh.
\end{theorem}
\leanchecked{BEDC.Derived.DyadicMeshUp.DyadicMeshPacket\_common\_refinement\_endpoint\_determinacy}
\begin{proof}
Start from \autoref{def:dyadicmesh-bhist-source}. Each accepted packet is a
finite tuple of level, cell-index, rational-interval endpoint, dyadic ledger,
transport, continuation, package, and name rows; no additional mesh object is
available to downstream consumers.

Apply \autoref{thm:dyadicmesh-refinement-stability} to both packets at the
displayed common level. The hypothesis that the transported cell-index rows
agree fixes the refined cell boundary through the shared $\hsame$ and
$\Cont$ coordinates.

Next use \autoref{thm:dyadicmesh-rationalinterval-coverage}. Agreement of
the transported endpoint $\hsame$ rows fixes the unique exported
$\RationalIntervalUp$ coverage row and the endpoint-radius-order ledger, and
therefore fixes each endpoint read seen by the $\DyadicApproximationUp$
consumer.

Now apply \autoref{thm:dyadicmesh-validatednumerics-handoff}. The enclosure
handoff reads exactly the finite cell, rational-interval endpoint, dyadic
ledger, continuation, package, and local naming rows already fixed by the
common refinement packet.

Finally use
\autoref{thm:dyadicmesh-standard-finite-mesh-bridge-boundary}. Its bridge
surface names the $\RealUp$ boundary token and the endpoint readback as
projections of the same finite $\mathsf{BHist}$ packet, so the two accepted
packets give the same consumer-visible endpoint and boundary rows. The
conclusion is only this finite-row determinacy claim and introduces no host
subset, completed real cell, solver state, floating-point semantics, or
infinite mesh.
\end{proof}

\begin{theorem}[DyadicMesh validated terminal readback determinacy]
\label{thm:dyadicmesh-validated-terminal-readback-determinacy}
Let $M$ and $M'$ be accepted $\DyadicMeshUp$ packets displayed at a common
refinement as in
\autoref{thm:dyadicmesh-common-refinement-endpoint-determinacy}. Suppose a
terminal accepted $\DyadicApproximationUp$ finite packet reads the same
$\RationalIntervalUp$ cell, endpoint-radius-order ledger,
$\mathsf{RegSeqRat}$ observation window, and $\RealUp$ boundary token, and
suppose a $\ValidatedNumericsUp$ enclosure consumes that same cell through the
visible $\hsame$ transports, $\Cont$ routes, $\Pkg$ provenance rows, and local
$\NameCert_{\DyadicMeshUp}$, $\NameCert_{\DyadicApproximationUp}$, and
$\NameCert_{\ValidatedNumericsUp}$ rows.

Then every mesh-cell consumer, terminal dyadic readback, and validated
enclosure read factors through one finite $\RealUp$ boundary surface. The
surface is the common refined cell with its rational interval endpoint,
endpoint-radius-order ledger, terminal dyadic packet readback, validated
enclosure row, observation window, transport rows, continuation routes,
package rows, and local certificate rows. It does not assert completed real
equality, a host subset of $\RealUp$, solver correctness, floating-point
hardware semantics, or an infinite refinement object.
\end{theorem}
\begin{proof}
First apply
\autoref{thm:dyadicmesh-common-refinement-endpoint-determinacy}. The common
refinement fixes the finite cell row, rational interval endpoint,
endpoint-radius-order ledger, $\RealUp$ boundary token, transport rows,
continuation routes, package rows, and local mesh certificate row seen by the
two displayed mesh packets.

Next read the terminal dyadic packet through
\autoref{thm:dyadicapproximation-terminal-finite-packet-real-readback}. The
terminal packet contributes only its finite $\mathsf{RegSeqRat}$ observation
window, rational interval cell, endpoint ledger, and boundary token, so it has
the same finite target surface fixed by the mesh refinement.

Now use
\autoref{thm:dyadicapproximation-common-refinement-enclosure-handoff} and
\autoref{thm:dyadicapproximation-shared-real-boundary-determinacy}. These
identify the terminal readback and the enclosure handoff as projections of the
same finite rational-interval boundary, with the shared $\RealUp$ token and
without a completed real equality claim.

Finally invoke the $\ValidatedNumericsUp$ bridge boundary used by the dyadic
handoff. Its enclosure read consumes exactly the fixed cell, endpoint ledger,
observation window, transport, continuation, provenance, and local certificate
rows. Repacking those rows gives the single finite $\RealUp$ boundary surface
claimed in the statement.
\end{proof}
