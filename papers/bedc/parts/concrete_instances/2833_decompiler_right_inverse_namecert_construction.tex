\chapter{A Concrete Naming Certificate for $\DecompilerRightInverseUp$}
\label{ch:concrete-instances-decompiler-right-inverse-namecert}
\origin{human}

The $\DecompilerRightInverseUp$ interface packages decompilation as a finite
right-inverse packet for compiler certificates. It does not assert that every
compiler is invertible. It records the special case where a target-to-source
decompiler graph, followed by the displayed compiler graph, lands back in the
target certificate up to the explicitly recorded optimisation row.

\begin{definition}[Decompiler right-inverse carrier]
\label{def:decompiler-right-inverse-carrier}
A $\DecompilerRightInverseUp$ carrier is a finite $\BHist$ packet
$$
D=(S,T,K,\Psi,O,R,H,C,P,N)
$$
with the following visible rows:
\begin{enumerate}
\item $S$ is the source $\NameCert$ row read by the compiler and produced by the decompiler;
\item $T$ is the target $\NameCert$ row produced by the compiler and read by the decompiler;
\item $K$ is the accepted $\CertificateCompilerUp$ packet carrying the finite compiler graph from $S$ to $T$;
\item $\Psi$ is the finite target-to-source decompiler graph row from $T$ to $S$;
\item $O$ is the target optimisation row witnessing that the composite $K\circ\Psi$ reads as a $\NameCert_T$ endomorphism modulo the displayed target classifier;
\item $R$ is the paired right-inverse ledger matching each displayed target row with its decompile-then-compile image through $O$;
\item $H$ records the componentwise $\hsame$ transports for source certificate rows, target certificate rows, compiler graph edges, decompiler graph edges, optimisation rows, right-inverse rows, package provenance, and the local naming row;
\item $C$ records the displayed $\Cont$ routes for target inspection, decompiler graph readback, source certificate readback, compiler graph readback, optimisation readback, right-inverse comparison, and NameCert exposure;
\item $P$ is the $\Pkg$ provenance row naming $\BHist$, $\hsame$, $\Cont$, $\Pkg$, $\NameCert$, $\NameCertMorphismUp$, $\CertificateCompilerUp$, target optimisation, and the decompiler right-inverse shape;
\item $N$ is the local $\NameCert_{\DecompilerRightInverseUp}$ row.
\end{enumerate}
The classifier compares accepted packets only through the displayed compiler
graph, decompiler graph, target optimisation row, paired right-inverse ledger,
$\hsame$ transports, $\Cont$ routes, $\Pkg$ provenance, and local naming row.
It has no coordinate for a host decompiler function, semantic equivalence
oracle, global compiler inverse, or unrecorded target behaviour.
The compiler graph row $K$ is consumed through the finite
certificate-compiler surface of
\autoref{ch:concrete-instances-certificatecompiler-namecert}; the decompiler
packet adds only the reverse graph, optimisation row, and paired right-inverse
ledger on top of that sibling surface.
\end{definition}

\begin{theorem}[Decompiler right-inverse NameCert obligations]
\label{thm:decompiler-right-inverse-namecert-obligations}
For the carrier of \autoref{def:decompiler-right-inverse-carrier}, the local
$\NameCert_{\DecompilerRightInverseUp}$ surface consists exactly of the source
certificate row $S$, target certificate row $T$, accepted compiler packet $K$,
finite compiler graph $K$, finite decompiler graph $\Psi$, target optimisation row $O$, paired
right-inverse ledger $R$, componentwise $\hsame$ transports $H$, displayed
$\Cont$ routes $C$, package provenance $P$, and local naming row $N$.
\end{theorem}
\begin{proof}
Project the carrier. Each row required to read the decompiler packet is a
coordinate of the finite $\BHist$ packet before any consumer can ask for the
right-inverse comparison.

The compiler side is read through $K$, so the source-to-target direction is the
same finite compiler-certificate graph already licensed by
$\CertificateCompilerUp$. The reverse direction is read through $\Psi$, and it
lands back in the displayed source certificate row $S$ rather than in a host
source semantics.

The right-inverse clause is the pair $(O,R)$. The row $O$ supplies the target
optimisation surface for the composite $K\circ\Psi$, while $R$ records the
componentwise target-row comparison that consumes that optimisation row.

Finally the rows $H,C,P,N$ expose transport, continuation, provenance, and local
naming for the same packet. There is no remaining coordinate from which a host
inverse, semantic equivalence oracle, or unrecorded compiler behaviour could be
exported.
\end{proof}

\begin{theorem}[Decompiler right-inverse non-escape boundary]
\label{thm:decompiler-right-inverse-non-escape-boundary}
Every accepted $\DecompilerRightInverseUp$ packet exports decompilation only as
the displayed target-to-source graph $\Psi$ together with the target
optimisation row $O$ and paired right-inverse ledger $R$. A consumer may use the
packet only through $S,T,K,\Psi,O,R,H,C,P,N$, and cannot read it as a global
compiler inverse, host semantic equality, quotient of programs, total
decompiler, or optimiser theorem absent from the displayed rows.
\end{theorem}
\begin{proof}
Inspect the carrier coordinates of
\autoref{def:decompiler-right-inverse-carrier}. The only reverse-direction data
are the finite graph row $\Psi$, its routes through $C$, and its transported
comparison rows through $H$.

The composite comparison is not a hidden inverse law. It is the displayed
target optimisation row $O$ together with the paired right-inverse ledger $R$,
so every target read returns to the target certificate through finite packet
data.

The package and local naming rows expose exactly those coordinates. Since the
carrier has no row for host compiler semantics, program quotients, global
invertibility, total decompiler behaviour, or unrecorded optimiser correctness,
none of those objects can cross the boundary.
\end{proof}

\begin{theorem}[Decompiler right-inverse target-row classifier]
\label{thm:decompiler-right-inverse-target-row-classifier}
Let
$$
D_i=(S_i,T_i,K_i,\Psi_i,O_i,R_i,H_i,C_i,P_i,N_i)\quad (i=0,1)
$$
be accepted $\DecompilerRightInverseUp$ packets in the sense of \autoref{def:decompiler-right-inverse-carrier}. For each $i$, let $G_i$ be the displayed compiler graph read from the accepted $\CertificateCompilerUp$ packet $K_i$, and let $\rho_i(t_i)$ be a displayed target-row right-inverse comparison in $R_i$: it starts from a displayed target row $t_i$ of $T_i$, follows the decompiler graph row $\Psi_i$ to the source certificate, follows $G_i$ back to the target certificate, applies the target optimisation row $O_i$, and then compares the result with $t_i$ through the paired ledger row $R_i$.

Assume the two reads expose the same classifier footprint: $G_0$ and $G_1$, $\Psi_0$ and $\Psi_1$, $O_0$ and $O_1$, and the ledger rows $R_0$ and $R_1$ are matched componentwise by the displayed $\hsame$ transports, and the target inspection and right-inverse comparison routes in $C_0$ and $C_1$ read those matched rows. Then $\rho_0(t_0)$ is accepted by the $\DecompilerRightInverseUp$ target-row classifier exactly when $\rho_1(t_1)$ is accepted. Moreover, the common accepted comparison exports only the displayed compiler graph, decompiler graph, target optimisation row, paired right-inverse ledger, $\hsame$ transports, $\Cont$ routes, $\Pkg$ provenance, and local $\NameCert$ rows; it exports no independent inverse, host decompiler function, semantic equality, quotient of programs, or unrecorded target behaviour.
\end{theorem}
\begin{proof}
Project the carrier in \autoref{def:decompiler-right-inverse-carrier}. A target-row right-inverse comparison has only one available route: target inspection through $T_i$, decompiler readback through $\Psi_i$, compiler readback through the graph displayed by $K_i$, optimisation readback through $O_i$, and paired comparison through $R_i$. These are precisely the rows named in the statement.

The compiler part of the route is finite graph data. By \autoref{thm:certificate-compiler-edge-classifier-exactness}, an accepted compiler-edge comparison is witnessed only by the displayed graph, landing, transported classifier, and displayed continuation rows of the $\CertificateCompilerUp$ packet. Thus the return from the decompiled source row to the target row is determined by the displayed compiler graph $G_i$ and its recorded route data, not by a host compiler meaning.

The right-inverse part is the pair $(O_i,R_i)$. By \autoref{thm:decompiler-right-inverse-namecert-obligations}, $O_i$ supplies the target optimisation surface for the composite $K_i\circ\Psi_i$, while $R_i$ records the componentwise target-row comparison that consumes that optimisation row. Hence, once $G_i$, $\Psi_i$, $O_i$, and $R_i$ are fixed up to the displayed componentwise $\hsame$ transports and are read through the corresponding routes in $C_i$, the classifier has no further coordinate on which the two target-row comparisons could differ.

Therefore the comparison in $D_0$ is accepted exactly when the corresponding comparison in $D_1$ is accepted: both read the same finite target-row classifier footprint, transported componentwise through $H_0$ and $H_1$ and replayed through $C_0$ and $C_1$.

Finally apply the non-escape boundary \autoref{thm:decompiler-right-inverse-non-escape-boundary}. It says that an accepted $\DecompilerRightInverseUp$ packet exports decompilation only as $\Psi$ together with $O$ and $R$, and that every consumer read remains inside $S,T,K,\Psi,O,R,H,C,P,N$. Since the classifier footprint above uses only those rows, the accepted comparison cannot export an independent inverse, host decompiler function, semantic equality, program quotient, or unrecorded target behaviour.
\end{proof}
