\chapter{A Concrete Naming Certificate for $\KernelAcceptanceAuditWitnessUp$}
\label{ch:concrete-instances-kernelacceptanceauditwitness-namecert}
\origin{human}

The $\KernelAcceptanceAuditWitnessUp$ surface turns a Lean acceptance event into a finite \NameCert{} packet. It does not certify the kernel from inside BEDC. It records the generated candidate, the accepted declaration, the environment ledger, the axiom-query boundary, the build replay route, transport, continuation, provenance, and local naming row that make the acceptance event available to audit consumers.

The witness packet consumes the finite audit-gate rows of \autoref{ch:concrete-instances-auditgateboundary-namecert}: source-token refusal, axiom-purity dependency report, paper-Lean target audit, origin and import conservativity ledger, and GAP boundary rows. The $\KernelAcceptanceAuditWitnessUp$ packet specializes that sibling gate surface to a single Lean acceptance event by adding generated-term, kernel-candidate, accepted-declaration, environment-ledger, query, replay, transport, continuation, provenance, and local $\NameCert{}$ rows.

\begin{definition}[KernelAcceptanceAuditWitness carrier]
\label{def:kernel-acceptance-audit-witness-carrier}
A $\KernelAcceptanceAuditWitnessUp$ carrier is a finite $\BHist$ packet
$$
W=(G,K,A,L,Q,R,H,C,P,N)
$$
with the following displayed rows:
\begin{enumerate}
\item $G$ is the generated-term row naming the elaborated proof term, definition body, inductive declaration, or recursor use submitted to the Lean kernel;
\item $K$ is the kernel-candidate row, pairing $G$ with its claimed type, universe data, binder context, recursion account, or constructor signature as applicable;
\item $A$ is the accepted-declaration row, present only when the candidate has been accepted under the current Lean environment;
\item $L$ is the environment-ledger row recording the accepted name and the module position at which the declaration is entered;
\item $Q$ is the axiom-query row naming the transitive dependency query surface used by the strict purity audit;
\item $R$ is the build-replay row naming the project replay that reconstructs the same acceptance event from the Lean sources;
\item $H$ records componentwise $\hsame$ transport for the generated term, candidate, accepted declaration, ledger row, axiom query, replay row, provenance, and local naming row;
\item $C$ records the displayed $\Cont$ routes through which consumers read kernel acceptance, inspect the ledger position, ask the axiom query, and replay the build route;
\item $P$ is the $\Pkg$ provenance over $\BHist$, generated terms, kernel candidates, accepted declarations, environment ledgers, axiom-query rows, build-replay rows, $\hsame$, $\Cont$, and $\NameCert$;
\item $N$ is the local $\NameCert_{\KernelAcceptanceAuditWitnessUp}$ row.
\end{enumerate}
The carrier exports no row that proves the Lean kernel correct, no row that turns a weak theorem statement into a stronger paper claim, and no row that hides a dependency outside the axiom-query boundary.
\end{definition}
