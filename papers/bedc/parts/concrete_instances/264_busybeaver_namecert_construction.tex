\chapter{A Concrete Naming Certificate for $\BusyBeaverUp$}
\label{ch:concrete-instances-busybeaver-namecert}

The busy-beaver interface $\BusyBeaverUp$ packages a finite-state-bounded halting maximum predicate parametrised by a displayed state count and a candidate output bound. It depends on $\NameCert_{\NatUp}$ (\autoref{ch:concrete-instances-nat-namecert}), $\NameCert_{\TuringMachineUp}$ (\autoref{ch:concrete-instances-turingmachine-namecert}), and $\NameCert_{\ComputableUp}$ (\autoref{ch:concrete-instances-computable-namecert}).

\section{Carrier definition}
\label{sec:busybeaver-carrier}

A $\BusyBeaverUp$ token carries a unary state-count $n$, a finite enumeration of $n$-state two-symbol $\TuringMachineUp$ codes, a unary candidate output bound $B$, and a halted-step ledger. The carrier accepts a witness when, for every machine code in the enumeration that halts on the empty tape, its halted output and step count are bounded by $B$ in the sense of $\NameCert_{\ComputableUp}$. The classifier is $\hsame$ on output bounds at fixed unary state count; the ledger keeps the enumeration, the halted bound $B$, the per-machine readback, and the non-halting witness rows visible.

\begin{theorem}[BusyBeaver halted-bound admissibility]
\label{thm:busybeaver-halted-bound-admissibility}
For a $\BusyBeaverUp$ token with unary state-count $n$, displayed finite
enumeration $E$ of $n$-state two-symbol $\TuringMachineUp$ codes, unary bound
$B$, and halted-step ledger $L$, every code $M$ in $E$ whose empty-input row is
recorded in $L$ as halted has a $\ComputableUp$ bounded-trace row in $L$. That
row starts from the displayed empty tape, follows the finite
$\TuringMachineUp$ transition trace for $M$, reads back the halted output row,
records the halted step-count row, and compares both rows with $B$ through the
$\NatUp$ order surface.
\end{theorem}
\leanchecked{BEDC.Derived.BusyBeaverUp.BusyBeaverHaltedBoundAdmissibility\_halted\_trace\_surface}
\begin{proof}
Project the accepted $\BusyBeaverUp$ packet to its finite enumeration and
halted-step ledger. Membership of $M$ in $E$ is a displayed finite row, and the
halted case is exactly the ledger branch that carries a finite
$\TuringMachineUp$ configuration trace from the empty input to a halted row.

Read that halted branch through the $\TuringMachineUp$ finite trace and tape
readback rows. They supply the endpoint configuration, output readback, and
unary step count without appealing to a host run or to a universal-machine
coding theorem.

The carrier clause for $\BusyBeaverUp$ requires this halted branch to be
repacked as a $\ComputableUp$ bounded trace. The same ledger row supplies the
bound comparisons against $B$ in the $\NatUp$ order, so the accepted witness is
admissible exactly for the halted machines already displayed in $E$.
\end{proof}

\begin{theorem}[BusyBeaver enumeration completeness]
\label{thm:busybeaver-enumeration-completeness}
The finite enumeration surface of a $\BusyBeaverUp$ token exhausts the displayed
$n$-state two-symbol $\TuringMachineUp$ code rows for its fixed unary state
count $n$. Every admitted machine row is either one of the listed enumeration
entries with its corresponding ledger branch, or it is outside the token's
carrier surface; no host universal-machine theorem, ambient coding theorem, or
external halting search is imported into the enumeration claim.
\end{theorem}
\begin{proof}
Unpack the source coordinate of the token. The enumeration is a finite BEDC row
whose entries are already typed as $n$-state two-symbol $\TuringMachineUp$
codes, and the carrier consumes only entries addressed by that row.

For a displayed code row at the same state count and alphabet, enumeration
completeness is the finite coverage statement internal to the token: the row is
found at one listed position, with the halted or non-halting ledger branch
attached there. If no listed position is supplied, the row has no carrier
membership witness for this $\BusyBeaverUp$ token.

Thus the proof is finite projection from the enumeration surface, not an
argument that a universal simulator can decide which machines halt. The only
readback used downstream is the per-entry $\TuringMachineUp$ or
$\ComputableUp$ ledger row already attached to the finite list.
\end{proof}

\begin{theorem}[BusyBeaver bound monotonicity]
\label{thm:busybeaver-bound-monotonicity}
Let a $\BusyBeaverUp$ token with enumeration $E$, unary state-count $n$, bound
$B$, and halted-step ledger $L$ be accepted. If $B'$ is another unary
$\NatUp$ bound with $B \leq B'$ in the displayed order surface, then replacing
$B$ by $B'$ preserves the accepted witness with the same enumeration and the
same halted-step ledger. The transport changes only the bound rows; it does not
add a halting-decision row and does not assert computability of the
busy-beaver function.
\end{theorem}
\begin{proof}
Start with an accepted witness at bound $B$. Each halted entry in the finite
enumeration has, by
\autoref{thm:busybeaver-halted-bound-admissibility}, a $\ComputableUp$
bounded trace whose output and step-count rows are bounded by $B$.

Transport those two comparisons along the supplied $\NatUp$ order proof
$B \leq B'$. The enumeration row, halted trace, output readback, and step-count
ledger are unchanged; only the terminal bound comparison is replaced by the
larger unary bound.

Non-halting ledger branches are also unchanged, since they carry no halted
output comparison. Repacking the same finite ledger with the transported
comparisons yields an accepted $\BusyBeaverUp$ witness at $B'$, with no new
halting oracle and no computability claim about the global busy-beaver
function.
\end{proof}
