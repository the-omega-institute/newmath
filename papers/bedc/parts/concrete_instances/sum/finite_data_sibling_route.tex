\begin{theorem}[Sum finite-data sibling route]
\label{thm:sum-finite-data-sibling-route}
The scoped finite-data exit surface of $\SumUp$ reads its visible branch tag through the same $b0/b1$ discipline licensed by \autoref{ch:concrete-instances-bool-namecert}. Once that visible tag has selected the left or right branch, the selected payload is carried only through the $\SumUp$ semantic certificate of \autoref{thm:concrete-sum-namecert} and through the existing branch exactness rows, including the single-valued visible readback of \autoref{thm:sum-history-carrier-single-valued-visible-readback}. Thus the sibling route identifies the finite tag surface consumed by $\SumUp$ without adding a host coproduct eliminator, a hidden branch coordinate, or any classifier beyond the displayed tagged finite-data rows.
\end{theorem}
\begin{proof}
The semantic certificate of \autoref{thm:concrete-sum-namecert} supplies the carrier and classifier for the tagged sum packet. Its two public entry forms are the displayed injections $\Ezero(l)$ and $\Eone(r)$, so the branch decision is already a visible finite tag rather than an ambient coproduct object.

The tag discipline is the same two-row $b0/b1$ surface recorded by \autoref{ch:concrete-instances-bool-namecert}. In the $\SumUp$ chapter this discipline is consumed only as the selected visible branch. The payload itself remains governed by the left or right source classifier inside the local sum certificate.

Apply the visible readback theorem \autoref{thm:sum-history-carrier-single-valued-visible-readback}. It says that the public history carrier has a single visible branch presentation, so the tag that selects the branch also selects the payload row used downstream. The branch exactness rows then compare only same-tag endpoints through the displayed branch classifier.

Therefore the route from finite tag to payload is a sibling finite-data route: it reads the branch through the BoolUp tag surface and then stays inside the SumUp semantic certificate and ledger rows. No hidden branch coordinate, host eliminator, or additional classifier is supplied.
\end{proof}

\begin{theorem}[Sum finite-data Option route]
\label{thm:sum-finite-data-option-route}
The optional value surface of \autoref{ch:concrete-instances-option-namecert} is the one-sided specialization of the same visible $b0/b1$ tag discipline used by $\SumUp$. $\OptionUp$ contributes the none/some finite tag route, while $\SumUp$ contributes the two-payload branch classifier through \autoref{thm:sum-finite-data-sibling-route}; neither route exports a host coproduct eliminator or hidden branch coordinate.
\end{theorem}
\begin{proof}
Project the $\SumUp$ semantic certificate of \autoref{thm:concrete-sum-namecert}. Its visible branch surface is the pair of displayed tags $\Ezero(l)$ and $\Eone(r)$, and the payload is read only after the tag selects the left or right source classifier.

Compare this with the $\OptionUp$ surface of \autoref{ch:concrete-instances-option-namecert}. The none/some distinction is a one-sided finite tag route: the none case carries no payload, while the some case carries the single payload behind its visible tag.

Use \autoref{thm:sum-finite-data-sibling-route}. The existing BoolUp $b0/b1$ discipline identifies the shared visible tag layer, so the Option route is a sibling finite-data route rather than a new classifier or hidden branch coordinate.
\end{proof}
