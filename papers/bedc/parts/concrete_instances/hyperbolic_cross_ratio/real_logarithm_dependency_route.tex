\begin{theorem}[Hyperbolic cross-ratio logarithm route]
\label{thm:hyperbolic-cross-ratio-real-logarithm-route}
Every accepted $\HyperbolicCrossRatioUp$ read that exports a logarithmic distance or invariant must first display the four endpoint rows $B_0,B_1,B_2,B_3$, the cross-ratio comparison ledger $Q$, positive-domain admission, the $\RealLogarithmUp$ inverse route of \autoref{ch:concrete-instances-real-logarithm-namecert}, the $\RealExponentialUp$ comparison row, $\RealAbsUp$ error control, componentwise $\hsame$ transport, displayed $\Cont$ replay, $\Pkg$ provenance, and local $\NameCert$. The logarithm is a sibling dependency read through the displayed RealLogarithm certificate; it is not a host logarithm primitive and it exports no quotient boundary equality, selected ideal endpoint, host global geometry, or unrecorded disk-map invariant.
\end{theorem}
\begin{proof}
Project the carrier of \autoref{def:hyperbolic-cross-ratio-carrier}. The endpoint rows $B_0,B_1,B_2,B_3$ and the cross-ratio comparison row $Q$ are already displayed before any downstream invariant can be read.

The logarithmic route is then admitted only as a positive-domain read. Invoke the $\RealLogarithmUp$ certificate of \autoref{ch:concrete-instances-real-logarithm-namecert}: its inverse route passes through the $\RealExponentialUp$ comparison row and the $\RealAbsUp$ error-control row before a logarithm endpoint is consumed.

Finally transport and replay the same carrier route through $H,C,P,N$. The sibling logarithm certificate supplies the inverse-route dependency, while the $\HyperbolicCrossRatioUp$ carrier supplies the four endpoint rows and comparison ledger. Neither surface contains a host logarithm, quotient boundary equality, selected ideal endpoint, global geometry oracle, or unrecorded disk-map invariant.
\end{proof}
