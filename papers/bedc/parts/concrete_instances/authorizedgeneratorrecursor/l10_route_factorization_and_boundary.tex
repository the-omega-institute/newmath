\begin{theorem}[AuthorizedGeneratorRecursor completion boundary exhaustion]
\label{thm:authorized-generator-recursor-completion-boundary-exhaustion}
Let $R=(I,E,M,B,D,O,A,H,C,P,G,N)$ be an accepted $\AuthorizedGeneratorRecursorUp$ packet. A completion-facing consumer of the recursor output has exactly two admissible boundary shapes: it either displays a separate completion-seal row before using completion data, or it remains bounded by the AuthorizedGeneratorRecursor packet through $I,E,M,B,D,O,A,H,C,P,G,N$. Without the separate completion-seal row, the consumer cannot read a completion object, quotient completion equality, selected limit, host completeness proof, or off-packet proof producer.
\end{theorem}
\begin{proof}
Begin with \autoref{thm:authorized-generator-recursor-open-phase-consumer-handoff}. Completion-facing use is treated as a downstream specialization of the same finite output route, so $O$ remains attached to $A,H,C,P,G,N$.

Apply \autoref{thm:authorized-generator-recursor-open-phase-ledger-nonescape}. The ledger exported by the packet is finite and local to $I,E,M,B,D,O,A,H,C,P,G,N$, and it does not export completion objects, real-completeness data, quotient completion equality, selected limits, or host completeness proofs.

If the consumer displays a separate completion-seal row, that row is outside the AuthorizedGeneratorRecursor packet and must carry its own boundary obligation. If it does not, the ground-compiler authorization theorem \autoref{thm:authorized-generator-recursor-ground-compiler-authorization} keeps the consumer on the accepted packet surface, exhausting the completion boundary at $G,N$ and the retained audit route.
\end{proof}
