\begin{theorem}[AuthorizedGeneratorRecursor root source]
\label{thm:authorized-generator-recursor-root-source}
Every accepted $\AuthorizedGeneratorRecursorUp$ recursor route exposes the generator source, authorization token, boundary row, displayed $\Cont$ route, $\Pkg$ provenance, ledger row, and local $\NameCert$ row before a consumer may read the output. Concretely, the route remains inside the finite $\BHist$ packet
$$
I,E,M,B,D,O,A,H,C,P,G,N
$$
where $I$ names the accepted source signature, $E$ names the emitted generator row, $B$ names the authorized branch package, $G$ names the boundary row, $C$ names the displayed continuation route, $P$ names package provenance, and $N$ names the local certificate.
\end{theorem}
\begin{proof}
Start from carrier admission in \autoref{thm:authorized-generator-recursor-carrier-admission}. It identifies the accepted recursor route with the displayed finite $\BHist$ packet and leaves no separate generator-source coordinate.

Use the route authorization surface of \autoref{thm:authorized-generator-recursor-route-authorization}. The source signature, emitted recursor, motive, branch package, descent row, output row, audit row, and local certificate are all read through the same accepted route.

Finally apply the boundary and terminal output rows. Theorems~\ref{thm:authorized-generator-recursor-root-boundary-ledger-totality} and~\ref{thm:authorized-generator-recursor-terminal-authorized-output-cover} force any output-facing read to pass through $G,C,P,N$ together with the displayed ledger and package rows. Hence generator source, authorization token, boundary, continuation, provenance, ledger, and local naming are exposed before output consumption.
\end{proof}
\leanchecked{BEDC.Derived.AuthorizedGeneratorRecursorUp.AuthorizedGeneratorRecursorRootSource}

\begin{theorem}[AuthorizedGeneratorRecursor boundary closure]
\label{thm:authorized-generator-recursor-boundary-closure}
Every closed substitution or generator-normalization consumer of an accepted $\AuthorizedGeneratorRecursorUp$ output must pass through the displayed source, authorization, boundary, $\Cont$, $\Pkg$, ledger, $\hsame$, and $\NameCert$ rows before reading $O$. The consumer may use the output only as a row of the packet $I,E,M,B,D,O,A,H,C,P,G,N$, not as a detached substitution rule or host generator.
\end{theorem}
\begin{proof}
First use the closed-substitution handoff. \autoref{thm:authorized-generator-recursor-closed-substitution-handoff} requires every closed-substitution consumer to remain inside the displayed $\BHist$, $\hsame$, $\Cont$, $\Pkg$, and $\NameCert$ route.

Next use terminal readiness. \autoref{thm:authorized-generator-recursor-output-readiness} states that $O$ is available only after the signature, emitted recursor, motive, branch package, descent, audit, transport, continuation, provenance, boundary, and local certificate rows have all been retained.

The generator-normalization side is therefore not a separate host route. By \autoref{thm:authorized-generator-recursor-terminal-branch-ledger-lock}, every terminal branch read is locked to the closed-substitution and ground-compiler authorization rows already displayed by the packet. Thus closed substitution and generator normalization pass through the displayed boundary before reading the output.
\end{proof}
\leanchecked{BEDC.Derived.AuthorizedGeneratorRecursorUp.AuthorizedGeneratorRecursorBoundaryClosure}

\begin{theorem}[AuthorizedGeneratorRecursor ledger nonescape]
\label{thm:authorized-generator-recursor-ledger-nonescape}
No ledger of an accepted $\AuthorizedGeneratorRecursorUp$ packet can export an unlicensed generator, host recursion oracle, or hidden substitution step. Every ledger-facing read of generator source, authorization, boundary evidence, continuation replay, package provenance, componentwise $\hsame$ transport, or local naming data remains inside the displayed $\BHist$, $\Cont$, $\Pkg$, and $\NameCert$ rows of the same packet.
\end{theorem}
\leanchecked{BEDC.Derived.AuthorizedGeneratorRecursorUp.AuthorizedGeneratorRecursorLedgerNonescape}
\begin{proof}
Begin with the root source theorem above. It exposes generator source, authorization, boundary, continuation, provenance, ledger, and local naming as displayed rows of one finite packet.

Apply boundary closure. Any closed substitution or generator-normalization consumer that reaches $O$ has already passed through the displayed boundary, $\hsame$, $\Cont$, $\Pkg$, and $\NameCert$ rows.

It remains to exclude hidden authority. Theorems~\ref{thm:authorized-generator-recursor-consumer-nonescape} and~\ref{thm:authorized-generator-recursor-terminal-reader-nonescape} rule out hidden generator state, host evaluation, unchecked recursion, and private audit routes. Therefore the ledger has no coordinate from which to export an unlicensed generator, host recursion oracle, or hidden substitution step.
\end{proof}

$$

$$

before any output-facing read. These rows are the accepted source signature, emitted generator, motive, branch package, structural descent, output, audit, componentwise transport, continuation, provenance, boundary, and local $\NameCert$ rows of the displayed $\BHist$ packet.
\end{theorem}

\begin{proof}

Use the root source theorem above to place generator source, authorization token, boundary, continuation, provenance, ledger, and local naming on the same admitted packet. Thus the output row $O$ is not visible before the surrounding admission rows have been displayed.

Finally apply boundary closure. The boundary, $\hsame$, $\Cont$, $\Pkg$, and $\NameCert$ rows stay attached to the same carrier before any closed-substitution or generator-normalization consumer may read $O$, so the carrier admission is exhausted by the stated finite packet.
\end{proof}

\begin{corollary}[AuthorizedGeneratorRecursor root source strict obstruction]
\label{cor:authorized-generator-recursor-root-source-strict-obstruction}
Let
$$
R=(I,E,M,B,D,O,A,H,C,P,G,N)
$$
be an accepted $\AuthorizedGeneratorRecursorUp$ recursor route. Suppose a displayed failure, refusal, separation, or countermodel row is attached to a named converse, reuse, or readback route at the root-source surface of $R$. Then the row is a strict local obstruction: it blocks only the named route whose consumer-facing use tries to read the generator source, authorization token, boundary row, displayed $\Cont$ route, $\Pkg$ provenance, ledger row, componentwise $\hsame$ transport, or local $\NameCert$ row away from the displayed packet coordinates
$$
I,E,M,B,D,O,A,H,C,P,G,N.
$$
Every permitted consumer-facing use factors through the root-source exposure rows and then through the ledger-facing $\BHist$, $\Cont$, $\Pkg$, $\hsame$, and $\NameCert$ rows of the same packet. The same displayed row cannot be reinterpreted as a global impossibility theorem for recursors, a host theorem, hidden external validation of the failed route, an unlicensed generator, a host recursion oracle, a detached substitution step, a private authorization channel, or any obstruction to a consumer route that still factors through the displayed rows of $R$.
\end{corollary}
\leanchecked{BEDC.Derived.AuthorizedGeneratorRecursorUp.AuthorizedGeneratorRecursorRootSourceStrictObstruction}
\begin{proof}
Fix an accepted recursor route $R$. By \autoref{thm:authorized-generator-recursor-root-source}, the generator source, authorization token, boundary row, displayed $\Cont$ route, $\Pkg$ provenance, ledger row, and local $\NameCert$ row are exposed before a consumer may read the output, and they are exposed only inside the finite $\BHist$ packet
$$
I,E,M,B,D,O,A,H,C,P,G,N.
$$
Thus a consumer-facing use of the root source has no independent source coordinate, no authorization token outside the route, no boundary row outside $G$, no continuation route outside $C$, no provenance row outside $P$, and no local certificate outside $N$.

Apply \autoref{thm:authorized-generator-recursor-ledger-nonescape} to the ledger side of the same route. Every ledger-facing read of generator source, authorization, boundary evidence, continuation replay, package provenance, componentwise $\hsame$ transport, or local naming data remains inside the displayed $\BHist$, $\Cont$, $\Pkg$, and $\NameCert$ rows of the packet. The ledger therefore exports no unlicensed generator, host recursion oracle, or hidden substitution step.

Now consider the displayed failure, refusal, separation, or countermodel row. If it is attached to a named converse, reuse, or readback route, the first cited theorem forces the attempted consumer-facing use to pass through the root-source packet coordinates, while the second cited theorem forces its ledger component to remain on the displayed transport, continuation, provenance, boundary, and local-name rows. Hence the row can block only the local route at the coordinate where that route tries to read, omit, replace, detach, or bypass one of those displayed rows.

Any stronger interpretation would require a coordinate not supplied by the packet. A global impossibility theorem would require a theorem surface beyond the finite root-source carrier; a host theorem would require host recursion or host evaluation data; hidden external validation would require an external certificate route; an unlicensed generator, host recursion oracle, detached substitution step, or private authorization channel would require exactly the exports ruled out by ledger nonescape. Since neither cited theorem exposes such a coordinate, the obstruction remains strict and local to the named converse, reuse, or readback route, and it does not block any consumer route that continues to factor through the displayed rows of $R$.
\end{proof}
