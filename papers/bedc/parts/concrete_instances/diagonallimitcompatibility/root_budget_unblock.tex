\begin{theorem}[Diagonal-limit compatibility root budget source]
\label{thm:diagonal-limit-compatibility-root-budget-source}
Every accepted $\DiagonalLimitCompatibilityUp$ packet reads its root budget
source through the displayed finite rows
$$
\BHist,\hsame,\Cont,\Pkg,\NameCert.
$$
The budget source is the same source exposed to
$\RegularCauchySelectorBudgetUp$ consumers: the diagonal row, dyadic ledger,
stream window, regular-rational handoff, seal route, componentwise transport,
route table, provenance, and local name must already be visible before the
compatibility packet is accepted.
\end{theorem}
\leanchecked{BEDC.Derived.DiagonallimitcompatibilityUp.DiagonalLimitCompatibilityCarrier\_root\_budget\_source}
\begin{proof}
Project the accepted compatibility carrier. The displayed packet contains the
diagonal source, triangle estimate, seal route, dyadic ledger, stream window,
regular readback, Real seal row, transports, continuations, provenance, and
local naming row.

Apply the NameCert obligation theorem and the window-ledger exactness theorem.
They force the budget-facing reads to use the same carrier, $\hsame$ rows,
$\Cont$ routes, $\Pkg$ provenance, and $\NameCert$ row rather than an
undisplayed budget source.

Finally use completion-boundary and non-escape. Since no host limit, quotient
stream, choice datum, host real equality, or ambient completeness coordinate is
exported, the accepted packet can source its budget only through the displayed
finite rows shared with the selector-budget route.
\end{proof}

\begin{theorem}[Diagonal-limit compatibility root window route]
\label{thm:diagonal-limit-compatibility-root-window-route}

$\StreamNameUp$ finite-window route is selected before any $\RegSeqRatUp$
readback or $\RealUp$ seal handoff is available. A downstream consumer reaches
the regular-rational row and the Real-facing seal only after reading the
displayed window row, dyadic ledger, transport rows, continuation rows,
provenance row, and local naming row.
\end{theorem}
\begin{proof}
Start with the carrier projection. The window row $W$ and dyadic ledger $D$
are displayed before the readback row $Q$ and seal row $L$ are consumed.

Use window-ledger exactness together with the triangle-seal route theorem.
These results place the route to $Q,L$ behind the same $\StreamNameUp$ window
row used by the diagonal source, triangle row, dyadic ledger, and seal route.

Apply completion boundary. It adds the componentwise $\hsame$ transport,
displayed $\Cont$ route, $\Pkg$ provenance, and local $\NameCert$ rows to the
same finite-window route, so no RegSeqRat or Real handoff precedes the window
selection.
\end{proof}

$R,T,S,D,W,Q,L,H,C,P,N$, not a read through host real equality, a selected host

\begin{proof}

dyadic ledger, and stream window have been fixed.

Now apply Real-seal exactness. It identifies the displayed route through
$R,T,S,D,W,Q,H,C,P,N$ as the boundary through which $L$ is reached.

The Real-seal non-escape theorem removes the possible host-side bypasses.
Thus the Real-facing read is exactly a finite regular-readback and seal-route
read inside the accepted compatibility packet.
\end{proof}

\begin{theorem}[Diagonal-limit compatibility root consumer non-escape]
\label{thm:diagonal-limit-compatibility-root-consumer-nonescape}
Every downstream $\RealUp$, $\CompletionUp$, or completion-seal consumer of an
accepted $\DiagonalLimitCompatibilityUp$ packet can inspect only the displayed
budget, window, diagonal, seal, transport, route, provenance, and local-name
rows. In packet notation the consumer-visible surface is exhausted by
$$
R,T,S,D,W,Q,L,H,C,P,N.
$$
\end{theorem}
\begin{proof}
Project the accepted packet and list the consumer-facing coordinates. The
budget and window information is carried by the dyadic ledger and stream
window, the diagonal and seal information by the source, triangle, seal,
readback, and Real rows, and the structural information by the transport,
route, provenance, and local-name rows.

Use the completion-consumer non-escape theorem. It already requires every
completion-facing or $\RealUp$ consumer to factor through the same listed
rows.

Finally replay the root budget source, root window route, and root seal
boundary results. Together they show that the consumer has no coordinate
outside the displayed budget, window, diagonal, seal, transport, route,
provenance, and local-name surface.
\end{proof}

\begin{theorem}[Diagonal-limit compatibility root budget-seal factorization]
\label{thm:diagonal-limit-compatibility-root-budget-seal-factorization}

In packet notation the consumer route is
$$
R,T,S,D,W,Q,L,H,C,P,N,
$$
and it excludes host real equality, selected host limits, quotient stream equality, choice data, and ambient completeness.
\end{theorem}
\begin{proof}
Start with the root budget source. \autoref{thm:diagonal-limit-compatibility-root-budget-source} identifies the budget-facing source with the displayed diagonal row, dyadic ledger, stream window, regular-rational handoff, Real seal route, transports, continuations, provenance, and local name.

Next apply the root window route. \autoref{thm:diagonal-limit-compatibility-root-window-route} forces the $\StreamNameUp$ finite-window row to be read before the $\RegSeqRatUp$ readback or the $\RealUp$ seal handoff is available.

Then use the root seal boundary. \autoref{thm:diagonal-limit-compatibility-root-seal-boundary} places each Real-facing read behind the regular readback and seal rows $Q,L$ on the displayed route through $R,T,S,D,W,Q,L,H,C,P,N$.

Finally apply root consumer non-escape. \autoref{thm:diagonal-limit-compatibility-root-consumer-nonescape} exhausts downstream Real, completion, and completion-seal consumers by the displayed packet rows, so none of the excluded host or quotient objects has a route into the accepted consumer surface.
\end{proof}

The exactness results of \autoref{thm:cauchy-tail-witness-envelope-finite-window-exactness} and \autoref{thm:cauchy-tail-witness-envelope-tail-comparison-exactness} keep the selected window and endpoint ledger on the same finite envelope surface.

Finally apply the Real-tail seal boundary and non-escape results of \autoref{thm:cauchy-tail-witness-envelope-real-tail-seal-boundary} and \autoref{thm:cauchy-tail-witness-envelope-nonescape}. \begin{theorem}[DiagonalLimitCompatibility terminal budget readback]
\label{thm:diagonal-limit-compatibility-terminal-budget-readback}
Let an accepted $\DiagonalLimitCompatibilityUp$ packet have a terminal $\RealUp$ or $\CompletionUp$ consumer read, and suppose that this read is later consumed by either an accepted $\CauchyTailWitnessEnvelopeUp$ packet or an accepted $\CauchyModulusRefinementUp$ packet. Then the terminal read first factors through the same displayed diagonal budget prefix
$$
R,T,S,D,W,Q,L,H,C,P,N.
$$
Only after this prefix may the route branch: the envelope case proceeds through
$$
S,F,Q,A,R,H,C,P,N,
$$
while the refinement case proceeds through
$$
M_0,M_1,U,V,T,W,Q,E,H,C,P,N.
$$
The common readback exports no quotient stream equality, selected host limit, hidden tail oracle, hidden selector, host real equality, choice datum, or ambient completeness coordinate.
\end{theorem}
\begin{proof}
First build the common diagonal prefix. Apply \autoref{thm:diagonal-limit-compatibility-root-budget-source}, \autoref{thm:diagonal-limit-compatibility-root-window-route}, and \autoref{thm:diagonal-limit-compatibility-root-seal-boundary} to place the terminal $\RealUp$ or $\CompletionUp$ read behind the root budget, window, seal, transport, route, provenance, and local-name rows.

Next close the prefix against escape. Use \autoref{thm:diagonal-limit-compatibility-root-consumer-nonescape}, \autoref{thm:diagonal-limit-compatibility-root-budget-seal-factorization}, and \autoref{thm:diagonal-limit-compatibility-root-provenance-exhaustion}. These identify the reusable consumer-facing tuple as $R,T,S,D,W,Q,L,H,C,P,N$ and rule out quotient stream equality, selected host limits, host real equality, choice data, and ambient completeness before any downstream branch is inspected.

Now split by the downstream consumer. If the read is consumed by $\CauchyTailWitnessEnvelopeUp$, apply \autoref{thm:diagonal-limit-compatibility-cauchy-tail-envelope-factorization}; the remaining route is exactly $S,F,Q,A,R,H,C,P,N$, and hidden tail oracles are excluded.

If the read is consumed by $\CauchyModulusRefinementUp$, apply \autoref{thm:diagonal-limit-compatibility-cauchy-modulus-refinement-route}; the remaining route is exactly $M_0,M_1,U,V,T,W,Q,E,H,C,P,N$, and hidden selectors are excluded. Thus both terminal branches share the same diagonal budget readback boundary before they diverge.
\end{proof}
