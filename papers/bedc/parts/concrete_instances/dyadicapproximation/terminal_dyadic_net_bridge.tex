\begin{theorem}[DyadicApproximation terminal dyadic-net bridge]
\label{thm:dyadicapproximation-terminal-dyadic-net-bridge}
Let $T$ be an accepted terminal $\DyadicApproximationUp$ finite packet, and
let $D$ be an accepted $\RealDiagonalCompletionUp$ packet matched to $T$ by
the displayed terminal diagonal handoff. Suppose a bridge-facing consumer
reads the matched pair as a finite $\RealUp$ and completion packet through
the terminal dyadic-net consumer route.

Then the exported bridge-facing surface is exactly the terminal dyadic-net
consumer surface of
\autoref{def:dyadicapproximation-terminal-dyadic-net-consumer-surface}. It is
the same finite packet through terminal precision, transported dyadic
endpoint, $\RationalIntervalUp$ cell, $\DyadicMeshUp$ selector,
$\ValidatedNumericsUp$ enclosure, $\StreamNameUp$ window, $\RegSeqRatUp$
readback, $\RealUp$ seal, $\RealDiagonalCompletionUp$ selector, visible
$\hsame$ rows, visible $\Cont$ routes, $\Pkg$ provenance, and local
$\NameCert$ rows.

The bridge exports no quotient streams, selected host limits, solver
correctness, metric completeness, or host real equality.
\end{theorem}
\leanchecked{BEDC.Derived.DyadicApproximationUp.DyadicApproximationCarrier\_terminal\_dyadic\_net\_bridge}
\begin{proof}
First read the matched pair through the terminal dyadic-net consumer
surface. Apply
\autoref{thm:dyadicapproximation-terminal-dyadic-net-consumption}. This
already exposes the terminal precision, dyadic endpoint, rational interval
cell, mesh selector, validated enclosure, stream window, regular-rational
readback, real seal, diagonal selector, transports, continuation routes,
package rows, and local certificates.

Next identify the completion side. Apply
\autoref{thm:dyadicapproximation-terminal-diagonal-completion-handoff}. The
matched $\RealDiagonalCompletionUp$ packet reads the same terminal
precision, endpoint, scheduled window, regular-rational readback, real seal,
transports, continuations, package provenance, and local certificate
provenance as the terminal dyadic packet.

Now compose the readback route. Apply
\autoref{thm:dyadicapproximation-terminal-diagonal-readback-composition}.
The diagonal consumer cannot replace the terminal rows by completed real
equality or by a selected completion point; it repacks the same finite
terminal rows for the completion-facing read.

Finally attach the stream seal source by
\autoref{thm:dyadicapproximation-finite-packet-streamname-real-seal-factorization}.
The scheduled $\StreamNameUp$ row, $\RegSeqRatUp$ readback, dyadic endpoint,
terminal precision, $\RealUp$ seal, diagonal selector, transports,
continuations, package provenance, and local dyadic certificate provenance
are therefore one finite surface.

The four cited surfaces name the same displayed coordinates. Repacking them
as the bridge-facing packet gives exactly
\autoref{def:dyadicapproximation-terminal-dyadic-net-consumer-surface}. Since
none of those surfaces contains a coordinate for quotient streams, selected
host limits, solver correctness, metric completeness, or host real equality,
the bridge cannot export such data.
\end{proof}
