\chapter{A Concrete Naming Certificate for $\CauchyExtensionOperatorUp$}
\label{ch:concrete-instances-cauchy-extension-operator-namecert}
\origin{human}

Following Bishop and Bridges and Kelley-style completion language, the $\CauchyExtensionOperatorUp$ packet records the finite operator boundary through which completion-extension consumers read a Cauchy-continuous source map before reaching the terminal real seal. It consumes the Cauchy-continuous extension route of \autoref{ch:concrete-instances-cauchy-continuous-extension-namecert}, the extension and uniqueness surface of \autoref{ch:concrete-instances-completionuniversalproperty-namecert}, the uniform handoff of \autoref{ch:concrete-instances-uniform-continuous-extension-namecert}, the finite stream windows of \autoref{ch:concrete-instances-streamname-namecert}, the regular rational source rows of \autoref{ch:concrete-instances-regseqrat-namecert}, the dyadic tolerance ledger of \autoref{ch:concrete-instances-dyadicratcore-namecert}, and the terminal real-name boundary of \autoref{ch:concrete-instances-real-namecert}. The packet names the operator-level service boundary; it does not identify host functions, build quotient completions, choose limits, or import ambient completeness.

\begin{definition}[Cauchy extension operator carrier]
\label{def:cauchy-extension-operator-carrier}
A $\CauchyExtensionOperatorUp$ carrier is a finite $\BHist$ packet
$$
O=(S,W,D,F,U,E,H,C,P,N)
$$
with the following displayed rows:
\begin{enumerate}
\item $S$ is the source regular-Cauchy row, read through $\RegSeqRatUp$ data and naming the source observations to which the operator is applied.
\item $W$ is the finite $\StreamNameUp$ window schedule opened for source reads, comparison requests, and operator replay.
\item $D$ is the $\DyadicRatCoreUp$ tolerance ledger controlling the source-window precision and the completion-facing tolerance request.
\item $F$ is the Cauchy-continuous map row, recording that matching source windows inside the ledger produce matching target completion requests.
\item $U$ is the completion-universal extension row, aligned with $\CompletionUniversalPropertyUp$ and with the $\UniformContinuousExtensionUp$ handoff when a uniform-modulus row is displayed.
\item $E$ is the target $\RealUp$ seal row through which the completed extension may be read only after $S,W,D,F,U$ have been replayed.
\item $H$ records componentwise $\hsame$ transport for $S,W,D,F,U,E$ and for the structural rows of the packet.
\item $C$ records displayed $\Cont$ replay of the route
$$
S,W,D\longmapsto F\longmapsto U\longmapsto E .
$$
\item $P$ is the $\Pkg$ provenance over $\CauchyExtensionOperatorUp$, $\CauchyContinuousExtensionUp$, $\CompletionUniversalPropertyUp$, $\UniformContinuousExtensionUp$, $\StreamNameUp$, $\RegSeqRatUp$, $\DyadicRatCoreUp$, $\RealUp$, $\BHist$, $\hsame$, $\Cont$, and $\NameCert$.
\item $N$ is the local $\NameCert_{\CauchyExtensionOperatorUp}$ row.
\end{enumerate}
The carrier stores no host function equality, quotient completion, selected limit, countable-choice schedule, ambient $\RealUp$ completeness coordinate, Lean verification row, or standard bridge.
\end{definition}

\begin{theorem}[Cauchy extension operator NameCert obligations]
\label{thm:cauchy-extension-operator-namecert-obligations}
For the carrier of \autoref{def:cauchy-extension-operator-carrier}, the local $\NameCert_{\CauchyExtensionOperatorUp}$ surface consists exactly of the following finite obligations:
\begin{enumerate}
\item carrier admission for the displayed packet $O=(S,W,D,F,U,E,H,C,P,N)$;
\item source-window control, requiring every operator read to inspect the regular-Cauchy source row $S$ through the finite windows $W$ and dyadic ledger $D$ before a map row is available;
\item Cauchy-continuous map control, requiring the row $F$ to be the only route from source Cauchy closeness to a target completion request;
\item completion-extension control, requiring the row $U$ to read the same $S,W,D,F$ coordinates through the completion-universal extension surface and any displayed uniform-extension handoff;
\item real-seal boundary, requiring every exported real-facing read to pass through $E$ only after $S,W,D,F,U$ have been replayed;
\item ledger non-escape, requiring $H,C,P,N$ to expose no host function equality, quotient completion, selected limit, countable choice, ambient $\RealUp$ completeness, Lean verification, or bridge checking.
\end{enumerate}
No NameCert obligation for this operator is stored outside these rows.
\end{theorem}
\begin{proof}
Carrier admission is projection from \autoref{def:cauchy-extension-operator-carrier}. The accepted packet displays exactly the source row, finite windows, tolerance ledger, map row, extension row, real seal, and structural rows.

Source-window control reads $S,W,D$. The regular source can be observed only through the finite window schedule and dyadic tolerance ledger, so an operator request cannot reach the map row before those finite source coordinates are present.

Map and extension control read $F$ and $U$. The row $F$ is the only displayed Cauchy-continuous passage from source closeness to a target completion request, and the row $U$ repacks that request through the completion-universal extension surface over the same source route.

The real-seal boundary is then the projection to $E$. Since $C$ replays the order $S,W,D,F,U,E$, any real-facing read is downstream of the finite extension route rather than an independent host real.

Finally inspect $H,C,P,N$. Transport is componentwise, replay follows the displayed coordinates, and provenance plus local naming mention only the rows listed in the carrier. Hence the non-escape clauses exhaust the remaining NameCert surface.
\end{proof}

\begin{theorem}[Cauchy extension operator source control]
\label{thm:cauchy-extension-operator-source-control}
Let
$$
O=(S,W,D,F,U,E,H,C,P,N)
$$
be an accepted $\CauchyExtensionOperatorUp$ carrier. Every extension read exported by the operator factors through the regular-Cauchy source row $S$, the finite $\StreamNameUp$ windows $W$, the $\DyadicRatCoreUp$ tolerance ledger $D$, and the Cauchy-continuous map row $F$ before it reaches the completion-universal extension row $U$.
\end{theorem}
\begin{proof}
Start with the source coordinates. The carrier offers no source observation except $S$ as read through the finite windows $W$ and the tolerance ledger $D$.

Next inspect the map coordinate. The Cauchy-continuous row $F$ consumes exactly the source-window and tolerance data, so a target completion request cannot be formed from an independent host map or from a selected source representative.

The extension row $U$ is downstream of that request. Its only advertised input route is the displayed tuple $S,W,D,F$, and the $\Cont$ row $C$ records the same order.

Therefore every exported extension read factors through the finite source-control route $S,W,D,F$ before the completion-universal extension row can be read.
\end{proof}

\begin{theorem}[Cauchy extension operator real seal boundary]
\label{thm:cauchy-extension-operator-real-seal-boundary}
Let
$$
O=(S,W,D,F,U,E,H,C,P,N)
$$
be an accepted $\CauchyExtensionOperatorUp$ carrier. Every $\RealUp$-facing read exported by $O$ reaches the target seal $E$ only after the operator has replayed the finite route
$$
S,W,D\longmapsto F\longmapsto U.
$$
Consequently the packet exports no host function equality, quotient completion, selected limit, countable choice, ambient $\RealUp$ completeness, Lean verification, or bridge checking as part of its real-facing boundary.
\end{theorem}
\begin{proof}
Project the seal coordinate $E$. It is listed after the source, window, tolerance, map, and completion-extension rows, so it is the terminal read of the carrier rather than a source of additional completion data.

Replay the displayed route with $C$. A real-facing consumer first reads $S,W,D$, then the Cauchy-continuous row $F$, and then the completion-universal extension row $U$ before $E$ can be inspected.

The exclusions follow from the carrier inventory. None of host function equality, quotient completion, selected limits, countable-choice schedules, ambient completeness, Lean verification, or bridge checking is a coordinate of $O$, and componentwise $\hsame$ transport cannot create such a coordinate.
\end{proof}
