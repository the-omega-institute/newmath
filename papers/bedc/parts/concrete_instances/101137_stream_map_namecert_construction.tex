Following Bishop and Bridges, the $\StreamMapUp$ packet records the finite pointwise route by which a certified countable name is mapped to another certified countable name before any regular rational or real-facing consumer reads the result. It sits after the finite window classifier of \autoref{ch:concrete-instances-streamname-namecert}, the tail-window handoff of \autoref{ch:concrete-instances-regseqrat-namecert}, the dyadic tolerance ledgers of \autoref{ch:concrete-instances-dyadicratcore-namecert}, and the terminal real-name seal of \autoref{ch:concrete-instances-real-namecert}. The packet names the map surface itself; it does not assert quotient stream equality, host function extensionality, selected limits, countable choice, or ambient $\RealUp$ completeness.

\begin{definition}[Stream map carrier]
\label{def:stream-map-carrier}
A $\StreamMapUp$ carrier is a finite $\BHist$ packet
$$
M=(S,T,F,W,Q,D,R,H,C,P,N)
$$
with the following displayed rows:
\begin{enumerate}
\item $S$ is the source $\StreamNameUp$ classifier row, exposing only finite source windows and their pointwise name cells.
\item $T$ is the target $\StreamNameUp$ classifier row, exposing only finite target windows and their pointwise name cells.
\item $F$ is the point-map row, assigning each displayed source cell in a requested window to the corresponding displayed target cell.
\item $W$ records the finite-window output observations produced by $F$ from $S$ and consumed by $T$.
\item $Q$ is the $\RegSeqRatUp$ tail-window handoff row attached to the same source and target windows when a regular rational consumer asks for mapped readback.
\item $D$ is the $\DyadicRatCoreUp$ tolerance ledger recording the dyadic budgets, endpoint readbacks, and comparison cells used by $S,T,F,W,Q$.
\item $R$ is the terminal $\RealUp$ seal boundary, visible only after $S,T,F,W,Q,D$ have fixed the finite map route.
\item $H$ records componentwise $\hsame$ transport for source windows, target windows, point-map rows, output observations, regular readback, dyadic ledgers, Real seals, replay rows, provenance, and local naming rows.
\item $C$ records displayed $\Cont$ replay from a stream-map request through source inspection, pointwise mapping, target-window inspection, regular rational handoff, dyadic tolerance inspection, and Real-seal consumption.
\item $P$ is the $\Pkg$ provenance over $\StreamMapUp$, $\StreamNameUp$, $\RegSeqRatUp$, $\DyadicRatCoreUp$, $\RealUp$, $\BHist$, $\hsame$, $\Cont$, and $\NameCert$.
\item $N$ is the local $\NameCert_{\StreamMapUp}$ row.
\end{enumerate}
The classifier compares accepted packets only through $S,T,F,W,Q,D,R,H,C,P,N$. It has no coordinate for quotient stream equality, host function extensionality, selected limits, countable choice, ambient Real completeness, or a detached semantic map between host functions.
\end{definition}

\begin{theorem}[Stream map pointwise stability]
\label{thm:stream-map-pointwise-stability}
Let $M=(S,T,F,W,Q,D,R,H,C,P,N)$ be an accepted $\StreamMapUp$ carrier. If two accepted requests inspect the same displayed finite source window through the $\StreamNameUp$ classifier row $S$ and transport its cells by the componentwise row $H$, then the point-map row $F$ produces the same displayed target cells through $T$ and the same output observations through $W$.
\end{theorem}
\begin{proof}
Project the accepted carrier. The source row $S$ exposes only finite windows and pointwise cells, so equality of inspected requests is read from the same finite classifier rows rather than from a quotient of streams.

The componentwise transport row $H$ moves those displayed cells into the point-map row $F$. Since $F$ has no access to an unbounded stream representative, it can act only on the cells opened by $S$ and named by the request.

The target row $T$ and output row $W$ are downstream of that same point-map read. Their cells are therefore determined by the displayed finite route $S,H,F$, and the packet contains no host function extensionality coordinate that could introduce a second comparison.
\end{proof}

\begin{definition}[Stream map pointwise carrier transport]
\label{def:stream-map-pointwise-carrier-transport}
For an accepted $\StreamMapUp$ carrier $M=(S,T,F,W,Q,D,R,H,C,P,N)$, the pointwise carrier transport datum is the finite relation
$$
\mathsf{Transport}_{M}(s,t)
$$
whose rows first read source $\StreamNameUp$ cell transport through $H$, then apply the point-map row $F$ to the transported source cell, and finally produce the target $\StreamNameUp$ cell transport in $T$ before any $W,Q,D,$ or $R$ row is read. The relation is stored only over displayed finite windows, transported source cells, point-map outputs, target-cell rows, replay, provenance, and local naming.
\end{definition}

\begin{theorem}[Stream map pointwise carrier transport stability]
\label{thm:stream-map-pointwise-carrier-transport-stability}
The datum of \autoref{def:stream-map-pointwise-carrier-transport} is stable under the finite-window functoriality route of \autoref{thm:stream-map-finite-window-functoriality}. In particular, two accepted reads related by the source-cell $\hsame$ rows of $H$ produce related target-cell $\hsame$ rows through $F$ and $T$, and the transport exports no host function extensionality, quotient stream equality, selected limit, or ambient $\RealUp$ completeness.
\end{theorem}
\begin{proof}
Project the pointwise transport relation. Its source side is exactly the source-cell component of $H$ over the finite window opened by $S$.

Apply \autoref{thm:stream-map-pointwise-stability}. The point-map row $F$ consumes the transported displayed source cell and determines the target cell exposed through $T$ and $W$.

Finally apply \autoref{thm:stream-map-finite-window-functoriality}. Since every downstream consumer factors through $S,F,T,W$ before reaching $Q,D,$ or $R$, the transport remains a finite StreamMap datum and cannot expose host function extensionality or quotient-stream data.
\end{proof}

\begin{theorem}[Stream map finite-window functoriality]
\label{thm:stream-map-finite-window-functoriality}
For an accepted $\StreamMapUp$ carrier $M=(S,T,F,W,Q,D,R,H,C,P,N)$, every finite-window consumer of the mapped stream factors through
$$
S \longmapsto F \longmapsto T \longmapsto W
$$
before it can inspect the regular rational handoff $Q$, the dyadic tolerance ledger $D$, or the terminal Real seal $R$.
\end{theorem}
\begin{proof}
Begin with a finite-window request for the mapped stream. Carrier admission exposes the source classifier row $S$ before any target-facing read is available.

The point-map row $F$ is next. It converts the displayed source cells into displayed target cells, and no row in the carrier selects a new infinite representative or compares host functions.

The target classifier $T$ and the output observations $W$ then package exactly the finite target window produced by $F$. Only after this finite route is visible may a consumer inspect the regular rational handoff $Q$, the dyadic ledger $D$, or the Real seal $R$.

Thus every mapped finite-window read has the displayed factorization $S \longmapsto F \longmapsto T \longmapsto W$, and downstream consumers inherit that route rather than a quotient-stream or selected-limit principle.
\end{proof}

\begin{theorem}[Stream map window functoriality]
\label{thm:stream-map-window-functoriality}
For an accepted $\StreamMapUp$ carrier $M=(S,T,F,W,Q,D,R,H,C,P,N)$, the pointwise carrier transport of \autoref{def:stream-map-pointwise-carrier-transport} preserves finite $\StreamNameUp$ windows through the route of \autoref{thm:stream-map-finite-window-functoriality}. Consequently every $\RegSeqRatUp$ or $\RealUp$ handoff reads the mapped window through the same $\BHist$, $\hsame$, $\Cont$, $\Pkg$, and $\NameCert$ route fixed by $S,F,T,W,Q,D,$ and $R$.
\end{theorem}
\begin{proof}
Project the carrier of \autoref{def:stream-map-carrier} and the pointwise transport datum of \autoref{def:stream-map-pointwise-carrier-transport}. The only source cells available to the map are the finite $\StreamNameUp$ window cells exposed by $S$ and transported through $H$.

Apply \autoref{thm:stream-map-finite-window-functoriality}. The mapped observation factors through $S,F,T,W$ before any regular rational, dyadic, or Real-facing row can be read.

Finally apply \autoref{thm:stream-map-regseq-real-handoff}. The handoff to $\RegSeqRatUp$ or $\RealUp$ consumes the same mapped window route and the same structural rows $H,C,P,N$, so it cannot import host function extensionality, quotient streams, selected limits, or ambient Real completeness.
\end{proof}

\begin{theorem}[Stream map constant-window stability]
\label{thm:stream-map-constant-window-stability}
Let $M=(S,T,F,W,Q,D,R,H,C,P,N)$ be an accepted $\StreamMapUp$ carrier applied to a constant certified point stream. Then the displayed finite observation window is preserved by the point-map row: the constant point row read from $S$ is routed through $F$ to the same target-window shape in $T$ and $W$, and any downstream $\RegSeqRatUp$ consumer must first read the same $\hsame$, $\Cont$, $\Pkg$, and local $\NameCert$ support rows.
\end{theorem}
\begin{proof}
Project the $\StreamMapUp$ carrier. The source classifier row $S$ exposes the constant point stream only through a displayed finite window and its point cells.

Use the constant point row together with the finite-window row $W$. Since every source cell in the inspected window is the certified constant point, the point-map row $F$ produces a target row with the same displayed window shape before $Q,D,$ or $R$ is available.

Componentwise transport is then read through $H$. The target cells, output observations, regular handoff, dyadic ledger, and Real seal boundary are moved only along the displayed source and target windows.

Finally the replay and provenance rows $C,P,N$ preserve that route for downstream consumers. A $\RegSeqRatUp$ consumer therefore receives the mapped constant window only after the finite StreamMap support has been displayed, and cannot replace it by quotient stream equality, host function extensionality, selected limits, countable choice, or ambient $\RealUp$ completeness.
\end{proof}

\begin{theorem}[Stream map composition-window route]
\label{thm:stream-map-composition-window-route}
Let two accepted $\StreamMapUp$ carriers be composable by identifying the first target window with the second source window. The composed mapped stream can be exposed only through the first source $\StreamNameUp$ row, the first point-map row, the intermediate target/source window, the second point-map row, and the final target window before any $\RegSeqRatUp$, $\DyadicRatCoreUp$, or $\RealUp$ row is consumed.
\end{theorem}
\begin{proof}
Project the two accepted carriers. The first carrier exposes its mapped finite-window read through its source row, point-map row, target row, and output row before any regular rational, dyadic, or Real-facing row is available.

Apply \autoref{thm:stream-map-finite-window-functoriality} to the first carrier. The intermediate target window is therefore a displayed $\StreamNameUp$ window, not a selected stream representative or a host-function value.

Apply the same theorem to the second carrier with that intermediate window serving as its source window. The second point-map row consumes only the displayed cells of the intermediate window and exports only the final displayed target window.

Thus the composed route is the concatenation of the two finite-window routes. It reaches the regular rational handoff, dyadic ledger, or Real seal only after the first source window, first point-map row, intermediate window, second point-map row, and final target window have all been displayed.
\end{proof}

\begin{theorem}[Stream map pointwise composition]
\label{thm:stream-map-pointwise-composition}
Let $M_1$ and $M_2$ be accepted $\StreamMapUp$ carriers whose first target $\StreamNameUp$ window is the second source $\StreamNameUp$ window. Every $\RealUp$-facing consumer of the pointwise composite must read the first source window, the first point-map row, the intermediate target window, the second point-map row, the final target window, the shared $\DyadicRatCoreUp$ tolerance ledger, and the $\RegSeqRatUp$ handoff before any terminal Real seal is available.
\end{theorem}
\begin{proof}
Apply \autoref{thm:stream-map-finite-window-functoriality} to $M_1$. The first source window is opened through the first source $\StreamNameUp$ row, the first point-map row consumes only those displayed cells, and the intermediate target window is exposed as a finite $\StreamNameUp$ row.

Apply \autoref{thm:stream-map-finite-window-functoriality} again to $M_2$. The second point-map row reads the same intermediate window as its source and exposes only the final displayed target window.

Now use \autoref{thm:stream-map-composition-window-route}. It concatenates the two finite-window routes, so the composite can reach a downstream consumer only after both point-map rows and both source/target window boundaries are visible.

Finally apply \autoref{thm:stream-map-tail-ledger-nonescape}. The dyadic tolerance ledger and $\RegSeqRatUp$ handoff remain downstream of the displayed windows, and the terminal Real seal cannot import host function extensionality, quotient streams, selected limits, countable choice, or ambient Real completeness.
\end{proof}

\begin{theorem}[Stream map tail-window determinacy]
\label{thm:stream-map-tail-window-determinacy}
For an accepted $\StreamMapUp$ carrier $M=(S,T,F,W,Q,D,R,H,C,P,N)$, any $\RegSeqRatUp$ or $\RealUp$ consumer of the mapped stream must first read the same source $\StreamNameUp$ row $S$, point-map row $F$, target $\StreamNameUp$ row $T$, output-window row $W$, and $\DyadicRatCoreUp$ tolerance ledger $D$ fixed by the finite-window route. If two consumer reads agree on those rows, then they agree on the exposed tail-window handoff $Q$ before either terminal Real seal $R$ is read.
\end{theorem}
\begin{proof}
Project the carrier of \autoref{def:stream-map-carrier}. The rows $S,F,T,W$ are the finite map route, while $Q$ and $D$ are downstream rows attached to that same displayed route.

Apply \autoref{thm:stream-map-finite-window-functoriality}. Every consumer read of the mapped stream factors through $S,F,T,W$ before it can inspect $Q,D,$ or $R$, so two reads agreeing on the source, point-map, target, and output-window rows have the same finite route.

Now read the tolerance and tail rows. The ledger $D$ records the dyadic budgets for the same finite output observations, and the handoff row $Q$ is the regular rational tail read attached to those observations.

Therefore agreement on $S,F,T,W,D$ fixes the classifier-facing handoff $Q$ before the terminal Real seal is available. Since no quotient-stream coordinate or selected-limit coordinate appears in the carrier, there is no second tail-window handoff for the consumer to select.
\end{proof}

\begin{theorem}[Stream map tail-ledger nonescape]
\label{thm:stream-map-tail-ledger-nonescape}
For an accepted $\StreamMapUp$ carrier $M=(S,T,F,W,Q,D,R,H,C,P,N)$, any consumer that reads the regular rational handoff $Q$, the dyadic tolerance ledger $D$, or the terminal $\RealUp$ seal $R$ must first read the same finite route $S,F,T,W$. The structural rows $H,C,P,N$ preserve only that displayed route and cannot introduce quotient stream equality, host function extensionality, selected limits, countable choice, or ambient $\RealUp$ completeness.
\end{theorem}
\begin{proof}
Apply \autoref{thm:stream-map-finite-window-functoriality}. Every finite-window consumer reaches $S,F,T,W$ before the downstream rows $Q,D,$ or $R$ are available, so a tail-facing read cannot bypass the displayed source window, point-map row, target window, or output-window row.

Apply \autoref{thm:stream-map-tail-window-determinacy}. Once two tail-facing reads agree on $S,F,T,W,D$, the exposed regular rational handoff $Q$ is fixed before the terminal Real seal is read.

It remains to inspect the structural rows. The rows $H,C,P,N$ transport, replay, cite provenance, and name the same finite route; none stores a quotient-stream comparison, host-function extensionality proof, selected-limit witness, countable-choice selector, or ambient Real-completeness principle. Thus every tail-ledger consumer remains inside the displayed StreamMap ledger.
\end{proof}

\begin{theorem}[Stream map NameCert obligation surface]

$$

the local $\NameCert_{\StreamMapUp}$ surface consists exactly of the following finite obligation rows:

\item carrier admission for source and target StreamName classifier rows, the point-map row, output observations, RegSeqRat handoff, DyadicRatCore ledger, Real seal boundary, transport, replay, provenance, and local naming;
\item source-window exactness, requiring every mapped request to project $S$ before $F,T,W,Q,D,$ or $R$ can be read;
\item pointwise stability, requiring $F$ to consume only displayed source cells and to export only displayed target cells;

\item ledger non-escape, requiring $D,R,H,C,P,N$ to export no quotient stream equality, host function extensionality, selected limit, countable choice, ambient Real completeness, or detached host map.

\begin{proof}

The tail-window handoff is the projection to $Q$. Since $Q$ is stored after $S,T,F,W$, it records the regular rational readback for the displayed windows and has no coordinate for choosing another stream representative.

Finally, ledger non-escape is exhausted by $D,R,H,C,P,N$. These rows record tolerances, seal boundaries, componentwise transport, displayed replay, provenance, and local naming for the same finite route, and none stores quotient stream equality, host function extensionality, selected limits, countable choice, ambient Real completeness, or a detached host map.
