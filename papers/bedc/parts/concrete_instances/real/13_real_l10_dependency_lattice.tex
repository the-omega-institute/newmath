\begin{definition}[Real L10 dependency lattice]
\label{def:real-l10-dependency-lattice}
The $\RealUp$ L10 dependency lattice is the ordered finite certificate route
whose source rows are read in this order:
\begin{enumerate}
\item the $\DyadicRatCoreUp$ observation rows supplied by
  \autoref{ch:concrete-instances-dyadicratcore-namecert};
\item the $\StreamNameUp$ finite stream-window rows supplied by
  \autoref{ch:concrete-instances-streamname-namecert};
\item the $\RegSeqRatUp$ regular-rational classifier rows supplied by
  \autoref{ch:concrete-instances-regseqrat-namecert};
\item the local $\RealUp$ seal, endpoint readback, and provenance rows.
\end{enumerate}
The route is finite: it is carried by the displayed $\mathsf{BHist}$ packet,
transported only by $\hsame$, read only through $\Cont$, and named only by the
local $\NameCert$ row before the $\RealUp$ seal is consumed.
\end{definition}

\begin{theorem}[Real L10 source route non-escape]
\label{thm:real-l10-source-route-nonescape}
Every accepted $\RealUp$ source-route consumer for the L10 dependency lattice
must read the $\DyadicRatCoreUp$ dyadic observation rows, then the
$\StreamNameUp$ finite stream-window rows, then the $\RegSeqRatUp$
regular-rational classifier rows, and only then the local $\RealUp$ seal rows.
The consumer may transport these rows by the displayed $\hsame$ coordinates,
read them through the displayed $\Cont$ route, and package them with the
displayed $\Pkg$ and local $\NameCert$ rows.

No accepted source-route consumer has a coordinate for a quotient stream, host
completion, countable choice, host real equality, selected host limit,
ambient completeness principle, arithmetic package, order package, or
metric-completeness row.
\end{theorem}
\begin{proof}
Start with the carrier and source specification in
\autoref{def:real-carrier} and \autoref{def:real-source-specification}. The
carrier is a sealed $\RegSeqRat$ name over $\DyadicRatCore$ observations, and
the source specification lists exactly the dyadic observation layer, the
stream-name layer, and the regular-sequence certificate layer.

Now apply the concrete seal handoff in
\autoref{thm:realup-regseqrat-streamname-seal-handoff}. It reads those layers
through the sibling chapters
\autoref{ch:concrete-instances-dyadicratcore-namecert},
\autoref{ch:concrete-instances-streamname-namecert}, and
\autoref{ch:concrete-instances-regseqrat-namecert}, then repacks the result as
the local $\RealUp$ seal handoff. This gives the ordered L10 route of
\autoref{def:real-l10-dependency-lattice}.

Finally use the exhaustion and boundary theorems for the Real RegSeqRat
StreamName dependency triple,
\autoref{thm:real-regseq-streamname-seal-route-exhaustion} and
\autoref{thm:real-regseq-streamname-cauchy-completion-interface}. They say
that every accepted consumer is exhausted by the finite rows of the displayed
packet, with only $\hsame$, $\Cont$, $\Pkg$, and $\NameCert$ structural data.
Since none of the cited source, handoff, or dependency-triple blocks exports a
quotient stream, host completion, countable choice, host real equality,
selected host limit, ambient completeness, arithmetic, order, or
metric-completeness coordinate, the consumer has no route to escape the L10
lattice.
\end{proof}

\begin{theorem}[Real L10 interface totality]
\label{thm:real-l10-dependency-lattice-interface-totality}
Every current Phase Real and Completion interface read of an accepted $\RealUp$
L10 dependency-lattice packet factors through exactly the four displayed
faces of \autoref{def:real-l10-dependency-lattice}: $\DyadicRatCoreUp$
observations, $\StreamNameUp$ finite-window rows, $\RegSeqRatUp$
regular-rational rows, and the local $\RealUp$ seal and readback rows. The
transport allowed between these faces is only the displayed $\hsame$ data,
the displayed $\Cont$ routes, the displayed $\Pkg$ provenance, and the local
$\NameCert$ data; no further quotient, completion, host-real, selector,
choice, arithmetic, order, or metric-completeness interface row is exported.
\end{theorem}
\begin{proof}
Start with the ordered source route of
\autoref{thm:real-l10-source-route-nonescape}. It confines accepted L10
consumers to the dyadic observation layer, the StreamName finite-window
layer, the RegSeqRat regular-rational layer, and the local Real seal layer,
with transport only through the displayed structural rows.

Next use the open-phase consumer theorem
\autoref{thm:realup-open-phase-four-face-consumer-totality}. It identifies
the current Phase Real and Completion reads with the same four faces, so an
interface consumer cannot insert a fifth source row between the dependency
lattice and the public open-phase surface.

The two sibling handoff theorems give the middle faces. By
\autoref{thm:streamname-finite-window-real-handoff-coverage}, finite
StreamName windows reach the Real handoff only through their displayed
window, RegSeqRat, Real-boundary, transport, continuation, provenance, and
naming rows. By \autoref{thm:regseqrat-real-seal-handoff}, the
regular-rational face supplies exactly the seal-facing rows consumed by
$\RealUp$.

Finally apply the current-surface theorem for the dependency triple,
\autoref{thm:real-open-phase-dependency-triple-current-surface}. It
repackages the same DyadicRatCore, StreamName, RegSeqRat, and local Real rows
as the current interface surface. Since every cited route exports only
$\hsame$, $\Cont$, $\Pkg$, and $\NameCert$ data besides those four faces, the
interface read is exhausted by the displayed L10 lattice.
\end{proof}

\begin{theorem}[Real L10 terminal face order]
\label{thm:real-l10-terminal-face-order}
Every accepted $\RealUp$ open-phase terminal read consumes the four faces of
\autoref{def:real-l10-dependency-lattice} in the order
$\DyadicRatCoreUp \to \StreamNameUp \to \RegSeqRatUp \to \RealUp$. The
terminal $\RealUp$ seal cannot be read before the dyadic observation rows, the
finite-window rows, and the regular-rational classifier rows have been
displayed through the same $\BHist$, $\hsame$, $\Cont$, $\Pkg$, and
$\NameCert$ surface.

Consequently a terminal read has no coordinate through which to consume
quotient stream equality, selected limits, host real equality, ambient
completeness, or any seal-facing row outside the displayed four-face route.
\end{theorem}
\begin{proof}
Start from the ordered route in \autoref{def:real-l10-dependency-lattice}.
The lattice lists the dyadic observation face, the StreamName finite-window
face, the RegSeqRat classifier face, and only then the local Real seal face.

Apply the non-escape theorem
\autoref{thm:real-l10-source-route-nonescape}. It prevents any accepted
source-route consumer from bypassing the dyadic, stream, or regular-rational
faces before the local seal is reached.

Next use interface totality,
\autoref{thm:real-l10-dependency-lattice-interface-totality}. Since every
open Phase Real and Completion interface read factors through exactly those
four faces, a terminal read cannot insert a further interface row between the
RegSeqRat classifier and the Real seal.

The middle dependency is exact by
\autoref{thm:streamname-dyadic-regseq-real-pullback-exactness}: the stream
window read is pulled back to the same dyadic and regular-rational source
surface. The final handoff is exact by
\autoref{thm:regseqrat-real-seal-handoff}: the Real seal consumes the
regular-rational source rows already displayed by the finite window.

Coordinate exhaustion then excludes all other terminal readings. The cited
routes expose only $\BHist$, $\hsame$, $\Cont$, $\Pkg$, and $\NameCert$ data
over the four named faces, so quotient stream equality, selected limits, host
real equality, ambient completeness, and outside seal data have no terminal
coordinate to consume.
\end{proof}

\begin{theorem}[Real L10 terminal comparison exhaustion]
\label{thm:real-l10-terminal-comparison-exhaustion}
Let $X$ and $Y$ be accepted terminal $\RealUp$ L10 reads. Every
completion-facing comparison between $X$ and $Y$ factors through the same
four faces
$$
\DyadicRatCoreUp \to \StreamNameUp \to \RegSeqRatUp \to \RealUp
$$
displayed in \autoref{def:real-l10-dependency-lattice}. The comparison may
use only the common $\BHist$ packet, the displayed $\hsame$ transports, the
displayed $\Cont$ routes, the displayed $\Pkg$ provenance, and the local
$\NameCert$ coordinates of those faces.

No completion-facing comparison of $X$ and $Y$ has a coordinate for quotient
stream equality, selected limits, host real equality, ambient completeness,
arithmetic or order packages, or metric-completeness rows.
\end{theorem}
\begin{proof}
Apply the terminal face order theorem,
\autoref{thm:real-l10-terminal-face-order}, to both accepted reads. It places
the dyadic observation face, the finite StreamName window face, the RegSeqRat
classifier face, and the Real seal face in the same terminal order for $X$
and $Y$.

Next apply the four-face route lock,
\autoref{thm:real-l10-four-face-route-lock}. A completion-facing comparison
cannot leave the displayed route once the two terminal reads have been
accepted on that route; its transport and readback data remain confined to
$\BHist$, $\hsame$, $\Cont$, $\Pkg$, and $\NameCert$ coordinates.

The shared finite surface is supplied by
\autoref{thm:streamname-regseqrat-real-dyadic-window-pullback}. It identifies
the StreamName window, RegSeqRat readback, DyadicRatCore ledger, and Real
seal boundary as one pullback surface for the two terminal reads.

Finally use the observation triangle
\autoref{thm:dyadicratcore-streamname-regseqrat-observation-triangle}. The
dyadic, stream, and regular-rational observations meet on the same finite
triangle before the Real seal is consumed. Since the cited routes export no
quotient stream equality, selected limit, host real equality, ambient
completeness, arithmetic or order package, or metric-completeness coordinate,
the comparison is exhausted by the four displayed faces.
\end{proof}

\begin{theorem}[Real L10 four-face canonical route]
\label{thm:real-l10-four-face-canonical-route}
Every accepted current Phase Real or Completion consumer of a $\RealUp$ L10
packet has a canonical finite route through exactly the four faces
$$
\DyadicRatCoreUp \to \StreamNameUp \to \RegSeqRatUp \to \RealUp
$$
of \autoref{def:real-l10-dependency-lattice}. The route is displayed only by
the common $\BHist$ packet, the shown $\hsame$ transports, the shown $\Cont$
routes, the shown $\Pkg$ provenance, and the local $\NameCert$ rows.

If two route presentations have the same four faces and the same displayed
$\BHist$, $\hsame$, $\Cont$, $\Pkg$, and $\NameCert$ rows, then they induce
the same consumer surface. No quotient stream, selected host limit, host real
equality, countable-choice selector, ambient completeness principle,
arithmetic or order package, or metric-completeness row can serve as an
alternate route.
\end{theorem}
\begin{proof}
The four faces are the faces of the dependency lattice in
\autoref{def:real-l10-dependency-lattice}. Apply the non-escape theorem,
\autoref{thm:real-l10-source-route-nonescape}, to exclude any route that
leaves the dyadic, StreamName, RegSeqRat, and local Real seal sequence.

Next apply interface totality,
\autoref{thm:real-l10-dependency-lattice-interface-totality}. It says that
every current Phase Real and Completion interface read factors through
exactly those four faces, using only the displayed $\hsame$, $\Cont$, $\Pkg$,
and $\NameCert$ rows. The terminal order theorem,
\autoref{thm:real-l10-terminal-face-order}, fixes their order as
$\DyadicRatCoreUp \to \StreamNameUp \to \RegSeqRatUp \to \RealUp$.

For comparisons of two accepted presentations, apply
\autoref{thm:real-l10-terminal-comparison-exhaustion}. It confines every
completion-facing comparison to the common $\BHist$, $\hsame$, $\Cont$,
$\Pkg$, and $\NameCert$ data over the same ordered four faces.

The shared finite consumer surface is then the pullback surface of
\autoref{thm:streamname-regseqrat-real-dyadic-window-pullback} together with
the observation triangle of
\autoref{thm:dyadicratcore-streamname-regseqrat-observation-triangle}. These
identify the dyadic observations, finite stream windows, regular-rational
readback, and Real seal boundary before the consumer surface is read. With no
exported coordinate for quotient streams, selected host limits, host real
equality, countable choice, ambient completeness, arithmetic or order
packages, or metric-completeness rows, no alternate route remains.
\end{proof}

\begin{theorem}[Real StreamName RegSeqRat seal lattice]
\label{thm:real-streamname-regseqrat-seal-lattice}
Every completion-facing $\RealUp$ consumer that also carries a $\CauchyLimitSealUp$ row factors through the ordered four faces
$$
\DyadicRatCoreUp \to \StreamNameUp \to \RegSeqRatUp \to \RealUp
$$
of \autoref{def:real-l10-dependency-lattice}. The $\CauchyLimitSealUp$ row is only the compatibility row for the already displayed seal route; it does not add a second completion face, quotient stream equality, selected limit, host real equality, ambient completeness, or metric-completeness row.
\end{theorem}
\begin{proof}
Start with \autoref{def:real-l10-dependency-lattice}. Its source route is the ordered finite lattice of dyadic observation rows, finite StreamName windows, regular-rational classifier rows, and the local Real seal.

Apply \autoref{thm:real-l10-four-face-canonical-route}. Every current Phase Real or Completion consumer already has a canonical route through exactly those four faces, with only the common $\BHist$, $\hsame$, $\Cont$, $\Pkg$, and $\NameCert$ rows as structural data.

For completion-facing comparisons, use \autoref{thm:real-l10-terminal-comparison-exhaustion}. It keeps both terminal reads on the same four-face route. The shared finite surface is the pullback of \autoref{thm:streamname-regseqrat-real-dyadic-window-pullback}, so the StreamName window, RegSeqRat readback, DyadicRatCore ledger, and Real seal boundary remain one displayed route.

The extra $\CauchyLimitSealUp$ row is therefore read as the compatibility surface supplied by \autoref{ch:concrete-instances-cauchylimitseal-namecert}. Since the canonical route and terminal comparison already exhaust the consumer surface, the seal row cannot add quotient stream equality, selected limits, host real equality, ambient completeness, or metric-completeness data.
\end{proof}
