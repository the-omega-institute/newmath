\begin{theorem}[Prod real-tail window forward link]
\label{thm:prod-real-tail-window-forward-link}
Any $\RealUniformTailComparisonUp$ packet using two $\StreamNameUp$ windows reads its pair of finite windows through the $\ProdUp$ componentwise interface before the $\RegSeqRatUp$ readback rows and $\DyadicRatCoreUp$ tolerance ledger are consumed. The two windows are therefore a displayed product route: first the product history carrier, componentwise classifier, displayed-pair ledger, source $\NameCert$ rows, $\hsame$ transport, $\Cont$ replay, and $\Pkg$ provenance of \autoref{thm:prod-public-finite-data-exit-package}; then the finite-window rows of \autoref{ch:concrete-instances-real-uniform-tail-comparison-namecert}; and only after that the regular rational readbacks and shared dyadic tolerance ledger.
\end{theorem}
\begin{proof}
Start with the product componentwise surface. The componentwise interface and public finite-data exit package expose the left and right component histories, the displayed pair ledger, source certificates, transport, replay, and provenance before a consumer receives a paired read.

Use the real-completion product-window route, \autoref{thm:prod-finite-data-real-completion-sibling-route}. It already places StreamName and RegSeqRat product-window reads downstream of the displayed $\ProdUp$ finite-data route rather than behind a host tuple or selected projection.

Now read the $\RealUniformTailComparisonUp$ carrier. Its two windows $W_A,W_B$ are consumed before the regular rational rows $Q_A,Q_B$ and before the shared $\DyadicRatCoreUp$ ledger. Therefore a comparison packet that uses two finite windows must first pass through the product component rows that make those windows a displayed pair.

The downstream handoff is exactly the finite route through $\ProdUp$, $\StreamNameUp$, $\RegSeqRatUp$, and $\DyadicRatCoreUp$. It exports no host tuple equality, quotient stream equality, selected projection, hidden product stream, or ambient real-completeness row.
\end{proof}
