\begin{theorem}[Prod Bool endpoint-discriminator route]
\label{thm:prod-bool-endpoint-discriminator-route}
Every public $\ProdUp$ component-route read uses the two endpoint rows licensed
by \autoref{ch:concrete-instances-bool-namecert} only as a finite discriminator
for component position. The left and right positions are tags for routing
through the displayed-pair ledger; the payload carriers, component classifiers,
source $\NameCert$ rows, $\hsame$ transports, $\Cont$ replays, and $\Pkg$
provenance remain owned by $\ProdUp$ and by the source certificates named in
\autoref{thm:prod-public-finite-data-exit-package}.

Consequently a public product consumer cannot replace the componentwise
interface by an untagged finite-data quotient or by a host ordered-pair
primitive.
\end{theorem}
\begin{proof}
Start with the componentwise interface. The displayed product route reads
components through the product history classifier and the displayed-pair ledger,
as in \autoref{thm:prod-component-history-classifier-displayed-ledger-endpoints}
and \autoref{thm:prod-component-history-classifier-descends-to-envelope}. These
rows identify which component position is being read, but the component payload
still comes from the source carriers and source classifiers.

Next apply the public finite-data exit package of
\autoref{thm:prod-public-finite-data-exit-package}. It lists exactly the rows
available to a public product consumer: the componentwise semantic certificate,
displayed-pair ledger endpoints, descent to the product history envelope, source
certificate rows, $\hsame$, $\Cont$, and $\Pkg$ provenance.

The two $\BoolUp$ endpoint rows therefore serve only as the finite discriminator
for left versus right component position. Since all payload, classifier,
transport, replay, and provenance rows remain the product rows displayed above,
there is no untagged finite-data quotient and no host ordered-pair primitive
available to the consumer.
\end{proof}
