\begin{theorem}[Prod option-sum component sibling route]
\label{thm:prod-option-sum-component-sibling-route}
Every accepted $\ProdUp$ component read factors through the finite-data sibling lattice containing \autoref{ch:concrete-instances-option-namecert} and \autoref{ch:concrete-instances-sum-namecert} before a downstream component consumer can treat the pair as a case-split or optional projection. The route reads the product carrier through the displayed component rows, then passes through the $\OptionUp$ possible-absence surface and the $\SumUp$ branch surface, and only then exposes the component to a finite-data consumer.
\end{theorem}
\begin{proof}
Start with the product public finite-data exit package of \autoref{thm:prod-public-finite-data-exit-package}. It exposes the componentwise semantic certificate, displayed-pair ledger endpoints, descent to the product history envelope, source $\NameCert$ rows, $\hsame$, $\Cont$, and $\Pkg$ provenance.

Next apply the finite-data sibling lattice of \autoref{thm:prod-finite-data-sibling-lattice}. That lattice requires the component read to pass through the finite-data siblings before the ordered pair is exported.

The relevant sibling route has two finite-data faces. The $\OptionUp$ chapter supplies the possible-absence projection discipline, while the $\SumUp$ chapter supplies the branch partition used when the downstream consumer treats a component as a case split.

Since the public product package has no host tuple equality and no outside-ledger projection, a component consumer receives only the product rows already routed through those two siblings. Thus every accepted component read factors through the option and sum surfaces before it can be used as an optional projection or case-split input.
\end{proof}
