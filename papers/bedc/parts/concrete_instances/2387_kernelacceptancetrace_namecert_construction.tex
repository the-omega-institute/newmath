\chapter{A Concrete Naming Certificate for $\KernelAcceptanceTraceUp$}
\label{ch:concrete-instances-kernelacceptancetrace-namecert}
\origin{human}

The $\KernelAcceptanceTraceUp$ surface records kernel acceptance as a finite trace packet: a submitted candidate, the type and classifier under which it is checked, the accepted stamp, the environment row, the displayed axiom ledger, transport, continuation, provenance, and local naming row. It is an audit witness for what the kernel admitted and which dependency rows the admission exposes, not a proof of kernel correctness.

The candidate row and accepted stamp are read as an inscription-event record surface in the sense of \autoref{ch:concrete-instances-inscriptionevent-namecert}: the submitted candidate, checked type, classifier, stamp, and environment give the event record that the trace consumes. The axiom ledger remains $\KernelAcceptanceTraceUp$'s own displayed row, so dependency exposure is still governed by the trace packet rather than by the inscription-event chapter.

\begin{definition}[KernelAcceptanceTrace carrier]
\label{def:kernel-acceptance-trace-carrier}
A $\KernelAcceptanceTraceUp$ carrier is a finite $\BHist$ packet
$$
K=(A,T,C,S,E,L,H,R,P,N)
$$
with the following displayed coordinates:
\begin{enumerate}
\item $A$ is the candidate row naming the term, declaration, inductive clause, or recursor instance submitted to the kernel;
\item $T$ is the type row against which the candidate is checked;
\item $C$ is the kernel classifier row recording the admitted declaration class and the local acceptance predicate for the candidate at type $T$;
\item $S$ is the kernel stamp row that records the accepted name, module position, and replay boundary for the acceptance event;
\item $E$ is the environment row in which the stamp is valid;
\item $L$ is the displayed axiom ledger for the accepted trace, including the dependency rows exposed by the strict audit query;
\item $H$ stores componentwise $\hsame$ transports for the candidate, type, classifier, stamp, environment row, axiom ledger, package provenance, and local naming row;
\item $R$ stores the displayed $\Cont$ routes through which consumers read the candidate, transport the classifier, inspect the stamp, query the environment ledger, and replay the axiom ledger;
\item $P$ is the $\Pkg$ provenance over $\BHist$, candidate rows, type rows, kernel classifiers, kernel stamps, environment ledgers, axiom ledgers, $\hsame$, $\Cont$, and $\NameCert$;
\item $N$ is the local $\NameCert_{\KernelAcceptanceTraceUp}$ row.
\end{enumerate}
The carrier has no hidden coordinate for ambient kernel soundness, host runtime trust, compiler trust, unqueried dependency ancestry, or a stronger paper-side theorem than the accepted declaration states.
\end{definition}

\begin{theorem}[KernelAcceptanceTrace NameCert obligations]
\label{thm:kernel-acceptance-trace-namecert-obligations}
For every accepted $\KernelAcceptanceTraceUp$ carrier
$$
K=(A,T,C,S,E,L,H,R,P,N),
$$
the local $\NameCert_{\KernelAcceptanceTraceUp}$ surface consists of five finite obligations:
\begin{enumerate}
\item carrier exposure for the displayed packet $K$ and its candidate, type, classifier, stamp, environment, axiom ledger, transport, continuation, provenance, and naming rows;
\item classifier transport, requiring every accepted comparison of traces to move $A,T,C,S,E,L,P,N$ only by the componentwise $\hsame$ rows in $H$ and the displayed $\Cont$ routes in $R$;
\item kernel-stamp exposure, requiring consumers to read acceptance through the accepted name, module position, and replay boundary in $S$;
\item environment-ledger exposure, requiring the accepted trace to route through the environment row $E$ and the displayed axiom ledger $L$ before any dependency claim is consumed;
\item non-escape, excluding hidden kernel soundness, host runtime trust, compiler trust, unqueried dependency ancestry, and theorem-content strengthening outside the accepted declaration.
\end{enumerate}
\end{theorem}
\leanchecked{BEDC.Derived.KernelAcceptanceTraceUp.KernelAcceptanceTraceCarrier\_namecert\_obligations}
\begin{proof}
Carrier exposure is projection from \autoref{def:kernel-acceptance-trace-carrier}. The packet lists the candidate, type, classifier, stamp, environment, axiom ledger, transport, continuation, provenance, and local naming rows as finite coordinates.

Classifier transport is componentwise. The $\hsame$ rows in $H$ preserve the displayed coordinates, and the $\Cont$ rows in $R$ give only the routes named in the packet, so the accepted trace cannot be compared through an unrecorded classifier.

The stamp and environment obligations then force every acceptance read through $S$ and $E$. The axiom ledger $L$ is the only displayed ancestry row for dependency classification.

Finally inspect the carrier inventory. Since no coordinate names kernel soundness, host runtime trust, compiler trust, unqueried ancestry, or theorem strengthening beyond the accepted declaration, no local $\NameCert_{\KernelAcceptanceTraceUp}$ consumer can export such data.
\end{proof}

\begin{theorem}[KernelAcceptanceTrace axiom ledger non-escape]
\label{thm:kernel-acceptance-trace-axiom-ledger-nonescape}
For every accepted $\KernelAcceptanceTraceUp$ carrier
$$
K=(A,T,C,S,E,L,H,R,P,N),
$$
the exported dependency surface is exactly the displayed axiom ledger $L$ together with the stamp, environment, transport, continuation, provenance, and local naming rows $S,E,H,R,P,N$. No consumer may obtain a hidden $\mathsf{Classical}$ source, $\mathsf{Quot}$ source, $\mathsf{propext}$ source, or any other unlisted dependency by reading the candidate row, type row, classifier row, or acceptance stamp alone.
\end{theorem}
\begin{proof}
Read the dependency side of the carrier through $L$. The ledger is a named coordinate of the finite packet and is the only row that classifies transitive dependency ancestry.

The stamp $S$ and environment row $E$ identify where the accepted declaration lives, while $H,R,P,N$ preserve and package the route by transport, continuation, provenance, and local naming. These rows can expose the displayed ledger but do not add a second ancestry channel.

Therefore a hidden $\mathsf{Classical}$, $\mathsf{Quot}$, $\mathsf{propext}$, or other unlisted source would require an extra coordinate outside $K$. The local trace certificate has no such coordinate, so the accepted trace exports only the displayed axiom ledger.
\end{proof}

\begin{theorem}[KernelAcceptanceTrace classifier transport]
\label{thm:kernel-acceptance-trace-classifier-transport}
For every accepted $\KernelAcceptanceTraceUp$ carrier
$$
K=(A,T,C,S,E,L,H,R,P,N),
$$
accepted trace comparison transports only the candidate row $A$, type row $T$, classifier row $C$, stamp row $S$, environment row $E$, axiom ledger $L$, provenance row $P$, and local name $N$ through the displayed $\hsame$ rows in $H$ and the displayed $\Cont$ routes in $R$.
\end{theorem}
\begin{proof}
Project the carrier coordinates and the local NameCert obligations. The comparison-facing data are exactly $A,T,C,S,E,L,P,N$.

The transport row $H$ is componentwise, and the route row $R$ is displayed in the packet. Thus accepted trace comparison can move the listed rows, but it has no separate classifier oracle, host runtime row, compiler row, or hidden dependency source.
\end{proof}

\begin{theorem}[KernelAcceptanceTrace stamp-environment exposure]
\label{thm:kernel-acceptance-trace-stamp-environment-exposure}
Every acceptance read from an accepted $\KernelAcceptanceTraceUp$ carrier factors through the stamp row $S$ and the environment row $E$ before the axiom ledger $L$ can be consumed. The ledger cannot be read as an ambient ancestry claim detached from the accepted name, module position, replay boundary, and environment ledger.
\end{theorem}
\begin{proof}
Begin with the carrier. The stamp row $S$ records the accepted name, module position, and replay boundary, while $E$ records the environment in which that stamp is valid.

Apply the axiom-ledger non-escape theorem. Dependency ancestry is exposed by $L$, but it is exported together with $S,E,H,R,P,N$ rather than through the candidate, type, or classifier rows alone.

Therefore any acceptance read must first locate the stamp and environment. Only then can it consume the displayed ledger as the dependency surface of that accepted trace.
\end{proof}

\begin{theorem}[KernelAcceptanceTrace accepted-declaration boundary]
\label{thm:kernel-acceptance-trace-accepted-declaration-boundary}
The accepted declaration boundary of a $\KernelAcceptanceTraceUp$ carrier cannot strengthen theorem content beyond the declared type row $T$ and the displayed dependency ledger $L$. It exports the submitted candidate, accepted type, classifier, stamp, environment, ledger, transport, continuation, provenance, and local name, but no stronger theorem, kernel-correctness claim, or host-runtime trust row.
\end{theorem}
\begin{proof}
Read the declaration through $A,T,C,S,E,L,H,R,P,N$. The type row $T$ is the only displayed statement boundary for the accepted candidate, and $L$ is the only displayed dependency ancestry.

Use classifier transport to keep comparisons inside the same rows. Transport may preserve the declared type and ledger, but it does not introduce a stronger theorem statement.

Finally apply stamp-environment exposure and axiom-ledger non-escape. Since the accepted trace must pass through $S,E$ and has no hidden ancestry coordinate, the boundary cannot add kernel correctness, host runtime trust, or theorem content beyond $T$ and $L$.
\end{proof}
