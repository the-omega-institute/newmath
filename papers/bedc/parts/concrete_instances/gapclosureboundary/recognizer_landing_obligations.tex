\begin{theorem}[GapClosureBoundary recognizer source admission]
\label{thm:gap-closure-boundary-recognizer-source-admission}
For every accepted $\GapClosureBoundaryUp$ carrier $B=(G,S,R,H,C,P,N)$, any recognizer-side source admission for a hard-problem sibling consumer must enter through the $\GroundCompilerRecognizerFlowUp$ acceptance route and land in the $\FiniteReflectionTupleUp$ tuple image before the boundary packet is consumed. The admitted read exposes the displayed request row $G$, the source row $S$, and the structural rows $H,C,P,N$, while the downstream consumers $\CausalCommitmentUp$, $\FiniteWitnessedRefutationUp$, $\OtherMindsCommitmentUp$, $\AspectChainUp$, and $\SolipsismGapRefusalUp$ receive no source coordinate outside those rows.
\end{theorem}
\begin{proof}
Start with the carrier of \autoref{def:gap-closure-boundary-carrier}. Its recognizer-facing boundary is already the seven-row packet $G,S,R,H,C,P,N$.

Use the $\GroundCompilerRecognizerFlowUp$ gate through \autoref{thm:groundcompiler-recognizer-flow-certificate-gate-exactness}. An accepted source read must pass through the event-flow, source-channel, generated-recognizer, and certificate-gate rows before it can be consumed as compiler-layer evidence.

Next apply \autoref{thm:finite-reflection-tuple-fieldfaithful-prediction}. The admitted recognizer row lands in the displayed $\FiniteReflectionTupleUp$ image and is read back from that tuple, so the source branch available to the boundary is the displayed route $G \longrightarrow S$ over $H,C,P,N$.

Finally use root source exactness, \autoref{thm:gap-closure-boundary-root-source-exactness}. The listed hard-problem consumers may consume the admitted source branch only through the displayed boundary rows, so no additional source coordinate is exported.
\end{proof}

\begin{theorem}[GapClosureBoundary recognizer refusal stability]
\label{thm:gap-closure-boundary-recognizer-refusal-stability}
For every accepted $\GapClosureBoundaryUp$ carrier $B=(G,S,R,H,C,P,N)$, if a recognizer-side request does not land in the displayed $\FiniteReflectionTupleUp$ tuple image after $\GroundCompilerRecognizerFlowUp$ acceptance, then the boundary routes the request to the refusal ledger $R$. The refusal route is stable under the structural rows $H,C,P,N$ and cannot become internal closure, hidden $\mathsf{T}$ supply, private access, causal necessity, direct aspect reduction, or an off-packet continuation for any listed sibling consumer.
\end{theorem}
\begin{proof}
Begin with the non-evidence refusal theorem for $\GroundCompilerRecognizerFlowUp$, \autoref{thm:groundcompiler-recognizer-flow-non-evidence-refusal}. It keeps report-like source, host compiler state, proof text, and raw classifier quotients outside accepted evidence when the generated recognizer route does not pass the gate.

The tuple boundary of \autoref{thm:finite-reflection-tuple-fieldfaithful-prediction} then prevents a failed landing from being reread as a reflected boundary tuple. Thus the request has no displayed route through $G \longrightarrow S$.

Apply refusal route exactness and refusal ledger stability, \autoref{thm:gap-closure-boundary-refusal-route-exactness} and \autoref{thm:gap-closure-boundary-refusal-ledger-stability}. The request is routed to $R$ through $H,C,P,N$ and the same rows preserve that refusal endpoint for all listed sibling consumers.
\end{proof}

\begin{theorem}[GapClosureBoundary recognizer consumer exhaustion]
\label{thm:gap-closure-boundary-recognizer-consumer-exhaustion}
For the downstream consumers $\CausalCommitmentUp$, $\FiniteWitnessedRefutationUp$, $\OtherMindsCommitmentUp$, $\AspectChainUp$, and $\SolipsismGapRefusalUp$, the combined $\GroundCompilerRecognizerFlowUp$ and $\FiniteReflectionTupleUp$ landing route exhausts the recognizer-facing $\GapClosureBoundaryUp$ consumer surface. A successful recognizer landing is consumed as $G \longrightarrow S$ with $H,C,P,N$; a failed or unsupported landing is consumed as $G \longrightarrow R$ with $H,C,P,N$; no third recognizer, tuple, or sibling-specific route is exported.
\end{theorem}
\begin{proof}
First account for successful recognizer landings. By \autoref{thm:gap-closure-boundary-recognizer-source-admission}, accepted recognizer evidence reaches the boundary only after the $\GroundCompilerRecognizerFlowUp$ gate and the $\FiniteReflectionTupleUp$ tuple readback, and it is then consumed through the displayed source branch.

Next account for failed or unsupported landings. By \autoref{thm:gap-closure-boundary-recognizer-refusal-stability}, these requests are routed to the refusal ledger $R$ over the same structural rows.

It remains to rule out another consumer path. Use hard-problem route stability and consumer ledger exhaustion, \autoref{thm:gap-closure-boundary-hard-problem-root-route-stability} and \autoref{thm:gap-closure-boundary-consumer-ledger-exhaustion}. They keep all five listed consumers inside the same source/refusal split, so the recognizer-facing route is exhausted by the two displayed branches.
\end{proof}
