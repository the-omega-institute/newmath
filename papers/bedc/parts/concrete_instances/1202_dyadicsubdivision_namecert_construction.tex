\chapter{A Concrete Naming Certificate for $\DyadicSubdivisionUp$}
\label{ch:concrete-instances-dyadicsubdivision-namecert}
\origin{human}

The $\DyadicSubdivisionUp$ interface names a finite subdivision certificate for one rational interval into dyadic mesh cells with validated enclosure rows attached. In the Moore--Kearfott--Cloud interval-analysis reading, subdivision refines an enclosure into finitely many checked sub-enclosures; in BEDC the object is only a finite $\mathsf{BHist}$ ledger over $\DyadicRatCoreUp$, $\RationalIntervalUp$, $\DyadicMeshUp$, $\ValidatedNumericsUp$, $\RealUp$, $\hsame$, $\Cont$, $\Pkg$, and $\NameCert$.

\begin{definition}[DyadicSubdivision source]
\label{def:dyadic-subdivision-source}
A $\DyadicSubdivisionUp$ source is a finite $\mathsf{BHist}$ packet
$$
S=(I,L,C,M,V,P,N)
$$
with the following visible rows:
\begin{enumerate}
\item $I$ is the parent $\RationalIntervalUp$ row whose endpoint and order data are the interval being subdivided;
\item $L$ is the unary dyadic level row used to read every child cell at one displayed mesh precision;
\item $C$ is a finite ordered cell-list row, with each entry carrying its displayed position inside the parent interval;
\item $M$ is the finite family of $\DyadicMeshUp$ cell packets indexed by $C$;
\item $V$ is the finite family of $\ValidatedNumericsUp$ containment rows attached to the same cells;
\item $P$ is the $\Pkg$ provenance row naming $\DyadicRatCoreUp$, $\RationalIntervalUp$, $\DyadicMeshUp$, $\ValidatedNumericsUp$, and $\RealUp$;
\item $N$ is the local $\NameCert_{\DyadicSubdivisionUp}$ row.
\end{enumerate}
The classifier compares accepted subdivision packets by transporting the parent interval, level row, ordered cell list, mesh cells, containment rows, package provenance, and local name row componentwise. The carrier has no coordinate for a host interval tree, infinite refinement process, solver state, or completed real subset.
\end{definition}

\begin{theorem}[DyadicSubdivision NameCert obligations]
\label{thm:dyadic-subdivision-namecert-obligations}
The seed $\NameCert_{\DyadicSubdivisionUp}$ obligation surface has five rows:
\begin{enumerate}
\item carrier: the finite packet of \autoref{def:dyadic-subdivision-source};
\item classifier: componentwise $\hsame$ transport of the parent interval, dyadic level, ordered cell list, $\DyadicMeshUp$ packets, $\ValidatedNumericsUp$ rows, provenance, and local name;
\item stability: appending displayed refinement routes to the mesh family preserves the transported parent interval and refines only the finite cell ledger;
\item exactness: every child boundary and containment read is one of the $\DyadicMeshUp$ or $\ValidatedNumericsUp$ rows indexed by the ordered list $C$;
\item ledger: the package and local certificate rows record exactly the dependency surface over $\DyadicRatCoreUp$, $\RationalIntervalUp$, $\DyadicMeshUp$, $\ValidatedNumericsUp$, and $\RealUp$.
\end{enumerate}
These rows are the full seed naming surface for finite dyadic subdivision certificates.
\end{theorem}
\leanchecked{BEDC.Derived.DyadicSubdivisionUp.DyadicSubdivisionSource\_namecert\_obligations}
\begin{proof}
Project the packet of \autoref{def:dyadic-subdivision-source}. The carrier row displays the parent interval, common level, finite ordered list, mesh-cell family, validated containment family, package provenance, and local certificate row.

The classifier is componentwise because the packet contains no hidden geometric object. Transporting a subdivision means transporting exactly those visible rows by $\hsame$ and replaying the stored $\Cont$ routes for the mesh and validated containment reads.

Stability and exactness are then read from the same finite ledger. Refinement appends routes to the mesh-family reads, while every boundary or enclosure claim remains indexed by $C$ and therefore belongs to one of the displayed $\DyadicMeshUp$ or $\ValidatedNumericsUp$ rows.
\end{proof}

\begin{theorem}[DyadicSubdivision refinement stability]
\label{thm:dyadic-subdivision-refinement-stability}
Let $S=(I,L,C,M,V,P,N)$ be an accepted $\DyadicSubdivisionUp$ packet. If $S'$ is obtained by replacing each mesh entry in $M$ by a finite displayed $\DyadicMeshUp$ refinement whose rational-interval boundary transports to the same parent row $I$, and by transporting the corresponding containment entries in $V$ along the same routes, then $S'$ is an accepted $\DyadicSubdivisionUp$ packet for the same parent interval.
\end{theorem}
\leanchecked{BEDC.Derived.DyadicSubdivisionUp.DyadicSubdivisionPacket\_refinement\_stability}
\begin{proof}
Start with the finite subdivision packet. The parent interval row $I$, package provenance $P$, and local name row $N$ are retained, so the outer naming surface is unchanged.

For each entry of the ordered list $C$, use the $\DyadicMeshUp$ refinement route to replace the displayed cell by a finer finite family. The endpoint comparison is still through the transported $\RationalIntervalUp$ boundary rows, so no host interval or completed real subset is introduced.

Transport the validated containment rows through the same cell routes. Repacking the parent row, refined level data, refined ordered cell ledger, refined mesh entries, transported containment entries, provenance, and local certificate gives exactly the carrier of \autoref{def:dyadic-subdivision-source}.
\end{proof}

\begin{theorem}[DyadicSubdivision mesh coverage]
\label{thm:dyadic-subdivision-mesh-coverage}
For every accepted $\DyadicSubdivisionUp$ packet $S=(I,L,C,M,V,P,N)$, coverage of the parent interval is represented solely by the finite ordered cell list $C$ together with the $\DyadicMeshUp$ packets in $M$. A consumer may read each child rational interval, dyadic endpoint ledger, and adjacency order through those rows, but it cannot read an infinite mesh, a host subset of $\RealUp$, or a global interval-tree object from $S$.
\end{theorem}
\begin{proof}
Read the parent interval row $I$ and the ordered list $C$. The list is the only row that arranges the children, and it is finite by construction of the packet.

Each child coverage object is then projected from the matching $\DyadicMeshUp$ entry in $M$. Its rational interval and dyadic endpoint ledger are already the finite coverage surface supplied by the mesh certificate.

Since the subdivision carrier contains no further coverage coordinate, the parent coverage surface is exhausted by $C$ and $M$. Infinite meshes, host subsets, and interval-tree data are outside the packet.
\end{proof}
\leanchecked{BEDC.Derived.DyadicSubdivisionUp.DyadicSubdivisionSource\_mesh\_coverage}

\begin{theorem}[DyadicSubdivision validated enclosure handoff]
\label{thm:dyadic-subdivision-validated-enclosure-handoff}
A $\ValidatedNumericsUp$ consumer of an accepted $\DyadicSubdivisionUp$ packet may read exactly the parent $\RationalIntervalUp$ row, common dyadic level row, ordered finite cell ledger, child $\DyadicMeshUp$ packets, attached $\ValidatedNumericsUp$ containment rows, $\hsame$ transports, $\Cont$ routes, package provenance, and local $\NameCert$ row. The handoff supplies a finite enclosure subdivision and does not assert solver correctness, floating-point hardware semantics, classical real completeness, or an infinite refinement theorem.
\end{theorem}
\begin{proof}
Project the carrier. The validated-numerics side first sees the parent interval, the level row, and the ordered cell list; these rows determine which finite child packets can be inspected.

For each child, read the matching $\DyadicMeshUp$ packet and the corresponding $\ValidatedNumericsUp$ containment row. The $\hsame$ transports and $\Cont$ routes ensure that the mesh boundary and containment ledger are read along the same finite cell route.

Thus the handoff is the finite enclosure subdivision formed by these rows. No coordinate supplies solver correctness, hardware semantics, classical completeness, or an infinite refinement principle.
\end{proof}

\begin{theorem}[DyadicSubdivision refinement composition]
\label{thm:dyadic-subdivision-refinement-composition}
Let $S=(I,L,C,M,V,P,N)$ be an accepted $\DyadicSubdivisionUp$ packet. Suppose $S'$ is obtained from $S$ by displayed finite $\DyadicMeshUp$ refinement routes over the same parent $\RationalIntervalUp$ row $I$, and suppose $S''$ is obtained from $S'$ by a second displayed finite family of $\DyadicMeshUp$ refinement routes over the same parent row. Then $S''$ is an accepted $\DyadicSubdivisionUp$ packet for $I$.

The mesh coverage and validated enclosure handoff exported by $S''$ are the composed finite ledger: first the refined child rows from $S$ to $S'$, then the refined child rows from $S'$ to $S''$, together with the transported $\ValidatedNumericsUp$, $\hsame$, $\Cont$, $\Pkg$, and $\NameCert$ rows. The composition asserts no interval-tree semantics, infinite refinement process, host real subset, or solver-correctness theorem.
\end{theorem}
\begin{proof}
Apply refinement stability to the first displayed refinement family. The result is the accepted packet $S'$ over the same parent interval row $I$, with only its finite cell ledger, mesh rows, and validated containment rows refined.

Apply refinement stability again to the second displayed refinement family. Since its routes are also over the same parent row, the second application yields an accepted packet $S''$ whose parent interval and provenance surface remain the displayed rows of the original packet.

It remains to identify the consumer surface. Mesh coverage reads only the finite ordered cell list and the $\DyadicMeshUp$ packets; validated enclosure handoff reads the matching $\ValidatedNumericsUp$ rows along the same routes. Chaining those two finite ledgers gives the composed mesh and enclosure surface named in the statement, with no additional infinite or host-level object.
\end{proof}

\begin{theorem}[DyadicSubdivision common refinement exhaustion]
\label{thm:dyadic-subdivision-common-refinement-exhaustion}
Let $S_0=(I,L_0,C_0,M_0,V_0,P_0,N_0)$ and $S_1=(I,L_1,C_1,M_1,V_1,P_1,N_1)$ be two accepted $\DyadicSubdivisionUp$ packets over the same displayed $\RationalIntervalUp$ parent row $I$. If both packets carry finite $\DyadicMeshUp$ refinement ledgers for the same parent interval, then there is a finite accepted child-ledger over $\DyadicMeshUp$ rows that refines both $C_0$ and $C_1$ and whose validated enclosure rows are transported through the corresponding $\ValidatedNumericsUp$, $\hsame$, $\Cont$, $\Pkg$, and $\NameCert$ routes.
\end{theorem}
\begin{proof}
Read both packets through \autoref{def:dyadic-subdivision-source}. Each subdivision has a displayed parent interval row, a dyadic level row, a finite ordered cell list, a finite family of mesh cells, and a finite family of validated enclosure rows.

Form the common child-ledger by taking the displayed finite intersections of the two ordered dyadic cell ledgers inside the same parent row $I$. Because every cell boundary is read from $\DyadicMeshUp$ rows over one rational interval parent, the resulting ledger is again finite and ordered.

Apply \autoref{thm:dyadic-subdivision-refinement-stability} to transport the first packet to this common ledger, and apply it again to transport the second packet to the same ledger. The validated rows move along the same cell routes, so the common ledger exhausts both displayed refinement families without adding an infinite mesh or host interval tree.
\end{proof}

\begin{theorem}[DyadicSubdivision cell ledger transport]
\label{thm:dyadic-subdivision-cell-ledger-transport}
For an accepted $\DyadicSubdivisionUp$ packet $S=(I,L,C,M,V,P,N)$, componentwise $\hsame$ and $\Cont$ transport of the parent interval $I$, the dyadic level row $L$, the ordered cell list $C$, the $\DyadicMeshUp$ cell family $M$, the $\ValidatedNumericsUp$ enclosure family $V$, the package row $P$, and the local $\NameCert$ row $N$ preserves the subdivision classifier.
\end{theorem}
\begin{proof}
The classifier is read directly from \autoref{def:dyadic-subdivision-source}. It compares only the parent interval, common level, ordered cell list, mesh cells, validated enclosure rows, and provenance rows.

Transport the parent interval and level first. Then transport each ordered cell-list entry and its matching $\DyadicMeshUp$ row along the displayed $\Cont$ route, preserving its position in the finite ledger.

Transport the $\ValidatedNumericsUp$ rows along the same cell routes. Since each containment row remains attached to the transported mesh cell indexed by the transported ordered list, the packet still satisfies the carrier conditions.

Repack the transported rows with $P$ and $N$. The resulting packet is classified with the original one, and no global interval tree, solver state, or completed real subset is introduced.
\end{proof}

\begin{theorem}[DyadicSubdivision NameCert package]
\label{thm:dyadic-subdivision-namecert-package}
The $\DyadicSubdivisionUp$ obligation package is assembled from the source row of \autoref{def:dyadic-subdivision-source}, the NameCert obligations of \autoref{thm:dyadic-subdivision-namecert-obligations}, refinement stability, mesh coverage, validated enclosure handoff, refinement composition, common refinement exhaustion, and cell-ledger transport. The package is finite over $\DyadicRatCoreUp$, $\RationalIntervalUp$, $\DyadicMeshUp$, $\ValidatedNumericsUp$, $\RealUp$, $\mathsf{BHist}$, $\hsame$, $\Cont$, $\Pkg$, and $\NameCert$.
\end{theorem}
\begin{proof}
The source row supplies the displayed carrier and dependency coordinates. The NameCert obligations record carrier, classifier, stability, exactness, and ledger rows for the finite subdivision packet.

Refinement stability is \autoref{thm:dyadic-subdivision-refinement-stability}; mesh coverage is \autoref{thm:dyadic-subdivision-mesh-coverage}; validated enclosure handoff is \autoref{thm:dyadic-subdivision-validated-enclosure-handoff}. Together these rows give finite coverage and consumer exactness for a single displayed subdivision ledger.

Refinement composition is \autoref{thm:dyadic-subdivision-refinement-composition}, common refinement exhaustion is \autoref{thm:dyadic-subdivision-common-refinement-exhaustion}, and cell-ledger transport is \autoref{thm:dyadic-subdivision-cell-ledger-transport}. Repacking these rows gives the stated obligation package while preserving the non-escape clauses for interval-tree semantics, solver correctness, hardware semantics, classical completeness, and infinite refinement.
\end{proof}
