\chapter{A Concrete Naming Certificate for $\FastCauchyUp$}
\label{ch:concrete-instances-fastcauchy-namecert}

The $\FastCauchyUp$ interface records a finite, modulus-normalized Cauchy-name surface feeding the regular-sequence and real-name layers. In Bishop-style constructive analysis, a Cauchy name is useful only together with visible rate data; the BEDC packet keeps that rate as finite certificate data over $\StreamNameUp$, $\CauchyModulusUp$, $\DyadicRatCoreUp$, $\RegSeqRatUp$, and $\RealUp$.

\section{Finite modulus-normalized Cauchy names}
\label{sec:fastcauchy-finite-modulus}

\begin{definition}[FastCauchy carrier]
\label{def:fastcauchy-carrier}
A $\FastCauchyUp$ carrier is a finite $\mathsf{BHist}$ packet
$$
C=(s,\mu,E,R,L,T,P,N)
$$
with the following visible rows:
\begin{enumerate}
\item $s$ is a scheduled $\StreamNameUp$ observation table whose displayed entries are dyadic-rational observations;
\item $\mu$ is a $\CauchyModulusUp$ threshold table normalized so that each requested precision row names the finite index after which all scheduled comparisons are read;
\item $E$ is the dyadic endpoint ledger read through $\DyadicRatCoreUp$ for every scheduled observation and precision row used by $\mu$;
\item $R$ is the late-pair radius ledger, recording for each precision row that any two scheduled observations beyond the threshold named by $\mu$ fit inside the displayed dyadic radius;
\item $L$ records the componentwise $\hsame$ transport rows for the stream, threshold, dyadic endpoint, and late-pair ledgers;
\item $T$ records the $\Cont$ windows through which those finite reads are consumed by downstream certificates;
\item $P$ is the $\mathsf{Pkg}$ provenance for the named source chapters $\DyadicRatCoreUp$, $\CauchyModulusUp$, $\StreamNameUp$, $\RegSeqRatUp$, and $\RealUp$;
\item $N$ is the local $\NameCert_{\FastCauchyUp}$ row for the displayed packet.
\end{enumerate}
The carrier is a finite explicit-rate Cauchy name. It contains no quotient stream, no host real equality, no selected limit, and no standard bridge datum.
\end{definition}

\begin{definition}[FastCauchy classifier]
\label{def:fastcauchy-classifier}
For accepted $\FastCauchyUp$ packets $C$ and $C'$, the classifier $\mathsf{FastCauchyClassifier}(C,C')$ holds when the scheduled observation windows, modulus threshold rows, dyadic endpoint ledgers, late-pair radius ledgers, $\hsame$ transports, $\Cont$ windows, $\mathsf{Pkg}$ provenance, and local $\NameCert$ rows compare componentwise through their displayed source classifiers.

The classifier is exact only for the finite modulus-normalized ledger. It does not assert equality of completed real numbers, equality of quotient Cauchy streams, or existence of a chosen limit.
\end{definition}

\begin{theorem}[FastCauchy modulus transport]
\label{thm:fastcauchy-modulus-transport}
Let $C$ be an accepted $\FastCauchyUp$ packet. If the scheduled observation rows of $C$ transport along the displayed $\hsame$ rows and the threshold table $\mu$ transports along the displayed $\Cont$ windows, then the transported packet preserves the same modulus-normalized late-pair radius obligations.

Concretely, every precision row in $\mu$ keeps its finite threshold, every scheduled pair beyond that threshold keeps its dyadic endpoint rows from $\DyadicRatCoreUp$, and the late-pair radius ledger remains readable by the same $\CauchyModulusUp$ classifier.
\end{theorem}
\leanchecked{BEDC.Derived.FastCauchyUp.FastCauchyFiniteCarrier\_modulus\_transport}
\begin{proof}
Project the carrier tuple of \autoref{def:fastcauchy-carrier}. The scheduled stream, threshold table, dyadic endpoint ledger, and late-pair ledger are separate finite rows, so each row can be transported through its displayed $\hsame$ coordinate.

The $\Cont$ row supplies the finite window in which downstream consumers may read the transported data. Because the threshold row is preserved, every pair of scheduled observations whose indices are beyond that threshold remains in the same late-pair comparison window.

Repack the transported stream, threshold, endpoint, radius, transport, window, provenance, and local certificate rows. The result has the same carrier shape and satisfies the classifier of \autoref{def:fastcauchy-classifier}, so the modulus-normalized obligations are preserved.
\end{proof}

\begin{theorem}[FastCauchy handoff to RegSeqRat]
\label{thm:fastcauchy-regseqrat-handoff}
Every accepted $\FastCauchyUp$ packet presents a $\RegSeqRatUp$ regular-sequence window by forgetting only the local $\NameCert_{\FastCauchyUp}$ row. The handoff keeps the scheduled $\StreamNameUp$ observation rows, the explicit $\CauchyModulusUp$ threshold table, the dyadic endpoint rows from $\DyadicRatCoreUp$, the late-pair radius ledger, the $\hsame$ and $\Cont$ transport rows, and the $\mathsf{Pkg}$ provenance.

The handoff therefore supplies regularity data to $\RegSeqRatUp$ without producing a completed $\RealUp$ object, quotienting Cauchy names, or selecting a limiting value.
\end{theorem}
\leanchecked{BEDC.Derived.FastCauchyUp.FastCauchyFinitePacket\_regseqrat\_handoff}
\begin{proof}
Start with the packet of \autoref{def:fastcauchy-carrier}. Remove the local row $N$ and retain the stream, modulus, endpoint, radius, transport, window, and provenance coordinates.

For each precision row, the $\CauchyModulusUp$ threshold names the finite index after which comparisons must be checked. The late-pair radius ledger supplies exactly those checks over the scheduled dyadic endpoint rows.

These retained rows are the regular-sequence surface consumed by $\RegSeqRatUp$. Since the construction never adds a quotient, host real equality, or selected limit coordinate, the resulting handoff is only the finite regularity window.
\end{proof}

\begin{theorem}[FastCauchy precision window restriction]
\label{thm:fastcauchy-precision-window-restriction}

Its $\RegSeqRatUp$ handoff is the restriction of
\autoref{thm:fastcauchy-regseqrat-handoff} to the selected window, and its
$\RealUp$ boundary is the restriction of
\autoref{thm:fastcauchy-real-boundary-window} to the same displayed
$\DyadicPrecisionUp$ row. The restriction introduces no selected limit, quotient
stream, host real equality, completeness datum, or standard bridge.
\end{theorem}
\begin{proof}

$\StreamNameUp$ observations, so the restricted packet keeps precisely the
matching threshold row, dyadic endpoint rows, late-pair radius entries,

Use \autoref{thm:fastcauchy-modulus-transport} on the selected rows. The
transport theorem keeps the modulus threshold, dyadic endpoints, $\hsame$

restriction has the same public coordinates as the parent packet, only with the

Apply \autoref{thm:fastcauchy-public-regseqrat-real-seal} to the restricted
surface. Its $\RegSeqRatUp$ handoff is therefore the selected part of
\autoref{thm:fastcauchy-regseqrat-handoff}, and its $\RealUp$ boundary is the
selected part of \autoref{thm:fastcauchy-real-boundary-window}.

The seal theorem also rules out any extra coordinate for a selected limit,

Thus the precision subwindow is only a finite public restriction of the

\end{proof}

\begin{theorem}[FastCauchy DyadicPrecision window cofinality]
\label{thm:fastcauchy-dyadicprecision-window-cofinality}
Let $F$ be an accepted $\FastCauchyUp$ packet and let $D$ be an accepted
$\DyadicPrecisionUp$ schedule selecting a finite precision window for $F$.

$\Cont$ transport rows, the restricted $\mathsf{Pkg}$ provenance rows, and the
local $\NameCert$ rows attached to the selected coordinates.

Conversely, every selected $\CauchyModulusUp$ threshold row, every selected
$\DyadicRatCoreUp$ endpoint row, and every late-pair radius ledger entry whose
indices lie in the finite $\DyadicPrecisionUp$ window appears in the restricted
$\RegSeqRatUp$ handoff and in the finite $\RealUp$ boundary. The coverage is a
coverage of public finite rows only: it does not assert a selected limit,
quotient stream, host real equality, completeness datum, or standard bridge.
\end{theorem}
\begin{proof}
Project $F$ through \autoref{def:fastcauchy-carrier}. This exposes the
scheduled observation rows, modulus thresholds, dyadic endpoints, late-pair
ledger entries, transport rows, provenance rows, and local name rows carried by
the accepted packet.

Project $D$ through \autoref{def:dyadic-precision-schedule-carrier}. The
schedule chooses a finite window of those carried coordinates, so
\autoref{thm:fastcauchy-precision-window-restriction} identifies the restricted
packet with exactly the rows whose indices are selected by $D$.

Apply \autoref{thm:fastcauchy-public-regseqrat-real-seal}. It identifies the
public $\RegSeqRatUp$ handoff and finite $\RealUp$ boundary with the same
selected threshold, endpoint, late-pair, transport, provenance, and naming rows.

The forward coverage follows because no other consumer-visible row survives the
restriction. The converse follows because each selected threshold, endpoint,
and late-pair row named by $D$ is one of the rows sealed into both public
consumer surfaces.
\end{proof}

\begin{theorem}[FastCauchy explicit-rate seal-window exhaustion]
\label{thm:fastcauchy-explicit-rate-seal-window-exhaustion}
Let $F$ be an accepted $\FastCauchyUp$ packet carrying its displayed
explicit-rate $\CauchyModulusUp$ schedule, and let $D$ be a displayed
$\DyadicPrecisionUp$ subwindow of that schedule. Every public $\RealUp$ seal
read over the restricted packet factors through the displayed explicit-rate
packet and the subwindow $D$: it is exactly one of the scheduled
$\StreamNameUp$ observation rows, $\CauchyModulusUp$ threshold rows,
$\DyadicRatCoreUp$ endpoint rows, late-pair radius rows, $\hsame$ rows,
$\Cont$ rows, $\mathsf{Pkg}$ provenance rows, or $\NameCert$ rows already
isolated by \autoref{thm:fastcauchy-precision-window-restriction},
\autoref{thm:fastcauchy-public-regseqrat-real-seal}, and
\autoref{thm:fastcauchy-standard-regseqrat-real-bridge}.

Conversely, each row of those displayed kinds whose index lies in the
$\DyadicPrecisionUp$ subwindow is a public $\RealUp$ seal read of the
restricted packet through the same explicit-rate factorization. The factorized
seal has no coordinate for a selected limit, quotient stream, host real
equality, completeness datum, hidden tail, or standard bridge datum beyond the
finite rows named above.
\end{theorem}
\begin{proof}
Project the carrier of $F$ from \autoref{def:fastcauchy-carrier}. The projected
packet exposes the scheduled stream observations, explicit modulus thresholds,
dyadic endpoints, late-pair radius ledger, transport rows, package rows, and
name-certificate rows; these are the only displayed coordinates available
before any consumer reads the packet.

Restrict this carrier along the displayed $\DyadicPrecisionUp$ row $D$. By
\autoref{thm:fastcauchy-precision-window-restriction}, the restriction keeps
exactly the rows whose indices lie in the finite precision subwindow, including
the explicit-rate thresholds and the dyadic endpoint rows used by the late-pair
radius ledger.

Apply the $\RegSeqRatUp$ handoff and the finite $\RealUp$ boundary. The handoff
identifies the regular-rational consumer rows with the same selected schedule,
and \autoref{thm:fastcauchy-public-regseqrat-real-seal} identifies the
$\RealUp$ seal reads with that finite public surface.

Finally use \autoref{thm:fastcauchy-standard-regseqrat-real-bridge}. The bridge
does not create an additional public coordinate; it only routes the displayed
RegSeqRat and Real consumers through the same explicit-rate window. Therefore
every public seal read is exhausted by the listed rows, and the omitted limit,
quotient, equality, completeness, tail, and bridge data remain unavailable.
\end{proof}
