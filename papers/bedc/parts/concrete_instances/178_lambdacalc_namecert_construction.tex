\chapter{A Concrete Naming Certificate for $\LambdaCalcUp$}
\label{ch:concrete-instances-lambdacalc-namecert}

The lambda-calculus interface $\LambdaCalcUp$ packages the inductive carrier of $\lambda$-terms together with $\beta$-reduction, $\alpha$-equivalence, and a stability classifier under capture-avoiding substitution. It depends on $\NameCert_{\TreeUp}$ (\autoref{ch:concrete-instances-tree-namecert}) and $\NameCert_{\NatUp}$ (\autoref{ch:concrete-instances-nat-namecert}).

\section{Carrier definition}
\label{sec:lambdacalc-carrier}

\begin{definition}[LambdaCalc BHist term carrier]
\label{def:lambdacalc-bhist-term-carrier}
The LambdaCalc BHist term carrier $\mathsf{LambdaCalcBHistTermCarrier}(h)$ is
the $\TreeUp$-backed syntax row whose visible spine is a $\mathsf{BHist}$ tree with
three constructor tags. A variable row has tag $\mathsf{lamVar}$ and a
$\NatUp$ unary variable index $i$ with $\mathsf{UnaryHistory}(i)$. An
abstraction row has tag $\mathsf{lamAbs}$, a unary binder index, and one body
subtree. An application row has tag $\mathsf{lamApp}$ and two subtrees. In each
case the spine is carried by a $\mathsf{TreeBHistCarrier}$ row, constructor
readback is expressed by $\hsame$ between the visible tag-and-subtree endpoint
and the displayed $\mathsf{BHist}$ syntax row, and the classifier compares carried
terms only through the corresponding $\mathsf{Cont}$ witness on these visible
spines.
\end{definition}

\begin{theorem}[LambdaCalc variable constructor carrier]
\label{thm:lambdacalc-variable-constructor-carrier}
For histories $i,h:\Hist$, if $\mathsf{UnaryHistory}(i)$, the visible variable
constructor row for $h$ is a $\mathsf{TreeBHistCarrier}$ row, and the displayed
constructor endpoint satisfies
$$
\hsame(h,\mathsf{lamVar}(i)),
$$
then $\mathsf{LambdaCalcBHistTermCarrier}(h)$. Moreover the carried row keeps
the $\NatUp$ index visible: any $\mathsf{Cont}$ witness transporting $h$ to
$h'$ along a constructor-preserving $\hsame(h,h')$ transports the same unary
index row and yields $\mathsf{LambdaCalcBHistTermCarrier}(h')$.
\end{theorem}
\begin{proof}
Read the variable row through the visible spine supplied by
$\mathsf{TreeBHistCarrier}$. The tag readback identifies the root as
$\mathsf{lamVar}$, and the sole payload is the unary history $i$.

NatUp unary index closure keeps $i$ in the unary fragment under the displayed
$\hsame$ transport. TreeUp visible-spine transport then carries the tagged
endpoint from $h$ to $h'$, so the transported row is again a LambdaCalc carrier
row of the variable form.
\end{proof}
\leanchecked{BEDC.Derived.LambdaCalcUp.LambdaCalcBHistTermCarrier\_variable\_constructor\_carrier}

\begin{theorem}[LambdaCalc abstraction and application carrier closure]
\label{thm:lambdacalc-abstraction-application-carrier-closure}
For histories $x,b,f,g,h:\Hist$, suppose the displayed syntax spines for $b$,
$f$, and $g$ are $\mathsf{TreeBHistCarrier}$ rows and
$$
\mathsf{LambdaCalcBHistTermCarrier}(b),\quad
\mathsf{LambdaCalcBHistTermCarrier}(f),\quad
\mathsf{LambdaCalcBHistTermCarrier}(g).
$$
If $\mathsf{UnaryHistory}(x)$ and
$$
\hsame(h,\mathsf{lamAbs}(x,b)),
$$
then $\mathsf{LambdaCalcBHistTermCarrier}(h)$. If instead
$$
\hsame(h,\mathsf{lamApp}(f,g)),
$$
then $\mathsf{LambdaCalcBHistTermCarrier}(h)$. Both conclusions are stable
under any constructor-preserving $\mathsf{Cont}$ transport of the displayed
$\mathsf{BHist}$ spine.
\end{theorem}
\begin{proof}
For abstraction, use the TreeUp visible-spine row to expose the
$\mathsf{lamAbs}$ root, the unary binder payload, and the single body subtree.
The body is already a LambdaCalc carrier row, and NatUp unary index closure
keeps the binder inside the admitted index surface.

For application, use the same TreeUp spine readback with root
$\mathsf{lamApp}$ and the two carried subtrees $f$ and $g$. No host syntax
operation is used: the two arguments are exactly the displayed $\mathsf{BHist}$
subtrees, and the carrier conclusion repacks them through the LambdaCalc
definition.

Constructor-preserving $\mathsf{Cont}$ transport acts componentwise on the
visible spine. It preserves the root tag and transports each carried subtree by
$\hsame$, so both constructor clauses remain inside
$\mathsf{LambdaCalcBHistTermCarrier}$.
\end{proof}
\leanchecked{BEDC.Derived.LambdaCalcUp.LambdaCalcBHistTermCarrier\_abstraction\_application\_closure}

\begin{theorem}[LambdaCalc substitution ledger scope]
\label{thm:lambdacalc-substitution-ledger-scope}
Let $x,s,t,u:\Hist$ be histories with $\mathsf{UnaryHistory}(x)$,
$\mathsf{LambdaCalcBHistTermCarrier}(s)$, and
$\mathsf{LambdaCalcBHistTermCarrier}(t)$. A LambdaCalc substitution ledger row
for substituting $s$ for the unary variable $x$ in $t$ is in scope only when it
is presented by a $\mathsf{BHist}$ visible-spine traversal whose nodes are carried by
$\mathsf{TreeBHistCarrier}$, whose recursive calls are connected by
$\mathsf{Cont}$ witnesses, and whose output endpoint $u$ satisfies
$\mathsf{LambdaCalcBHistTermCarrier}(u)$ together with constructorwise
$\hsame$ readback. The row exports no host substitution function and no
unmentioned capture-avoidance primitive.
\end{theorem}
\begin{proof}
Traverse the displayed TreeUp spine of $t$. In the variable case, compare the
visible NatUp unary index with $x$ and choose either the carried substituent
$s$ or the original variable row, with the choice recorded as a BEDC ledger
branch rather than a host-language function.

In the abstraction and application cases, use the constructor closure theorem
above after recursively transporting the displayed child rows. The
$\mathsf{Cont}$ witnesses record the recursive route through the visible
$\mathsf{BHist}$ spine, and every endpoint comparison is an $\hsame$ comparison
between displayed constructor rows.

Thus the substitution ledger remains a scoped LambdaCalc carrier obligation:
it names only $\mathsf{BHist}$, $\mathsf{UnaryHistory}$, $\mathsf{TreeBHistCarrier}$,
$\mathsf{Cont}$, $\hsame$, and $\mathsf{LambdaCalcBHistTermCarrier}$ rows.
\end{proof}

\begin{definition}[LambdaCalcUp term packet carrier]
\label{def:lambdacalc-term-packet-carrier}
A $\LambdaCalcUp$ term packet carrier is a finite $\mathsf{BHist}$ packet
$$
\mathcal L=(T,I,a,b,s,\lambda,\rho)
$$
over the source rows $\NameCert_{\TreeUp}$ and $\NameCert_{\NatUp}$. The row
$T$ is the displayed $\TreeUp$ syntax row for variables, abstraction nodes,
and application nodes. The row $I$ is the displayed $\NatUp$ binder-index row
used to name de Bruijn levels, binder depth, and free-variable positions.
The row $a$ is the alpha-equivalence ledger comparing syntax only through the
carried tree and index rows. The row $b$ is the beta-redex ledger naming
application-abstraction pairs together with their substitution endpoint. The
row $s$ is the capture-avoidance $\Cont$ ledger recording renaming, binder
shift, and substitution continuations. The row $\lambda$ orders the finite
term-construction and reduction steps, and $\rho$ is the $\Pkg$ provenance row
for the displayed syntax, index, alpha-equivalence, beta-redex,
capture-avoidance, and continuation entries.
\end{definition}

\section{The certificate}
\label{sec:lambdacalc-certificate}

\begin{definition}[LambdaCalc free-variable ledger packet]
\label{def:lambdacalc-free-variable-ledger-packet}
A LambdaCalc free-variable ledger packet is a finite $\mathsf{BHist}$ packet
$$
\mathcal F=(T,I,F,A,B,S,\lambda,\rho)
$$
over $\NameCert_{\TreeUp}$ and $\NameCert_{\NatUp}$. The row $T$ is the
TreeUp syntax spine for variables, abstraction nodes, and application nodes.
The row $I$ is the NatUp binder-index ledger naming de Bruijn levels and binder
depth. The row $F$ records the free-variable endpoint rows exposed by the
syntax traversal. The row $A$ records alpha-renaming comparisons, and $B$
records beta-redex source and endpoint rows. The row $S$ is the finite
$\Cont$ substitution provenance for renaming, binder shift, and
capture-avoiding replacement. The row $\lambda$ orders the displayed traversal,
and $\rho$ is the $\Pkg$ provenance tying the TreeUp syntax, NatUp indices,
free-variable rows, alpha rows, beta-redex rows, and substitution ledger into a
single LambdaCalc certificate packet.
\end{definition}

\begin{theorem}[LambdaCalc free-variable ledger coverage]
\label{thm:lambdacalc-free-variable-ledger-coverage}
Every accepted free-variable endpoint of a $\LambdaCalcUp$ term is projected
from a packet of \autoref{def:lambdacalc-free-variable-ledger-packet}. More
precisely, if $v:\Hist$ is accepted as a free variable of a carried
$\mathsf{LambdaCalcBHistTermCarrier}$ term $t$, then there are rows
$T,I,F,A,B,S,\lambda,\rho$ forming $\mathcal F$ such that $t$ occurs in the
TreeUp spine $T$, $v$ occurs in the NatUp index ledger $I$, the endpoint row is
listed in $F$, and the projection is justified only by the displayed
$\mathsf{BHist}$, $\Cont$, $\hsame$, $\NameCert$, and $\Pkg$ rows. No host
variable set is exported.
\end{theorem}
\begin{proof}
Induct on the displayed TreeUp spine of the carried LambdaCalc term. In the
variable case, the NatUp unary index row is the exposed free-variable endpoint,
and the packet records it in $F$ with the same TreeUp and NatUp provenance rows.

For an abstraction node, read the binder index from $I$ and compare it with the
free-variable row by the carried $\hsame$ ledger. The binder branch removes only
the displayed endpoint row from the packet; it does not form a host set of
variables.

For an application node, use the two recursive packet projections and concatenate
their finite $\Cont$ provenance through $\lambda$. Repacking the two child rows
with the common $\Pkg$ provenance gives exactly the accepted endpoint rows in
$F$, so every accepted free-variable endpoint is covered by the finite packet.
\end{proof}

\begin{theorem}[LambdaCalc capture-avoidance Cont closure]
\label{thm:lambdacalc-capture-avoidance-cont-closure}
Let $x,s,t,u:\Hist$ be histories with $\mathsf{UnaryHistory}(x)$,
$\mathsf{LambdaCalcBHistTermCarrier}(s)$, and
$\mathsf{LambdaCalcBHistTermCarrier}(t)$. If a packet of
\autoref{def:lambdacalc-free-variable-ledger-packet} displays a
capture-avoiding substitution $\Cont$ ledger from $t$ with substituent $s$ at
the unary variable $x$ to endpoint $u$, then
$\mathsf{LambdaCalcBHistTermCarrier}(u)$. The closure statement is stable under
the same TreeUp and NatUp $\hsame$ rows recorded by the packet.
\end{theorem}
\begin{proof}
Induct on the TreeUp spine of $t$. The variable branch compares the visible
NatUp index with $x$ inside the finite ledger and chooses either the carried
substituent $s$ or the original variable constructor row.

For abstraction, use the binder row in $I$ and the alpha ledger $A$ to rename
the binder when the displayed free-variable row would otherwise be captured.
The substitution provenance remains a $\Cont$ row, and the body endpoint is
obtained by the induction hypothesis.

For application, run the induction hypothesis on the two displayed child
spines and combine their endpoints with the application constructor closure of
\autoref{thm:lambdacalc-abstraction-application-carrier-closure}. The resulting
endpoint $u$ is therefore a LambdaCalc BHist term carrier, transported only
through the listed $\hsame$ and $\Cont$ rows.
\end{proof}

\begin{theorem}[LambdaCalc alpha-beta carrier transport]
\label{thm:lambdacalc-alpha-beta-carrier-transport}
Suppose two beta-redex packets in
\autoref{def:lambdacalc-free-variable-ledger-packet} have TreeUp syntax rows
and NatUp binder-index rows related by the alpha ledger $A$. Then the source
application, abstraction body, argument, substitution endpoint, and
beta-contractum endpoints transport as
$\mathsf{LambdaCalcBHistTermCarrier}$ rows along the displayed TreeUp and NatUp
$\hsame$ comparisons. Thus alpha-related beta-redex packets determine
equivalent carrier endpoints without exporting quotient alpha classes.
\end{theorem}
\begin{proof}
Read the two beta-redex packets through the TreeUp application spine. The alpha
ledger supplies constructor-preserving $\hsame$ rows for the application root,
the abstraction child, the binder index, and the argument subtree.

Transport the capture-avoidance ledger $S$ along these rows. The preceding
closure theorem keeps the substitution endpoint inside
$\mathsf{LambdaCalcBHistTermCarrier}$ after binder renaming and NatUp index
transport.

Finally repack the transported source and endpoint rows as the beta-redex
carrier data. Since every comparison is a displayed TreeUp or NatUp $\hsame$
row and every route is recorded in $\Cont$, the transported endpoints are
carrier-equivalent in the LambdaCalc certificate surface.
\end{proof}

\begin{theorem}[LambdaCalc carrier reflexivity]
\label{thm:lambdacalc-carrier-reflexive}
Every accepted $\LambdaCalcUp$ term packet over a $\TreeUp$ syntax row and a
$\NatUp$ binder ledger classifies with itself. The reflexive classifier is the
componentwise $\hsame$ row on the syntax tree, binder indices, free-variable
ledger, and package provenance.
\end{theorem}
\begin{proof}
\begin{enumerate}
\item Traditional target: reflexivity of the lambda-term equality classifier.
BEDC source: an accepted $\mathsf{LambdaCalcBHistTermCarrier}$ packet whose
visible syntax row is supplied by $\TreeUp$ and whose binder indices are
supplied by $\NatUp$.

\item Carrier: keep the same $\mathsf{BHist}$ term packet and the same
$\Pkg$ provenance row. Classifier: use reflexive $\hsame$ on the visible
TreeUp skeleton, each unary binder index, the free-variable ledger, and the
package provenance.

\item Stability and ledger: the identity $\mathsf{Cont}$ ledger transports no
constructor payload and no binder row. The carried rows are therefore read back
through the same visible spine.

\item Core theorem and exactness: the accepted packet classifies with itself,
and no endpoint outside the displayed syntax, binder, free-variable, and
provenance rows is read. Standard bridge: absent. Status: paper obligation for
the future $\BEDC$ namespace $\mathsf{Derived.LambdaCalcUp}$.
\end{enumerate}
\end{proof}

\begin{theorem}[LambdaCalc alpha classifier symmetry]
\label{thm:lambdacalc-alpha-classifier-symmetric}
If an accepted alpha-classifier row relates two accepted $\LambdaCalcUp$ term
packets, then the reverse alpha-classifier row is accepted. The reversed row is
obtained by reversing the carried $\hsame$ rows on the $\TreeUp$ skeletons and
the $\NatUp$ binder indices.
\end{theorem}
\begin{proof}
\begin{enumerate}
\item Traditional target: symmetry of alpha equivalence. BEDC source: an
accepted alpha-classifier row between two $\mathsf{BHist}$ lambda-term packets,
with TreeUp skeleton certificates and NatUp binder-index certificates exposed
as visible rows.

\item Carrier: both endpoints remain the original accepted LambdaCalc packets.
Classifier: reverse each carried $\hsame$ comparison on the TreeUp skeleton,
the unary binder indices, and the provenance rows that identify the two
packets as LambdaCalc terms.

\item Stability and ledger: the alpha row already carries a constructorwise
ledger for the binder-renaming route. Reading that ledger in reverse preserves
the same constructor tags and consumes the same NatUp unary index rows.

\item Core theorem and exactness: the reversed row is an accepted alpha
classifier and exports no quotient of alpha classes. Standard bridge: absent.
Status: paper obligation for the future $\BEDC$ namespace
$\mathsf{Derived.LambdaCalcUp}$.
\end{enumerate}
\end{proof}

\begin{theorem}[LambdaCalcUp substitution ledger obligation]
\label{thm:lambdacalc-substitution-ledger-obligation}
Substitution in $\LambdaCalcUp$ is admitted only from a carried
\autoref{def:lambdacalc-term-packet-carrier}: the $\TreeUp$ syntax row names
the term spine, the $\NatUp$ binder-index row names the binder depth and free
positions, and the capture-avoidance $\Cont$ ledger supplies the renaming,
shift, and substitution continuations. No host syntax equality, ambient
variable set, or quotient alpha-class is part of the substitution admission
surface.
\end{theorem}
\begin{proof}
Project the syntax row $T$ and binder-index row $I$ from
\autoref{def:lambdacalc-term-packet-carrier}. These rows give the only term
spine and binder positions that the substitution endpoint may read.

Project the capture-avoidance ledger $s$ and the finite continuation ledger
$\lambda$. Their entries record the renaming and binder-shift steps needed to
thread substitution through abstraction and application nodes.

Repack the substitution endpoint using exactly these projected rows and the
$\Pkg$ provenance row $\rho$. Because the carrier contains no row for host
syntax equality, ambient variable sets, or quotient alpha-classes, those data
cannot be exported by the substitution obligation.
\end{proof}

\begin{theorem}[LambdaCalcUp beta-reduction classifier obligation]
\label{thm:lambdacalc-beta-reduction-classifier-obligation}
Beta reduction in $\LambdaCalcUp$ is classifier-stable under $\hsame$
transport of the carried $\TreeUp$ syntax row, the carried $\NatUp$
binder-index row, and the substitution ledger of
\autoref{thm:lambdacalc-substitution-ledger-obligation}. A transported
beta-redex row therefore repacks as a beta-reduction row with the transported
syntax, binder index, capture-avoidance ledger, and $\Pkg$ provenance; it does
not identify terms by host syntax equality or by quotient alpha-classes.
\end{theorem}
\begin{proof}
Start from the beta-redex ledger $b$ in
\autoref{def:lambdacalc-term-packet-carrier}. Its source application,
abstraction body, argument, and endpoint are named through the carried syntax
row and binder-index row.

Transport the syntax and binder-index rows by $\hsame$, then transport the
substitution ledger supplied by
\autoref{thm:lambdacalc-substitution-ledger-obligation}. The transported
ledger still records renaming, shift, and capture-avoidance continuations
inside the displayed $\mathsf{BHist}$ packet.

Repack the transported redex, substitution endpoint, continuation ledger, and
provenance as the beta-reduction classifier row. Since each field came from a
displayed carrier component or its $\hsame$ transport, no host equality of
terms and no quotient alpha-class enters the classifier.
\end{proof}

\begin{theorem}[LambdaCalcUp constructor case exhaustion]
\label{thm:lambdacalc-constructor-case-exhaustion}
Every accepted $\mathsf{LambdaCalcBHistTermCarrier}$ row in a displayed
$\Hist$ endpoint is witnessed by exactly one visible
$\mathsf{TreeBHistCarrier}$ constructor tag among variable, abstraction, and
application. More precisely, for any $\mathsf{BHist}$ packet whose
$\LambdaCalcUp$ carrier predicate and term-packet predicate are accepted, the
carried $\TreeUp$ spine supplies a $\mathsf{UnaryHistory}$ constructor tag
whose $\Cont$ row and $\hsame$ endpoint rows select one of those three cases,
and the constructor-tag ledger excludes the other two cases.
\end{theorem}
\begin{proof}
Read the $\mathsf{TreeBHistCarrier}$ row from the
$\mathsf{LambdaCalcBHistTermCarrier}$ packet. The row exposes the visible
$\Hist$ spine, the $\mathsf{BHist}$ constructor ledger, and the unary tag used
by the $\TreeUp$ syntax endpoint.

Perform constructor inspection on that tag. The variable, abstraction, and
application rows are the only admitted tags in the term-packet predicate, and
each row carries its own $\Cont$ continuation data and $\hsame$ endpoint
alignment.

The tag ledger is single-valued inside the displayed $\mathsf{BHist}$ packet.
Thus an accepted row lands in exactly one constructor case, and the two
remaining constructor tags are excluded by the same ledger field.
\end{proof}

\begin{theorem}[LambdaCalcUp alpha beta ledger separation]
\label{thm:lambdacalc-alpha-beta-ledger-separation}
For every carried $\Hist$ endpoint satisfying
$\mathsf{LambdaCalcBHistTermCarrier}$ and the LambdaCalc term-packet
predicates, alpha-classifier rows and beta-redex rows consume disjoint visible
$\mathsf{BHist}$ ledger fields except for the shared carried syntax row and
the shared binder-index rows. The alpha side reads only the
$\mathsf{TreeBHistCarrier}$ spine, $\mathsf{UnaryHistory}$ binder names,
$\Cont$ renaming ledger, and $\hsame$ binder transport rows, while the beta
side reads the application-abstraction redex row, the substitution packet, its
$\Cont$ continuation ledger, and the output $\hsame$ endpoint row.
\end{theorem}
\begin{proof}
Project the common syntax and binder-index rows from the
$\mathsf{LambdaCalcBHistTermCarrier}$ packet. These are the only fields shared
by alpha conversion and beta reduction, because both classifiers must address
the same displayed $\TreeUp$ term and binder coordinates.

For the alpha classifier, read the renaming and binder-transport ledger. Its
continuation rows name the permitted capture-avoiding binder changes and do
not mention an application redex or a substitution output endpoint.

For the beta-redex classifier, read the application, abstraction body,
argument, substitution packet, and output endpoint rows. These fields are
separate from the alpha renaming ledger, so the two classifiers can overlap
only at carried syntax and binder-index data.
\end{proof}

\begin{theorem}[LambdaCalcUp substitution carrier preservation]
\label{thm:lambdacalc-substitution-carrier-preservation}
Let a substitution ledger be admitted by
\autoref{thm:lambdacalc-substitution-ledger-obligation} for a carried input
$\Hist$ endpoint satisfying $\mathsf{LambdaCalcBHistTermCarrier}$ and the
LambdaCalc term-packet predicates. If the ledger records its output endpoint
inside $\mathsf{BHist}$ with the displayed $\mathsf{TreeBHistCarrier}$ syntax
row, $\mathsf{UnaryHistory}$ binder-index row, $\Cont$ substitution
continuation, and $\hsame$ endpoint transport rows, then the output endpoint
again satisfies $\mathsf{LambdaCalcBHistTermCarrier}$.
\end{theorem}
\begin{proof}
Apply
\autoref{thm:lambdacalc-substitution-ledger-obligation} to obtain the syntax,
binder-index, capture-avoidance, and provenance rows that the substitution
endpoint is allowed to read.

The ledger transports the input tree spine and binder coordinates through the
renaming, shift, and substitution continuations. Each transported row remains
a displayed $\mathsf{BHist}$ row with the required $\hsame$ endpoint alignment.

Repack the transported $\mathsf{TreeBHistCarrier}$ row, unary binder row,
substitution $\Cont$ ledger, and provenance row as the LambdaCalc carrier for
the output endpoint. No host lambda syntax equality or quotient alpha-class is
used in the preservation step.
\end{proof}

\begin{theorem}[LambdaCalcUp substitution output determinacy]
\label{thm:lambdacalc-substitution-output-determinacy}
Let two LambdaCalc substitution ledgers act on the same carried input endpoint
and suppose they expose the same $\TreeUp$ syntax row, the same $\NatUp$ binder
index row, the same substituent carrier row, and the same capture-avoidance
$\Cont$ provenance inside $\mathsf{BHist}$. Then their output endpoints are
classifier-equivalent LambdaCalc carrier endpoints through the displayed
$\hsame$ and $\Pkg$ rows.
\end{theorem}
\begin{proof}
Apply \autoref{thm:lambdacalc-substitution-ledger-obligation} to both ledgers.
The two readings expose the same TreeUp syntax row, NatUp binder index row,
substituent carrier, capture-avoidance continuation, and provenance packet.

Use \autoref{thm:lambdacalc-substitution-carrier-preservation} for each output.
Both endpoints are repacked from the same $\mathsf{TreeBHistCarrier}$ syntax
row, the same $\mathsf{UnaryHistory}$ binder-index row, and the same
substitution $\Cont$ ledger.

The classifier comparison is therefore the displayed $\hsame$ endpoint
transport between two outputs of one visible ledger packet. Repack this
comparison with the common $\Pkg$ provenance row; no host substitution
function, alpha quotient, or unlisted syntax equality is used.
\end{proof}

\begin{theorem}[LambdaCalcUp alpha beta constructor transport]
\label{thm:lambdacalc-alpha-beta-constructor-transport}
For an application-abstraction beta redex carried by
$\mathsf{LambdaCalcBHistTermCarrier}$, alpha transport along the displayed
TreeUp syntax row, NatUp binder-index row, $\Cont$ renaming ledger, and
$\hsame$ binder rows preserves the visible beta constructor row and its
substitution endpoint. Thus alpha-related redex packets expose the same
application constructor, abstraction body, argument row, and beta-contractum
endpoint as LambdaCalc classifier data.
\end{theorem}
\begin{proof}
Read the redex through
\autoref{thm:lambdacalc-constructor-exhaustion}. The application tag,
abstraction tag, body row, and argument row are the visible constructor fields
of one $\mathsf{TreeBHistCarrier}$ packet.

Apply \autoref{thm:lambdacalc-alpha-beta-ledger-separation}. Alpha transport
may change only the renaming and binder-transport rows attached to the common
syntax and binder coordinates; it does not replace the beta-redex constructor
fields.

Finally apply \autoref{thm:lambdacalc-substitution-carrier-preservation} to
the transported beta substitution ledger. The substitution endpoint remains a
LambdaCalc carrier endpoint over the same $\mathsf{BHist}$ packet, so the beta
constructor row and its contractum read back unchanged at the public
classifier surface.
\end{proof}

\begin{theorem}[LambdaCalcUp free-variable ledger exactness]
\label{thm:lambdacalc-free-variable-ledger-exactness}
For a carried LambdaCalc term endpoint, the accepted free-variable endpoints
are exactly the endpoints projected from a packet of
\autoref{def:lambdacalc-free-variable-ledger-packet}. The packet rows are the
displayed $\TreeUp$ syntax spine, $\NatUp$ binder-index ledger,
free-variable row list, alpha row, beta-redex row, substitution $\Cont$
provenance, traversal order, and $\Pkg$ provenance; no host variable set or
external syntax predicate is part of the accepted surface.
\end{theorem}
\begin{proof}
For soundness, apply
\autoref{thm:lambdacalc-free-variable-ledger-coverage}. Every accepted
free-variable endpoint projects from the displayed packet rows
$T,I,F,A,B,S,\lambda,\rho$, and the endpoint appears in the finite
free-variable row list $F$.

For completeness, start from a row of
\autoref{def:lambdacalc-free-variable-ledger-packet}. The TreeUp syntax spine
and NatUp binder-index ledger determine the variable, abstraction, and
application branches of the traversal, while the $\Cont$ provenance records
the finite route to each endpoint listed in $F$.

Use \autoref{thm:lambdacalc-capture-avoidance-cont-closure} when the endpoint
is transported through substitution provenance, and use
\autoref{thm:lambdacalc-alpha-beta-carrier-transport} when alpha or beta rows
share the syntax endpoint. Repacking these rows with the common $\Pkg$
provenance gives exactly the accepted free-variable endpoints.
\end{proof}
