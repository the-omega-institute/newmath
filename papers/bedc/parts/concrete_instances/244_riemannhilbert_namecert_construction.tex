\chapter{A Concrete Naming Certificate for $\RiemannHilbertUp$}
\label{ch:concrete-instances-riemannhilbert-namecert}

The Riemann-Hilbert interface $\RiemannHilbertUp$ packages the equivalence between regular holonomic D-modules on a complex manifold and perverse sheaves of $\mathbb{C}$-vector spaces. It depends on $\NameCert_{\DerivedFunctorUp}$ (\autoref{ch:concrete-instances-derivedfunctor-namecert}) and $\NameCert_{\SheafUp}$ (\autoref{ch:concrete-instances-sheaf-namecert}), and bridges differential and topological certificates through a common derived classifier.

\section{Carrier definition}
\label{sec:riemannhilbert-carrier}

\begin{definition}[Riemann-Hilbert BHist bridge packet]
\label{def:riemannhilbert-bhist-bridge-packet}
A $\RiemannHilbertUp$ carrier is a finite $\mathsf{BHist}$ bridge packet whose rows are the following data.

\begin{enumerate}
\item A derived-functor source row from $\NameCert_{\DerivedFunctorUp}$, including the displayed source object, derived endpoint, degree or cohomological index row, and exact-triangle provenance carried by the derived-functor ledger.

\item A sheaf target row from $\NameCert_{\SheafUp}$, read as a perverse-sheaf or constructible-sheaf target over the same displayed base histories, with its section, restriction, locality, and gluing rows visible as $\mathsf{BHist}$ data.

\item A regular-holonomic source classifier row saying that the differential source lies on the regular-holonomic branch of the derived-functor packet and can be compared only through the displayed classifier and $\hsame$ transport rows.

\item Transport rows matching the derived source, sheaf target, de Rham readback, local-system branch, and regular-holonomic classifier by $\hsame$.

\item $\Cont$ provenance rows explaining how the derived source is read to the sheaf target, how local horizontal sections are read as local-system rows, and how gluing reads back through the $\SheafUp$ exactness ledger.

\item $\mathsf{Pkg}$ rows naming the $\DerivedFunctorUp$ and $\SheafUp$ dependency certificates, together with the public dependency boundary consumed by the bridge packet.
\end{enumerate}

The carrier records only this displayed BEDC packet. It does not contain an external derived category, analytic D-module model, sheaf quotient, monodromy equivalence, or host-level Riemann-Hilbert theorem.
\end{definition}
\leandef{BEDC.Derived.RiemannHilbertUp.RiemannHilbertBHistBridgePacket}

\section{The certificate}
\label{sec:riemannhilbert-certificate}

\begin{theorem}[Riemann-Hilbert derived-sheaf source]
\label{thm:riemannhilbert-derived-sheaf-source}
Every accepted $\RiemannHilbertUp$ packet exposes a derived-functor source row and a sheaf target row exactly as in \autoref{def:riemannhilbert-bhist-bridge-packet}. The source side is the displayed $\DerivedFunctorUp$ row with its exact-triangle and degree provenance, and the target side is the displayed $\SheafUp$ row with its section, restriction, locality, and gluing ledger. No external derived category, analytic solution functor, or host sheaf quotient is projected by the source boundary.
\end{theorem}
\leanchecked{BEDC.Derived.RiemannHilbertUp.RiemannHilbertBHistBridgePacket\_derived\_sheaf\_source}
\leanchecked{BEDC.Derived.RiemannHilbertUp.RiemannHilbertBHistBridgePacket\_source\_boundary}
\begin{proof}
Project the accepted packet to its first two carrier coordinates. The definition gives the $\DerivedFunctorUp$ source row, including the derived endpoint and exact-triangle provenance, and the $\SheafUp$ target row, including section, restriction, locality, and gluing data.

The remaining carrier coordinates are comparison, continuation, and package rows attached to those two projections. Since the carrier has no coordinate for an ambient derived category, analytic solution functor, or host sheaf quotient, the source boundary is exactly the displayed pair of BEDC rows.
\end{proof}

\begin{theorem}[Riemann-Hilbert flat-connection classifier]
\label{thm:riemannhilbert-flat-connection-classifier}
The flat-connection branch of a $\RiemannHilbertUp$ packet is classified only through the regular-holonomic source classifier, the local-system sheaf row, and the displayed $\hsame$ transports between the derived source, de Rham readback, and sheaf target. If two accepted packets agree on those rows, then they have the same visible flat-connection classifier surface.
\end{theorem}
\leanchecked{BEDC.Derived.RiemannHilbertUp.RiemannHilbertBHistBridgePacket\_flat\_connection\_classifier}
\begin{proof}
Read the regular-holonomic classifier coordinate from \autoref{def:riemannhilbert-bhist-bridge-packet}. It is the only row that marks the differential source branch and it is already tied to the derived-functor packet.

Next project the $\hsame$ rows matching the derived source, de Rham readback, local-system branch, and sheaf target. Agreement on these rows forces the same visible comparison surface after transport, because every local-system read is mediated by the displayed $\Cont$ provenance rather than by a host flat-connection object.
\end{proof}

\begin{theorem}[Riemann-Hilbert local-system ledger]
\label{thm:riemannhilbert-local-system-ledger}
The local-system ledger exported by a $\RiemannHilbertUp$ packet consists exactly of the sheaf target row, the local horizontal-section read rows, the $\hsame$ comparisons tying them to the derived source and de Rham readback, and the $\Cont/\mathsf{Pkg}$ provenance naming the $\DerivedFunctorUp$ and $\SheafUp$ dependencies. It exports no monodromy representation, covering-space quotient, or external equivalence row.
\end{theorem}
\begin{proof}
Project the local-system branch from the sheaf target and continuation coordinates of the carrier. The carrier definition lists the horizontal-section reads, their comparison rows, and the dependency packages that make the read visible to downstream consumers.

Constructor inspection of the displayed packet gives the exclusion claim. The rows generate local-system information only through $\SheafUp$ sections, restrictions, locality, gluing, and $\Cont$ provenance; there is no constructor slot for a monodromy representation, covering-space quotient, or external equivalence.
\end{proof}

\begin{theorem}[Riemann-Hilbert regular-holonomic soundness]
\label{thm:riemannhilbert-regular-holonomic-soundness}
The regular-holonomic soundness boundary of $\RiemannHilbertUp$ is the conjunction of the derived-functor source row, the regular-holonomic source classifier row, the de Rham readback transport, and the sheaf target classifier row. A downstream consumer can read a regular-holonomic conclusion only by projecting these rows from the bridge packet.
\end{theorem}
\leanchecked{BEDC.Derived.RiemannHilbertUp.RiemannHilbertBHistBridgePacket\_regular\_holonomic\_soundness}
\begin{proof}
Use \autoref{thm:riemannhilbert-derived-sheaf-source} to identify the source and target rows available to the packet. The source contributes the derived-functor endpoint and exact-triangle provenance, while the target contributes the sheaf classifier and gluing ledger.

Use \autoref{thm:riemannhilbert-flat-connection-classifier} for the classifier transport. The regular-holonomic branch is stable only along the displayed $\hsame$ and de Rham readback rows, so any consumer conclusion must be a projection from those rows together with the source and target packets.
\end{proof}

\begin{theorem}[Riemann-Hilbert sheaf-gluing exactness]
\label{thm:riemannhilbert-sheaf-gluing-exactness}
For a $\RiemannHilbertUp$ bridge packet, compatible local-system rows glue exactly through the $\SheafUp$ target ledger carried in \autoref{def:riemannhilbert-bhist-bridge-packet}. The resulting global sheaf row remains tied to the same derived-functor source, regular-holonomic classifier, $\hsame$ transport rows, $\Cont$ provenance, and $\mathsf{Pkg}$ dependency rows.
\end{theorem}
\begin{proof}
Project the $\SheafUp$ locality and gluing coordinates from the target row. Compatible local-system rows are sheaf rows in this packet, so gluing is performed by the displayed sheaf exactness ledger rather than by an external sheafification operation.

After gluing, project the comparison and provenance coordinates again. They still name the same derived-functor source, regular-holonomic classifier, $\hsame$ transports, continuation routes, and dependency packages, so the glued target remains inside the original $\RiemannHilbertUp$ carrier.
\end{proof}

\begin{theorem}[Riemann-Hilbert de Rham readback scope]
\label{thm:riemannhilbert-de-rham-readback-scope}
Every de Rham readback row in an accepted $\RiemannHilbertUp$ bridge packet is sourced only from the displayed $\DerivedFunctorUp$ source row, the regular-holonomic classifier row, the $\SheafUp$ target row, the local-system ledger, the displayed $\hsame$ transports, the $\Cont$ provenance, and the $\mathsf{Pkg}$ dependency rows. It exports no analytic solution functor, monodromy equivalence, host derived-category quotient, or external perverse-sheaf equivalence.
\end{theorem}
\begin{proof}
Use \autoref{thm:riemannhilbert-derived-sheaf-source} to identify the dependency rows available to the packet: the $\DerivedFunctorUp$ source row and the $\SheafUp$ target row with their displayed provenance. Use \autoref{thm:riemannhilbert-flat-connection-classifier} for the regular-holonomic classifier and its de Rham transport row.

Next use \autoref{thm:riemannhilbert-local-system-ledger} to bind the local-system ledger to the same sheaf target, horizontal-section read rows, $\hsame$ comparisons, and $\Cont/\mathsf{Pkg}$ provenance. Use \autoref{thm:riemannhilbert-sheaf-gluing-exactness} to keep those rows stable under sheaf gluing.

Finally inspect the constructors of \autoref{def:riemannhilbert-bhist-bridge-packet}. The carrier has slots only for the displayed source, classifier, target, transport, continuation, and package rows, so a de Rham readback row cannot export an analytic solution functor, monodromy equivalence, host derived-category quotient, or external perverse-sheaf equivalence.
\end{proof}
