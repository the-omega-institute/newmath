\section{Carrier definition}
\label{sec:belief-carrier}

A finite $\BeliefUp$ packet is the observation-update carrier suggested by
\texttt{BEDC.Derived.BeliefUp.taste\_gate}; the Lean touchpoint fixes the
chapter name and TasteGate carrier, while the paper surface below records the
certificate obligations without marking them as checked.
\leanchecked{BEDC.Derived.BeliefUp.taste\_gate}

\begin{definition}[Belief observation-update carrier]
\label{def:belief-observation-update-carrier}
A $\BeliefUp$ observation-update carrier is a finite packet over
$\mathsf{BHist}$ consisting of the following rows.
\begin{enumerate}
\item A prior row naming the displayed belief state before the observation.
\item An observation row naming the finite evidence read by the update.
\item An update trace recording the visible constructor route from the prior
row and observation row to the posterior row.
\item A probability row supplied by $\NameCert_{\ProbSpaceUp}$ and classified
only through the displayed probability certificate.
\item An evidence ledger that exposes exactly the prior, observation, update
trace, and probability rows used by the packet.
\end{enumerate}
The packet is accepted when those rows are present in the same finite
$\mathsf{BHist}$ ledger, the update trace is a $\SeqUp$ row, and the posterior
belief row is the row named by that trace.
\end{definition}

\begin{theorem}[Belief classifier transport obligation]
\label{thm:belief-classifier-transport-obligation}
Let $p$ be an accepted $\BeliefUp$ packet with prior row, observation row,
update trace, and probability row displayed in its evidence ledger. If
$\hsame$ transports each of those four rows to corresponding displayed rows,
then the transported rows repack as an accepted $\BeliefUp$ packet with the
same update-trace shape and a posterior row classified by the transported
probability certificate.
\end{theorem}
\leanchecked{BEDC.Derived.BeliefUp.BeliefObservationUpdateCarrier\_transport\_obligation}
\begin{proof}
Read the accepted packet through its finite $\mathsf{BHist}$ ledger. The prior
and observation rows are transported first, so the update trace has the same
source rows after transport.

Next transport the update trace as a $\SeqUp$ row. Since the trace records a
visible constructor route, the posterior row is obtained by replaying the same
finite route on the transported prior and observation rows.

Finally transport the probability row through the displayed
$\NameCert_{\ProbSpaceUp}$ certificate. The four transported rows fill the same
carrier fields, so the packet is accepted with no hidden evidence row.
\end{proof}

\begin{theorem}[Belief ledger exactness obligation]
\label{thm:belief-ledger-exactness-obligation}
For every accepted $\BeliefUp$ packet, the evidence ledger exposes exactly the
prior row, observation row, update trace, and probability row admitted by the
carrier. A downstream consumer may read those four rows and the posterior row
named by the update trace, but no additional hidden evidence, prior source, or
probability witness is part of the packet.
\end{theorem}
\begin{proof}
Inspect the carrier definition. Acceptance requires the prior row, observation
row, update trace, probability row, and evidence ledger to be displayed in the
same finite $\mathsf{BHist}$ packet.

The ledger exactness clause then identifies every exported evidence entry with
one of those four carrier rows. The posterior is readable only as the endpoint
of the displayed update trace, so it does not add an independent hidden
evidence coordinate.
\end{proof}

\begin{theorem}[Belief TasteGate carrier recognition]
\label{thm:belief-tastegate-carrier-recognition}
Every accepted $\BeliefUp$ packet is constrained by the TasteGate carrier named
by the Lean definition
\texttt{BEDC.Derived.BeliefUp.taste\_gate}. In particular, the accepted packet
may expose only the prior row, observation row, update trace, probability row,
posterior endpoint, evidence ledger, and displayed provenance rows admitted by
that carrier boundary.
\end{theorem}
\begin{proof}
Read the accepted packet through the TasteGate carrier boundary named above. The
carrier admits the observation-update rows as a finite $\mathsf{BHist}$ packet
and supplies no constructor for a host belief state, hidden likelihood object, or
external update operation.

Projecting the packet therefore leaves only the displayed prior, observation,
update trace, probability, posterior endpoint, evidence ledger, and provenance
rows. Since every accepted packet must pass through the same carrier boundary,
consumer recognition is restricted to those rows.
\end{proof}

\begin{theorem}[Belief posterior trace transport obligation]
\label{thm:belief-posterior-trace-transport-obligation}
Let $p$ be an accepted $\BeliefUp$ packet. If its prior row, observation row,
update trace, and probability row are transported through displayed $\hsame$
comparisons and $\Cont$ routes, then the transported rows repack as a
$\BeliefUp$ packet whose posterior endpoint is the endpoint named by the
transported update trace.
\end{theorem}
\begin{proof}
Start with the accepted packet and project its four transported input rows. The
prior and observation endpoints move through the displayed $\hsame$ comparisons,
while the update trace moves along the displayed $\Cont$ route.

The posterior endpoint is not an independent input. It is read as the endpoint
named by the update trace after transport, so replaying the transported trace
with the transported prior and observation rows determines the posterior row.

Transport the probability row through the displayed probability certificate and
place it with the transported trace data in the same finite ledger. These rows
fill the carrier fields again, so the packet repacks under $\hsame/\Cont$.
\end{proof}

\begin{theorem}[Belief finite evidence ledger coverage]
\label{thm:belief-finite-evidence-ledger-coverage}
For every accepted $\BeliefUp$ packet, the finite evidence ledger covers exactly
the prior row, observation row, update trace, probability row, and posterior
endpoint. No additional likelihood row, hidden prior source, external
observation stream, or host update primitive is exported by the ledger.
\end{theorem}
\begin{proof}
Inspect the carrier rows. Acceptance places the prior, observation, update
trace, probability row, and posterior endpoint in one finite $\mathsf{BHist}$
ledger with their displayed provenance.

The ledger coverage clause identifies each readable evidence entry with one of
those five rows. Since the posterior endpoint is already the trace endpoint, it
does not create a second evidence source.

Thus a downstream consumer can cite exactly the listed rows and no hidden
belief-update data. Every permitted read is a projection of the displayed finite
ledger.
\end{proof}
