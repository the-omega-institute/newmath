The $\BanachFixedPointUp$ interface names the finite BEDC contraction packet that carries a Banach fixed-point construction as displayed data. Its human reference is Banach's contraction principle, but the certificate records only the rows consumed by BEDC: a Banach carrier row, a complete-metric distance row, a contraction-map row, a rational Lipschitz ledger below the unit bound, a finite Picard ledger, an explicit Cauchy modulus, a regular-rational convergence handoff, a seal-facing real-name endpoint row, transport and continuation rows, package provenance, and a local $\NameCert$ row.

Its dependency list is $\BanachUp$, $\CompleteMetricUp$, $\MetricSpaceUp$, $\PicardContractionUp$, $\CauchyModulusUp$, $\RegSeqRatUp$, and $\RealUp$.

\begin{definition}[BanachFixedPoint carrier]
\label{def:banach-fixed-point-carrier}
A $\BanachFixedPointUp$ carrier is a finite $\mathsf{BHist}$ packet
$$
F=(B,d,T,\lambda,I,M,R,E,H,C,P,N)
$$
with the following visible coordinates:
\begin{enumerate}
\item $B$ is the $\BanachUp$ carrier row whose state histories provide the domain inspected by the fixed-point packet;
\item $d$ is the $\CompleteMetricUp$ and $\MetricSpaceUp$ distance row used to read finite iterate distances and the carried completeness surface;
\item $T$ is the contraction-map row displaying the map-code history and the continuation route from a carried state row to the next Picard state;
\item $\lambda$ is a rational Lipschitz ledger with a displayed positivity row and a displayed unit-bound row;
\item $I$ is the finite Picard-iterate ledger, recording $x_0,\ldots,x_n$ together with the application rows for $x_{k+1}=T(x_k)$;
\item $M$ is the explicit $\CauchyModulusUp$ row built from the iterate ledger and the Lipschitz-ratio row;
\item $R$ is the $\RegSeqRatUp$ and $\CompleteMetricUp$ convergence handoff, exposing the regular rational name and the carried completion route used by consumers;
\item $E$ is the $\RealUp$ seal-facing endpoint row attached to the handoff;
\item $H$ stores componentwise $\hsame$ transports for the Banach, distance, contraction, ratio, iterate, modulus, handoff, endpoint, package, and naming rows;
\item $C$ stores the displayed $\Cont$ routes for metric reads, contraction application, finite Picard concatenation, modulus readback, completion handoff, and endpoint sealing;
\item $P$ is the $\Pkg$ provenance row naming the $\BanachUp$, $\CompleteMetricUp$, $\MetricSpaceUp$, $\PicardContractionUp$, $\CauchyModulusUp$, $\RegSeqRatUp$, and $\RealUp$ dependencies;
\item $N$ is the local $\NameCert_{\BanachFixedPointUp}$ row.
\end{enumerate}
The carrier has no coordinate for host real equality, a choice-selected limit, a global nonlinear-analysis theorem, or ambient completeness outside the carried $\CompleteMetricUp$ row.
\end{definition}

\begin{theorem}[BanachFixedPoint iteration ledger]
\label{thm:banach-fixed-point-iteration-ledger}
Every accepted $\BanachFixedPointUp$ carrier exposes a finite $\PicardContractionUp$ ledger $I$ over the carried $\BanachUp$ row $B$, distance row $d$, contraction row $T$, and rational Lipschitz ledger $\lambda$. For each displayed adjacent pair $x_k,x_{k+1}$ in $I$, the application row reads $x_{k+1}$ from $T$ and $x_k$, and the distance comparison is read through $d$ and $\lambda$.
\end{theorem}
\begin{proof}
Read the $I$ coordinate of \autoref{def:banach-fixed-point-carrier}. It is a finite Picard ledger, so every adjacent pair in the ledger is a visible $\mathsf{BHist}$ row rather than a completed orbit.

The contraction-map coordinate $T$ supplies the application route for each step. The distance row $d$ supplies the metric read, and the rational ledger $\lambda$ supplies the displayed Lipschitz comparison below the unit bound.

Thus the finite iteration surface is exactly the $\PicardContractionUp$ surface carried inside $F$. No infinite sequence object or host fixed-point value is read from the packet.
\end{proof}

\begin{theorem}[BanachFixedPoint uniqueness boundary]
\label{thm:banach-fixed-point-uniqueness-boundary}
