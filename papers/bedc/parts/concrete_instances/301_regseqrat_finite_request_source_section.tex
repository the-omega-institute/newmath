\begin{theorem}[RegSeqRat finite-request source section]
\label{thm:regseqrat-finite-request-source-section}
Let $R$ be an accepted $\RegSeqRatUp$ packet, and let $S$ be a public finite
request to the accepted packet. Then $S$ factors through one displayed
$\RegSeqRatUp$ source section carried by a finite $\BHist$ packet. The visible
rows of that section are exactly the following rows:
\begin{enumerate}
\item the $\StreamNameUp$ schedule rows selected by the request;
\item the $\DyadicRatCoreUp$ endpoint, radius, and finite dyadic-distance rows
read at those schedule positions;
\item the $\RegSeqRatUp$ classifier row and the regularity rows certifying the
accepted packet;
\item the componentwise $\hsame$ rows tying the schedule, dyadic endpoint,
radius, distance, classifier, and regularity rows to the same request packet;
\item the displayed $\Cont$ routes from the StreamName schedule through the
dyadic rows and the RegSeqRat classifier surface;
\item the $\mathsf{Pkg}$ provenance over $\StreamNameUp$, $\DyadicRatCoreUp$,
$\RegSeqRatUp$, $\BHist$, $\hsame$, $\Cont$, and $\NameCert$;
\item the local $\NameCert_{\RegSeqRatUp}$ rows for the request.
\end{enumerate}
No other row is part of the RegSeqRat source section.
\end{theorem}
\begin{proof}
Begin with the StreamName finite-observation section of
\autoref{thm:streamname-finite-observation-section}. It gives the displayed
schedule rows named by the finite request and places them in one finite source
packet.

Apply the RegSeqRat tail-budget section lock,
\autoref{thm:streamname-finite-observation-regseqrat-tail-budget-section-lock}.
The lock prevents the request from replacing the displayed schedule by an
off-packet tail selector, so every later RegSeqRat row is read over the same
finite window.

Next use the exact RegSeqRat handoff,
\autoref{thm:regseqrat-streamname-dyadic-real-handoff-exactness}. It supplies
the dyadic endpoint, radius, finite distance, classifier, and regularity rows
over the displayed schedule.

Now apply \autoref{thm:regseqrat-common-dyadic-tail-ledger}. The common ledger
identifies the componentwise $\hsame$ rows, $\Cont$ routes, package
provenance, and local naming rows as part of the same finite RegSeqRat packet.
This gives one $\BHist$ source section and no second source coordinate.
\end{proof}

\begin{theorem}[RegSeqRat Real request section nonescape]
\label{thm:regseqrat-real-request-section-nonescape}
Let $R$ be an accepted $\RegSeqRatUp$ packet, and let $S$ be a public finite
request to $R$ that is consumed by a $\RealUp$ seal. Then the same displayed
source section of
\autoref{thm:regseqrat-finite-request-source-section} is the entire source
section seen by the seal. In particular the section cannot export quotient
stream equality, host sequence equality, selected limits, ambient completeness,
metric-completeness data, choice data, or a second $\RealUp$ seal coordinate.
\end{theorem}
\begin{proof}
First factor the request through
\autoref{thm:regseqrat-finite-request-source-section}. The factorization has
only the StreamName schedule rows, DyadicRatCore endpoint, radius, and distance
rows, RegSeqRat classifier and regularity rows, and the displayed transport,
route, package, and local naming rows.

Apply the Real finite-request fusion theorem,
\autoref{thm:real-regseqrat-finite-request-fusion}. It says that a Real seal
which consumes the same request reads through one finite $\BHist$ packet with
the same schedule, dyadic, classifier, regularity, $\hsame$, $\Cont$,
$\mathsf{Pkg}$, and local $\NameCert$ rows.

Finally compare the two packets through
\autoref{thm:regseqrat-streamname-dyadic-real-handoff-exactness} and
\autoref{thm:regseqrat-common-dyadic-tail-ledger}. These theorems keep the
StreamName, DyadicRatCore, and RegSeqRat faces as the only source faces
available to the seal-facing request. Since those faces carry no row for
quotient stream equality, host sequence equality, selected limits, ambient
completeness, metric-completeness data, choice data, or a second Real seal
coordinate, the consumed section cannot export any such datum.
\end{proof}

\begin{theorem}[RegSeqRat finite-request Real seal correspondence]
\label{thm:regseqrat-finite-request-real-seal-correspondence}
Let $R$ be an accepted $\RegSeqRatUp$ packet, and let $S$ be a public finite
request to $R$ that is consumed by a $\RealUp$ seal. The finite request source
section of \autoref{thm:regseqrat-finite-request-source-section} determines
the seal-visible source section of
\autoref{thm:regseqrat-real-request-section-nonescape}: its rows restrict
exactly to the StreamName schedule, the $\DyadicRatCoreUp$ endpoint, radius,
and dyadic-distance rows, the $\RegSeqRatUp$ classifier and regularity rows,
the componentwise $\hsame$ rows, the displayed $\Cont$ routes, the
$\mathsf{Pkg}$ provenance, and the local $\NameCert_{\RegSeqRatUp}$ rows.
Conversely any seal-visible section for the same request restricts to exactly
those rows of the finite request source section. The correspondence carries no
quotient stream equality, host sequence equality, selected limits, ambient
completeness, metric-completeness data, choice data, or second seal coordinate.
\end{theorem}
\begin{proof}
First build the finite source packet with
\autoref{thm:regseqrat-finite-request-source-section}. This gives the unique
displayed request section whose visible rows are the StreamName schedule rows,
the dyadic endpoint, radius, and finite distance rows, the RegSeqRat classifier
and regularity rows, and the displayed $\hsame$, $\Cont$, $\mathsf{Pkg}$, and
local $\NameCert$ rows.

Next route the same public request to the Real consumer by
\autoref{thm:real-regseqrat-finite-request-fusion}. The fusion theorem forces
the seal to read through the finite $\BHist$ packet already named by the
request, so the consumer has no independent source packet from which additional
rows could be drawn.

The common visible rows are then fixed by
\autoref{thm:regseqrat-streamname-dyadic-real-handoff-exactness} and
\autoref{thm:regseqrat-common-dyadic-tail-ledger}. These theorems identify the
schedule, dyadic, classifier, regularity, $\hsame$, $\Cont$, package, and
naming rows on both sides of the handoff, giving the restriction from the
finite source section to the seal-visible section.

For the converse, start with a seal-visible section for the same request and
apply \autoref{thm:regseqrat-real-request-section-nonescape}. Nonescape says
that the seal-visible section is already the displayed RegSeqRat finite source
section, so restricting it back along the request recovers exactly the same
StreamName schedule, dyadic endpoint, radius, distance, classifier,
regularity, $\hsame$, $\Cont$, $\mathsf{Pkg}$, and local $\NameCert$ rows.
The excluded rows are absent on both sides, so the correspondence introduces
no quotient stream equality, host sequence equality, selected limit, ambient or
metric completeness, choice datum, or second seal coordinate.
\end{proof}
