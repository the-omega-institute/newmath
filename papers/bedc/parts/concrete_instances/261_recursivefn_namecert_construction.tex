\chapter{A Concrete Naming Certificate for $\RecursiveFnUp$}
\label{ch:concrete-instances-recursivefn-namecert}

The partial-recursive interface $\RecursiveFnUp$ packages partial functions $\NatUp \to \NatUp$ generated by zero, successor, projection, composition, primitive recursion, and bounded minimisation. It depends on $\NameCert_{\NatUp}$ (\autoref{ch:concrete-instances-nat-namecert}), $\NameCert_{\ListUp}$ (\autoref{ch:concrete-instances-list-namecert}), and $\NameCert_{\ComputableUp}$ (\autoref{ch:concrete-instances-computable-namecert}).

\section{Carrier definition}
\label{sec:recursivefn-carrier}

A $\RecursiveFnUp$ token carries an inductive constructor history of generation steps: zero, successor, projection $\pi^k_i$, composition row, primitive-recursion row with explicit base and step subterms, and bounded-minimisation row with an explicit unary bound. The carrier accepts a partial-graph value $F(n,m)$ when the constructor-history evaluation on the $\NatUp$-input row $n$ returns $\NatUp$-output $m$ within the displayed bound, and the evaluation trace is a $\SeqUp$ object. The classifier is $\hsame$ on output rows after the constructor history is fixed; the ledger keeps the constructor sequence, intermediate evaluation rows, and bounded-minimisation witness visible.

\begin{theorem}[RecursiveFn zero successor projection obligation]
\label{thm:recursivefn-zero-successor-projection-obligation}
The $\RecursiveFnUp$ constructor-history surface contains visible zero,
successor, and projection rows over $\NatUp$ inputs. For a displayed finite
constructor history, the zero row returns the displayed $\NatUp$ zero output,
the successor row transports the input to its displayed successor output, and
the projection row $\pi^k_i$ selects the $i$th $\NatUp$ input from the finite
input tuple. In each case the accepted graph row is classified by $\hsame$ on
the output once the constructor history is fixed.

These base constructor rows are finite $\mathsf{BHist}$ and $\SeqUp$ rows only.
They do not assert unbounded $\mu$-recursion totality, oracle queries, or a
Church-Turing bridge.
\end{theorem}
\leanchecked{BEDC.Derived.RecursiveFnUp.RecursiveFnZeroSuccessorProjection\_obligation\_surface}
\begin{proof}
Inspect the head constructor of the displayed finite constructor history.

In the zero case, the ledger contains no recursive subtrace; the accepted
output is the visible $\NatUp$ zero row, and $\hsame$ compares that output row
with itself.

In the successor case, the ledger contains the input $\NatUp$ row and its
successor output row. The accepted graph row is obtained by the constructor
route from the input to the displayed successor.

In the projection case, the finite input tuple supplies the selected coordinate
$\pi^k_i$. The ledger records the coordinate choice and the selected $\NatUp$
row, so no oracle or host computation is introduced.
\end{proof}

\begin{theorem}[RecursiveFn composition and primitive-recursion obligation]
\label{thm:recursivefn-composition-primitive-recursion-obligation}
For a displayed $\RecursiveFnUp$ constructor history, composition and primitive
recursion are compatible with the visible $\SeqUp$ evaluation traces. A
composition row accepts exactly when the inner constructor histories evaluate
on the displayed $\NatUp$ inputs to intermediate $\NatUp$ rows and the outer
history evaluates on those rows. A primitive-recursion row accepts exactly when
the base row and each displayed step row appear in order in the finite
$\SeqUp$ trace, with $\hsame$ comparing only the final $\NatUp$ output after
the constructor history is fixed.

The obligation is a constructor-history compatibility statement over finite
$\mathsf{BHist}$ and $\SeqUp$ ledgers. It excludes unbounded
$\mu$-recursion totality, oracle queries, and Church-Turing bridge claims.
\end{theorem}
\leanchecked{BEDC.Derived.RecursiveFnUp.RecursiveFnCompositionPrimitiveRecursion\_obligation\_surface}
\begin{proof}
Proceed by inspecting the displayed constructor history.

For composition, first read the inner histories. Each one consumes the same
visible $\NatUp$ input tuple and produces an intermediate output row in the
$\SeqUp$ evaluation trace. The outer constructor history then consumes exactly
those intermediate rows, so the composed graph row is the final row already
present in the trace.

For primitive recursion, read the base row at the displayed zero input and
then inspect each successor step in the finite recursion trace. The step row
uses the previous output row and the current $\NatUp$ index already recorded in
the ledger. The final accepted output is therefore the final visible trace row.

No branch of the inspection adds an unbounded search, an oracle row, or a
standard recursion-theory bridge; all movement is through the displayed finite
constructor history.
\end{proof}

\begin{theorem}[RecursiveFn bounded minimisation ledger exactness]
\label{thm:recursivefn-bounded-minimisation-ledger-exactness}
For a displayed bounded-minimisation constructor in $\RecursiveFnUp$, the
accepted output row is exactly the first witness row recorded in the finite
$\mathsf{BHist}$ ledger under the visible $\mathsf{UnaryHistory}$ bound, or the
displayed failure row when the bounded trace records no witness. The
$\SeqUp$ evaluation trace contains every tested $\NatUp$ input up to the bound,
and $\hsame$ compares only the resulting bounded output row after the
constructor history is fixed.

The exactness statement is bounded and ledger-local. It does not claim
unbounded $\mu$-recursion totality, oracle access, or a Church-Turing bridge.
\end{theorem}
\leanchecked{BEDC.Derived.RecursiveFnUp.RecursiveFnBoundedMinimisationLedger\_exactness}
\begin{proof}
Inspect the bounded-minimisation constructor row. The
$\mathsf{UnaryHistory}$ coordinate gives the finite bound, and the
$\mathsf{BHist}$ ledger lists the tested $\NatUp$ inputs and their
$\SeqUp$ evaluation rows in order.

If a witness row appears, take the first such displayed row. Constructor
inspection shows that every earlier tested row is recorded as a non-witness,
so ledger exactness identifies the accepted output with that first witness.

If no witness row appears, the same finite inspection reaches the displayed
failure row at the end of the bounded trace. Since the trace is bounded by the
visible unary history, no unbounded search or oracle query is used.
\end{proof}

\begin{theorem}[RecursiveFn Nat input stability obligation]
\label{thm:recursivefn-nat-input-stability-obligation}
Let $x$ and $y$ be displayed $\NatUp$ input rows related by the $\NatUp$
classifier, and fix a finite $\RecursiveFnUp$ constructor history. If the
displayed $\mathsf{BHist}$ and $\SeqUp$ evaluation trace evaluates the history
at $x$ to an accepted output row, then the transported trace evaluates the same
history at $y$ to an output row classified by $\hsame$. The obligation uses
only $\mathsf{BHist}$ constructor rows, $\SeqUp$ evaluation rows, $\NatUp$
input and output rows, and the fixed constructor ledger.
\end{theorem}
\leanchecked{BEDC.Derived.RecursiveFnUp.RecursiveFnNatInputStability\_obligation}
\begin{proof}
Inspect the fixed constructor history. For zero, successor, and projection,
apply \autoref{thm:recursivefn-zero-successor-projection-obligation}; the
transported input row changes only the displayed $\NatUp$ input component, and
the output comparison is the same $\hsame$ row recorded by the constructor
ledger.

For composition and primitive recursion, use
\autoref{thm:recursivefn-composition-primitive-recursion-obligation}. The
inner traces, base row, and step rows are already visible $\SeqUp$ rows, so
classifier transport repacks each intermediate $\NatUp$ output and leaves the
final output classified by $\hsame$.

For bounded minimisation, apply
\autoref{thm:recursivefn-bounded-minimisation-ledger-exactness}. The
transported trace uses the same visible bound and the same finite list of
tested rows, so the first displayed witness, or the displayed failure row, is
transported without adding any unbounded search.
\end{proof}

\begin{theorem}[RecursiveFn constructor ledger coverage obligation]
\label{thm:recursivefn-constructor-ledger-coverage-obligation}
Every accepted $\RecursiveFnUp$ graph row is covered by exactly one visible
constructor case among zero, successor, projection, composition, primitive
recursion, and bounded minimisation. In that case the ledger exposes the
corresponding finite subtrace: the direct base-constructor row, the inner and
outer composition rows, the primitive-recursion base and step rows, or the
bounded-minimisation test rows under the displayed unary bound.
\end{theorem}
\leanchecked{BEDC.Derived.RecursiveFnUp.RecursiveFnConstructorLedger\_coverage\_obligation}
\begin{proof}
Proceed by the head constructor of the accepted constructor history.

The zero, successor, and projection heads are covered by
\autoref{thm:recursivefn-zero-successor-projection-obligation}. Each head has
a distinct constructor tag, so its accepted row cannot also be a composition,
primitive-recursion, or bounded-minimisation row.

The composition and primitive-recursion heads are covered by
\autoref{thm:recursivefn-composition-primitive-recursion-obligation}. Their
visible $\SeqUp$ subtraces distinguish the inner-and-outer composition case
from the base-and-step recursion case.

The bounded-minimisation head is covered by
\autoref{thm:recursivefn-bounded-minimisation-ledger-exactness}. Its
$\mathsf{UnaryHistory}$ bound and finite test ledger are part of the displayed
row, so no other constructor case supplies the same ledger shape.
\end{proof}

\begin{theorem}[RecursiveFn bounded search scope obligation]
\label{thm:recursivefn-bounded-search-scope-obligation}
For a displayed bounded-minimisation constructor in $\RecursiveFnUp$, the
search reads only the $\mathsf{UnaryHistory}$ bound, the listed $\NatUp$ test
rows, and their $\SeqUp$ evaluation rows. The accepted row has no unbounded
search coordinate, no oracle row, and no hidden host predicate beyond the
finite constructor ledger.
\end{theorem}
\leanchecked{BEDC.Derived.RecursiveFnUp.RecursiveFnBoundedSearchScope\_obligation}
\begin{proof}
Start with the bounded-minimisation constructor and apply
\autoref{thm:recursivefn-bounded-minimisation-ledger-exactness}. The theorem
identifies the accepted output with the first visible witness row, or with the
displayed failure row when the finite test ledger contains no witness.

The tested rows themselves are ordinary $\NatUp$ rows in the same finite
constructor history. The base-constructor, composition, and
primitive-recursion obligations,
\autoref{thm:recursivefn-zero-successor-projection-obligation} and
\autoref{thm:recursivefn-composition-primitive-recursion-obligation}, account
for the evaluations of those rows inside the listed $\SeqUp$ trace.

Thus every read used by the bounded search is already named in the
$\mathsf{UnaryHistory}$, $\NatUp$, and $\SeqUp$ ledger, and there is no
exported row for an unbounded search or oracle.
\end{proof}
