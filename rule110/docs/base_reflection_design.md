## 1. What the Lean side does

`BaseReflection` is an abstract reflection contract over histories, generated
signature objects, packages, policy indices, domains, and evidence.  The core
record is:

```text
BaseReflectionSetup:
  Hist, SigObj, Pkg, Pi, Domain, Evidence
  hsame      : SigObj -> SigObj -> Prop
  InDom      : Domain -> Hist -> Prop
  SigGen     : Pi -> Hist -> SigObj -> Evidence -> Prop
  TokIntro   : Pi -> SigObj -> Pkg -> Prop
  InGapSig   : Pi -> Domain -> Pkg -> Hist -> Prop
```

The files then prove that token uniqueness transports sameness through
packages.  `PsameBase` is the single-constructor relation generated by two
token introductions plus an `hsame` witness.  `PsameEqClosure` adds reflexive,
symmetric, and transitive closure.  `GeneratedSameSig` packages two generated
signatures, their evidence, and an `hsame` witness between the generated
signature objects.  `ExactGlobalizeBase` states coverage of histories by
gap-signature packages and an exact classification:

```text
PsameBase P p q <-> Nonempty (GeneratedSameSig P h k)
```

whenever `p` and `q` are the gap-signature packages attached to `h` and `k`.

The current Lean `BaseReflection` files do not encode full CIC syntax by
themselves.  They do not define a universe hierarchy, inductive declarations,
conversion, dependent products, or typing contexts.  Those live in the
`MetaCIC` layer, whose minimal term grammar has variables, application,
lambda, Pi, and a single `sort`; `inferTypeClosed` decides the closed atomic
typing fragment for that grammar.  `MetaCIC.BHistSubstrate` already gives a
BHist-to-term embedding and injectivity theorem for BHist payloads.  Thus the
Lean-side BaseReflection surface is best read as the package/signature bridge
that could carry a CIC fragment once `Hist`, `SigObj`, `Pkg`, `Evidence`, and
`SigGen` are instantiated by a MetaCIC encoding.

## 2. Why the rule110 substrate is harder

The existing rule110 mirrors are finite GroundCompiler-event manifests plus C
decoders.  They check representative rows for FKernel and thin derived
surfaces.  That is much weaker than a substrate-resident type checker.

Under the present BEDC substrate discipline, Rule 110 is not used as a native
general-purpose machine.  The route to universality runs through Cook-style
cyclic-tag systems and then through a Rule 110 row encoding of those systems.
The direct `.ct` manifests in this directory are not yet a complete compiler
from arbitrary cyclic-tag productions to phase-exact Rule 110 evolution.

Lean reflection is comparatively natural because Lean already supplies the
ambient dependent type theory.  The proofs in `BaseReflection` are folds over
Lean propositions and inductive evidence.  A rule110 mirror would instead need
to encode all relevant data as bit patterns and then run a substrate-level
checker over those patterns:

```text
Rule 110 cells
  -> Cook cyclic-tag simulation
  -> cyclic-tag program state
  -> GroundCompiler event stream / BHist stream
  -> MetaCIC term, context, and type encodings
  -> closed-fragment typing or proof-checking judgment
  -> BaseReflection package/signature classification row
```

Every arrow in that chain needs its own soundness contract.  In particular,
dependent binding and substitution are not local cellular-automaton features.
They become string-rewriting protocols over encoded de Bruijn terms, contexts,
and finite derivation trees.  The current Level 2 rows already leave universal
unbounded recognizers for simpler relations as explicit blockers.  A full CIC
checker is above that difficulty class.

## 3. Three possible paths forward

### A. Direct CT encoding

Encode the MetaCIC syntax, closedness predicate, context lookup, shifting,
substitution, syntactic equality, and `inferTypeClosed` as cyclic-tag
productions.  A BaseReflection manifest would then assert that the CT program
halts in an accepting state exactly for encoded closed judgments such as:

```text
empty-context |- sort : sort
empty-context |- pi sort sort : sort
empty-context |- (lambda sort (var 0)) : pi sort sort
```

This path is conceptually direct, but it is not a Level 5.5 deliverable under
the current substrate state.  Even for the closed atomic checker, the CT system
would need unbounded tree traversal, de Bruijn arithmetic, binder-sensitive
shift/substitution, and comparison of encoded terms.  A realistic production
set is likely in the thousands before Cook lowering.  It may be
semidecidable-as-acceptance for a bounded closed fragment, but arbitrary
conversion and proof checking must be specified carefully as a halting
protocol, not as a total negative decision unless the fragment is syntactically
bounded.

### B. Bounded CIC fragment

Mirror only the closed normal-form and atomic-judgment fragment already visible
in `MetaCIC.Decidable.CheckClosedAtom`.  This would encode a finite corpus of
closed terms and their inferred types as GroundCompiler event rows, with C
tests checking that the rows decode to the intended constructor tags and
bounded judgment outcomes.

This is feasible as a manifest family, but it would not be BaseReflection in
the strong sense.  It would be a finite fixture mirror of a MetaCIC checker
slice.  The result would resemble `TopologyUp`: representative rows and clear
scope, with no claim of universal cyclic-tag recognition.  It is useful only
after the L5.6 MetaCIC rule110 mirror chooses stable term, context, and
judgment encodings.

### C. Bridge via L5.6 MetaCIC

Build L5.5 on top of the L5.6 MetaCIC mirror.  The dependency order is:

```text
1. L5.6 chooses the concrete rule110/CT encoding of MetaCIC terms.
2. L5.6 defines the closed judgment assertion shape and bounded checker rows.
3. L5.5 instantiates BaseReflection fields with those encoded judgments:
   Hist     := encoded BHist or encoded MetaCIC trace
   SigObj   := encoded inferred signature object
   Pkg      := encoded package row
   Evidence := encoded checker trace
   SigGen   := closed checker accepts the history/signature pair
   TokIntro := package carries the checker signature object
   InGapSig := domain admits the package for the encoded history
4. L5.5 mirrors `ExactGlobalizeBase` as a classification relation over those
   bounded closed judgments.
```

This path preserves the separation between CIC syntax engineering and the
BaseReflection package/signature theorem surface.  It also matches the existing
trust-loop shape: rule110 manifests provide finite substrate rows; Lean
cross-checking can later verify that each row is a ground instance of the
intended theorem surface.

## 4. Recommended scope

Level 5.5 at this row covers path C: a BaseReflection design that depends on a
separate MetaCIC rule110 mirror for the actual term and judgment encoding.
Path A remains a research-grade cyclic-tag compiler project.  Path B remains a
possible finite fixture family, but only after L5.6 fixes the concrete
MetaCIC-on-BHist encoding.

The status should be `[!]`, with the following exact scope:

```text
Covered:
  Design boundary for BaseReflection over the rule110 substrate.
  Mapping from Lean BaseReflection fields to the needed MetaCIC mirror.
  Explicit obstruction list for direct CIC self-reflection.

Not covered:
  A cyclic-tag CIC type checker.
  A Rule 110 lowering of such a checker.
  A manifest claiming a closed-context type judgment.
  Lean cross-check registration for BaseReflection manifests.
```

No manifest is included with this design.  A manifest row for
`empty-context |- sort : sort` would be meaningful only after the MetaCIC
encoding, judgment tags, and accept/reject protocol are stable.  Before that,
it would test only a local text format, not a substrate-level reflection
claim.

## 5. Obstruction note

BaseReflection on a sub-Turing direct substrate is a research-grade target.
The ROADMAP already marks this area as potentially obstructed because Rule 110
has no native dependent types, no native syntax trees, no native contexts, and
no native universe hierarchy.  All of those must be represented as bit
patterns and manipulated through a cyclic-tag bridge before they can be lowered
to phase-exact Rule 110 evolution.

The current design document is therefore the Level 5.5 deliverable.  Executable
manifests await either a published reduction strategy from the selected
MetaCIC fragment to cyclic-tag productions, or a substrate amendment that
defines a smaller accepted target such as bounded closed-atom judgments with
finite trace certificates.
