# Rule 110 `.r110` Manifest Format

An `.r110` manifest records concrete Rule 110 initial rows that mirror a
cyclic-tag `.ct` manifest case. It is a line-oriented text format.

The first mark-family manifest is intentionally narrow: it covers
`mark/msame_refl.enum.ct`, whose cyclic-tag program has zero productions. The
payload is therefore a direct carrier for the GroundCompiler input bits, not a
claim that the Cook packet composition is complete for nonempty cyclic-tag
programs.

## Header

```
R110_MANIFEST 1
SOURCE_CT rule110/manifests/mark/msame_refl.enum.ct
CONSTRUCTION one_step_preimage
INITIAL_LENGTH 48
ASSERTIONS 2
```

Fields:

- `R110_MANIFEST 1` identifies the format version.
- `SOURCE_CT` names the `.ct` manifest mirrored by this file.
- `CONSTRUCTION one_step_preimage` means each `rule110_initial` row evolves for
  the listed number of steps and exposes the listed payload window.
- `INITIAL_LENGTH` is the fixed row width used by every case.
- `ASSERTIONS` is the number of case lines.

## Case Lines

```
case b0_b0: input=011011 ; rule110_initial=<48 bits> ; steps=1 ; payload_start=16 ; payload_len=6 ; expected_payload=011011 ; expected_reflexive=yes ; encoding=groundcompiler_event_pair ; construction=one_step_preimage
```

Fields:

- `input` is the original `.ct` input bit string.
- `rule110_initial` is the fixed-boundary Rule 110 row at time 0.
- `steps` is the number of Rule 110 steps to run.
- `payload_start` and `payload_len` identify the window read after evolution.
- `expected_payload` is compared byte-for-byte against that evolved window.
- `expected_reflexive=yes` says the payload must decode as two equal
  GroundCompiler events.
- `encoding=groundcompiler_event_pair` fixes the payload interpretation.
- `construction=one_step_preimage` repeats the header construction at case
  granularity.

Whitespace around semicolons is insignificant. Lines beginning with `#` are
comments.

## Example

```
R110_MANIFEST 1
SOURCE_CT rule110/manifests/mark/msame_refl.enum.ct
CONSTRUCTION one_step_preimage
INITIAL_LENGTH 48
ASSERTIONS 2
case b0_b0: input=011011 ; rule110_initial=... ; steps=1 ; payload_start=16 ; payload_len=6 ; expected_payload=011011 ; expected_reflexive=yes ; encoding=groundcompiler_event_pair ; construction=one_step_preimage
case b1_b1: input=10111011 ; rule110_initial=... ; steps=1 ; payload_start=16 ; payload_len=8 ; expected_payload=10111011 ; expected_reflexive=yes ; encoding=groundcompiler_event_pair ; construction=one_step_preimage
```

## Trust Path

The verifier only needs the primitive Rule 110 evaluator in
`evaluator/rule110.c`, plus the existing GroundCompiler decoder used elsewhere
in the mark-family tests. For this empty cyclic-tag case, no cyclic-tag
interpreter step and no Cook construction packet composition is part of the
claim.

## Algorithm Cook Diagnostic Manifests

Nonempty `.algo.ct` programs use a separate opt-in diagnostic format with the
suffix `.algo.r110.ct`. These rows are Cook packet rows generated by
`encoder/generate_r110_algo`, not compact direct carriers.

The header is:

```
ALGO_R110_MANIFEST 1
SOURCE_CT manifests/hist/hsame_refl.algo.ct
CONSTRUCTION cook_phase_exact_packet_diagnostic
EVOLUTION_STEPS 128
ASSERTIONS 5
```

Each case is a multi-line block:

```
case empty
INPUT 1111
CT_STEPS 4
CT_FINAL 20
10000100011001010011
RULE110_INITIAL 21532
<21532 bits>
RULE110_FINAL 21532
<21532 bits>
ENDCASE
```

`make test-algo-r110` replays every `RULE110_INITIAL` row for
`EVOLUTION_STEPS` using only `evaluator/rule110.c` and checks that the result is
exactly `RULE110_FINAL`. The `CT_FINAL` block records the cyclic-tag reference
certificate that was checked while generating the manifest. It is diagnostic
metadata until the Cook collision schedule exposes a decoded CT output window in
the evolved Rule 110 pattern.
