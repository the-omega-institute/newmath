## No nursery-shell growth (HARD GATE)

Automated rounds can add syntactically non-trivial but theoretically shallow growth by introducing an abstract carrier/spec shell and immediately returning assumptions supplied as parameters. This is forbidden for target selection.

A target is INVALID if its main contribution is any of these shapes:

1. Parameter echo: a theorem that takes `le_refl`, `le_trans`,
   `carrier_transport`, `lower_transport`, `upper_transport`, `normalized`, or a
   similar semantic law as an input and returns the same law unchanged or merely
   bundled with a `NameCert`. Such a theorem is an adapter, not theory growth.

2. Abstract-shell classifier: a new `Carrier : BHist → Prop`,
   `lower upper : BHist → Prop`, `Le : BHist → BHist → Prop`, or
   `normalized : ... → Prop` interface with no concrete BEDC-derived predicate
   tying it to existing constructors/relations beyond `hsame` bookkeeping.

3. Isolated derived `def`: a new `RatUp`, `IntervalUp`, `PreorderUp`, or similar
   horizon object that only adds carrier/spec definitions. A valid target for a
   horizon object must also add at least one real closure theorem, transport
   theorem, constructor/inversion theorem, or certificate whose proof uses the
   concrete definition.

4. Unclosed certificate shell: a `NameCert` theorem for a derived object whose
   proof works for arbitrary uninterpreted predicates supplied by parameters and
   does not prove any object-specific closure from a BEDC construction.

   When proving a chapter's canonical Lean witness for a paper-side
   `closurestatus` block, prefer one of these standard names so paper-side
   closure proposals can cite it as `\leantarget{BEDC.<...>.<theorem>}`:
     1. `<region>_semantic_name_certificate` (or `<X>_semanticNameCert` —
         CamelCase variant) — the canonical object-completion proof
     2. `<region>_name_certificate` — older convention, still acceptable
     3. `<region>_stability_certificate_fields` — when the closure is a
         field-aggregate proof rather than a single object certificate
     4. `<region>_laws` — when the closure bundles laws into one theorem
   `bedc_ci.py audit` rejects a paper `closurestatus` block whose
   `\leantarget` doesn't exist in lean4/BEDC, so pick the canonical form
   once and reuse it as the chapter's verification-axis citation.

Prefer targets that close a concrete block:
- Define a carrier over existing BEDC constructors, then prove its transport or
  closure theorem in the same round.
- For RatUp, prove denominator positivity transport / numerator transport /
  normalized stability from concrete assumptions already present in the file.
- For IntervalUp, prove a theorem about a specific boundary predicate or a real
  transport lemma that consumes the interval carrier definition.
- For PreorderUp, prove a concrete preorder instance or derived composition law,
  not a restatement of arbitrary `Le` assumptions.

This supplements the anti-hollow rules: non-hollow code can still be rejected if it does not
make the theory more closed, composable, or concrete.

For new `lean_name` under `BEDC.Derived.<X>Up.*`, if the conclusion shape is one of `<X>HistoryClassifier_exactness`, `<X>HistoryEndpoint_pair`, `<X>HistoryLedgerPolicy_envelope_closure`, `<X>HistoryClassifier_branch_inversion`, `<X>HistoryCarrier_*_readback`, the candidate is **discouraged** — it is the per-domain copy of an already-saturated shape. Prefer either: a closure / determinism / inversion / no-confusion theorem specific to this domain's underlying inductive (e.g. constructor disjointness for `SumUp`, denominator-positivity transport for `RatUp`, distributivity for `RingUp`), OR an abstraction step that hoists the shared shape into a `BEDC.Derived` typeclass.

In `strategy`, briefly state which NAMING.md rules you checked and the grep result confirming no collision.

## Try-close before adding (HARD GATE)

Before emitting a Phase B target, FIRST verify the candidate's stated `lean_signature` cannot be closed by an existing tactic combo applied to existing primitives. Use `lean_multi_attempt` or open a scratch `lean_run_code` and try, in order:

```
by simp [<3-5 nearby existing lemmas>]
by induction h <appropriate variable>; simp [<lemmas>]
by exact <existing_theorem>.mp <h> -- when conclusion is one direction of an iff
by omega -- for arithmetic-ish goals
by decide -- for finite-case goals
```

If any of these closes the goal in under ~30s, the candidate is INVALID — the "theorem" is a one-liner that belongs INLINE inside whatever proof needs it, not as a public theorem with its own name. Discard the candidate and pick something that requires a multi-line constructive argument or a genuinely new combinator.

This rule applies especially to:
- `_iff` directions where one side is `T.mp h` for a single existing `T`;
- arithmetic facts about `length`, `append`, `cons` that are visible after one `simp`;
- carrier transports that are `.transport` projection of an existing classifier;
- `_endpoint_pair` / `_branch_partition` shapes that decompose into one constructor case + reflexivity.

The library has roughly 1500 declarations; further mass-production of one-line lemmas degrades the average information density. Phase B's job is to find theorems that are NOT one-step-from-existing.

## Layered closure planning (READ FIRST — drives target shape)

Each open chapter has TWO axes of closure progress, both read from the chapter's `closurestatus` block:

- **theory_closure axis (6 grades)**: `seedClosure` → `obligationClosure` → `scopedClosure` → `publicClosure` → `bridgedClosure` → `matureClosure`. Owned mostly by paper P-rounds.
- **formal_status axis (8 grades)**: `unformalizedV` → `formalTargetV` → `encodedDefV` → `scaffoldCheckedV` → `theoremCheckedV` → `auditCleanV` → `axiomCleanV` → `bridgeCheckedV`. **Owned by lean R-rounds (THIS phase)**.

A chapter is "retired" from the critical path only when it reaches `(scopedClosure, theoremCheckedV)` simultaneously. Until then it remains in `top`.

The `critical_path.py` JSON output now includes per-chapter `theory_grade` / `formal_grade` / `theory_grade_next` / `formal_grade_next` / `next_axis` / `next_grade_transition`. Use these fields to pick a target SHAPE that advances the chapter's formal grade by exactly one step. **Lean R-rounds focus on the formal axis**; paper P-rounds handle theory closure.

### Formal-axis transition templates (target shape per step)

| Current → Next | Required target shape |
|---|---|
| `(none) → formal work` | Chapter has NO `closurestatus` block yet (`theory_grade` and `formal_grade` are both null). This is the majority of open chapters today and is a NORMAL lean target. Treat as `unformalizedV → formalTargetV` or `formalTargetV → encodedDefV` depending on what's already in the lean file: if `file_lean` does not yet exist or has no carrier `def`, add a `\leandef{X}` marker for a kernel-anchored carrier; if a carrier exists, add `\leanchecked{X}` markers for one or more unmarked paper theorems. Do NOT skip just because the closurestatus block is missing — paper P-rounds register the block on their own schedule, and lean does not need to wait for it before adding markers. |
| `unformalizedV → formalTargetV` | Add a `\leanstmt{X}` marker for at least one paper theorem in the chapter, where X is a NEW Lean declaration with body `:= by sorry`-equivalent (statement only) — but BEDC has 0-sorry invariant, so use `theorem T : <prop> := <existing_thm>` if a related thm exists, else skip and let formalTargetV transition wait for a real theorem-checked round. |
| `formalTargetV → encodedDefV` | Add a `\leandef{X}` marker for chapter's CARRIER definition. X should be a `def` or `inductive` whose body uses BEDC kernel constructors (`BHist`, `Cont`, `hsame`, `UnaryHistory`, `NameCert`). Provides the chapter's mathematical encoding. |
| `encodedDefV → scaffoldCheckedV` | Add 1-3 `\leanchecked{X}` markers for theorems that depend on the chapter's carrier definition. Each X is a real theorem whose proof uses BEDC kernel + the carrier's `def`. Multiple sibling files are fine; pick siblings with high `sibling_effective_unmarked`. |
| `scaffoldCheckedV → theoremCheckedV` | Add `\leanchecked{X}` markers for the chapter's REMAINING unmarked theorems (close out coverage). When all paper theorems in the chapter have a `\leanchecked` marker pointing to a real proof, paper-side `closure_mark` proposal will register the transition. |
| `theoremCheckedV → auditCleanV` | Run `python3 lean4/scripts/bedc_ci.py audit` and resolve any drift / dangling marker / dup label issue this round. Mostly paper-side cleanup; lean rounds can address `unresolved Lean markers` by ensuring the named declaration exists. |
| `auditCleanV → axiomCleanV` | Refactor any theorem in the chapter that pulls in `Classical.choice` / `Quot.sound` / `propext` via `simp [iff]` / classical reasoning. Replace with explicit `Iff.mp` / `Iff.mpr` / case analysis. Verify with `bedc_ci.py axiom-purity --strict`. |
| `axiomCleanV → bridgeCheckedV` | Prove a bridge theorem connecting the chapter's concrete BHist instance to its abstract interface (`<X>Up` schema). Last grade — chapter is mature. |

### Skip rule (narrow — applies only when both grades are non-null)

The skip rule is for chapters that already HAVE a `closurestatus` block (`theory_grade` and `formal_grade` both non-null). Two cases:

- If `next_axis == "theory_closure"` AND `formal_grade` is non-null, the chapter's formal axis is already ahead of its theory axis — paper P-rounds need to catch up first. Skip and pick a chapter where `next_axis == "formal_status"`, OR a chapter where both grades are null (those are always lean-eligible per the `(none) → formal work` row above).
- If `formal_grade` is null but `theory_grade` is non-null (rare — paper wrote a closurestatus block without any lean marker yet), this counts as `next_axis == "formal_status"`; treat as `unformalizedV → formalTargetV` or `formalTargetV → encodedDefV` per the table.

Empty-targets escape `{"targets": []}` is for the genuine corner case where every accessible chapter has both grades non-null AND every accessible front is theory-bottlenecked. This is exceedingly rare today (most chapters are still `(none, none)`). DO NOT emit empty targets just because top[0..2] have null grades — null grades are normal lean work.

Each target's JSON MUST include a `grade_transition` field that quotes the chapter's `next_grade_transition` from critical_path output. Phase D will not enforce this currently but Phase C uses it to pick the right marker family (`\leandef` for encodedDefV, `\leanchecked` for theoremCheckedV, etc). For null-grade chapters, set `grade_transition` to `"(no block) -> formalTargetV"` or `"(no block) -> encodedDefV"` depending on whether the round adds `\leanstmt`/`\leanchecked` or `\leandef`.

### Sibling-file granularity (`granularity: "sibling"`)

Each `top[i]` entry is now a **single sibling file** within a chapter, not the chapter as a whole. Multiple `top[i]` entries can share the same `name` (chapter) but with different `file_paper` and `sibling_id`. Concurrent workers each lock on their chosen `file_paper` (per-sibling cooldown), so this is the unit of contention. Practical implications:

- Pick your 3 targets from 3 DIFFERENT `file_paper` paths — never from the same sibling (would just pile thms onto one front and waste a slot). Different `sibling_id` of the same chapter is fine and often useful.
- The `score`/`downstream` field still measures the **chapter's** transitive impact, not the sibling's, so all sibling fronts of a high-leverage chapter rank above any sibling of a low-leverage chapter. This is intentional.
- The `sibling_thms` field is the leanchecked-marker count *inside that one sibling file* (vs `thms` which is the chapter total). Prefer siblings with low `sibling_thms` — they are closer to a fresh closure surface for the chapter.
- `file_lean` is shared across all siblings of a chapter (the chapter has one `<X>Up.lean` parent), but you can put the new theorem under `BEDC/Derived/<X>Up/<Subdir>/...` to mirror the paper sibling structure when the chapter Lean file would otherwise grow past ~600 lines.

### Top-1 magnet rule (HARD GATE)

When `top[0].downstream` dominates the rest by a wide margin, "1 of 3 from top[0..2]" lets codex pick top[1] or top[2] (which may carry only 1 or 0 downstream horizons) and ignore the actual bottleneck. Compute:

```
top0_dwn  = top[0].downstream
top1_dwn  = top[1].downstream
magnet    = top0_dwn >= max(5, 5 * top1_dwn)
```

If `magnet == True`, **at least 2 of 3 selected targets MUST come from `top[0]`'s chapter** (i.e. `target_file` matches `top[0].file_lean`, OR the target is in the same `BEDC.Derived.<X>Up` namespace as `top[0].name`). The remaining one slot may be from `top[1..2]` or any other valid front. Trigger observed (R2347-R2358 window): `top[0]=continuousmap dwn=71` vs `top[1]=homology dwn=1` (71x), and zero of the last ten rounds attacked continuousmap — exactly the failure mode this gate fixes.

If `magnet == False` (no chapter dominates), the original "1 of 3 from top[0..2]" rule applies unchanged.

The "blocked" three-condition test still applies: if `top[0]` itself is genuinely blocked (all three conditions in §"Mechanical blocker definition" hold), fall back to "1 of 3 from top[0..2]" without the magnet enforcement, and the round may emit `{"targets": []}` if all three top entries are blocked.

### Pre-emit parameter-echo gate (HARD GATE — mirrors phase_d_lint)

Run this same regex against your proposed `lean_signature` BEFORE emitting JSON. The phase D lint rejects on the conjunction:

1. The signature has a hypothesis binder `(<name> : forall ... hsame ...)` (matches Python regex `\(\s*(\w+)\s*:\s*(?:∀|forall)\b[^)]*hsame\b`).
2. The conclusion (text after the last top-level `:`) also has `forall ... hsame ...` (matches Python regex `(?:∀|forall)\b[^,]*,[^.]*hsame\b` after stripping `BHist`/`hsame` tokens for the residual-anchor check).

If both fire, the target is INVALID — do not emit it. Two recent rounds (R1363, R1365) burned ~600s each formalising `field_nonzero_factors_exclude_zero_product` with this exact shape; the lint catches it after Phase C and the round FAILS. Verify locally before commit:

```bash
python3 -c '
import re
sig = """<your lean_signature here, including hypothesis binders and conclusion>"""
bind = re.search(r"\(\s*(\w+)\s*:\s*(?:∀|forall)\b[^)]*hsame\b", sig, re.DOTALL)
concl_part = sig.rsplit(":", 1)[-1] if ":" in sig else sig
concl = re.search(r"(?:∀|forall)\b[^,]*,[^.]*hsame\b", concl_part, re.DOTALL)
print("PARAMETER-ECHO REJECT" if (bind and concl) else "OK")
'
```

If this prints `PARAMETER-ECHO REJECT`, replace the target. Acceptable replacements anchor on a concrete BEDC kernel object: a constructor of `BHist` (`Empty`, `cons`), `Cont`, `UnaryHistory`, the `:= emp` value of a singleton instance, or a specific `\leandef`-marked function from the paper's concrete-singleton instance section.

### Mechanical blocker definition (no soft fallback)

The previous prompt allowed "if technically blocked, use `top[3..]`" — which was wide enough that codex took it on every round and produced a 0/7 hit rate. A top-3 node counts as "blocked" only when **ALL THREE** of the following are true:

1. The chapter file `top[i].file_paper` contains fewer than 3 `\begin{definition}` blocks (the paper has not yet specified the carrier sufficiently to formalize). Verify with:
   ```bash
   grep -c '\\begin{definition}' <top[i].file_paper>
   ```
2. The implementation would require a Lean inductive type or `import` that does not currently exist anywhere under `lean4/BEDC/`. Verify with `grep -rE '(inductive|import) <name>' lean4/BEDC/`.
3. The `critical_path.py` JSON shows `depsReady = false` for this node. By construction the script already excludes deps-not-ready nodes from `top`, so this condition is always false for any node IN `top`.

If condition 3 fails alone (or any single condition fails), the node is NOT blocked. Pick a different top-3 node where all three conditions are absent. The previous 0/7 round-misses were all rationalised through the soft "blocked" escape; a single-condition rationalisation is invalid.

**If ALL of `top[0..2]` claim blocked** under the three-condition test above, output `{"targets": []}` and let the gate skip this round. Do NOT silently fall back to depth-refinement of saturated domains. Empty rounds are preferred over silent fallback.

### Self-check before emitting JSON

Before returning your JSON output, compute:

```
hit = any(t.target_file matches one of top[0..2].file_lean for t in targets)
```

If `hit == false` AND no genuine three-condition blocker exists for all of `top[0..2]`, replace your `targets` list with `[]` and emit an empty round. Do not silently emit non-top-3 candidates with a hand-waved blocker.

## Input Hygiene (kernel basis is locked)
The kernel "input set" — sum of `inductive` types + `class` declarations + the `BaseReflectionSetup` structure — is intentionally minimal (~16). All theorem growth must be derived from this fixed basis, not by adding parallel inputs.

- If a candidate target would require introducing a NEW `inductive` type, FIRST grep `lean4/BEDC/**/*.lean` for any existing `inductive` whose constructors match (e.g. `BHist (Empty / e0 / e1)` already covers any sequence-of-binary-marks type — a new `BWord (nil / bit0 / bit1)` is redundant). Re-use the existing type via `def Alias := Existing` or use it directly. Do NOT propose a target that introduces a parallel inductive isomorphic to one that already exists.
- If a candidate target would require a new `class` setup, check whether an existing setup class can carry the new field. Adding a new class is permitted only when the field is genuinely orthogonal to all four existing setup interfaces.
- When both a `def` and an `inductive` would express the same closed-generation idea, prefer `def` — `def` is conservative (no new logical commitment), `inductive` is a structural input.
- A target whose primary effect is to add a parallel input duplicating an existing input MUST be skipped. Note this in `strategy` and pick a different open obligation.

   ```
   git worktree list | grep -F .worktrees/round_R | awk '{print $1}'
   ```

   For each path returned (excluding this round's own worktree), run:
   ```
   git -C <path> diff codex-auto-dev -- lean4/BEDC/ | grep -E '^\+(theorem|lemma|def) [A-Za-z_]+ ' | sed -E 's/^\+(theorem|lemma|def) ([A-Za-z_]+).*/\2/' | sort -u
   ```

   Collect the union of these names — they are *in-flight*. Any candidate whose `lean_name` (after the namespace prefix is stripped) appears in this set is INVALID. Replace it with a different candidate before emitting. If after this filter you cannot fill 3 slots from `critical_path top[0..2]`, prefer 2 valid `top` targets over 3 candidates that include in-flight collisions — the round will reach Phase C with 2 targets and a soft warning, which is strictly better than aborting on `All targets duplicated by other rounds`.

   Newly-unlocked horizons (post schema-only unlock: e.g. CommRingUp, FieldUp, ModuleUp) are common in-flight targets across 7 parallel workers; this check is mandatory there.

4. **No `discovery` targets — pure formalization mode (HARD BAN).** Lean rounds MUST NOT propose new paper theorem / definition / lemma blocks. The paper side (P-rounds + external bedc-deep supervisor) owns all proposal of new `\label{thm:...}` / `\label{def:...}` blocks. Lean rounds register `\leanchecked` markers on theorems the paper has ALREADY captured.

   Every emitted target MUST have `paper_block_status: "exists_without_marker"`. The `paper_label` field MUST resolve to a `\label{...}` that already exists somewhere under `papers/bedc/parts/` and lacks a `\leanchecked` marker. Verify with:
   ```bash
   git grep -nE "\\\\label\\{<your-label>\\}" -- 'papers/bedc/parts/'
   git grep -nE "leanchecked\\{<your-target-decl>\\}" -- 'papers/bedc/parts/'
   ```
   First grep must return ≥1 hit; second must return 0. If first returns 0 (paper hasn't captured the claim yet), DROP that target and pick a different one whose paper site exists.

   If after applying this filter you cannot fill 3 slots from `critical_path top[0..2]`, emit `{"targets": []}` — empty round is preferred over creating-from-scratch on the paper side. Paper rounds will catch up; the next dispatched lean round will find more `exists_without_marker` candidates.

   This eliminates the R-round vs P-round label-collision race at its source: only the paper pipeline ever introduces new labels.
