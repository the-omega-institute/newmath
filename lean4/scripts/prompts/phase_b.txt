## v2.1 — Anti-hollow semantic discipline (HARD GATE)

A prior project review discovered that the `NameCert` license abstraction was
structurally hollow: certificate fields (SourceSpec, PatternSpec, ClassifierSpec,
StabilityCert, LedgerPolicy) were declared as bare `Type` and instantiated with
`Unit`, so `nat_up_name_certificate := NameCert.mk () () () () ()` typechecked
without saying anything about Unary histories or `hsame`. Markers pointed to
real Lean targets but the Lean targets carried no semantic content.

Effective from v2.1:

1. NEVER pick a target whose Lean implementation would be `Unit`, `True`,
   `True.intro`, `()`, or any other trivially-inhabited placeholder. The
   implementation must non-trivially relate to a BEDC kernel construct
   (BMark, BHist, msame, hsame, Ext, Cont, ProbeBundle, SigRel, sameSig,
   Pkg, InGap, NameCert, BaseReflectionSetup, or a derived predicate over
   them).

2. When implementing a `NameCertSetup` instance for a derived interface
   `<Object>Up`, the carrier and classifier MUST be tied to the BEDC
   kernel. For example:
   - NatUp's carrier is the `Unary : BHist → Prop` predicate; the classifier
     is `hsame` restricted to unary histories.
   - AddUp's pattern is `ContR` between unary histories, classifier is `hsame`
     on continuation results.
   - Future objects (IntUp, RatUp, BoolUp, etc.) must define their own
     carrier predicate over BHist (or over an existing BEDC inductive) and
     classifier as a restricted `hsame` / structural sameness, not a fresh
     opaque Type.

3. NEVER use `MinimalNameCertSetup` for new license proofs. That instance
   sets every field to `Unit` and is preserved only for legacy compatibility.
   New license proofs must use the `SemanticNameCert` shape introduced in
   `lean4/BEDC/FKernel/NameCert.lean`.

4. If the new target file requires importing `Init.Data.Unit` or otherwise
   constructs the proof from `()` exclusively, the target is INVALID — abort
   and pick a different target.

This is a stricter superset of the existing HARD GATE in Step 2 below.

## Input Hygiene (kernel basis is locked)
The kernel "input set" — sum of `inductive` types + `class` declarations + the `BaseReflectionSetup` structure — is intentionally minimal (~16). All theorem growth must be derived from this fixed basis, not by adding parallel inputs.

- If a candidate target would require introducing a NEW `inductive` type, FIRST grep `lean4/BEDC/**/*.lean` for any existing `inductive` whose constructors match (e.g. `BHist (Empty / e0 / e1)` already covers any sequence-of-binary-marks type — a new `BWord (nil / bit0 / bit1)` is redundant). Re-use the existing type via `def Alias := Existing` or use it directly. Do NOT propose a target that introduces a parallel inductive isomorphic to one that already exists.
- If a candidate target would require a new `class` setup, check whether an existing setup class can carry the new field. Adding a new class is permitted only when the field is genuinely orthogonal to all four existing setup interfaces.
- When both a `def` and an `inductive` would express the same closed-generation idea, prefer `def` — `def` is conservative (no new logical commitment), `inductive` is a structural input.
- A target whose primary effect is to add a parallel input duplicating an existing input MUST be skipped. Note this in `strategy` and pick a different open obligation.
