This is the field-level faithfulness check: `ChapterTasteGate.round_trip` + `layer_separation` only force `toEventFlow` to be injective on inhabitants, NOT that the encoding reflects every declared sub-field of the carrier. A 9-field `XUp.mk a b c ... i` could pass ChapterTasteGate while `toEventFlow` only encodes `(a, b)` and discards the other 7 fields. `FieldFaithful` closes that loophole.

**Pattern for `FieldFaithful <X>Up`** (chapter's `.mk` has BHist fields `a b c ... i`):

```lean
instance <x>FieldFaithful : FieldFaithful <X>Up where
  fields := fun x =>
    match x with
    | <X>Up.mk a b c d e f g h i => [a, b, c, d, e, f, g, h, i]
  field_faithful := by
    -- BEDC touchpoint anchor: BHist BMark
    intro x y h
    cases x with
    | mk aŌéü bŌéü cŌéü dŌéü eŌéü fŌéü gŌéü hŌéü iŌéü =>
      cases y with
      | mk aŌéé bŌéé cŌéé dŌéé eŌéé fŌéé gŌéé hŌéé iŌéé =>
        simp only [] at h
        -- h : [aŌéü, bŌéü, cŌéü, ...] = [aŌéé, bŌéé, cŌéé, ...]
        -- Extract pairwise equalities from list-equality
        injection h with ha t1
        injection t1 with hb t2
        injection t2 with hc t3
        injection t3 with hd t4
        injection t4 with he t5
        injection t5 with hf t6
        injection t6 with hg t7
        injection t7 with hh ti_rest
        injection ti_rest with hi _
        subst ha; subst hb; subst hc; subst hd; subst he; subst hf
        subst hg; subst hh; subst hi
        rfl
```

For inductive carriers with multiple constructors, write a `match` over both constructors and prove field equality per constructor; the `fields` projection is a flat List BHist concatenated across the carrier's BHist sub-fields.

### HARD GATE ŌĆö AI chapter carrier alignment (single carrier, not two)

If the target touches an `\origin{ai}` chapter (currently `BeliefUp`; in
the future, any chapter whose paper closurestatus block carries
`\origin{ai}`), the chapter MUST have a SINGLE Lean carrier `Type` that
serves both the paper-side carrier definition and the
`ChapterTasteGate` instance. A chapter with two parallel carriers
("the real one in `<X>Up.lean`, the placeholder in
`<X>Up/TasteGate.lean`") is a structural defect that causes the
TasteGate to gate the placeholder while the real mathematics is gated
by nothing. Empirically observed on `BeliefUp` (commit `a43da0c58` lean
side and onwards): the real carrier `BeliefObservationUpdateCarrier`
on `BHist` was unrelated to the placeholder `BeliefUp` `Type` that the
`ChapterTasteGate` instance used.

How to satisfy the gate:

1. Define ONE Lean `Type` (or `inductive Type`) that is the chapter's
   carrier ŌĆö call it `<X>Up` (matching the paper macro `\<X>Up`).
2. Provide `BHistCarrier <X>Up` with `toEventFlow : <X>Up ŌåÆ EventFlow`
   and `fromEventFlow : EventFlow ŌåÆ Option <X>Up`. The `toEventFlow`
   embedding must encode the carrier's actual rows (prior /
   observation / classifier rows etc), not a constructor-name
   placeholder.
3. Provide `ChapterTasteGate <X>Up` with a real `round_trip` proof
   (`fromEventFlow Ōłś toEventFlow = some` on every chapter token) and a
   real `layer_separation` proof (distinct tokens ŌåÆ distinct
   `EventFlow`s).
4. The chapter's other Lean theorems (transport / ledger exactness /
   carrier obligations) must operate on the SAME `<X>Up` `Type`. Do
   not declare a separate `<X>ObservationUpdateCarrier` predicate or
   relation that lives in parallel ŌĆö extend `<X>Up`'s constructors or
   add member theorems instead.

**TOUCHPOINT ANCHOR (HARD REQUIREMENT ŌĆö read carefully)**: the Phase
D `no-touchpoint` gate uses regex `\bBHist\b` etc., and `BHistCarrier`
does NOT match `\bBHist\b` (no word boundary between `BHist` and
`Carrier`). Therefore EVERY new declaration in `TasteGate.lean`
(including the `BHistCarrier <X>Up` instance, the `ChapterTasteGate
<X>Up` instance, and every helper `*FromEventFlow` /
`*ToEventFlow_injective` / `*_round_trip` / `taste_gate`) MUST contain
a bare kernel token. Do this by adding ONE comment line as the first
line of every new declaration body:

```lean
instance dyadicPrecisionBHistCarrier : BHistCarrier DyadicPrecisionUp where
  -- BEDC touchpoint anchor: BHist BMark
  toEventFlow := fun x => ...
  fromEventFlow := fun ef => ...

instance dyadicPrecisionChapterTasteGate :
    ChapterTasteGate DyadicPrecisionUp where
  -- BEDC touchpoint anchor: BHist BMark
  round_trip := ...
  layer_separation := ...

private theorem dyadicPrecision_round_trip (x : DyadicPrecisionUp) :
    dyadicPrecisionFromEventFlow (dyadicPrecisionToEventFlow x) = some x := by
  -- BEDC touchpoint anchor: BHist BMark
  ...
```

The gate inspects the raw block (comments included), so this passes.
This is not "gaming" the gate ŌĆö it is the prescribed satisfaction
mechanism for typeclass-instance declarations whose type signature
references typeclass names that contain `BHist` but are not the bare
`BHist` token. Skipping this comment is the #1 cause of R-round
failure observed 2026-05-11 (R4670/R4668/R4673/R4671/R4667/R4674/
R4675/R4686/R4694). Do not skip it.

**PROPEXT TRAP (HARD REQUIREMENT ŌĆö read carefully)**: do NOT write a
standalone (non-`private`, non-instance-field) theorem whose statement
is `BHistCarrier.fromEventFlow (BHistCarrier.toEventFlow x) = some x`
or similar typeclass-projection equality. The typeclass projection
forces Lean to unfold the `BHistCarrier <X>Up` instance via
propositional equality, which transitively depends on `propext` and
fails `axiom-purity --strict`. The `ChapterTasteGate <X>Up` instance
already carries the equivalent `round_trip` field; any *public* form
of the round trip should reference the concrete underlying defs
(`dyadicPrecisionFromEventFlow` / `dyadicPrecisionToEventFlow`, not
`BHistCarrier.fromEventFlow` / `BHistCarrier.toEventFlow`). If you
need a helper theorem at all, declare it `private` and phrase it in
terms of the concrete defs, as in the example above. Observed 6+ R
rounds (R4696/R4700/R4704/R4705/R4706/R4708, 2026-05-12) all failed
axiom-purity from helper theorems of this exact shape. Do not repeat
this.

If the round adds a new theorem to an `\origin=ai` chapter and the
chapter currently has the two-carrier defect, the round MUST include
the unification: drop one of the carriers (whichever has fewer
downstream callers) and migrate its theorems onto the other.

This gate is what makes "AI chapters use the compiler formalisation"
mean something concrete: the same `Type` flows through ground
compiler `EventFlow`, through `BHistCarrier`, into `ChapterTasteGate`,
and through the chapter's own obligations. Without the alignment,
TasteGate is decorative.

## Step 5: Pre-commit sync with origin BASE (proactive label-collision resolution)

Before staging and committing, fetch the latest BASE and merge it into the worktree. This catches concurrent rounds' commits that landed during your Phase C session, and lets you resolve any label / declaration collision INLINE while you still have full context ŌĆö much cheaper than the outer pipeline's reactive `_codex_resolve_post_rebase_audit` flow which runs after the audit gate already failed.

```bash

git fetch origin "$BASE"
git merge --no-ff --no-edit "origin/$BASE"
```

If `git merge` reports conflicts:

1. List the conflicted files: `git diff --name-only --diff-filter=U`.
2. For each `.tex` conflict: keep BOTH sides' additions when they are independent. When two sides added the SAME `\label{thm:<X>}` (`bedc_ci.py audit` would catch this as `duplicate paper labels`), drop YOUR side's `\label{thm:<X>}` block and ALL of its content (the `\begin{...}...\end{...}` block plus any marker line on the next non-blank line). The other side's claim is canonical because they got there first. The Lean side of your target still proceeds ŌĆö only the paper marker registration is dropped.
3. For each `.lean` conflict in `lean4/BEDC/`: if both sides added the same `theorem` / `def` name, drop yours and use the other side's. If both added different names but same conclusion shape, drop yours per "duplicate theorem" discipline of Step 5.6.
4. After resolution: `git add <each-file>`, then `git commit --no-edit` to finalize the merge. The merge commit is fine ŌĆö Step 6's own-round commit is separate.
5. After the merge commit, re-run Step 4's full verification (`lake_gate.py build` + `make` + `check-axioms.py`). The merge introduced new content that could break compilation. If verification fails on the merged state, fix the proof against the merged BASE; do not `git merge --abort`.

## Step 5.3: Paper-side discovery anti-parameter-echo (HARD GATE)

When this round inserts a `\\begin{{theorem}}` / `\\begin{{definition}}` / `\\begin{{lemma}}` block into `papers/bedc/parts/` (a `discovery` target, or a paper insertion produced during conflict resolution), the inserted block MUST NOT be a parameter-echo theorem. Forbidden shapes:

1. **Source-equivalence echo.** A theorem of the form: assume $A \\Leftrightarrow B$ as carriers and $\\sim_A \\Leftrightarrow \\sim_B$ as classifiers, conclude that every previously proved $A$-side property holds on the $B$-side. Look for paired carrier/pattern/ledger $\\Leftrightarrow$ chains, fields-by-fields equivalence lists, or "Restate certificate $\\mathsf{{cert}}_A$ for $\\mathsf{{cert}}_B$" structures.

2. **Indexed restatement of an existing theorem.** Inserting `[<X> source equivalence]`, `[<X> under source equivalence]`, `[<X> source-equivalence with fields]`, `[<X> classifier source equivalence]`, or any sibling whose only difference from a label already present in the same chapter directory is the `_with_fields` / `_under` / `_alt` / `_v2` / `_again` suffix.

3. **Pure $A$-vs-$B$ parametric statement** with no concrete BEDC kernel construction (BMark, BHist, ProbeBundle, SigRel, specific Pkg / NameCert instance, specific cons-boundary witness) on the conclusion side.

If the existing chapter has 2 or more theorems whose label or title contains `source-equivalence` or `under-source-equivalence`, do NOT insert another. Either pick a different paper site (different chapter, different concrete kernel object), or drop the discovery target entirely from this round and let the round commit Lean code only ŌĆö register-only is forbidden, but Lean-only is fine when the Lean code is meaningful.

The Lean side of this round must still proceed: write the formal theorem in `lean4/BEDC/`, but skip the paper insertion if the only available paper site would be a parameter-echo block.

## Step 5.4: Paper-side label collision pre-check (HARD GATE)

When this round adds new paper-side blocks (from a `discovery` target ŌĆö a `\\begin{{theorem}}ŌĆ”\\end{{theorem}}` block inserted into `papers/bedc/parts/`), the parallel paper pipeline may be inserting blocks at the same time. Before commit, run a fast collision scan:

```bash
git diff --cached -- papers/bedc/ \
  | grep -nE "^\+\\\\label\{" \
  | sed -E 's/.*label\{([^}]+)\}.*/\1/' \
  | while read L; do
      cnt=$(grep -rE "\\\\label\{${L}\}" papers/bedc/ | wc -l | tr -d ' ');
      if [ "$cnt" -gt 1 ]; then echo "DUPLICATE: $L"; fi;
    done
```

Any DUPLICATE line means this round is about to introduce a `\\label{}` that is already present in `papers/bedc/parts/` on the current worktree. Even if the merged base branch does not yet have this label, a sibling paper round could land it before this round merges, and the combined audit would fail. Fix one of:

1. Drop the offending paper insertion entirely from this round and let the paper pipeline own that block; the round still keeps its Lean additions.
2. Rename the `\\label{}` to something that genuinely encodes new content (not a `_v2`, `_alt`, `_under`, `_again` suffix ŌĆö those are version-narrative and disallowed). Update any internal `\\autoref{}` consumers in the same patch.

Do NOT commit until the duplicate scan returns no DUPLICATE lines.

Phase D's `phase_d_lint.py` rejects rounds in two cases. Self-check both before commit:

**(a) Round-internal**: two newly-added theorems with conclusions that are alpha-equivalent (after stripping forall binders and iff direction). ```bash

  | awk '/^\+\s*(protected )?(theorem|lemma|def|abbrev) /{
            sub(/^\+\s*(protected )?(theorem|lemma|def|abbrev)[ \t]+/,""); print $1
