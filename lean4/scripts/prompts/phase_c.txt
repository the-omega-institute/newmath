## Script-backed Phase D gates

2. Stale marker target: a new `\leanchecked`, `\leanstmt`, or `\leandef` marker points to a Lean declaration that was not introduced in this round.
3. New `\leanvariant`: automated rounds must not add variants. If your target only yields a wrapper, projection, weakened form, pair-bundle, or forwarding lemma for an already checked theorem, abandon it and implement a genuinely new statement.
4. Hollow construction: added Lean lines using `Unit`, `()`, `Unit.unit`, `True.intro`, `trivial`, `NameCert.mk ()`, `NameCert.mk Unit.unit`, `MinimalNameCertSetup`, or `import Init.Data.Unit` are rejected.
5. ## No nursery-shell growth (HARD GATE)

Do not implement targets that only make a horizon module look larger. A round
must make BEDC theory more closed, composable, or concrete.

The following implementation shapes are INVALID even if they compile and pass
the current anti-hollow checks:

1. Parameter echo: a theorem that accepts semantic laws such as `le_refl`,
   `le_trans`, `carrier_transport`, `lower_transport`, `upper_transport`, or
   `normalized` and returns those same laws unchanged or simply bundles them.
   If the proof is mostly `exact le_refl`, `exact le_trans`, or constructor
   packaging of input assumptions, abandon the target.

2. Abstract-shell classifier: adding `Carrier : BHist → Prop`,
   `lower upper : BHist → Prop`, `Le : BHist → BHist → Prop`, or
   `normalized : ... → Prop` without also proving a concrete closure theorem
   about a BEDC-defined predicate. Do not fabricate general Prop interfaces just
   to manufacture a `NameCert`.

3. Isolated derived definitions: adding only carrier/spec definitions for
   `RatUp`, `IntervalUp`, `PreorderUp`, or another horizon object. If you add a
   new derived carrier/spec, the same round must also prove at least one theorem
   that uses that concrete definition: transport, closure, constructor
   inversion, determinacy, composition, or a certificate whose object-specific
   content is not supplied entirely as parameters.

4. Unclosed certificates: a `NameCert` theorem over arbitrary uninterpreted
   predicates is invalid unless the theorem also proves an object-specific BEDC
   property from concrete constructors/relations in the file.

Valid examples:
- `IntCarrier` transport across `hsame` because it uses the concrete carrier
  fields and `unary_transport`.
- List classifier transitivity because it performs induction over list structure
  and uses `hsame_trans` elementwise.
- Composite gap coverage because it constructs a new witness chain from two
  coverage layers.

Invalid examples:
- A generic preorder theorem that takes arbitrary `Le` plus `le_refl` and
  `le_trans`, then returns those assumptions.
- An interval certificate over arbitrary `lower` and `upper` predicates where all
  boundary transport is supplied as parameters and no concrete boundary theorem
  is proved.
- A rational carrier/spec file with no transport or certificate theorem.

If all selected targets reduce to these invalid shapes, abandon them and pick
new targets rather than committing shallow growth.

### HARD GATE — every new `.lean` file MUST be imported into `BEDC.lean`

If this round CREATES a new file under `lean4/BEDC/` (i.e. `git status --porcelain lean4/BEDC/ | grep '^A'` lists it), you MUST add a corresponding `import BEDC.<dotted.module.path>` line to `lean4/BEDC.lean` in the same round. Failure to do so produces an "orphan module" — `lake build` from `BEDC.lean` won't build it, and downstream `bedc_ci.py axiom-purity --strict` invocations will report `"no parsed #print axioms result"` for every theorem in the orphan, failing the audit gate and triggering recovery.

Run this check after Step 2 build succeeds, before proceeding to Step 3:

```bash
# List newly-added .lean files under lean4/BEDC/
new_lean=$(git -C "${REPO_ROOT:-$(git rev-parse --show-toplevel)}" status --porcelain lean4/BEDC/ \
  | awk '/^\?\?|^A / {print $2}' \
  | grep '\.lean$' \
  | grep -v 'BEDC\.lean$')

# For each, derive the dotted module path and check BEDC.lean has the import
for f in $new_lean; do
  mod=$(echo "$f" | sed 's|/|.|g; s|\.lean$||; s|^lean4\.||')
  if ! grep -qF "import $mod" lean4/BEDC.lean; then
    echo "ORPHAN: $f → missing 'import $mod' in lean4/BEDC.lean"
  fi
done
```

If the script prints any `ORPHAN:` line, append the corresponding `import BEDC.<module>` line to `lean4/BEDC.lean` BEFORE Step 3. Place the new line near the existing imports for the same `BEDC.Derived.<X>Up.*` namespace (alphabetic ordering preferred but not strictly enforced — just keep related imports clustered). Re-run `lake_gate.py build` after the edit to confirm the import compiles cleanly. The build verifies the new file's transitive deps; the orphan check verifies the new file is reachable from the toplevel module.

This gate catches the failure mode where a Lean round adds new theorems that build standalone but don't enter the main `#print axioms` graph — a salvage-merge or solo-run gap that produces transient `axiom-purity` failures across downstream rounds until manually fixed.

### HARD GATE — case-insensitive filename conflict (PascalCase enforcement)

BEDC chapter modules are PascalCase: `<X>Up.lean` where `<X>` is the chapter name (`TopGroupUp`, `DeRhamUp`, `LambdaCalcUp`, `RingOfIntegersUp`, …). Before creating a new file under `lean4/BEDC/Derived/`, run a **case-insensitive** existence check:

```bash
# Replace TopgroupUp with the file you intend to create
intended="lean4/BEDC/Derived/TopgroupUp.lean"
basename_lc=$(basename "$intended" | tr 'A-Z' 'a-z')
existing=$(find lean4/BEDC -iname "$basename_lc" 2>/dev/null)
if [ -n "$existing" ]; then
  echo "CASE-CONFLICT: $intended would shadow $existing on case-insensitive filesystems"
fi
```

**If the check prints `CASE-CONFLICT`, do NOT create the new file.** Instead, edit the existing file (whose path the find command printed) and place your new `theorem` / `def` inside its existing namespace. macOS APFS and Windows NTFS are case-insensitive by default — `TopgroupUp.lean` and `TopGroupUp.lean` are the SAME inode. Adding both to `BEDC.lean` produces:

```
error: BEDC.lean:N:0: import BEDC.Derived.TopGroupUp failed,
environment already contains 'BEDC.Derived.TopGroupUp.<thm>'
from BEDC.Derived.TopgroupUp
```

Lean's import resolver treats the two casings as DISTINCT modules but the OS hands back the SAME file twice, so identical declarations get registered twice and the build fails. R3806 spent a full round renaming `TopgroupUp.lean` → `TopGroupUp.lean` to canonicalize the casing; R3809 then re-introduced the lowercase variant in the same hour, triggering the failure again. Do not repeat this — always run the case-insensitive find before adding a new chapter file. If you find an existing chapter under any casing, **use that exact casing** for both the filename and every `import` / `namespace` reference; never create a sibling file with a different case.

  | while read L; do
      cnt=$(grep -rE "\\\\label\{${L}\}" papers/bedc/ | wc -l | tr -d ' ');
      if [ "$cnt" -gt 1 ]; then echo "DUPLICATE: $L"; fi;
    done
```

Any DUPLICATE line means this round is about to introduce a `\\label{}` that is already present in `papers/bedc/parts/` on the current worktree. Even if the merged base branch does not yet have this label, a sibling paper round could land it before this round merges, and the combined audit would fail. Fix one of:

1. Drop the offending paper insertion entirely from this round and let the paper pipeline own that block; the round still keeps its Lean additions.
2. Rename the `\\label{}` to something that genuinely encodes new content (not a `_v2`, `_alt`, `_under`, `_again` suffix — those are version-narrative and disallowed). Update any internal `\\autoref{}` consumers in the same patch.

Do NOT commit until the duplicate scan returns no DUPLICATE lines.

## Step 5.5: HARD GATE — round must contain real formalization

git diff --cached --stat lean4/BEDC/ | tail -1
```

If the staged changes show ZERO insertions in `lean4/BEDC/` (i.e., the round only modified `papers/bedc/` to register markers, with no new Lean theorems / defs / inductives / structures), the round is INVALID — it is a "register-only round" and is forbidden. Do NOT commit. Instead:

A valid round produces a non-trivial Lean diff. Marker registration alone is not formalization.

## Step 5.6: HARD GATE — duplicate theorem detection (round-internal AND tree-wide)

Phase D's `phase_d_lint.py` rejects rounds in two cases. Self-check both before commit:

**(a) Round-internal**: two newly-added theorems with conclusions that are alpha-equivalent (after stripping forall binders and iff direction). Six rounds (R1948 / R1949 / R1950 / R1951 / R1953 / R1954 / R1955) FAILed in 8 minutes by re-implementing `SubgroupCentralizerQuotientKernel_witness_classifier_iff` together with `SubgroupCentralizerNormalizerQuotientClassifier_kernel_iff` and `_kernel_iff_direct` — three names, one conclusion.

**(b) Tree-wide**: a newly-added theorem whose name already exists somewhere under `lean4/BEDC/` outside this round's diff. The same set of failures was caused by codex re-defining theorems that already live in `lean4/BEDC/Derived/SubgroupUp.lean`, just placed in newly-created files like `SubgroupUp/Quotient.lean`. Lean's namespace flattening means `BEDC.Derived.SubgroupUp.<name>` resolves to ONE declaration regardless of file — duplicates would break the build.

Self-check sequence:

```bash
# 1. List every new theorem name introduced by this round's staged diff
new_names=$(git diff --cached lean4/BEDC/ \
  | awk '/^\+\s*(protected )?(theorem|lemma|def|abbrev) /{
            sub(/^\+\s*(protected )?(theorem|lemma|def|abbrev)[ \t]+/,""); print $1
         }' | sort -u)

# 2. For each, check if the name ALREADY EXISTS elsewhere in lean4/BEDC/
for n in $new_names; do
  # Search for the SAME name in any committed (non-staged) location.
  if git grep -nE "^\s*(protected )?(theorem|lemma|def|abbrev)\s+${n}\b" -- lean4/BEDC/ \
     | grep -v ':+' | head -3; then
    echo "  -> existing declaration found for $n; this round's addition is a DUPLICATE"
  fi
done

# 3. List the conclusion types of new theorems for round-internal alpha-equivalence
git diff --cached lean4/BEDC/ \
  | awk '/^\+theorem |^\+lemma /{flag=1; print; next} /^\+/ && flag {print} /^[^+]/{flag=0}' \
  | head -200
```

If a new declaration's name (case (b)) appears anywhere else under `lean4/BEDC/`, DROP this round's redefinition: `git restore --staged <file>` + re-edit to remove the redundant block.

If two new declarations (case (a)) share a conclusion under different names (e.g. `<X>_iff` vs `<X>_iff_direct`, `<X>` vs `<X>_witness`, `<X>_kernel_iff` vs `<X>QuotientClassifier_kernel_iff` when the second's conclusion proves the same biconditional/equation), KEEP only one — drop the others.

**Left/right cancel and commutativity sibling rule**: a round must NOT introduce both `<X>_left_<Y>` AND `<X>_right_<Y>` (or `_left_cancel_iff` + `_right_cancel_iff`, or `_append_comm` + `_append_context_swap`) when their conclusions reduce to the same biconditional/equation under the BEDC `Cont` / `append` framework. In BEDC, left and right cancellation are typically alpha-equivalent because `Cont(x, y, out)` is symmetric under context permutation; Phase D's normalizer treats them as duplicates and rejects. Four R-rounds (R2220 / R2234 / R2254 / R2329) FAILed in 4 hours under the MagmaUp chapter by adding this exact pair. Pick one orientation (default: `_left_<Y>`), drop the other, and rely on a separate `_left_to_right_iff` lemma if the orientation symmetry needs explicit witness.

The round commits whichever ONE new theorem is canonical.

**Mechanical preview (HARD GATE)**: before staging the commit, run

```bash
python3 lean4/scripts/phase_d_lint.py --worktree . --base-branch codex-auto-dev --include-shallow
```

This is the SAME check `codex_formalize.py` runs after merge. If it exits non-zero (any of: mechanical-arity / parameter-echo / anchor-missing / SHALLOW GROWTH PATTERN duplicate-conclusion), the round is doomed — Phase D will reject the merge and the round goes into the recovery queue costing 10-15 minutes per failure cycle. SELF-HEAL NOW instead: read the violation message, identify the offending declaration, `git restore --staged <file>` for the file containing it, edit the file to drop the duplicate / re-target the conclusion, re-stage, re-run the lint. Repeat until the lint exits 0. THEN proceed to Step 6.

For SHALLOW GROWTH PATTERN: the violation lists the file path and the two theorem names whose conclusions match. The standard fix is to delete the second theorem and inline its references (typically there will be 0-1 references inside the same file). The boundary_open_laws / semantic_name_certificate type pair (where one theorem's conclusion is a strict subset of another's) is the most common shape — drop the subset.
